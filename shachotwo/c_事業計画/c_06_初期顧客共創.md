# Phase A' — 初期顧客共創プログラム

> **Phase Aと同時並行で実施する。Phase A完了を待たない。**
> 自社ドッグフーディング（Phase A）は「箱」の検証。初期顧客共創（Phase A'）は「中身」の検証。
> 両方が揃って初めてシャチョツーは製品として成立する。

---

## 1. Phase A'の戦略的位置づけ

### なぜPhase Aと同時並行でなければならないか

| 問題 | 説明 |
|---|---|
| 自社だけではゲノム1個分 | 自社はSaaS業のゲノムしか作れない。建設・製造・歯科のゲノムは、それぞれの業界の社長からしか取れない |
| ネットワーク効果の検証不能 | 「遺伝子合成」（業種間のナレッジ転用）は複数業種のゲノムが揃わないと検証できない |
| プロダクトの「中身」がない状態でのローンチはミスリード | Phase Bで4業種ゲノム入りの状態でローンチするには、Phase A'はPhase Bより前に始まっていないと間に合わない |
| 初期顧客のリレーションに時間がかかる | 3社との信頼関係構築・ゲノム品質の達成には最低3ヶ月。Phase A完了後に着手では遅すぎる |
| Series A前の参照顧客要件 | VCは「使っている顧客がいる」を求める。Phase A'の3社が最初の参照顧客になる |

### Phase A' が実現すること

```
Phase A（自社）                        Phase A'（初期顧客3社）
    ↓                                       ↓
SaaS業のゲノム                         建設業 + 製造業 + 歯科のゲノム
    ↓                                       ↓
「箱」が動くことの証明              「ゲノムの価値」が業種をまたいで機能することの証明
    ↓                                       ↓
                    ↘             ↙
                      Phase B MVP
                 4業種ゲノム入りでローンチ
                 「遺伝子合成」デモ可能
                 参照顧客3社がいる状態
```

### Phase A'の本質はセールスではなく共創

Phase A'は「売りに行く」フェーズではない。「一緒に磨く」フェーズである。

- 先方は「顧客」ではなく「共同開発パートナー」
- こちらが提供するのは「プロダクト」ではなく「70点のテンプレートを御社に合わせて90点にするサービス」
- 先方が提供するのは「お金」ではなく「業種の暗黙知・フィードバック・紹介」
- ゴールは「テンプレートの品質を上げること」であり「契約を取ること」ではない

### 共創の前提: テンプレートファースト

```
従来の共創イメージ（NG）:
  Day 1: ゲノム完成度 0.00 → 「ゼロから一緒に作りましょう」
  → 社長の負荷が高い / 初期体験が貧弱 / 価値を感じるまでに時間がかかる

テンプレートファースト共創（正しい姿）:
  Day 1: ゲノム完成度 0.50-0.70 → 「もう7割入っています。御社固有の部分を一緒に磨きましょう」
  → 社長は差分だけ教えればいい / 初日からQ&Aが使える / 「もうこんなに分かってるの？」という驚き

テンプレートの構成:
  共通ベーステンプレート（全企業自動適用）:
    → 経営管理・人事・経理・営業・法務・IT・総務の基本ルール・フロー
    → どの業種でも共通する経営の基礎部分（b_01_ブレイン編 §genome参照）

  業種別スターターテンプレート（業種選択時に自動ロード）:
    → 建設業・製造業・歯科 等の頻出パターン
    → 15業種のスターターを Phase 1 で初期搭載

  共創で磨く部分:
    → テンプレートに無い固有ルール（社長の暗黙知）
    → テンプレートの修正（「うちではこう違う」）
    → 例外ケース・判断基準の精緻化
```

## 3. 価値提案 — なぜ無償 or 激安で協力してもらえるか

### パートナーへの提供価値（こちらが提供するもの）

```
1. 完全無償 or ¥10,000/月以下
   → 通常価格は¥30,000/月予定。Phase A'パートナーは永久に優遇

2. 自社の業務ナレッジがAIに整理される（それ自体に価値がある）
   → 「うちのルールが文書化されたことがない」という課題が解決する
   → 社長の頭の中が見える化される

3. 「シャチョツー共同開発パートナー」としての対外的な位置づけ
   → Webサイト・ピッチ資料へのロゴ・社名掲載（希望する場合）
   → 事例インタビュー・登壇機会（業界での信頼度向上）

4. 製品化後の特典
   → Phase B以降1年間無料（¥360,000相当）
   → 機能要望の優先実装（ロードマップ上位保証）
   → パートナー専用サポート（チャット直接対応）

5. 業界全体が賢くなる
   → 同業の他社が使うほど、業界ゲノムが充実する
   → 業界標準のノウハウがAIに蓄積され、全員が恩恵を受ける
   → 競合が使い始める前に自社が先行できる

6. 将来の参照顧客・登壇機会
   → VCへのピッチ・メディア取材で「最初から使っている企業」として登場できる
   → Series A以降の知名度向上に繋がる可能性
```

### パートナーへの要求（先方に求めるもの）

```
1. 週1〜2時間 × 3ヶ月のコミット（合計約12〜24時間）
   → 月1〜2回のオンラインセッション（30〜60分）
   → セッション間に質問シートへの回答（非同期・30分以内）

2. 業務ルール・フローを正直に教えてくれること
   → 「美化しない」こと。現場の実態・例外・ルール破りの実情も含めて

3. 使ってみた感想を率直に言ってくれること
   → 「良かった」だけでなく「これは役に立たない」「ここが違う」も必要

4. テンプレートへのフィードバック（「合ってる/違う/足りない」の評価）
   → これが50%OFFの対価の一部。共創企業はテンプレート品質向上の貢献者
   → フィードバックは匿名化して全テンプレートの改善に活用

5. 条件達成後に同業者を1社紹介してくれること
   → Q&A精度75%達成、本人満足度4/5以上を達成した後
   → 紹介先のクオリティは問わない（あくまで1社）
```

## 4. アプローチ・獲得プロセス（12週間プラン）

### Week 1〜2: ターゲット候補リスト作成

```
作業:
- 各業種5社ずつ、計15社のターゲット候補をリストアップ
- リストアップ基準: 選定基準の「必須要件」を満たすか確認
- 接点の整理: 直接接点 / 紹介可能な第三者 / 完全コールド の3分類

成果物:
.context/sales/partners/candidates.md
  ├── 建設業 候補5社（接点・Pain Point・優先順位）
  ├── 製造業 候補5社
  └── 歯科   候補5社
```

### Week 2〜3: ファーストコンタクト

```
アプローチ順位:
  1. 直接接点がある先（最優先）
  2. 信頼できる第三者からの紹介
  3. SNS・DM経由のコールドアプローチ（最後の手段）

コンタクトメッセージの骨格:
  - 「自社の経営AIツールを共同開発するパートナーを探しています」
  - 「売りに来ているのではなく、一緒に作ることへのご協力をお願いしたい」
  - 「30分、お話を聞かせていただけますか？」

禁止事項:
  - 「製品のご紹介を」という売り込みトーン
  - 機能を羅列したピッチ資料を最初に送る
  - 料金プランを先に提示する
```

### Week 3〜4: 初回ミーティング（30分・オンライン）

```
アジェンダ:
  1. 自己紹介・シャチョツーのコンセプト説明（5分）
     → 「社長の暗黙知をAIに移す」という一言で説明する
  2. 相手の現状ヒアリング（15分）
     → 「今、業務でいちばん頭を使っていること・面倒なことは何ですか？」
     → 「人に業務を教えるとき、いちばん伝えにくいことは何ですか？」
     → 「ベテランが辞めたとき、困ったことはありますか？」
  3. 共創プログラムの提案（10分）
     → 「3ヶ月間、一緒に御社の業務ナレッジをAIに覚えさせませんか？」
     → 提供価値・要求・スケジュール概要を伝える
     → 「次回、詳しく説明させてください」で終わる（即決を求めない）

判断基準（進める / 見送り）:
  進める: ヒアリング中に「それは確かに困ってる」という反応が1回以上ある
  見送り: 「特に困っていることはない」「今はシステム入れ替えを考えていない」
```

### Week 4〜5: 詳細説明 → MOU（覚書）締結

```
詳細説明ミーティング（45分）:
  1. ゲノム構築の具体的な手順を説明
  2. 「何を提供し、何をお願いするか」を明文化して共有
  3. 患者データ・機密情報を扱わないことを明確に説明（特に歯科）
  4. MOU（覚書）の内容確認・署名

MOUの主要内容:
  - 共同開発パートナーとしての位置づけ（顧客契約ではない）
  - ナレッジの帰属（業種テンプレートはシャチョツーが活用 / 個社固有ナレッジは当該企業に帰属）
  - 守秘義務（双方向）
  - 期間（3ヶ月）・コミット内容
  - 事例・ロゴ使用の許諾範囲

成果物:
.context/sales/partners/[company_id]/mou.md
```

### Week 5〜6: Phase 1 — テンプレート適用・ギャップ分析

```
各社への対応スケジュール（並行して3社同時進行）:

  Week 5: キックオフ + テンプレート適用セッション（60分）
    → 共通ベーステンプレート + 業種別スターターテンプレートを自動適用
    → 適用結果を画面共有: 「御社にはこのルール・フローが入っています」
    → 社長と一緒に確認: 「合ってる / 違う / 足りない」を1項目ずつ

  Week 6: ギャップ分析 + 固有ナレッジ初期収集（45分）
    → テンプレートに無い固有ルール・フローをヒアリング
    → 「テンプレートでは○○ですが、御社ではどうですか？」形式（ゼロからより答えやすい）
    → ギャップリスト作成: テンプレート修正項目 + 新規追加項目

  目標: ゲノム完成度 0.50-0.70 → 0.65-0.75（テンプレート修正 + 初期固有ナレッジ）
```

### Week 7〜9: Phase 2 — 固有ナレッジ補完

```
  Week 7: 暗黙知の明示化セッション（45分）
    → 「社長にしかわからない判断」「例外の扱い」を重点的にヒアリング
    → テンプレートのデフォルト値を御社固有の値に上書き
    → CLIベースでナレッジ構造化を実施

  Week 8: 追加インタビュー + 例外ケース収集（30分）
    → 「いちばん判断が難しいケース」を重点的に聞く
    → 業務フローの詳細ウォークスルー（テンプレートと実態の差分に集中）

  Week 9: 社長確認セッション（30分）
    → 構造化ナレッジの正確性を確認
    → 「このルールは正しいですか？」「抜けているものはありますか？」
    → テンプレート改善フィードバック収集（匿名化して他社向けテンプレートにも反映）

  目標: ゲノム完成度 0.75-0.85 + テンプレート改善FB 10件以上
```

### Week 10〜12: Phase 3 — 検証・精緻化

```
  Week 10: Q&Aテストセット作成 + 初回検証
    → 社長に「AIに聞いてみたい質問20問」を作ってもらう
    → テンプレート由来の回答 vs 固有ナレッジ由来の回答を区別して精度測定
    → BPOシャドウラン: テンプレートルールで動かしてみる（shadow modeで観察のみ）

  Week 11: Q&A精度改善 + 2回目検証（30分）
    → 低スコア回答を分析してナレッジ補強
    → 不足カテゴリを特定して追加ヒアリング

  Week 12: 最終評価 + 移行確認 + テンプレートFB最終回収
    → Q&A精度・カバレッジスコア・社長満足度を測定
    → 「このまま有料でも使いたいですか？」を確認
    → テンプレート改善最終レポート: 「御社のFBでテンプレートがこう改善されました」
    → Phase B移行に向けた同業者紹介依頼

  目標: Q&A精度75%以上 + ゲノム完成度0.85+ + 社長満足度4/5以上
```

## 9. フェーズ全体を通じたリスクと対策

### リスク1: 3社が揃わない（採用できない）

```
発生確率: 中
影響: 大（Phase B移行が遅れる）

対策:
  - 各業種5社の候補を持つ（1社目がNGでも次がある）
  - 接点経由を最優先（紹介ゼロのコールドは採用率が低い）
  - 採用に2ヶ月以上かかる場合は「1業種でも始める」判断をする
  - 代替業種を用意する（小売業・飲食業 → 建設/製造/歯科と同様のPain Pointを持つ業種）
```

### リスク2: 社長のコミット不足（忙しくてフィードバックが来ない）

```
発生確率: 高（特に建設・製造の繁忙期）
影響: 中（ゲノム品質に直結）

対策:
  - セッションの頻度を「月2回」に設定（週1は負荷が高い）
  - 非同期フィードバックを活用（質問シートをLINEやメールで送る）
  - 「30分で完結」を徹底する（1時間以上にしない）
  - コミット低下を検知したら即座にアジェンダを削減して続けることを優先
```

### リスク3: ナレッジの機密性への懸念

```
発生確率: 中（特に製造業・歯科）
影響: 大（パートナー離脱 or 情報提供が表面的になる）

対策:
  - MOUに「個社固有ナレッジは当該企業の資産」を明記
  - 「業種テンプレートとして使うのは一般化・匿名化したもの」を丁寧に説明
  - 歯科については「患者情報・診療記録は一切扱わない」を初回から繰り返し伝える
  - 必要に応じてNDA（機密保持契約）を別途締結
```

### リスク4: ゲノム品質が基準に達しない

```
発生確率: 中
影響: 中（Phase B移行が遅れる）

対策:
  - 追加インタビューセッションを最大2回まで許容（スケジュールに余裕を持つ）
  - 低精度の回答カテゴリを早期に特定してナレッジを補強
  - 「完璧を求めない」: 75%で移行し、Phase B以降も継続改善する
```

### リスク5: 競合他社に動きを察知される

```
発生確率: 低
影響: 中

対策:
  - Phase A'は「共同開発パートナー」として進める（パブリックな告知はPhase B後）
  - パートナー企業にも「Phase B前は対外的に非公開でお願い」と合意を取る
  - MOU に機密保持条項を含める
```

### 完成度スコア計算

```python
def calculate_genome_coverage(genome: dict) -> float:
    """
    ゲノムの完成度スコアを0.0〜1.0で算出。
    0.7以上で「実用レベル」、0.9以上で「完成」と判定。
    """
    required_score = count_filled(genome, "必須") / count_total(genome, "必須")
    recommended_score = count_filled(genome, "推奨") / count_total(genome, "推奨")
    optional_score = count_filled(genome, "任意") / count_total(genome, "任意")

    # 重み: 必須60% + 推奨30% + 任意10%
    coverage = required_score * 0.6 + recommended_score * 0.3 + optional_score * 0.1
    return round(coverage, 2)

# 判定基準（テンプレートファースト型）
# 0.0:     未適用（テンプレート適用前）
# 0.50-0.70: テンプレート適用直後（共通ベース+業種別スターター）
# 0.70-0.85: 共創Phase 2完了（固有ナレッジ補完済み）
# 0.85-0.90: 共創Phase 3完了（検証・精緻化済み）→ Phase B移行基準
# 0.90-1.0:  運用成熟（継続利用+フィードバックで到達）
```

## 13. フィードバックループ仕様

### 頻度設計

| 期間 | 頻度 | 形式 | 主な内容 |
|---|---|---|---|
| 1ヶ月目 | **週1回** | LINE + 30分Zoom | ゲノム品質確認、Q&Aテスト、バグ報告 |

```
LINE（速報・日常）:
  ・Q&Aの回答が間違っていた時 → 即座に「違います、正しくは○○」
  ・新しいルールができた時 → 「今日から見積り基準が変わりました」
  ・バグ発見時 → スクリーンショット + 説明

月次対面/Zoom（振り返り・深掘り）:
  ・ゲノム充足度スコアの変化レビュー
  ・Q&A精度の定量評価（正答率の推移）
  ・次月の優先ナレッジ領域の合意
  ・新しい業務フロー・ルールの追加インタビュー
```

### フィードバック分類と対応SLA

| 分類 | 具体例 | 対応SLA | 担当 |
|---|---|---|---|
| **バグ** | 「Q&Aがエラーを返す」「ログインできない」 | **24時間以内** に修正 | 開発 |
| **ゲノム修正** | 「このルールは間違い」「この業務フローは古い」 | **3営業日以内** に反映 | 開発 + 先方確認 |
| **テンプレート修正** | 「テンプレートのこのルールが業界標準と違う」 | **1週間以内** に検討・反映 | 開発（全テンプレートへの影響を評価） |
| **Q&A精度改善** | 「この質問に正しく答えてほしい」 | **1週間以内** に改善 | 開発（プロンプト/ナレッジ調整） |
| **機能要望** | 「○○もできたらいいな」 | **次Sprint** で検討 | 開発（バックログ追加） |
| **UI/UX** | 「使いにくい」「わかりにくい」 | **次Sprint** で改善 | 開発 |

### フィードバック→改善のフロー

```
[先方からのフィードバック]

  ↓

  ↓
[対応] → 修正・改善実施
  ↓
[通知] → 「ご指摘の件、修正しました」とLINEで報告
  ↓
[確認] → 先方に修正結果を確認してもらう
  ↓
[クローズ] → フィードバックログに記録
```

## 14. 共創→有料顧客 移行基準

### 移行トリガー条件（全て満たした時に移行提案）

```
■ 定量基準（システムが自動判定）
  □ ゲノム完成度スコア ≥ 0.85（テンプレート + 固有ナレッジの合計）
  □ Q&A精度 ≥ 80%（直近20件の質問に対する正答率）
  □ 週次アクティブ利用 ≥ 3回（Q&A or ナレッジ追加を週3回以上使用）
  □ 共創期間 ≥ 2ヶ月（最低限の検証期間を確保）
  □ テンプレートFB ≥ 10件（テンプレート品質向上への貢献）

■ 定性基準（対面ヒアリングで確認）
  □ 先方社長の主観評価: 「これは使える」（5段階で4以上）
  □ 継続利用意思の明示的確認
  □ 社内の他メンバーへの展開意欲があるか
```

### 移行プロセス

```
Step 1: 移行提案（全条件達成後1週間以内）
  ・ゲノム充足度レポート + Q&A精度レポートを共有
  ・「共創フェーズの成果」をまとめた1ページサマリー
  ・有料プランの説明（共創価格 → 正規料金のスケジュール）

Step 2: 料金合意
  ・共創価格: 正規料金の50%OFF（最初の3ヶ月間）
    例: Starter ¥30,000/月 → 共創価格 ¥15,000/月
  ・4ヶ月目以降: 正規料金に移行
  ・年額契約の場合: さらに10%OFF

Step 3: 契約締結
  ・利用規約への同意
  ・個人情報取扱い同意（Phase A'での暗黙的同意から正式同意へ）
  ・SLAの明示（Starterプラン基準）

Step 4: 正式利用開始
  ・ダッシュボード（Web UI）へのアクセス権発行
  ・チーム招待（他メンバーの追加）
  ・サポートチャネルの案内
```

### 移行しない場合の対応

```
■ 条件未達の場合
  ・共創期間を1ヶ月延長（最大1回）
  ・未達項目に集中してナレッジ追加セッションを追加
  ・2回目の延長後も未達 → 共創終了（データは先方にエクスポート提供）

■ 先方が移行を希望しない場合
  ・理由をヒアリング（価格/品質/タイミング/社内事情）
  ・データエクスポート提供（JSON形式）
  ・ゲノムテンプレートは匿名化して活用（事前同意済み）
  ・将来の再開は歓迎（6ヶ月以内ならデータ保持）

■ 共創の学びは必ずプロダクトに還元
  ・「なぜ移行しなかったか」をPMF検証の重要データとして記録
  ・ゲノムスキーマの改善、インタビュー手法の改善に活用
```

## 15. フィードバック→仕様変更の意思決定プロセス

> **§13のフィードバックループは「受信→修正→通知」の運用フロー。**
> **このセクションは「何を変えるか・何を変えないか」の判断基準を定義する。**

### 判断の原則

```
1人開発 × PMF前 = 「全てのフィードバックを実装する」は不可能。
判断基準は1つ: 「この変更はパイロット3社の継続利用に直結するか？」
```

### 意思決定フレームワーク

```
[フィードバック受信]
  ↓
[分類] → §13の6カテゴリ（バグ/ゲノム修正/テンプレート修正/Q&A精度/機能要望/UI・UX）
  ↓
[緊急度判定]
  ├── ブロッカー（業務が止まる）→ 24時間以内に対応。他の作業を止める
  ├── 精度問題（回答が間違い）→ 3営業日以内。ナレッジ/プロンプト修正
  └── 改善要望（あったら嬉しい）→ バックログに記録。週次で優先度判定
  ↓
[矛盾チェック] → 複数社から矛盾するFBが来た場合
  ├── 業種固有の違い → 各社のgenome_customizationsで個別対応（テンプレートは変えない）
  ├── 好みの違い → 多数派を採用。少数派にはカスタマイズオプションを検討
  └── 本質的な設計矛盾 → 3社全員にヒアリングして合意形成。1週間以内に決定
  ↓
[実装判断] → 以下の基準で判断
  ├── YES: パイロット3社中2社以上が「これがないと使えない」と言った
  ├── YES: 積算精度に直接影響する（NPS/PMFゲートに直結）
  ├── LATER: 1社だけの要望 + 積算精度に無関係 → Phase B以降
  └── NO: 現在の建設業フォーカスから外れる要望 → 丁重に断る
  ↓
[実装→通知→確認] → §13のフローに合流
```

### 週次バックログ棚卸し（毎週月曜・15分）

```
対象: 未対応のフィードバック全件
作業:
  1. 新規FBを上記フレームワークで分類
  2. 今週実装するもの（最大3件）を決定
  3. 来週以降に回すものを先方に連絡（「認識しています。○月対応予定です」）
  4. 対応不要と判断したものは理由を記録してクローズ

記録先: バックログはGitHub Issues（ラベル: pilot-feedback）
判断ログ: shachotwo/h_意思決定記録/ に月次でサマリーを残す
```

### 「仕様を変える」vs「運用で回す」の判断基準

```
仕様変更（コード修正）すべきケース:
  ・同じ問題が2社以上で発生
  ・手動回避策がない or 回避策が複雑
  ・積算精度・Q&A精度に影響

運用で回すべきケース:
  ・1社だけの固有事情
  ・テンプレートのカスタマイズで対応可能
  ・UI上の好みの問題（慣れで解消される可能性）
```

### なぜSlackではなくLINEか

```
建設業の社長（20-100名規模）の実態:
  ・Slackを使っていない（100%に近い）

  ・LINEは全員が毎日使っている
  ・「新しいツールを入れてくれ」は摩擦になる

結論: パイロット3社が建設業である限り、チャネルはLINE一択。
将来IT/SaaS企業に展開する時にSlack対応を検討する。
```

### 現在のフロー vs 修正後のフロー

```
【現在】
  登録 → 業種選択 → テンプレート適用 → 「Q&Aを試してみてください」
  問題: Q&Aの価値がピンとこない。テンプレートの回答は一般的すぎる
  離脱リスク: 「ふーん、まあそうだよね」で終わる

【修正後】
  登録 → 業種選択 → テンプレート適用 → 「見積書PDFをアップロード」
  効果: 「自分の見積書」が即座に積算される → 自分ごと化
  感情: 「え、もう単価まで出るの？」→「ここの単価違うな、修正…」→「次はもっと正確になるのか」
  結果: 初回セッションで見積業務の価値を体感 + 学習データ1件目が蓄積
```
