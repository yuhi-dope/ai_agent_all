# シャチョツー 成長戦略サマリー（PLGメイン）

> **本ドキュメントの位置づけ**: Salesforce/Anthropic/Cursor/HubSpot/Ripplingの戦略分析を経て策定した、
> シャチョツーの成長戦略の統合サマリー。各設計ドキュメントへの反映指針として使用する。
>
> **作成**: 2026年3月
> **前提**: 2年で対象ターゲットの30%に導入 = 高速成長が必須

---

## 0. 戦略の結論（1ページサマリー）

```
■ 成長モデル: 「Anthropicの速さ」×「HubSpotの中小企業適合性」

  PLG（プロダクト主導成長）がメインエンジン（70%）
  パートナーは温まったリードの刈り取り役（20%）
  プラットフォーム連携は既存流通に乗る手段（10%）

■ 最大の武器: データネットワーク効果
  「使うほど全員が賢くなる」— これが最大の訴求であり、最大の堀

■ PLG成立の前提条件: Day 1 Value
  Genomeテンプレートの品質が全ての前提。ここが崩れるとPLGは不可能。

■ 設計の方向性:
  ・Compound Startup: 元の設計（brain + workers + proactive）が正しい
  ・Genome学習型: マーケットプレイスではなく、ユーザー使用データから自動学習
  ・グローバル: 設計はi18n対応、実行は日本ファースト（裏でアジアテスト）
  ・エンタープライズ: 設計で対応準備、実装はYear 2-3
```

### 1.2 採用する戦略要素

```
■ Anthropicから採用:
  ① プロダクト品質 > 営業の量（品質が口コミを生む）
  ② 無料をケチらない（無料枠を広げたら有料も増えた）
  ③ 既存プラットフォームに乗る（自分で売らない）
  ④ パートナーは「結果として」集まる（口説かない）

■ Cursorから採用:
  ⑤ マーケ費ゼロでも伸びるプロダクト（バイラルが自然発生）
  ⑥ 少人数で異常な生産性（AI活用での開発効率）

■ HubSpotから採用:
  ⑦ Free → Starter → Growth の3段階（心理的ジャンプを最小化）
  ⑧ パートナーがGrowth以上の大型案件を刈り取る分業

■ Ripplingから採用:
  ⑨ 共通データ基盤（9次元Twin）の上に複数プロダクトを載せる
  ⑩ クロスセルが自動的に発生する設計

■ 不採用:
  ✗ Salesforce型の大規模フィールドセールス（単価が合わない）
  ✗ Salesforce型のパートナー育成（時間がかかりすぎる）
  ✗ テンプレートマーケットプレイス（Genome自動学習の方が強い）
```

## 2. 成長エンジンの優先順位

```
■ リソース配分

  1位（30%）: プロダクト品質（Genome + Day 1 Value）
    → Genomeテンプレートの精度がPLG成立の絶対条件
    → Day 1で「おっ」と思わせないと全てが無駄

  2位（25%）: データネットワーク効果
    → 「1,000社の経営者の知恵が詰まったAI」が最大の訴求
    → 使うほど全員が賢くなる → 参加する動機が生まれる
    → Network Intelligence層（匿名化ベンチマーク）がこれを実現

  3位（15%）: PLGフロー（Free → Starter → Growth）
    → 意思決定コストゼロで始められる
    → 自然なアップグレードトリガー設計
    → バイラルループ（レポートフッター、ベンチマーク、紹介）

  4位（10%）: サプライチェーン伝播
    → 元請→下請の連鎖（建設業で特に強力）
    → 業界内の横伝播（同業者の口コミ）

  5位（10%）: パートナー（温まったリードの刈り取り）
    → 税理士/コンサルが有料転換・大型化を支援
    → Free→Starter はセルフサーブ、Growth以上でパートナー介入
    → パートナーは「口説く」のではなく「プロダクトの成功で自然に集まる」

  6位（5%）: プラットフォーム連携
    → LINE WORKS Bot / kintone プラグイン / freee連携
    → 既存の流通網に乗る（自分で集客しない）

  7位（5%）: コンテンツ / コミュニティ
    → YouTube / X / 業界メディア
    → 成功事例の発信（ユーザーをヒーローにする）
```

## 3. データネットワーク効果（最大の武器）

```
■ 定義
  ユーザーが増えるほど、全ユーザーの体験が良くなる仕組み。
  Wazeの「ドライバーが増えるほど渋滞情報が正確になる」と同じ。

■ シャチョツーでの実現
  建設業A社が「工期遅延の判断基準」を入力
  建設業B社が「下請け選定の基準」を入力
  建設業C社が「安全管理のフロー」を入力
  → 建設業の暗黙知データベースが自動的に形成
  → 新規D社は Day 1 から業界知見付きで使える
  → ユーザーが増えるほど Day 1 Value が向上 → PLGが加速

■ 技術的な実現方法（既存設計の活用）
  ・brain/knowledge/: 個社ナレッジのベクトル化・構造化
  ・brain/genome/: 業界テンプレートの自動精緻化（遺伝子合成モデル）
  ・network/anonymizer.py: k-匿名化（個社データを匿名化して集約）
  ・network/benchmark.py: 業界ベンチマーク計算
  ・network/patterns.py: 業界パターン集約
  → これらは全て 02_プロダクト設計.md に既に設計済み

■ 競合優位性（堀）
  ・データ量は時間とともに蓄積 → 後発は追いつけない
  ・業界特化の暗黙知は汎用AI（ChatGPT等）では代替不可能
  ・匿名化ベンチマークは「参加しないと損」の構造 → ロックイン

■ 訴求メッセージ
  旧: 「経営のナレッジを管理するAI」
  新: 「1,000社の経営者の知恵が詰まったAI。使うほど業界全体が賢くなる」
```

## 4. PLG成立の前提条件: Day 1 Value

```
■ なぜDay 1 Valueが全ての前提か

  Anthropicが爆速で伸びた理由:
  → Claude.aiを開いて質問 → 3秒で回答 → Day 1 Value = 即体験

  シャチョツーの課題:
  → 登録 → ナレッジ入力 → 構造化 → Q&A → 数日〜数週間後にようやく価値
  → これではPLGが機能しない（途中で離脱する）

■ 解決策: Genomeテンプレートで「80%完成状態」からスタート

  理想フロー:
  1. 登録（LINE連携 or メールのみ。30秒）
  2. 業種選択（建設業）
  3. Genomeテンプレートが即ロード（業界の暗黙知が80%入っている）
  4. 「杭工事の坪単価は？」→ テンプレートから即答
  5. 「おっ、すごい。でも自社の基準とちょっと違うな」
  6. ファイルアップロード開始（自社データでカスタマイズ）
  7. 使い込むほど自社専用に → 手放せない → 有料課金

  → Day 1 Value = Genomeテンプレートの品質に完全依存
  → テンプレートが良ければPLG成立、悪ければ全て崩壊

■ Day 1 Value 3機能（Genomeがなくても即使える保険）
  ① ファイルチェック（アップロード → 整合性チェック・異常検知）
  ② 議事録自動作成（音声 → 構造化議事録）
  ③ 文書下書き（用途指定 → LLM下書き生成）
  → これらはナレッジゼロでも動く。Genomeと合わせて二重の安全網。
```

## 5. Compound Startup設計（確認済み）

```
■ 現在の設計は既にCompound Startup
  共通データ基盤: 9次元Twin（社長の経営判断モデル）

  Product 1: 経営ナレッジ AI（brain/knowledge + brain/extraction）
  Product 2: 能動提案エンジン（brain/proactive + brain/twin）
  Product 3: BPO自動実行（workers/bpo + workers/connector）
  Product 4: 要件定義AI（workers/engineer）
  Product 5: 業務フロー可視化（brain/visualization）
  Product 6: What-ifシミュレーター（brain/twin）
  Product 7: 業界ベンチマーク（network/）

  → Ripplingの「Employee Graph」 = シャチョツーの「9次元Twin」
  → 共通基盤が強固なので、新プロダクト追加が高速

■ 将来の拡張（Year 2-3以降）
  Product 8: AI会計（社長の判断基準で仕訳 → freee/MF連携）
  Product 9: AI人事（採用基準・評価を社長の方針で）
  Product 10: AI営業（提案書・見積を社長の方針で自動生成）
  → ARPU: ¥5万 → ¥30-50万/月（6-10倍）
```

## 6. エンタープライズ対応方針

```
■ 方針: 設計で準備、実装はYear 2-3

■ なぜ今やらないか
  ① データの堀を先に作る → エンプラが「このデータが欲しい」と来る
  ② 中小→エンプラは自然に起きる（成長企業、大企業の新規事業部）
  ③ エンプラ営業は金と時間を食う（1社6-12ヶ月、今のフェーズでは致命的）

■ 設計レベルで今やること
  ・マルチユーザー設計（CEO + CFO + COO が各自のTwinを持てる拡張性）
  ・RLSのスコープ拡張（company_id + department_id の2層分離を見越す）
  ・監査ログの詳細化（SOC2/ISMS対応を見越す）
  ・SSO/SAML対応の設計余地

■ 実装レベルでYear 2-3にやること
  ・マルチユーザー + 部門別ナレッジ分離
  ・承認ワークフロー
  ・SSO/SAML実装
  ・SOC2取得
  ・SAP/Oracle連携コネクタ
  ・カスタムSLA
```

## 7. グローバル対応方針

```
■ 方針: 設計はグローバル対応、実行は日本ファースト

■ 表（リソース90%）: 日本全取り
  ・PLG + Genome学習 + サプライチェーン伝播で高速拡大
  ・初期3業種 → 全業種（Genomeの自動学習で対応業種が自然拡大）

■ 裏（リソース10%）: アジアテスト
  ① アーキテクチャのi18n対応（DB・UI・通貨・税制をconfigで切替可能）
  ② タイ or インドネシアでLINE Bot版βテスト（LINEが強い国）
  ③ 英語のLP + waitlist（海外反応の測定、コストほぼゼロ）

■ なぜアジアが有望か
  ・東南アジアの中小企業 = 日本と同じ課題（属人経営、ナレッジ流出）
  ・ITリテラシーが低い → AI代替の需要が日本以上
  ・LLMは多言語対応済み → ローカライズコスト低
  ・LINEが主要メッセンジャー（タイ、台湾、インドネシア）
```
