## Section 1: チャーン防止設計

> 目標: 月次チャーン 2%以下（年間解約率 21%以下）、理想 1%以下（年間 11%以下）
> Net Revenue Retention 100%以上（解約分をアップセル+追加エージェントで補う）

### 1.1 ヘルススコア（0-100）

#### スコア算出式

```
Health Score = Σ (Signal_i × Weight_i)

┌────────────────────────┬────────┬──────────────────────────────────────────────┐
│ シグナル                │ 重み    │ 計算方法                                      │
├────────────────────────┼────────┼──────────────────────────────────────────────┤
│ ① ログイン頻度          │ 20%    │ 直近14日間のDAU/WAU比率を0-100にスケーリング     │
│ ② Q&A利用頻度          │ 25%    │ 月間Q&A回数の推移（前月比・トレンド）で0-100評価  │
│ ③ ナレッジ更新頻度      │ 20%    │ 直近30日間の新規/更新ナレッジ数（0件=0, 10件+=100）│
│ ④ 機能利用深度          │ 20%    │ 利用機能カテゴリ数 ÷ 全カテゴリ数 × 100        │
│ ⑤ NPS（直近回答）       │ 15%    │ NPS値を0-100に変換（Detractor=0, Passive=50,    │
│                        │        │ Promoter=100）                                │
└────────────────────────┴────────┴──────────────────────────────────────────────┘
```

#### シグナル詳細定義

```
① ログイン頻度（Weight: 20%）
  ・データソース: Supabase Auth の last_sign_in_at + セッションログ
  ・計算: DAU/WAU Ratio（Sticky Factor）
    - DAU/WAU ≥ 0.7 → スコア100（毎日使っている）
    - DAU/WAU 0.4-0.7 → 線形補間
    - DAU/WAU < 0.2 → スコア0（ほぼ使っていない）
  ・閾値: 7日間ログインなし → イエローシグナル発火

② Q&A利用頻度（Weight: 25%）— 最重要シグナル
  ・データソース: qa_logs テーブル
  ・計算: 従量課金のため「プラン基本枠」は存在しない。
    代わりに月間Q&A回数のトレンドで評価:
    - 月100回以上 & 前月比維持/増加 → スコア100
    - 月30-100回 → 線形補間
    - 月10回未満 → スコア20以下
    - 前月比 50%以上減少 → 利用量に関わらずスコア半減
  ・閾値: 前月比 50%以上減少 → イエローシグナル発火
  ・根拠: Q&Aはプロダクトの中核価値。従量課金では利用量=売上に直結
  ・チャネル別追跡: LINE経由/メール経由/Web経由 を区別し、
    どのチャネルの利用が減ったかを特定可能にする

③ ナレッジ更新頻度（Weight: 20%）
  ・データソース: knowledge_items の created_at / updated_at
  ・計算:
    - 30日間で10件以上の新規/更新 → スコア100
    - 5-9件 → 線形補間
    - 0件 → スコア0
  ・閾値: 60日間更新なし → オレンジシグナル発火
  ・根拠: ナレッジが育たない = スイッチングコストが低い = 解約障壁が低い

④ 機能利用深度（Weight: 20%）
  ・データソース: feature_usage_logs テーブル
  ・カテゴリ定義:
    [Q&A] [ナレッジ入力] [業務フロー可視化] [意思決定ルール] [能動提案確認]
    [BPOタスク実行] [What-if分析] [ダッシュボード] [エクスポート]
  ・計算: 直近30日間で1回以上利用したカテゴリ数 ÷ プランで利用可能な全カテゴリ数
  ・閾値: 2カテゴリ以下の利用 → 「機能未活用」アラート

⑤ NPS（Weight: 15%）
  ・収集タイミング:
    - 導入3ヶ月後（オンボーディング完了時）
    - 以降6ヶ月ごと
    - 価格移行タイミング（3→4ヶ月目、6→7ヶ月目）
  ・計算:
    - Promoter (9-10) → スコア100
    - Passive (7-8) → スコア50
    - Detractor (0-6) → スコア0
  ・閾値: Detractor回答 → 即時CS通知
```

#### スコアのリアルタイム更新

```
更新頻度: 日次バッチ（深夜3:00 JST）
保存先: company_health_scores テーブル（company_id, score, signals_json, calculated_at）
履歴保持: 365日間（トレンド分析用）
```

#### 1.3.4 従量課金モデルにおけるチャーン対策

```
従量課金の特徴:
  ・基本料¥30,000/月は低い → 解約障壁も低い
  ・利用が減る → 請求額が下がる → 「使っていない」と感じやすい
  ・逆に利用が増える → 価値を実感 → 自然にリテンション

チャーンリスクが高いタイミング:
  ① 導入1ヶ月目: Day 1 Value 3機能以外に進展がない場合
  ② 利用量急減時: 月間従量が前月比50%以下に落ちた場合
  ③ 月額¥500K超過時: 固定プラン提案のタイミング（高額感で解約検討）

対策:
  ① 導入1ヶ月目:
     ・Day 1 Value → Level 1（データ収集）への移行を積極的にガイド
     ・「次はこのExcelを自動チェックしてみませんか？」とプロダクト内で提案
     ・チャネル別利用率を監視（LINEのみ→メール/ファイルにも拡大促進）

  ② 利用量急減時:
     ・自動検知（日次バッチ） → CS通知
     ・「先月はQ&A 200回でしたが今月は30回です。お困りごとはありませんか？」
     ・利用量に応じたROIレポートを自動送付

  ③ 月額¥500K超過時:
     ・固定プランへの移行提案を自動送付
     ・「固定プランなら月額¥○○で上限なし」のシミュレーション表示
     ・移行率モニタリング（目標: 80-90%）

  ④ 低利用・低額顧客の離脱防止:
     ・月額¥30K（基本料のみ）が3ヶ月続いたら → CS介入
     ・「無料で活用支援セッションを実施します」を案内
     ・チャネル変更提案（Web未利用→LINE提案、等）
```

## Section 2: マルチテナント運用設計

### 2.1 リソース公平性（Noisy Neighbor 防止）

#### 2.1.1 LLM呼び出しレートリミット

```
従量課金モデルのため「月間上限」は原則なし。
代わりにバースト制御（分単位）と異常利用検知で公平性を担保。

┌──────────────┬──────────────┬──────────────────────────────────────────┐
│ リソース      │ バースト制限  │ 備考                                      │
├──────────────┼──────────────┼──────────────────────────────────────────┤
│ Q&A          │ 30回/分      │ 従量: ¥30/回。上限なし（バースト制限のみ） │
│ BPOタスク    │ 5タスク/分   │ 従量: ¥200/件。上限なし                    │
│ ファイル処理  │ 10件/分      │ 従量: ¥50/件。上限なし                     │
│ 月額上限アラート│ —          │ ¥500K/月超過時に固定プラン提案を自動送付  │
└──────────────┴──────────────┴──────────────────────────────────────────┘

異常利用検知:
  ・月間利用額が前月の5倍を超えた場合 → 自動アラート + CS確認
  ・API経由の大量自動リクエスト → Bot検知 + 一時制限
  ・同一質問の反復（DoS的パターン） → 自動ブロック

実装方式:
  ・Token Bucket アルゴリズム（Redis による分散カウンタ）
  ・キー: rate_limit:{company_id}:{resource_type}:{window}
  ・窓: per-minute（バースト制御のみ。月間上限はなし）
  ・超過時レスポンス: HTTP 429 + Retry-After ヘッダ
  ・利用額通知: ¥100K/¥300K/¥500K 到達時にメール通知
```

#### 2.1.2 ベクトル検索クォータ

```
pgvector検索は全テナントが同一DBインスタンスを共有するため、
1テナントの大量検索が他テナントの応答速度を劣化させるリスクがある。

対策:
  ① クエリタイムアウト: 5秒（HNSW検索は通常50ms以内）
  ② 同時検索クエリ数制限: テナントあたり最大10並列
  ③ ナレッジ件数上限（従量課金モデルでは利用額ベースで段階設定）:
     - 基本（月額¥100K未満）: 10,000件
     - 中規模（月額¥100K-300K）: 50,000件
     - 大規模（月額¥300K超 or 固定プラン）: 100,000件
  ④ HNSW インデックスパラメータ:
     - ef_construction: 128（構築時の精度）
     - ef_search: 64（検索時の精度 vs 速度トレードオフ）
     - m: 16（グラフの接続数）
  ⑤ コネクションプール: pgBouncer（Supabase標準搭載）で最大接続数管理
     - per-tenant接続数: 最大5本（トランザクションモード）
```

### 2.2 テナント分離の多層防御

```
┌─────────────────────────────────────────────────────────────────┐
│                    テナント分離 3層防御                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Layer 1: DB層（RLS — 最終防衛線）                               │
│    ・全テーブルに company_id ベースの RLS ポリシー適用            │
│    ・RLSはDBエンジンレベルで強制 → アプリのバグでは迂回不可能     │
│    ・auth.uid() → users.company_id の間接参照で自動フィルタ      │
│    ・superuser権限のアプリケーション接続を禁止                    │
│      （service_role キーはバッチ処理のみ、APIからは使用しない）    │
│                                                                 │
│  Layer 2: API層フィルタ（FastAPI ミドルウェア）                   │
│    ・JWTから company_id を抽出し、リクエストコンテキストに注入     │
│    ・全エンドポイントのクエリに company_id フィルタを自動付与      │
│    ・company_id がリクエストパスのテナントIDと一致しなければ 403   │
```

### 3.2 BPO実行中の障害対応

#### 3.2.1 途中停止時のロールバック

```
BPOタスクはSaaS操作（kintone レコード作成、freee 仕訳登録等）を伴うため、
途中停止時に「中途半端な状態」を残さない設計が必須。

■ ロールバック戦略

  BPOStep 定義（02_プロダクト設計.md 10.2項より）:
    class BPOStep(TypedDict):
        step_id: int
        connector: str
        action: str
        parameters: dict
        rollback_action: str | None    # ← この設計が鍵

  各ステップに rollback_action を事前定義:
  ┌──────────────────────────┬──────────────────────────────┐
  │ 正方向のアクション        │ ロールバックアクション         │
  ├──────────────────────────┼──────────────────────────────┤
  │ kintone レコード作成      │ 作成したレコードの削除         │
  │ freee 仕訳登録           │ 登録した仕訳の削除             │
  │ Slack メッセージ送信      │ ロールバック不可（送信済み）   │
  │ Google Calendar 予定作成 │ 作成した予定の削除             │
  │ ファイルアップロード       │ アップロードしたファイルの削除  │
  └──────────────────────────┴──────────────────────────────┘

  ロールバック実行順序:
    障害発生 → 現在ステップを中断
    → 完了済みステップを逆順にロールバック
    → ロールバック結果を bpo_execution_logs に記録
    → テナントに通知（「タスクが中断されました。実行済み操作は取り消しました」）

  ロールバック不可能なケース:
    ・メール/Slack送信（送信取り消し不可）
    ・外部API側の制約で削除不可能な操作
    → これらは「ロールバック不可能」フラグを付与し、手動対応を促す通知を送信
```

#### 3.2.2 冪等性の担保

```
■ 冪等性キー（Idempotency Key）

  全BPOタスクに一意の idempotency_key を付与:
    idempotency_key = f"{company_id}:{task_id}:{step_id}:{attempt_number}"

  処理フロー:
    ① リクエスト受信 → idempotency_key で bpo_execution_logs を検索
    ② 既に成功記録あり → 同じ結果を返す（再実行しない）
    ③ 記録なし or 失敗記録 → 実行して結果を記録

  SaaS側の冪等性対応:
    ・kintone: レコードID指定更新（PUT は冪等）
    ・freee:  deal_id 指定（重複作成防止）
    ・Slack:  ts（タイムスタンプ）でメッセージ特定
    → 各コネクタが SaaS ごとの冪等性パターンを実装

■ ステップ実行の Exactly-Once 保証

  LangGraph のチェックポイント機能を活用:
    ・各ステップ完了時に state を永続化（Supabase）
    ・障害復旧後、最後のチェックポイントから再開
    ・完了済みステップはスキップ（idempotency_keyで判定）
```

### 3.3 Graceful Degradation 設計

```
■ 段階的機能縮退の設計原則

  「全機能停止」ではなく「使える範囲で使い続けられる」状態を維持する。
  顧客のデータ損失は絶対に発生させない。

■ 機能縮退レベル

  Level 0: 正常稼働（全機能利用可能）
      ↓ LLM障害
  Level 1: LLM縮退モード
    ・Q&A → セマンティックキャッシュからのキャッシュ応答
    ・能動提案 → 最新の生成済み提案を表示（新規生成停止）
    ・ナレッジ構造化 → 入力を受け付けてキューに保存（LLM復旧後に処理）
    ・BPO → 新規タスクの計画生成を一時停止（承認済みタスクは実行継続）
    ・表示: 「AI分析が一時的に制限されています。キャッシュされた情報でお答えします」
      ↓ Embedding障害
  Level 2: ベクトル検索縮退モード
    ・Q&A → キーワード全文検索のみ（PostgreSQL ts_vector）
    ・ナレッジ新規登録 → 受け付けるがベクトル化はキューに積む
    ・表示: 「検索精度が一時的に低下しています」
      ↓ DB障害
  Level 3: 読み取り専用モード（リードレプリカが生きている場合）
    ・Q&A → 最後の同期時点のデータで応答
    ・ナレッジ閲覧 → 可能
    ・書き込み系 → 全て一時停止
    ・BPO → 全停止
    ・表示: 「現在メンテナンス中です。閲覧のみ可能です」
      ↓ 全コンポーネント障害
  Level 4: 静的ページモード
    ・Cloud Storage にホストした静的HTMLを表示
    ・内容: 障害状況 + 復旧見込み + 連絡先
    ・ステータスページ URL: status.shachotwo.com
    ・表示: 「現在システム障害が発生しています。復旧作業中です」

■ 縮退判定ロジック（自動）

  ヘルスチェックエンドポイント: GET /health
  チェック項目:
    ・DB接続: SELECT 1（3秒タイムアウト）
    ・LLM接続: Gemini API ping（5秒タイムアウト）
    ・Embedding接続: Voyage AI ping（5秒タイムアウト）
    ・Redis接続: PING（1秒タイムアウト）

  各チェックの結果に応じて degradation_level を自動設定:
    全OK → Level 0
    LLM NG → Level 1
    LLM + Embedding NG → Level 2
    DB NG (レプリカOK) → Level 3
    DB NG (レプリカNG) → Level 4

  判定頻度: 30秒ごと（Cloud Runインスタンス内）
  状態通知: degradation_level 変更時に #ops-alerts へ即時通知
```
