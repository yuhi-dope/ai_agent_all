# 共通 BPO フロー設計 — 業種横断の再利用可能部品

> **目的**: 6業種×37ドメインで共通する BPO フローを部品化し、開発量を削減する。
> 08_BPOタスクカタログ Section 7.2 で特定した共通パターンの詳細設計。
>
> **設計原則**:
> - 共通フローは `domain/{ドメイン}/` レイヤーに配置（業種非依存）
> - 業種固有の差分は `specialist/{業種}_{ドメイン}/` で上書き
> - 全フローに ChannelResolver を適用（SaaS/Excel/manual を自動切替）
> - **チャネルファースト**: 入力は LINE Bot / Email(IMAP) / ファイルアップロード / 音声(Whisper) / FAX(eFAX) が主要。SaaS APIはオプション
> - **Level 0-4 段階的自動化**: 各フローはLevel 0（通知のみ）から開始し、実績に応じてLevel昇格
>   - Level 0: リマインダー / Level 1: データ収集 / Level 2: 下書き作成 / Level 3: 承認付き実行 / Level 4: 自律実行
> - **統合ファイル処理エンジン**: Excel/PDF/Word/PowerPoint/CSV/画像/音声/動画/CAD を統一パイプラインで処理
> - **Day 1 Value**: ★マーク付きフローは導入初日から Level 0-1 で即利用可能
> - **従量課金**: BPO実行 ¥200/件、ファイル処理 ¥50/件

---

## 1. 経理系共通フロー

### 1.1 請求書発行フロー

```
トリガー: 月次スケジュール or 納品完了イベント or LINE「請求書作って」
対象ドメイン: 全業種の経理
Day 1 対応: ★（Level 0-1 で即利用可能。Excelテンプレートから請求書ドラフト生成）
従量課金: BPO実行 ¥200/件

■ チャネルファースト入力
  ・LINE Bot: 「今月の請求書作って」→ フロー開始
  ・メール: 請求データExcel添付 → 自動検出・フロー開始
  ・ファイルアップロード: Excel/CSV → 統合ファイル処理エンジンで構造化
  ・音声: 「A社に50万、B社に30万」→ Whisper→構造化

■ Level対応
  Level 0: 「請求書作成のリマインド」を月末に通知
  Level 1: 請求データを自動収集・一覧化して提示
  Level 2: 請求書ドラフトを自動生成（人間が確認・送信）
  Level 3: 承認ボタンで請求書確定・送信
  Level 4: 過去パターンに基づき自動発行（異常値のみ確認）

■ 意図レベルの計画（ChannelResolver が実行手段を自動選択）

  Step 1: 請求対象データの取得
    意図: 「当月の請求対象データを取得する」
    → Excelファイルアップロード → 統合ファイル処理エンジンで構造化（最も一般的）
    → kintone接続あり → kintone_search_records
    → Sheets接続あり → sheets_read_range
    → LINE/音声入力 → LLM構造化
    → 何もなし → manual「請求対象のリストを入力してください」

  Step 2: 請求書ドラフト作成
    意図: 「請求書を作成する」
    → freee接続あり → freee_create_invoice（ドラフト状態）
    → MF接続あり → mf_create_invoice
    → 何もなし → sheets にテンプレート出力 or manual

  Step 3: 承認チェックポイント ★必須
    リスクレベル: Medium（金額変更を含む）
    → 承認者に通知（slack/アプリ内）
    → 承認画面: 件数・合計金額・前月比を表示
    → [承認] [修正] [却下]

  Step 4: 確定・送信
    意図: 「請求書を確定して送信する」
    → freee → freee_send_invoice（メール送信）
    → manual → 「以下の請求書を送信してください」タスク発行

  Step 5: 結果記録
    → 処理件数・合計金額・所要時間を記録
    → ROI計算（手動時間との差分）

■ 業種別 specialist での上書き例

  建設業: Step 1 で「出来高ベースの請求」ロジックを追加
    → 工事台帳の出来高率 × 契約金額 で請求額を算出
    → JV工事は持分按分を自動計算

  製造業: Step 1 で「納品書ベースの請求」ロジックを追加
    → 納品確認済みの受注データのみを対象

  士業: Step 1 で「タイムシートベースの請求」ロジックを追加
    → 案件別の稼働時間 × 単価 で請求額を算出

  不動産: Step 1 で「仲介手数料 + 管理手数料」の2種類を処理
  医療/歯科: 保険請求（レセプト）と自費請求を分離
```

### 1.2 入金消込フロー

```
トリガー: 銀行入金データの取得（日次 or 週次）
対象ドメイン: 全業種の経理

  Step 1: 入金データ取得
    意図: 「銀行の入金データを取得する」
    → 銀行API接続あり → bank_api_get_transactions
    → CSV手動アップロード → OCR + パース
    → manual → 「入金一覧を入力してください」

  Step 2: 売掛金との照合（名寄せ）
    意図: 「入金を請求書と紐付ける」
    処理:
      ① 完全一致: 振込名 = 取引先名 AND 金額一致 → 自動消込
      ② 部分一致: 振込名の揺れ（(株)→株式会社 等）→ 候補提示
      ③ 不一致: → manual「この入金の取引先を教えてください」
    学習: ②③の結果を名寄せ辞書に自動追加

  Step 3: 消込実行
    → freee → freee_match_payment
    → manual → 「以下の消込を記帳してください」

  Step 4: 未消込アラート
    残った未消込 → 経理担当に通知
    「3件の入金が未消込です。確認してください」

■ 業種別差分
  不動産: 入居者名と物件の紐付けが追加（部屋番号で照合）
  士業: 顧問料の定額入金 → 定額マッチングロジック
```

### 1.3 経費精算フロー

```
トリガー: レシート/領収書のアップロード or LINE写真送信
対象ドメイン: 全業種の経理
Day 1 対応: ★（Level 0-1 で即利用可能。写真→OCR→仕訳候補）
従量課金: ファイル処理 ¥50/件

■ チャネルファースト入力
  ・LINE Bot: レシート写真を送信 → 即OCR処理
  ・メール: レシート画像添付 → 自動検出
  ・ファイルアップロード: PDF/画像 → Document AI

  Step 1: レシート取得
    意図: 「レシートを読み取る」
    → LINE写真 / メール添付 / カメラ撮影 / スキャン → Document AI
    → 出力: {日付, 店名, 金額, 税率, 品目}

  Step 2: 仕訳候補生成
    → LLM が品目から勘定科目を推定
    → 過去の仕訳パターン（learned ルール）を参照
    → 「交通費 ¥1,200 / 現金」候補を提示

  Step 3: 承認チェックポイント
    リスクレベル: Low（少額経費）→ manager 以上
    → [承認] [科目修正] [却下]

  Step 4: 記帳
    → freee → freee_create_transaction
    → manual → 「以下の仕訳を記帳してください」

■ 学習ポイント
  ・科目修正があった場合 → 修正駆動学習
    「コンビニ購入の文房具 → 消耗品費（旅費交通費ではない）」
  ・3回同じ修正 → 自動ルール化
```

### 1.4 月次決算チェックリストフロー

```
トリガー: 月末 + 5営業日（締め日の設定可能）
対象ドメイン: 全業種の経理

  Step 1: チェック項目の自動検証
    □ 未処理仕訳がないか → freee APIで検出
    □ 仮勘定残高が0か → freee APIで確認
    □ 銀行残高と帳簿残高が一致するか → 銀行API × freee
    □ 売掛金/買掛金の残高確認 → 前月比で異常値検出
    □ 固定資産の減価償却計上 → スケジュール照合
    □ 消費税の仮払/仮受バランス確認

  Step 2: 結果レポート生成
    → 「5/6項目クリア。未処理仕訳が3件あります」
    → 未クリア項目の詳細と対処方法を提示

  Step 3: 通知
    → 経理担当に結果通知
    → 全クリアなら「月次決算準備完了」

■ 業種別差分
  建設業: 工事進行基準の出来高確認を追加
  製造業: 棚卸資産の評価チェックを追加
  医療/歯科: レセプト請求額との照合を追加
```

### 1.5 消費税区分チェックフロー

```
トリガー: 月次決算チェックの一部 or 随時
対象ドメイン: 全業種の経理

  Step 1: 仕訳データ取得
    → freee/MF から当月の全仕訳を取得

  Step 2: 区分チェック
    □ インボイス番号の存在確認（仕入税額控除の対象か）
    □ 課税/非課税/不課税の区分が正しいか
    □ 軽減税率（8%）の適用が正しいか

  Step 3: 異常通知
    → 「5件の仕訳でインボイス番号が未登録です」
    → 「2件の仕訳で税率区分が不一致の可能性があります」

■ 業種別差分
  不動産: 居住用/事業用の区分チェックが重要（非課税判定）
  医療: 保険診療は非課税、自費診療は課税の区分
```

## 2. 総務/人事系共通フロー

### 2.1 勤怠集計フロー

```
トリガー: 月末（給与計算の前）
対象ドメイン: 全業種の総務/人事

  Step 1: 勤怠データ取得
    意図: 「全社員の勤怠データを取得する」
    → 勤怠SaaS → API取得
    → Excel/紙 → OCR or 手動入力

  Step 2: 集計処理
    ・総労働時間 / 残業時間 / 深夜勤務時間 / 休日出勤
    ・有給取得日数 / 残日数
    ・遅刻・早退の回数

  Step 3: 法定チェック ★重要
    □ 36協定の上限チェック（月45h / 年360h）
    □ 特別条項の上限（月100h / 2-6ヶ月平均80h）
    □ 年5日の有給取得義務チェック
    □ 60時間超の割増率チェック（50%）

  Step 4: アラート
    → 上限に近い社員をリスト化
    → 管理者に通知
    → 「鈴木さんの今月残業が42h。上限まであと3hです」

  Step 5: 給与計算用データ出力
    → 給与SaaS → API連携
    → CSV → 給与計算担当に送付

■ 業種別差分
  建設業: 現場ごとの集計が必要（直行直帰対応）
  医療: 夜勤72時間ルール・当直の扱い
  歯科: 半日勤務パターンの集計
  製造: 交代勤務（2交代・3交代）の計算
```

### 2.2 入退社手続きフロー

```
トリガー: 入社日 or 退社日の X 日前
対象ドメイン: 全業種の総務/人事

  ■ 入社時チェックリスト

  Step 1: 書類収集リマインド
    → 入社 14日前に必要書類リストを本人に送信
    必須書類:
      □ マイナンバー
      □ 住民票 or 住所確認書類
      □ 年金手帳 or 基礎年金番号通知書
      □ 給与振込先口座
      □ 扶養控除等申告書
      □ 雇用保険被保険者証（転職の場合）

  Step 2: 社会保険手続き（入社後5日以内）
    → 手続きリストを通知
    → 期限カウントダウン
    → 電子申請対応: e-Gov or 社労士に依頼

  Step 3: アカウント発行
    → SaaS/メール/チャットのアカウント作成リスト
    → manual タスクとして IT 担当に発行

  Step 4: 完了確認
    → 全チェック項目の完了を確認
    → 未完了項目のリマインド

  ■ 退社時チェックリスト（逆のフロー）
    → 社保喪届 / 離職票 / 源泉徴収票
    → アカウント停止リスト
    → 備品回収リスト

■ 業種別差分
  建設業: 建設キャリアアップシステム(CCUS)の登録/抹消
  医療: 保健所への届出（医師・看護師の異動届）
  士業: 所属弁護士会等への届出
```

### 2.3 シフト表作成フロー

```
トリガー: 月末（翌月分の作成）
対象ドメイン: 全業種の総務/人事

  Step 1: 制約条件の収集
    ・希望休の収集（Google Forms / slack）
    ・固定シフト（曜日固定の社員）
    ・法定制約: 週40h / 連続6日以内 / 深夜制限
    ・業種固有制約: specialist ルールから取得

  Step 2: ドラフト自動生成
    → LLM + 制約ソルバーでシフト候補を生成
    → 「全制約を満たすパターンを3案生成しました」

  Step 3: 管理者確認
    → 3案から選択 or 手動修正
    → 修正内容を学習（「この人は月曜NGが多い」等）

  Step 4: 配信
    → 確定シフトを全員に通知（slack / LINE WORKS）
    → カレンダーに自動登録

■ 業種別差分
  医療: 夜勤あり。日勤→深夜→準夜のローテーション。法定の夜勤回数制限
  歯科: 半日診療あり。ユニット数 × 必要人員の制約
  製造: 交代勤務。設備稼働に合わせたライン配置
  建設: 現場ごとの配置。天候による変動
```

### 2.4 資格・法定研修管理フロー

```
トリガー: 月次チェック（更新期限の X ヶ月前にアラート）
対象ドメイン: 全業種の総務/人事

  Step 1: 資格DB照合
    → 全社員の保有資格と有効期限をチェック
    → 3ヶ月前 / 1ヶ月前 / 期限切れ の3段階アラート

  Step 2: 法定研修の受講チェック
    → 業種ごとの法定研修リスト（specialist ルールから取得）
    → 未受講者の検出

  Step 3: アラート通知
    → 本人 + 管理者に通知
    → 「鈴木さんの1級施工管理技士の更新が2ヶ月後です」

■ 業種別差分
  建設: 施工管理技士、建設業経理士、安全衛生教育
  製造: 特定化学物質取扱、フォークリフト、クレーン
  不動産: 宅地建物取引士（5年更新）、法定講習
  士業: 弁護士会費、税理士会の研修単位、社労士の更新研修
  医療: 医師免許（更新なし）、看護師の継続教育、BLS/ACLS
  歯科: 歯科医師免許、歯科衛生士の研修、感染対策講習
```

## 3. 営業系共通フロー

### 3.1 見積書作成フロー

```
トリガー: 営業担当からのリクエスト
対象ドメイン: 建設・製造・不動産・士業の営業

  Step 1: 見積条件の入力
    → チャット or フォームで条件入力
    → 「○○工事の見積を作りたい。RC造3階建て、延床500㎡」

  Step 2: 過去実績検索
    意図: 「類似案件の見積実績を検索する」
    → ナレッジDB（pgvector）でセマンティック検索
    → 類似案件 TOP5 を提示（金額・条件・結果）

  Step 3: 見積ドラフト生成
    → specialist ルール（業種固有の原価計算ロジック）を適用
    → テンプレートに自動入力
    → 利益率チェック（最低粗利率を下回っていないか）

  Step 4: 承認チェックポイント
    → 金額別の決裁ルート（ナレッジから取得）
    → 「200万円の見積 → 部長承認が必要です」

  Step 5: 出力
    → PDF生成 → メール送信 or 印刷

■ 業種別の原価計算ロジック（specialist で定義）
  建設: 歩掛 × 単価 + 諸経費率（材工共）
  製造: BOM × 材料単価 + 労務費 + 製造間接費
  不動産: 仲介手数料の上限計算（売買価格 × 3% + 6万円）
  士業: タイムチャージ × 見積工数 + 実費
```

### 3.2 顧客フォロー通知フロー

```
トリガー: 週次バッチ
対象ドメイン: 全業種の営業

  Step 1: 接触履歴の分析
    → CRM / kintone / メール / チャットから最終接触日を取得
    → 「接触間隔がN日を超えた顧客」を抽出（Nは specialist ルールで定義）

  Step 2: フォロー理由の判定
    → 単純な未接触 → 「ご無沙汰フォロー」
    → 直近のクレーム/問合せ後 → 「フォローアップ」
    → 契約更新が近い → 「更新提案」
    → 季節イベント → 「季節挨拶 + 提案」

  Step 3: 通知
    → 営業担当に通知（slack / アプリ内）
    → 「A社への最終接触が45日前です。フォローしましょう」
    → 提案トークの候補も提示（ナレッジから）

■ 業種別差分
  建設: アフター点検のタイミング、協力会社への発注フォロー
  歯科: リコール（定期検診）管理 = 顧客フォローの一種
  士業: 顧問先の月次巡回スケジュール
```

## 4. インフラ系共通フロー

### 4.1 在庫/消耗品 発注点アラートフロー

```
トリガー: 日次バッチ or 在庫変動イベント
対象ドメイン: 製造(購買/設備保全)、医療(薬剤)、歯科(助手)

  Step 1: 現在在庫の取得
    → zaico / kintone / 在庫管理SaaS → API
    → Excel → 定期読み込み

  Step 2: 発注点との照合
    → 品目ごとの安全在庫（specialist ルール or learned ルール）
    → 現在在庫 < 安全在庫 → アラート対象

  Step 3: 発注ドラフト生成
    → 過去の発注パターンから数量・仕入先を自動提案
    → 「○○を△△個、A社に発注しますか？」

  Step 4: 承認 → 発注
    → Semi: 承認後に発注書を自動生成 → FAX/メール/API
    → Manual: 「以下を発注してください」タスク発行

■ 業種別差分
  製造: BOMベースの所要量計算（MRP的）
  医療: 麻薬・劇薬の在庫管理は別フロー（法規制）
  歯科: 材料の使用期限管理を追加
```

### 4.2 期限管理アラートフロー

```
トリガー: 日次バッチ
対象ドメイン: 全業種

  Step 1: 期限データの取得
    → kintone / calendar / DB から期限付きデータを収集

  Step 2: アラート判定
    → X日前（設定可能）にアラート発火
    → 緊急度: 期限切れ > 1週間以内 > 1ヶ月以内 > 3ヶ月以内

  Step 3: 通知
    → 担当者 + 管理者に通知
    → 期限切れは即アラート（赤）

■ 業種別の期限対象（specialist で定義）
  建設: 安全書類、建設業許可更新、資格更新
  製造: ISO認証更新、設備点検期限、計測器校正
```

```
  Excel(.xlsx/.xls): 構造推論→テンプレート学習→異常検知→マクロ対応
  PDF: pdfplumber + Document AI OCR
  Word(.docx): python-docx → テキスト+構造抽出
  PowerPoint(.pptx): python-pptx → スライド別テキスト抽出
  CSV: pandas → 自動型推論
  画像(JPG/PNG): Document AI OCR → テキスト化
  音声(MP3/WAV): Whisper → テキスト化
  CAD(DWG/DXF): メタデータ抽出（図面内容は将来対応）

  Step 1: ファイル取得
    → LINE写真/ファイル送信 / メール添付 / ファイルアップロード / カメラ撮影 / スキャン / FAX(eFAX)

  Step 2: OCR処理
    → Google Document AI でテキスト抽出
    → テーブル構造の認識

  Step 3: LLM構造化
    → specialist ルール（業種固有の書類フォーマット知識）を参照
    → JSON に構造化
    → 例: 請求書 → {取引先, 日付, 品目[], 金額, 税額}

  Step 4: 確認
    → 「以下の内容で合っていますか？」
    → 修正があれば学習

  Step 5: DB格納 or 後続フローへ渡す
    → 請求書OCR → 入金消込フローへ
    → 見積書OCR → ナレッジDBへ
```
