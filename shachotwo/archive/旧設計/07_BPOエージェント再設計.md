# BPO エージェント再設計 — チャネルファースト × Level 0-4 段階的自動化

> **背景**: 現行 bpo_agent は SaaS API オーケストレーション専用。
> 本ドキュメントは「チャネルファースト（LINE/メール/ファイル/音声が主入口）で、
> 全業界×全職種の業務を段階的に自動化する汎用BPOエージェント」への再設計を定義する。
> 02_プロダクト設計.md の Company Genome（遺伝子合成モデル）と統合し、
> 高精度な初期テンプレートの生成パイプラインまでを設計範囲とする。
>
> **価格モデル**: 基本料 ¥30K/月 + 従量課金（タスク実行量に応じたスケール）
> **ターゲット**: 売上 10億〜100億の中堅企業
> **マルチユーザー**: 参謀(部長以上) / アシスタント(課長) / ヘルパー(一般) × 部門別

---

## 1. 設計思想: チャネルファースト × Level 0-4

### 1.1 チャネルファーストの原則

```
旧設計: SaaS API が前提 → API を持たない企業は対象外
新設計: LINE/メール/ファイル/音声が主入口 → SaaS API は「オプション」

  ユーザーの現実:
    ・社長はLINEで指示を出す
    ・現場は写真をLINEで送る
    ・経理はExcelを添付してメールで送る
    ・会議は音声で記録される

  → これらをそのまま BPO の起動トリガーにする
  → SaaS API 連携は「あればさらに便利」のオプション
```

### 1.2 BPO Level 0-4: 段階的自動化モデル

```
Level 0: リマインダー
  ・期限通知、定期リマインド
  ・シャチョツーから一方向の通知のみ
  ・リスク: ゼロ（情報を出すだけ）
  ・例: 「明日は安全書類の提出期限です」

Level 1: データ収集・整理
  ・ファイル受信→構造化→集計
  ・人間が送ったデータを整理して返す
  ・リスク: 極低（データを読むだけ、書き込まない）
  ・例: 「日報写真5枚を受け取りました。工程進捗表を更新しました」

Level 2: 下書き・ドラフト作成
  ・請求書ドラフト、レポート生成、見積書案
  ・人間が確認・修正してから使う前提
  ・リスク: 低（人間が最終確認する）
  ・例: 「今月の請求書ドラフトを作成しました。確認してください」

Level 3: 承認付き実行
  ・承認を得てからSaaS操作/ファイル送信を実行
  ・人間の承認がゲートになる
  ・リスク: 中（承認があるが、実行は自動）
  ・例: 「freeeに仕訳を登録します。[承認] [修正] [却下]」

Level 4: 自律実行（信頼獲得後）
  ・承認なしで自動実行
  ・事後レポートのみ
  ・リスク: 高（完全自動。信頼構築が前提）
  ・例: 「今月の勤怠集計を完了し、給与計算用CSVを生成しました」
  ・昇格条件: 同一タスクで Level 3 を5回連続無修正承認
```

### 1.3 Level 昇格の仕組み

```
Level 0 → 1: 即時（データ収集は導入直後から可能）
Level 1 → 2: 1-2週間（データの構造理解が安定した後）
Level 2 → 3: 2-4週間（ドラフト精度が安定した後）
Level 3 → 4: 個別判定（同一タスクで5回連続無修正承認が条件）

昇格はタスク単位:
  ・「日報集計」は Level 4 だが「請求書発行」は Level 3 のまま
  ・タスクごとに独立して Level が進化する
  ・降格: エラー発生時は自動で1段階降格
```

## 2. エージェントは1つ、ルールスタックで特化

### 新設計: 責務ドメイン × 企業プロファイル

```
企業プロファイル（大分類×中分類×規模）が起点
  → 各社員に「責務ドメイン」をアサイン
  → 1人が複数ドメインを持てる（中小企業の兼務に対応）
  → エージェントのコードは共通、ルールスタックだけが違う

BPOAgentContext:
  industry:     "建設業"              ← 大分類（業界）
  sub_industry: "総合工事業"          ← 中分類
  scale:        "small"               ← 規模 (micro/small/medium/large)
  domains:      ["経理", "総務"]      ← 責務ドメイン（複数可 = 兼務対応）
  channels:     ["line", "email", "excel", "freee"]  ← 入力チャネル + 実行チャネル
  user_tier:    "参謀"               ← 参謀/アシスタント/ヘルパー
  company_id:   "uuid"                ← 会社固有
```

### マルチユーザー × BPO Level の関係

```
■ ユーザーティア別のBPO権限

  参謀（部長以上）:
    ・全 Level のタスクを起動可能
    ・Level 3-4 の承認権限あり
    ・Level 昇格の承認権限あり
    ・部門横断のデータ閲覧可能

  アシスタント（課長）:
    ・Level 0-3 のタスクを起動可能
    ・Level 2 の承認権限あり（自部門のみ）
    ・Level 3 は参謀に承認エスカレーション

  ヘルパー（一般）:
    ・Level 0-1 のタスクを起動可能
    ・Level 2 以上は上長に自動エスカレーション
    ・データ入力・ファイル送信が主な操作
```

### 責務ドメインと職種の関係

```
「職種」ではなく「責務ドメイン」を最小単位にする理由:

  中小企業の現実:
    田中さん（建設業20人・バックオフィス1人体制）:
      └── [経理] + [総務] + [人事] の3ドメインを兼務
          → 3ドメインのルールが合成された「田中さん専用エージェント」

    佐藤さん（建設業80人・経理専任）:
      └── [経理] のみ
          → 経理特化エージェント

  → ドメインの組み合わせは自由。規模に応じた職種分岐ルール不要。

ゲノムテンプレートに典型的な兼務パターンをデフォルトとして持たせる:

  typical_domain_bundles:
    micro (1-10人):
      - バックオフィス: [経理, 総務, 人事]
      - 現場兼営業: [現場監督, 営業]
      - 社長: [経営, 営業, 現場監督]  ← 社長が全部やるパターン
    small (10-50人):
      - 経理総務: [経理, 総務]
      - 人事: [人事]
      - 営業: [営業]
      - 現場監督: [現場監督]
    medium (50-300人):
      - 全ドメイン個別配置（1人1ドメイン）

  → 導入時に「あなたの会社の典型パターンはこれです」と提案
  → ユーザーがカスタマイズ（ドメインの追加・削除・移動）
  → 将来「この人の業務を分割して新人に渡したい」→ ドメインの付け替えだけ
```

### ルール合成: 6層スタック

```
優先度（低→高）:

Layer 1: universal/
  └── 全エージェント共通ルール（安全性・API制約・承認フロー）

Layer 2: domain/{責務ドメイン}/
  └── 経理・営業・人事・現場管理・受付... のドメイン共通知識
  └── 複数ドメインの場合は全ドメインのルールを合成

Layer 3: industry/{大分類}/
  └── 建設業・製造業・医療... の業界共通知識（法令・用語・商習慣）

Layer 4: specialist/{大分類}_{ドメイン}/
  └── 「建設業の経理」「医療の受付」等の専門ルール ★差別化の核
  └── 複数ドメインの場合は各 specialist を合成

Layer 5: scale/{規模}/
  └── 小規模（兼務多い・紙中心）/ 中規模（SaaS導入済み）等

Layer 6: learned/{company_id}_{ドメイン}/
  └── 会社固有の自動蓄積ルール（学習ループ出力）
```

**合成ロジック**:
- 各レイヤーは任意（ファイルが無ければスキップ）
- 上位レイヤーが下位レイヤーを上書き（会社固有 > specialist > industry > domain > universal）
- 複数ドメインの場合: Layer 2/4/6 を各ドメイン分マージ（矛盾時は先に指定されたドメインを優先）

**ルール注入方針**:
- 6層フィルタ後の合成結果を**全てプロンプトに注入**する（選択的注入はしない）
- 合成後のルール量は最大 25-50KB（10,000-20,000トークン）で、LLMコンテキスト内に余裕で収まる
- LLMが全ルールを把握した上で曖昧な指示（「月末のあれやっといて」等）を解釈する
  → ITリテラシーの低いユーザーでも適切なタスク計画が生成される
- 将来 learned ルールが肥大化して50KB超になった場合のみ、最適化を検討（YAGNI）

## 3. チャネルファーストの入力アーキテクチャ

### 3.1 主要入力チャネルと BPO 起動

```
■ LINE（最優先チャネル）
  ・テキストメッセージ → 意図解析 → BPO タスク起動
  ・写真送信 → OCR/画像解析 → データ構造化
  ・ファイル送信 → ファイル処理エンジン
  ・音声メッセージ → STT → 意図解析
  ・制約: ファイルサイズ上限あり → 大容量はアップロードURL誘導

■ メール
  ・本文テキスト → 意図解析
  ・添付ファイル → ファイル処理エンジン
  ・転送メール → 取引先とのやり取りを構造化
  ・制約: なし（大容量ファイルも対応可能）

■ ファイルアップロード（Web/アプリ）
  ・Excel/CSV → 構造推定 → 集計/変換/レポート
  ・PDF → テキスト抽出 → 構造化
  ・Word → テキスト抽出 → 要約/テンプレート適用
  ・画像（写真/スキャン）→ OCR → データ構造化
  ・音声ファイル → STT → テキスト化 → 構造化
  ・動画 → フレーム抽出 + 音声抽出 → 複合処理

■ 音声（リアルタイム）
  ・会議録音 → STT → 議事録自動生成
  ・電話メモ → STT → タスク抽出
  ・ボイスコマンド → 意図解析 → BPO 起動

■ SaaS API（オプション・あれば活用）
  ・kintone/freee/ANDPAD 等の連携
  ・Webhook → イベント駆動の BPO 起動
  ・定期ポーリング → データ同期
```

### 3.2 統一ファイル処理エンジン

```
BPO の主要ユースケースとしてファイル処理を位置づける。

■ Excel/CSV 処理（最頻出）
  入力: Excel ファイル（LINE送信/メール添付/アップロード）
  処理:
    1. シート構造の自動推定（ヘッダー行検出、データ型推定）
    2. specialist ルール参照（業種固有のフォーマット知識）
    3. LLM による意図理解（「これ集計して」→ 何の集計か推定）
    4. 処理実行（集計/変換/マージ/フィルタ）
    5. 結果を元の形式で返却 or レポート生成
  例:
    ・「この日報Excelを今月分まとめて」→ 月次集計
    ・「原価表と予算表を突き合わせて」→ 差異分析
    ・「この売上データからグラフ作って」→ 可視化

■ PDF 処理
  入力: PDF（請求書/契約書/安全書類/行政文書）
  処理:
    1. テキスト抽出（テキストPDF）or OCR（スキャンPDF）
    2. Document AI によるレイアウト解析
    3. LLM による構造化（JSON変換）
    4. 後続フローへの接続（入金消込/ナレッジ登録等）

■ 画像処理
  入力: 写真（現場写真/レシート/名刺/ホワイトボード）
  処理:
    1. 画像種別の自動判定
    2. OCR or 画像認識
    3. 構造化データ出力
  例:
    ・レシート写真 → 経費精算データ
    ・現場写真 → 工程記録台帳
    ・名刺写真 → 連絡先データベース

■ 音声/動画処理
  入力: 音声ファイル/動画ファイル
  処理:
    1. Whisper STT でテキスト化
    2. 話者分離（diarization）
    3. LLM で構造化（議事録/要約/タスク抽出）
  例:
    ・会議録音 → 議事録 + アクションアイテム
    ・現場動画 → 安全チェック報告書
```

## 4. エラー防止4層設計

### 4.1 概要

```
BPO が業務を自動化する以上、エラーは致命的。
「間違った請求書を送った」「二重支払した」は信頼を一瞬で失う。

4層のエラー防止で、実行前に問題を検出・防止する:

  Layer 1: 確信度判定（Confidence Gate）
  Layer 2: クリティカル判定（Critical Operation Detector）
  Layer 3: 差分ハイライト（Change Highlighter）
  Layer 4: ロールバック（Undo Capability）
```

### 4.2 Layer 1: 確信度判定（Confidence Gate）

```
全タスクステップの実行前に確信度を算出。
閾値未満なら自動で Level を下げる（自動→手動確認）。

  確信度の算出:
    ・入力データの品質（欠損率・フォーマット一致率）
    ・過去の同一タスクの成功率
    ・specialist ルールのカバレッジ
    ・ChannelResolver の解決確実性

  閾値:
    0.9以上 → そのまま実行
    0.7-0.9 → 「確認推奨」マーク付きで続行
    0.5-0.7 → Level を1段階下げる（Level 4→3、Level 3→2）
    0.5未満 → 人間に委譲（manual フォールバック）

  例:
    日報集計の確信度 0.95 → Level 4 で自律実行
    請求書作成の確信度 0.72 → 「確認推奨」で Level 3 に降格
    未知フォーマットのExcel → 確信度 0.4 → 人間に「このファイルの構造を教えてください」
```

### 4.3 Layer 2: クリティカル判定（Critical Operation Detector）

```
操作の「不可逆性」と「影響範囲」でクリティカル度を判定。
クリティカル操作は Level に関係なく承認を強制。

  クリティカル判定基準:
    金額操作: 10万円以上の支払い/請求
    外部送信: メール/FAX/APIでの外部データ送信
    削除操作: データの削除/上書き
    法令関連: 法定帳票/届出に関わる操作
    個人情報: PII を含むデータの操作

  Level 4（自律実行）でもクリティカル操作は承認必須:
    例: 「勤怠集計は Level 4 で自動だが、
         給与振込データの生成は金額操作なので承認を求める」
```

### 4.4 Layer 3: 差分ハイライト（Change Highlighter）

```
承認画面で「何が変わるか」を明確に表示。
承認者が一目で判断できるようにする。

  表示項目:
    ・変更前 vs 変更後の差分（赤/緑ハイライト）
    ・前回実行との比較（「前月は5件、今月は8件。3件増加」）
    ・異常値のハイライト（「A社の請求額が前月比3倍」）
    ・影響範囲のサマリ（「3つのSaaSに書き込みます」）

  例: 請求書発行の承認画面
    ┌────────────────────────────────────┐
    │ 請求書発行（2026年3月分）           │
    ├────────────────────────────────────┤
    │ 件数: 12件（前月: 10件 △+2）       │
    │ 合計: ¥4,800,000（前月比 +15%）     │
    │                                      │
    │ ⚠️ 注意:                            │
    │ ・A社: ¥1,200,000（前月比 +200%）   │
    │   → 理由推定: Q4まとめ請求          │
    │                                      │
    │ [全件承認] [個別確認] [却下]          │
    └────────────────────────────────────┘
```

### 4.5 Layer 4: ロールバック（Undo Capability）

```
実行後に問題が発覚した場合のロールバック手段。

  チャネル別のロールバック能力:
    freee API  : 仕訳の取消/修正が可能 → 自動ロールバック対応
    kintone    : レコード更新の履歴あり → 自動ロールバック対応
    Excel/CSV  : 処理前のバックアップを自動保存 → 手動復元
    メール送信 : 送信取消不可 → 訂正メール自動生成
    LINE送信   : 取消不可 → 訂正メッセージ自動送信

  ロールバックフロー:
    1. 全タスク実行時に「ロールバックポイント」を自動保存
    2. 実行後24h以内は「元に戻す」が可能
    3. ロールバック不可の操作は事前に警告
       「この操作は取り消せません。実行しますか？」
```

## 5. 実行手段の抽象化: チャネルファースト版

### 5.1 意図レベルの計画 + 実行戦略の自動選択

```
核心: タスクプランナーは「何をしたいか」を抽象的に計画する。
      チャネルの選択は実行時に ChannelResolver が自動で行う。

  プランナー出力（意図レベル）:
    「受注データを会計システムに転記する」

  ChannelResolver が接続済みチャネルから最適手段を選択:
    ├── freee API接続あり → freee_create_invoice
    ├── freee未接続 + Google Sheets あり → sheets_append_row
    └── 何もなし → manual タスク発行（人間に依頼）

  → 同じタスク計画が、SaaS企業でもExcel企業でも動く
```

```python
# channel は実行手段の抽象。複数可。企業の接続状況から自動構成。
channels: list[str] = ["line", "email", "excel", "freee", "google_sheets"]

# チャネル種別の分類
input_channels = ["line", "email", "file_upload", "voice"]       # 入力チャネル
output_channels = ["freee", "kintone", "google_sheets", "manual"] # 実行チャネル

# ツール選択は channel 単位
for channel in channels:
    tools += ToolRegistry.list_tools(channel)

# 実行時のチャネル解決
class ChannelResolver:
    def resolve(self, intent: str, available_channels: list[str]) -> Tool:
        """
        intent: "会計システムにデータ登録"
        available_channels: ["freee", "google_sheets"]
        → freee_create_invoice を返す（freee優先）
        → freee未接続なら sheets_append_row にフォールバック
        → 何もなければ manual タスク発行
        """
```

### 5.2 チャネル一覧（入力 + 実行）

| チャネル種別 | 例 | 方向 | 実装方法 |
|---|---|---|---|
| **LINE** | LINE Messaging API | 入力+出力 | Webhook + Reply API |
| **メール** | Gmail, 独自ドメイン | 入力+出力 | IMAP受信 + SMTP送信 |
| **ファイル** | Excel, PDF, 画像, 音声, 動画 | 入力 | 統一ファイル処理エンジン |
| **音声** | Whisper STT | 入力 | リアルタイム or ファイル |
| **SaaS API** | kintone, freee, ANDPAD | 実行 | REST API（オプション） |
| **Google Workspace** | gmail, sheets, calendar, drive | 実行 | Google API |
| **GAS** | Google Apps Script | 実行 | Apps Script Execution API |
| **コミュニケーション** | slack, LINE WORKS, chatwork | 出力 | 各API |
| **手動タスク** | manual | 出力 | 人間にタスクを割り当てるキュー |

### 5.3 `manual` チャネル（重要）

SaaS化されていない業務（紙・電話・対面）のために、**人間にタスクを発行する**チャネル。
`manual` は常に最終フォールバックとして暗黙的に利用可能:

```
タスク: 「取引先Aに電話して納期確認」
チャネル: manual
→ 担当者にSlack/LINE/プッシュ通知でタスク発行
→ 担当者が結果を入力（「3/15納品確定」）
→ 結果をBPOフローに戻して次ステップ続行
```

これにより「全てがデジタルでなくても」BPOフローに組み込める。
**何も接続していない企業でも、LINE + manual チャネルだけでBPOの価値が出る。**

### 6.2.1 ルールライフサイクル管理

```
■ ライフサイクル状態遷移

  draft ──→ active ──→ review ──→ active（更新後）
                         │
                         └──→ deprecated（廃止）

■ 自動レビュートリガー

  review_trigger の種類:
    annual      : 毎年自動で review 状態に → 管理者に通知
    quarterly   : 四半期ごと
    on_feedback : ユーザーFBがあった場合のみ
    on_law_change: e-Gov法令APIで法改正を検知した場合 ★法令ルール専用

■ 法改正の自動検知

  定期バッチ（月1回）:
    1. specialist ルールから source が法令名のセクションを抽出
    2. e-Gov法令APIで該当法令の最終改正日をチェック
    3. 改正があれば → status: review に自動変更
    4. LLMが改正内容を反映したドラフトを自動生成
    5. 管理者が承認 → active に戻す

■ auto_updatable フラグ

  true  : 学習ループが自動で内容を更新可能（業界標準・ツール・リスク等）
  false : 学習ループでは上書き不可（法令ルール等）。人間承認が必須。

■ deprecated ルールの扱い

  status: deprecated のルールはプロンプト注入から自動除外。
  削除はせず履歴として保持（監査・ロールバック用）。
```

## 7. SIer 共存→侵食→置換の3フェーズ戦略

### 7.1 概要

```
多くの中堅企業は SIer にシステム運用を依存している。
シャチョツーは SIer を「敵」ではなく「共存→段階的置換」する。

Phase 1: 共存（導入初期）
  ・SIer の既存システムはそのまま
  ・シャチョツーは SIer が対応しない「隙間業務」を自動化
  ・例: Excel手作業、LINE→SaaS転記、ファイル集計
  ・SIer と競合しないので導入障壁が低い

Phase 2: 侵食（導入半年〜）
  ・シャチョツーの BPO が SIer 業務の一部を代替し始める
  ・例: 月次レポート自動化（SIer に毎月依頼していたもの）
  ・企業側が「SIer に頼まなくてもできる」と気づく
  ・月額コスト比較: SIer ¥500K → シャチョツー ¥30K + 従量

Phase 3: 置換（導入1年〜）
  ・SIer 依存の業務がシャチョツーに完全移行
  ・残るのは「本当にカスタム開発が必要なもの」だけ
  ・シャチョツーの Engineer Worker が要件定義→スペック化を支援
```

### 7.2 SIer 業務の分類と置換可能性

```
■ 置換可能（シャチョツー BPO で対応）
  ・定型レポート作成
  ・データ集計・整形
  ・SaaS間のデータ転記
  ・帳票フォーマット変換
  ・定期バッチ処理の管理

■ 侵食可能（段階的に置換）
  ・SaaS の設定変更・カスタマイズ
  ・業務フロー改善の提案
  ・新ツール導入の検討・選定

■ 置換困難（SIer が必要）
  ・基幹システムのカスタム開発
  ・オンプレサーバーの保守
  ・ハードウェア/ネットワーク管理
  ・セキュリティ監査
```

### 8.2 ルールローダーの変更

```python
def load_bpo_rules(
    rules_dir: str,
    domains: list[str],      # 責務ドメイン（複数可）
    industry: str | None,    # 業界
    scale: str | None,       # 規模
    channels: list[str],     # 実行チャネル
    company_id: str | None,  # 会社固有
) -> str:
    """6層ルールスタックを合成して返す。全量をプロンプトに注入。"""
    layers = []

    # Layer 1: universal
    layers.append(load_if_exists(f"{rules_dir}/universal/general_rules.md"))

    # Layer 2: domain（責務ドメイン — 複数ドメイン対応）
    for domain in domains:
        layers.append(load_if_exists(f"{rules_dir}/domain/{domain}_rules.md"))

    # Layer 3: industry（業界）
    if industry:
        layers.append(load_if_exists(f"{rules_dir}/industry/{industry}_rules.md"))

    # Layer 4: specialist（業界×ドメイン — 複数ドメイン対応）
    if industry:
        for domain in domains:
            layers.append(load_if_exists(
                f"{rules_dir}/specialist/{industry}_{domain}_rules.md"
            ))

    # Layer 5: scale（規模）
    if scale:
        layers.append(load_if_exists(f"{rules_dir}/scale/{scale}_rules.md"))

    # Layer 6: channel-specific（チャネル固有ルール）
    for ch in channels:
        layers.append(load_if_exists(f"{rules_dir}/channel/{ch}_rules.md"))

    # Layer 7: learned（会社固有 — 各ドメインの learned を合成）
    if company_id:
        for domain in domains:
            layers.append(load_if_exists(
                f"{rules_dir}/learned/{company_id}_{domain}_learned.md"
            ))

    # deprecated ルールを除外（status: deprecated のセクションをフィルタ）
    combined = "\n\n---\n\n".join([l for l in layers if l])
    return filter_deprecated_rules(combined)
```

## 9. 遺伝子モデルとの統合

### 現行の遺伝子 = 業務プロセス単位のモジュール

```
会計遺伝子、シフト管理遺伝子、積算遺伝子 ...
→ 企業テンプレート = 遺伝子の組み合わせ
```

### 新設計: 遺伝子 + specialist ルール + ドメインバンドルの3層

```
Company Genome（企業レベル）:
  = 遺伝子の組み合わせ（既存設計そのまま）
  = 「この会社はどんな業務があるか」の定義

ドメインバンドル（人員配置レベル）:
  = 「誰がどの責務ドメインを担当するか」の定義
  = 規模に応じた典型パターンをデフォルト提供

specialist ルール（BPOエージェントレベル）:
  = 「その責務ドメインでどう振る舞うか」の詳細
  = BPOエージェントが実際にタスクを実行するときのルール

関係:
  遺伝子「会計遺伝子（個別原価/月次）」
    → ドメインバンドル: [経理] ドメインに含まれる
    → specialist「建設業_経理」ルールに展開:
      「出来高に基づく個別原価計算を行う。
       工事台帳から実行予算と実績を対比する。
       JV工事は持分法で処理する。」
```

つまり:
- **遺伝子** = What（何の業務があるか）
- **ドメインバンドル** = Who（誰がどの業務群を担当するか）
- **specialist ルール** = How（その業務をどうやるか）

遺伝子が Company Genome のセットアップ時に選ばれ、
ドメインバンドルで人員に割り当てられ、
specialist ルールが BPO エージェントの実行時に参照される。

## 10. 学習ループの拡張

### 現行の3つの学習ループ（維持）

1. **実行ナレッジ学習**: 成功パターン → ベストプラクティスルール
2. **失敗学習**: 失敗パターン → 回避ルール
3. **修正駆動学習**: ユーザー修正 → 意図解釈ルール

### 追加: specialist ルールへのフィードバック

```
学習ループの出力先を拡張:

現行:
  learned/{saas}_{genre}_learned.md（フラット）

新設計:
  learned/{company_id}_{domain}_learned.md（会社固有）
    ↓ 同一パターンが複数社で発生
  specialist/{industry}_{domain}_rules.md に昇格提案
    ↓ 承認
  全社共通ルールに反映（auto_updatable: true のセクションのみ）

例:
  A建設の経理エージェントが「JV精算で○○に注意」を学習
  → A建設固有の learned ルールに保存
  B建設の経理エージェントも同じパターンを学習
  → 「建設業_経理の specialist ルールに追加しますか？」と提案
  → 承認 → 全建設業の経理エージェントに反映
  → 昇格されたルールの status: active, review_trigger: on_feedback
```

### 追加: チャネル/ファイル処理の学習

```
ファイル処理パターンの学習:

  1. Excel構造の学習
    ・「この会社の日報Excelはこの構造」を記憶
    ・2回目以降は構造推定のステップをスキップ
    ・フォーマット変更を検知 → 「日報の形式が変わりましたか？」

  2. チャネル選好の学習
    ・「この人はLINEで送ってきたものをメールで返すのを好む」
    ・入力チャネル × 出力チャネルの組み合わせパターン

  3. Level 昇格データの蓄積
    ・タスクごとの承認履歴 → 自動昇格判定
    ・エラー履歴 → 自動降格判定
```

## 12. Supabase スケーラビリティ

### データ量見積もり

```
specialist_templates:
  40テンプレート × 10KB = 400KB（重点6業種）
  全業種展開時: 300テンプレート × 10KB = 3MB
  → Supabase Free でも余裕

execution_knowledge:
  100社 × 月100タスク × 12ヶ月 = 120,000行
  × 2KB/行 = 240MB
  → Supabase Pro (8GB) で余裕

learned ルール（会社固有）:
  100社 × 8職種 × 20KB = 16MB
  → 余裕

ファイル処理キャッシュ:
  100社 × 月50ファイル × 500KB = 2.5GB/年
  → Supabase Storage に格納（定期クリーンアップ）

権威知識キャッシュ（Stage 1 出力）:
  300組み合わせ × 100KB = 30MB
  → Supabase Storage に格納

結論: 500社規模まで Supabase Pro で問題なし
```
