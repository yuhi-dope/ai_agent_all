# BPO 例外処理設計 — 失敗分類×対処×学習

> **目的**: BPO タスク実行中の失敗を分類し、対処パターンと学習ループへの接続を定義する。
> 07_BPOエージェント再設計.md の Self-Healing 概念を具体化。
>
> **設計原則**:
> - 失敗は学習の機会。全失敗を記録し、パターン化して予防に転換する
> - 安全側に倒す。迷ったら止める。勝手にリトライしない（Type D/E）
> - ChannelResolver のフォールバックを最大限活用（API失敗→manual）
> - **チャネルファースト対応**: LINE Bot / Email / ファイルアップロード / 音声 / FAX の各チャネル固有エラーを Type F として追加
> - **Level別対処方針**: Level 0-1はアラート通知のみ、Level 2は下書き破棄、Level 3-4はロールバック実行
> - **エラー防止4層**との整合: 確信度スコア → 重要判断(2段階承認) → 差分ハイライト → ロールバック保証

---

## 1. 失敗分類（5 Type）

```
Type A: 一時的エラー（リトライで解決）
  │
Type B: データ不整合（人間確認で解決）
  │
Type C: 認証/権限エラー（管理者対応で解決）
  │
Type D: ビジネスロジックエラー（即停止・法令リスク）
  │
Type E: 未知のエラー（記録→manual フォールバック）
  │
Type F: チャネル固有エラー（チャネルファースト対応・NEW）
```

---

## 2. Type A: 一時的エラー

### 定義
API の一時的な障害。リトライで解決する可能性が高い。

### 検出パターン

| エラー | 例 | HTTP Status |
|---|---|---|
| Rate Limit | freee API の呼出上限 | 429 |
| タイムアウト | kintone の応答遅延 | 408 / 504 |
| 一時的サーバーエラー | SaaS側のメンテナンス | 500 / 502 / 503 |
| ネットワークエラー | DNS解決失敗、接続拒否 | - |

### 対処フロー

```
Step 1: 指数バックオフでリトライ（自動）
  1回目: 5秒後
  2回目: 15秒後
  3回目: 60秒後
  → 最大3回。3回失敗 → Step 2 へ

Step 2: チャネルフォールバック（自動）
  ChannelResolver に代替チャネルを問い合わせ
  例: freee API → freee iPaaS連携 → Google Sheets → manual
  → 代替チャネルで成功 → 結果記録 + 「代替手段で完了しました」通知

Step 3: 全チャネル失敗 → タスク一時停止
  → 担当者に通知: 「○○タスクが一時停止中。APIの復旧を待っています」
  → 30分後に自動リトライ（最大3回）
  → それでも失敗 → manual フォールバック
    「以下の操作を手動で実行してください」タスク発行
```

### 学習

```
・同一APIの失敗頻度を記録
  → 「freee APIは毎月25日にrate limitに達する」パターン検出
  → 予測スケジューリング: 25日は早朝に処理を分散する
  → specialist ルールに反映（auto_updatable: true）

・チャネルフォールバックの成功率を記録
  → フォールバック優先順位の最適化
```

## 3. Type B: データ不整合

### 定義
入力データが想定と異なる。自動判断できないため人間の確認が必要。

### 検出パターン

| パターン | 例 | 検出方法 |
|---|---|---|
| 金額異常 | 請求額が前月比200%超 | 統計的異常値検出 |
| 必須フィールド欠損 | 取引先名が空 | バリデーション |
| 日付矛盾 | 納品日が受注日より前 | ロジックチェック |
| マスタ不一致 | 存在しない取引先コード | マスタ照合 |
| 重複データ | 同一請求書が2件 | ユニーク制約 |
| フォーマット不正 | 口座番号の桁数不正 | 正規表現チェック |

### 対処フロー

```
Step 1: タスクを一時停止（該当ステップのみ）

Step 2: 担当者に確認通知
  [シャチョツー → 田中さん]
  「BPOタスク #045 の Step 2 で確認が必要です:

   ⚠️ A社への請求額が ¥1,200,000 です。
   前月: ¥450,000（2.7倍）

   以下のいずれかを選んでください:
   [金額は正しい（理由を入力）]
   [金額を修正する]
   [タスクをキャンセル]」

Step 3: 回答に基づいて続行 or 修正

Step 4: 24時間未回答 → リマインド
  → 48時間未回答 → タスクキャンセル + 管理者にエスカレーション
```

### 学習

```
・「金額は正しい」+ 理由が入力された場合:
  → 理由をパターン化: 「A社は四半期まとめ請求のため3倍になることがある」
  → learned ルールに追加
  → 次回は自動でスキップ（閾値を A社は4倍まで許容に変更）

・「金額を修正」の場合:
  → 修正前後の差分を記録
  → 修正パターンが3回蓄積 → ルール化提案
```

## 4. Type C: 認証/権限エラー

### 定義
SaaS への接続が認証レベルで失敗。管理者の対応が必要。

### 検出パターン

| パターン | 例 | HTTP Status |
|---|---|---|
| OAuth トークン期限切れ | freee のリフレッシュトークン失効 | 401 |
| 権限不足 | kintone のアプリ権限変更 | 403 |
| アカウント停止 | SaaS の契約期限切れ | 403 |
| API キー無効 | キーのローテーション | 401 |
| 2FA 要求 | 追加認証が必要 | 401 |

### 対処フロー

```
Step 1: 該当チャネルの全タスクを一時停止
  → 認証が切れた状態で他タスクを実行しない

Step 2: 管理者に再認証リクエスト
  [シャチョツー → 管理者]
  「⚠️ freee との接続が切れました。
   原因: OAuthトークンの期限切れ

   再接続してください:
   [freee を再接続する] → OAuth再認証フロー

   影響:
   ・一時停止中のタスク: 3件
   ・次回予定タスク: 月末請求書発行（3日後）」

Step 3: 再認証完了 → 一時停止タスクを再開

Step 4: 72時間未対応 → manual フォールバック
  → 「freee が使えないため、手動で実行してください」
  → 全該当タスクを manual チャネルに切り替え
```

### 予防

```
・OAuth トークンの有効期限を監視
  → 期限 7日前に「もうすぐ再認証が必要です」通知
  → 自動リフレッシュ（リフレッシュトークンが有効な場合）

・SaaS の契約期限を管理
  → 契約更新の1ヶ月前にアラート
```

## 5. Type D: ビジネスロジックエラー

### 定義
法令違反やビジネスルール違反のリスクがある操作。**即座に停止**。自動リトライ禁止。

### 検出パターン

| パターン | 例 | 根拠 |
|---|---|---|
| 支払期日違反 | 下請代金の支払が60日超 | 下請法第2条の2 |
| 36協定超過 | 残業が月45hを超える承認なし勤務 | 労基法第36条 |
| インボイス不備 | 適格請求書の必須項目欠損 | 消費税法 |
| 個人情報の不正送信 | PII を含むデータの外部送信 | 個人情報保護法 |
| 金額承認未取得 | 100万超の支出が未承認のまま実行 | 社内決裁規程 |
| 二重支払 | 同一請求書への重複支払 | — |
| 法定帳票不備 | 安全書類の期限切れ状態での作業指示 | 労安法 |

### 対処フロー

```
Step 1: 即座に実行停止（該当ステップ以降を全停止）
  → 自動リトライしない
  → チャネルフォールバックもしない（legal risk なので manual もNG）

Step 2: 即座にアラート（承認者 + 管理者 + 社長）
  [シャチョツー → 社長 / 管理者]
  「🔴 法令リスク検出 — 即時対応が必要です

   検出: 下請代金の支払期日超過の可能性
   対象: 協力会社○○への3月分支払（¥2,400,000）
   期限: 2026/03/31（下請法: 60日以内）
   現在: 2026/03/28（残り3日）

   ⚠️ BPOタスク #062 を停止しました。
   手動で対応してください。

   [対応方法を見る] [社長に相談]」

Step 3: 能動提案に連携
  → 同種のリスクを今後も検出できるよう、能動提案パターンに追加
  → 「下請代金の支払期日が近づいたら早めにアラート」ルール化

Step 4: 人間が解決 → 結果を記録
  → 解決方法を学習（ただし auto_updatable: false のルールは変更しない）
```

### 予防（Type D は予防が最重要）

```
■ 事前チェック機能（Pre-execution Validation）

  全 BPO タスクの実行前に法令チェックを走らせる:

  1. タスク計画生成時に specialist ルールの [法令] セクションと照合
  2. 法令違反の可能性がある操作を事前に検出
  3. 検出 → タスク計画に ⚠️ マークを付与
  4. 承認画面で「法令リスクあり」として表示

  例:
    タスク計画: 「協力会社Aに発注書を送付」
    事前チェック: 「A社の前回支払が遅延していました。今回は期日内ですか？」
    → 承認者が確認した上で実行
```

## 6. Type E: 未知のエラー

### 定義
上記 A-D に分類できないエラー。原因不明。

### 対処フロー

```
Step 1: 詳細ログ記録
  ・エラーメッセージ全文
  ・リクエスト/レスポンスの全内容（PII マスク後）
  ・実行コンテキスト（BPOState のスナップショット）
  ・直前の成功ステップの結果

Step 2: タスク一時停止 + manual フォールバック
  → 「予期しないエラーが発生しました。
     以下の操作を手動で実行してください」
  → manual タスク発行

Step 3: 開発チームに自動報告
  → エラーログを Slack #dev-alerts に送信
  → Sentry / エラートラッキングに記録

Step 4: 同一エラーの再発監視
  → 同一パターンが3回発生:
    → specialist ルールの [リスク] セクションに
      「既知の問題: ○○の場合にエラーが発生する」を追加
    → 回避策が判明 → Type A-D のいずれかに再分類
```

## 6.5 Type F: チャネル固有エラー（NEW — チャネルファースト対応）

### 定義
LINE Bot / Email / ファイルアップロード / 音声(Whisper) / FAX(eFAX) の各入力チャネルに固有のエラー。

### 検出パターン

| チャネル | エラー例 | 検出方法 |
|---|---|---|
| LINE Bot | Webhook切断、メッセージ配信失敗、リッチメニュー表示不可 | LINE Messaging API エラーコード |
| Email(IMAP) | IMAP接続失敗、メール解析失敗、添付ファイル取得不可 | IMAP接続タイムアウト / MIME解析エラー |
| ファイル | ファイル破損、未対応形式、パスワード付きExcel、マクロ付き | ファイル検証バリデーション |
| 音声(Whisper) | 音声品質不良（ノイズ・無音）、言語認識失敗 | Whisper confidence < 0.5 |
| FAX(eFAX) | FAX受信失敗、画質不良でOCR不可、FAX番号不正 | eFAX API エラー / OCR confidence < 0.3 |

### 対処フロー

```
■ LINE Bot エラー
  Step 1: Webhook切断 → 自動再接続（3回リトライ）
  Step 2: 配信失敗 → プッシュ通知からリプライメッセージに切替
  Step 3: LINE障害 → メール/アプリ内通知にフォールバック
  → 「LINEでの通知が一時的に利用できません。メールでお知らせします」

■ Email(IMAP) エラー
  Step 1: IMAP接続失敗 → 15分後に再接続（最大5回）
  Step 2: メール解析失敗 → 生テキストとして保存 + 担当者に「手動確認」通知
  Step 3: 添付ファイル取得不可 → 「添付ファイルが開けませんでした。再送をお願いします」自動返信

■ ファイル処理エラー（統合ファイル処理エンジン）
  Step 1: ファイル破損 → 「ファイルが破損しています。再アップロードをお願いします」
  Step 2: 未対応形式 → 「この形式は現在対応していません。Excel/PDF/CSV等に変換してください」
  Step 3: パスワード付きExcel → 「パスワードを入力してください」（LINE/アプリで問い合わせ）
  Step 4: マクロ付きExcel → マクロ無効化して処理 + 「マクロの計算結果は反映されていません」警告
  Step 5: Excel構造推論失敗 → 「このExcelの構造を教えてください」対話型学習を開始

■ 音声(Whisper) エラー
  Step 1: 品質不良 → 「音声が聞き取れませんでした。もう一度お話しください」
  Step 2: 部分認識 → 認識できた部分を表示 + 「この内容で合っていますか？」確認
  Step 3: 完全失敗 → テキスト入力へフォールバック

■ FAX(eFAX) エラー
  Step 1: 受信失敗 → 送信元に再送依頼（自動FAX返信）
  Step 2: 画質不良 → 高解像度再送を依頼 + 受信済み画像で最善のOCR試行
  Step 3: OCR不可 → 手動入力へフォールバック
```

### Level別チャネルエラー対処

```
■ Level 0-1（通知・データ収集）
  → チャネルエラー時: 別チャネルでリトライ通知を送信
  → 影響: 低（データ収集が遅延するのみ）

■ Level 2（下書き作成）
  → チャネルエラー時: 処理中の下書きを一時保存 + チャネル復旧後に再開
  → 影響: 中（下書きの遅延）

■ Level 3（承認付き実行）
  → チャネルエラー時: 承認待ち状態で停止 + 別チャネルで承認依頼を再送
  → 影響: 高（承認フロー中断）→ 全チャネル失敗なら24hタイムアウト

■ Level 4（自律実行）
  → チャネルエラー時: 入力チャネル障害はキューイング、出力チャネル障害はロールバック
  → 影響: 最高 → 即座にLevel 3に降格（人間承認を要求）
```

## 7. エスカレーションマトリクス

```
■ 通知先の自動振り分け

  Type A (一時的):
    1回目: ログのみ（通知なし）
    2回目: 担当者に通知
    3回目: 管理者 + 担当者に通知

  Type B (データ不整合):
    即時: 担当者に確認依頼
    24h未回答: 管理者にエスカレーション
    48h未回答: 社長にエスカレーション + タスクキャンセル

  Type C (認証):
    即時: 管理者（IT担当）に通知
    72h未対応: 社長にエスカレーション

  Type D (法令リスク):
    即時: 社長 + 管理者 + 担当者に同時通知 ★全員に即通知
    対応完了まで: 関連タスク全停止

  Type E (未知):
    即時: 担当者にmanualフォールバック通知
    同時: 開発チームに報告
    3回再発: 管理者にエスカレーション

  Type F (チャネル固有):
    LINE障害: 即時メール/アプリ通知にフォールバック + 運営チームにSlack通知
    Email障害: 即時LINE/アプリ通知にフォールバック
    ファイル破損: 即時アップロード者に再送依頼
    音声認識失敗: テキスト入力へ誘導（通知なし）
    FAX障害: 送信元にFAX再送依頼 + 管理者にメール通知
```

## 8. 失敗 → 学習 → 予防 のサイクル

```
          失敗発生
             │
     ┌───────┴───────┐
     │  Type 分類     │
     └───────┬───────┘
             │
     ┌───────┴───────┐
     │  対処実行      │ ← 上記フロー
     └───────┬───────┘
             │
     ┌───────┴───────┐
     │  結果記録      │ ← execution_knowledge テーブル
     └───────┬───────┘
             │
     ┌───────┴───────┐
     │  パターン分析   │ ← 同一エラーの集約（業種×ドメイン×チャネル）
     └───────┬───────┘
             │
     ┌───────┴────────────────┐
     │  learned ルール生成      │ ← 会社固有の回避ルール
     └───────┬────────────────┘
             │
     ┌───────┴────────────────┐
     │  specialist 昇格判定    │ ← 複数社で同一パターン → 共通ルール化
     └───────┬────────────────┘
             │
     ┌───────┴────────────────┐
     │  予防チェックに追加     │ ← 次回の事前チェックで検出可能に
     └──────────────────────┘

→ 失敗が0になることはない。
   しかし「同じ失敗を2回しない」ことで、成功率が段階的に向上する。
   目標: 月次成功率 82% → 89% → 94%（09_導入ジャーニーの実績目標と一致）
```

## 9. DB設計への影響

```sql
-- bpo_task_executions テーブルに失敗情報を追加

ALTER TABLE bpo_task_executions
    ADD COLUMN error_type TEXT,                -- 'type_a' / 'type_b' / 'type_c' / 'type_d' / 'type_e'
    ADD COLUMN error_detail JSONB DEFAULT '{}', -- エラー詳細
    ADD COLUMN fallback_channel TEXT,           -- フォールバック先チャネル
    ADD COLUMN resolution TEXT,                 -- 'auto_retry' / 'channel_fallback' / 'human_resolved' / 'cancelled'
    ADD COLUMN resolution_detail JSONB DEFAULT '{}', -- 解決詳細
    ADD COLUMN learned_rule_id UUID;            -- 学習ルールへの参照（生成された場合）

-- 失敗パターン分析用インデックス
CREATE INDEX idx_bpo_errors
    ON bpo_task_executions(error_type, company_id)
    WHERE error_type IS NOT NULL;
```
