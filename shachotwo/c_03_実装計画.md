### 並列化の鉄則

## Agent間状態管理設計（LangGraph Checkpointer）

### 背景と課題

ブレイン編・BPO編・自社システム編の3フェーズは独立したLangGraphとして設計されているが、「HearingStateで集めたナレッジがBPOExecutionStateにどう渡るか」「途中でエラーが起きたときに状態を巻き戻せるか」の設計が未定義。これがないと、フェーズ間でデータが欠落し、BPO誤実行やシステム生成の要件ミスが発生する。

### 共通コンテキストスキーマ（CompanyContext）

全LangGraphが参照する共通コンテキスト:

```python
class CompanyContext(TypedDict):
    company_id: str
    plan_type: str                    # starter/growth/enterprise
    active_agents: List[str]          # 起動中のエージェントID
    knowledge_coverage: float         # ナレッジ充足率 0.0-1.0
    bpo_levels: Dict[str, int]        # workflow_id → 現在のBPO Level
    connected_saas: List[str]         # 接続済みSaaS名リスト
    cost_remaining_yen: int           # 今月の残LLMコスト枠
    security_flags: Dict[str, bool]   # kill_switch / throttled / high_risk_cooldown
```

This context is loaded at the start of every LangGraph execution and is read-only during execution (mutations go through dedicated state transitions only).

### フェーズ間データ受け渡しAPI

3つのフェーズ間の受け渡しを明確に定義:

**ブレイン → BPO の受け渡し**
```python
class BrainToBPOHandoff(TypedDict):
    company_id: str
    source_workflow_ids: List[str]    # 発見されたworkflow IDs
    knowledge_snapshot_id: str        # どの時点のナレッジかを記録
    confidence_threshold: float       # このナレッジでBPOを開始してよい閾値
    handoff_reason: str               # 'workflow_discovered' | 'coverage_reached' | 'manual'
    created_at: datetime
```

**BPO → 自社システム の受け渡し**
```python
class BPOToSystemHandoff(TypedDict):
    company_id: str
    crystallization_point_id: str     # system_crystallization_pointsのID
    bpo_execution_log_ids: List[str]  # 要件定義の根拠となった実行ログ
    replacement_saas: List[str]       # 置換対象SaaS
    roi_simulation: dict              # モンテカルロROI結果
    handoff_reason: str               # 'crystallization_detected' | 'manual'
    created_at: datetime
```

DB tables:
- phase_handoff_log (id, company_id, from_phase, to_phase, handoff_data JSONB, triggered_by, created_at)

### LangGraph Checkpointer設計（途中再開保証）

Using Supabase as the checkpointer backend:

```python
from langgraph.checkpoint.postgres import PostgresSaver

# Supabase PostgreSQL接続
checkpointer = PostgresSaver.from_conn_string(
    settings.SUPABASE_DB_URL
)

# 各LangGraphに適用
brain_graph = brain_workflow.compile(
    checkpointer=checkpointer,
    interrupt_before=["ask_question"]  # 社長への質問前に必ず一時停止
)

bpo_graph = bpo_workflow.compile(
    checkpointer=checkpointer,
    interrupt_before=["request_approval", "execute_external_write"]  # 承認・実行前に停止
)
```

Thread ID設計 (=実行セッションの識別子):
- ヒアリングセッション: `hearing_{company_id}_{session_date}`
- BPO実行: `bpo_{company_id}_{rule_id}_{execution_date}`
- システム生成: `system_{company_id}_{crystallization_id}`

### エラー時の状態巻き戻し（Rollback保証）

3段階のロールバック:

Level 1 — LangGraph内ロールバック（ステップ単位）:
- Checkpointerから直前の安全なステップに戻る
- BPOの場合: SaaS書き込みステップの前まで巻き戻し

Level 2 — DBトランザクションロールバック:
- 全BPO実行はDBトランザクション内で実行
- 途中失敗 → 全操作をロールバック + 監査ログに記録

Level 3 — SaaS操作ロールバック:
- 各コネクタにrollback()メソッドを実装（例: freee.delete_draft(draft_id)）
- rollback不可能な操作（メール送信等）は実行前に必ず人間承認を挟む

  ・3回リトライ（exponential backoff）
  ・失敗時: 「手動対応」フラグを立ててユーザーに通知
  ・API変更検知: レスポンス構造の変化を検知→アラート

■ LLM構造化出力（JSONパース）
  ・パースエラー時: LLMに再生成を依頼（最大2回）
  ・2回失敗時: 非構造化テキストとして保存+手動レビューフラグ

■ Embedding生成（Voyage AI）
  ・3回リトライ
  ・失敗時: キューに入れて後続バッチで再処理
  ・レート制限: バッチ処理（最大128テキスト/リクエスト）

■ LINE Bot Webhook
  ・LINE Platformからのリトライに対応（べき等性保証）
  ・処理タイムアウト: 先に200応答 → 非同期で処理 → 結果をプッシュ送信
  ・ファイルダウンロード失敗: 3回リトライ後にユーザーに再送信を依頼

■ メール受信（IMAP）
  ・接続断: 自動再接続（exponential backoff）
  ・添付ファイル抽出失敗: エラー通知メールを返信
  ・スパムフィルタ: 登録済みドメインからのみ処理

■ ファイル処理エンジン
  ・ファイルサイズ上限: 50MB（超過時はエラー通知）
  ・破損ファイル: 自動検知 → ユーザーに再送信を依頼
  ・OCR精度不足: 信頼度スコアが閾値以下の場合、手動確認フラグ
```

## 解約・事業承継・データポータビリティ設計

### 背景と課題

「解約したらデータはどうなるか」「社長が変わったら（M&A・事業承継）どうするか」が未設計。中小企業の社長は事業承継を常に意識しており、「蓄積したナレッジが会社の資産として引き継げるか」は導入判断の重要ポイント。また「いつでも出られる」安心感がチャーン防止にも逆説的に寄与する（ベンダーロックインへの恐怖が導入阻害要因になるため）。

### データエクスポート（いつでも持ち出せる設計）

エクスポート対象と形式:
- ナレッジ全件: JSON（knowledge_items + knowledge_relations）+ Excel（人間可読版）
- 業務フロー: Mermaid形式 + PDF（可視化済み）
- BPO実行ログ: CSV（過去全履歴）
- 意思決定ルール: JSON + 人間可読テキスト
- 9次元スナップショット: JSON + PDF診断レポート
- 自社生成システム: ソースコード一式（ZIP）+ DB定義SQL

### 幻覚検出・防止設計

#### RAG幻覚検出

```python
class HallucinationDetector:
    async def check(self, response: str, source_chunks: list[str]) -> HallucinationReport:
        """
        回答の各文を抽出し、ソースチャンクに根拠があるか検証
        根拠なし → hallucination flagged
        """
        claims = await self.extract_claims(response)
        for claim in claims:
            grounded = await self.verify_claim_in_sources(claim, source_chunks)
            if not grounded:
                return HallucinationReport(detected=True, claim=claim)
```

- Q&A応答の全文を「クレーム単位」に分解し、各クレームが参照ナレッジチャンクに根拠を持つかを検証する
- 根拠なしクレームが検出された場合は回答を差し戻し、「情報が見つかりませんでした」へ自動フォールバック
- 幻覚が検出された場合は `hallucination_logs` テーブルに記録し、改善サイクルのインプットとして活用する

#### 信頼度キャリブレーション測定

- **Expected Calibration Error（ECE）** を月次で計算する
- 「85%信頼度と言ったQ&Aの正答率は実際に85%か？」という問いを定期的に検証する
- **ECE ≤ 0.10** を品質基準とし、超過時はプロンプト・温度パラメータを見直す

```python
def compute_ece(confidences: list[float], accuracies: list[bool], n_bins: int = 10) -> float:
    """Expected Calibration Error を計算する"""
    bin_boundaries = [i / n_bins for i in range(n_bins + 1)]
    ece = 0.0
    for lo, hi in zip(bin_boundaries[:-1], bin_boundaries[1:]):
        in_bin = [(c, a) for c, a in zip(confidences, accuracies) if lo <= c < hi]
        if in_bin:
            avg_conf = sum(c for c, _ in in_bin) / len(in_bin)
            avg_acc = sum(a for _, a in in_bin) / len(in_bin)
            ece += abs(avg_conf - avg_acc) * len(in_bin) / len(confidences)
    return ece
```

```python
class PhaseProgressTracker:
    """
    顧客のPhase到達状況をリアルタイム追跡
    CS介入のトリガーとして使用
    """

    PHASE_THRESHOLDS = {
        "phase1": {
            "min_knowledge_items": 5,
            "min_qa_accuracy": 0.60,
            "min_qa_count": 3,
        },
        "phase2": {
            "min_knowledge_items": 20,
            "min_qa_accuracy": 0.80,
            "min_qa_count": 20,
            "knowledge_coverage_score": 0.40,  # 業務領域の40%をカバー
        },
        "phase3": {
            "min_knowledge_items": 50,
            "min_qa_accuracy": 0.90,
            "bpo_shadow_enabled": True,
            "knowledge_coverage_score": 0.70,
        },
    }

    async def calculate_current_phase(self, company_id: str) -> PhaseStatus:
        metrics = await self.get_company_metrics(company_id)
        for phase in ["phase3", "phase2", "phase1"]:
            if self._meets_threshold(metrics, self.PHASE_THRESHOLDS[phase]):
                return PhaseStatus(current_phase=phase, metrics=metrics)
        return PhaseStatus(current_phase="phase0", metrics=metrics)
```

```sql
-- Phase進捗の履歴記録
CREATE TABLE company_phase_history (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  company_id uuid REFERENCES companies(id),
  previous_phase varchar(10),
  new_phase varchar(10),
  knowledge_items_count int,
  qa_accuracy float,
  knowledge_coverage_score float,
  transitioned_at timestamptz DEFAULT now()
);

-- Phase到達率ダッシュボードビュー
CREATE VIEW phase_reach_rates AS
SELECT
  DATE_TRUNC('month', c.created_at) as cohort_month,
  COUNT(*) as total_companies,
  COUNT(CASE WHEN ph1.company_id IS NOT NULL THEN 1 END) as reached_phase1,
  COUNT(CASE WHEN ph2.company_id IS NOT NULL THEN 1 END) as reached_phase2,
  COUNT(CASE WHEN ph3.company_id IS NOT NULL THEN 1 END) as reached_phase3,
  ROUND(COUNT(CASE WHEN ph1.company_id IS NOT NULL THEN 1 END)::decimal / COUNT(*), 2) as phase1_rate,
  ROUND(COUNT(CASE WHEN ph2.company_id IS NOT NULL THEN 1 END)::decimal / COUNT(*), 2) as phase2_rate
FROM companies c
LEFT JOIN (SELECT DISTINCT company_id FROM company_phase_history WHERE new_phase = 'phase1') ph1 ON c.id = ph1.company_id
LEFT JOIN (SELECT DISTINCT company_id FROM company_phase_history WHERE new_phase = 'phase2') ph2 ON c.id = ph2.company_id
LEFT JOIN (SELECT DISTINCT company_id FROM company_phase_history WHERE new_phase = 'phase3') ph3 ON c.id = ph3.company_id
GROUP BY 1;
```

## Feature Flag 戦略

> LaunchDarkly等の外部サービスは使わない。DB-driven feature flags で十分。
> 1人開発でコストを抑えつつ、安全なロールアウトを実現。

### Feature Flag テーブル

```sql
CREATE TABLE feature_flags (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  flag_name       VARCHAR(100) NOT NULL UNIQUE,      -- 例: "bpo_level3_enabled"
  description     TEXT,
  enabled_global  BOOLEAN DEFAULT FALSE,             -- グローバルON/OFF
  rollout_percent INTEGER DEFAULT 0 CHECK (rollout_percent BETWEEN 0 AND 100),
  company_ids     UUID[] DEFAULT '{}',               -- 特定企業のみ有効
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
);

-- 使用例
INSERT INTO feature_flags (flag_name, description, enabled_global, company_ids)
VALUES
  ('proactive_proposals', '能動提案機能', FALSE, '{uuid-of-beta-company}'),
  ('bpo_level3', 'BPO Level 3 自動実行', FALSE, '{}'),
  ('semantic_cache', 'セマンティックキャッシュ', TRUE, '{}');
```

### Feature Flag チェックロジック

```python
async def is_feature_enabled(flag_name: str, company_id: str) -> bool:
    flag = await get_flag(flag_name)
    if flag is None:
        return False
    if flag.enabled_global:
        return True
    if company_id in flag.company_ids:
        return True
    if flag.rollout_percent > 0:
        # company_idのハッシュ値でパーセンテージ判定（決定論的）
        hash_val = int(hashlib.md5(f"{flag_name}:{company_id}".encode()).hexdigest(), 16)
        return (hash_val % 100) < flag.rollout_percent
    return False
```

### ロールアウト手順

```
1. 開発完了 → flag_name を作成（enabled_global=FALSE）
2. 社内テスト → company_ids に自社IDを追加
3. β顧客テスト → company_ids にβ顧客IDを追加
4. 段階ロールアウト → rollout_percent: 10% → 25% → 50% → 100%
5. 全展開 → enabled_global=TRUE、rollout_percent削除
6. コード整理 → フラグチェックのif文を削除（dead code除去）
```

  各ファイルの構成:
    -- Migration: 001_initial_schema
    -- Created: 2026-03-15
    -- Description: MVP初期スキーマ（12テーブル + RLS + インデックス）

    BEGIN;

    -- UP（適用）
    CREATE TABLE IF NOT EXISTS ...;
    CREATE POLICY ...;
    CREATE INDEX ...;

    COMMIT;

  ロールバック: 各マイグレーションに対応する down_{連番}.sql を用意
    -- DOWN: 001_initial_schema
    BEGIN;
    DROP TABLE IF EXISTS ... CASCADE;
    COMMIT;

■ マイグレーション実行

  開発: python -m server.migrate up
  本番: GitHub Actions → Supabase CLI (supabase db push)
  ロールバック: python -m server.migrate down 1  -- 直近1つを戻す

  安全策:
    ・本番適用前に staging で検証必須
    ・破壊的変更（DROP COLUMN等）は2ステップ:
      Step 1: 新カラム追加 + データ移行
      Step 2: 旧カラム削除（次回リリースで）

■ テストデータシーディング

  db/seeds/
    dev_seed.sql    -- 開発用（テスト企業2社 + テストユーザー4名 + サンプルナレッジ20件）
    test_seed.sql   -- テスト用（RLS検証用の2社データ）

  シーディング: python -m server.migrate seed dev
```

### DB接続プーリング（Supabase Pooler経由）

```
■ 接続方式: Supabase Connection Pooler（PgBouncer）

  MVPではSupabase組み込みのPgBouncerを使用。カスタムプーラーは不要。

  接続文字列: postgresql://[user]:[pass]@[host]:6543/postgres
    ※ ポート6543 = Pooler経由（5432はダイレクト接続）

■ プーリング設定

  ┌───────────────────┬─────────────┬──────────────────────────┐
  │ パラメータ          │ 値           │ 理由                      │
  ├───────────────────┼─────────────┼──────────────────────────┤
  │ pool_mode          │ transaction  │ リクエスト間で接続を共有    │
  │ pool_size/instance │ 10           │ Cloud Run 10インスタンス   │
  │                   │              │ × 10 = 最大100接続        │
  │ statement_timeout  │ 30s          │ 長時間クエリの強制終了      │
  │ idle_timeout       │ 60s          │ アイドル接続の回収          │
  └───────────────────┴─────────────┴──────────────────────────┘

  ※ Supabase Free/Pro プランのDB接続上限: 直接60 + Pooler経由200
  ※ max_instances(10) × pool_size(10) = 100 < 200 で安全マージンあり

■ Python側の設定（db/supabase.py）

  from supabase import create_client, Client

  # Pooler経由接続（推奨）
  SUPABASE_URL = os.environ["SUPABASE_URL"]
  SUPABASE_KEY = os.environ["SUPABASE_SERVICE_ROLE_KEY"]

  _client: Client | None = None

  def get_supabase_client() -> Client:
      global _client
      if _client is None:
          _client = create_client(SUPABASE_URL, SUPABASE_KEY)
      return _client

  ※ supabase-py は内部で httpx の接続プールを管理
  ※ 追加のコネクションプール設定は不要（Supabase Pooler側で制御）
```
