# 社内スタート計画 — シャチョツーで自社を回す

> **シャチョツーMVP完成前から、CLIベースで11エージェントを即座に稼働させる。**
> MVP完成後にシャチョツーに移行し、そのままドッグフーディング + 最強の営業デモにする。
>
> **Phase Aの11エージェントは暫定構成。シャチョツーの動的エージェントモデル（BPO編）が正の設計であり、Phase B以降はそちらに合流する。Phase Aの経験は「ゲノムの推奨初期エージェントセット」としてプロダクトに還元する。**

---

## 1. 戦略概要

### 実行順序

```
Step 1: 現職（コンサル・人材紹介会社）で Phase A を実行
        → 11エージェントを CLI で構築・運用
        → 「コンサル・人材紹介業ゲノム」が1つ目のゲノムになる

Step 2: 現職を「1社目の顧客モデル」として実績化
        → シャチョツー MVP に Phase B 移行
        → 「うちで回してます」が営業トーク

Step 3: スピンアウト → 独立
        → 新会社（シャチョツー社）自体もシャチョツーで回す
        → 「SaaS業ゲノム」が2つ目のゲノム
        → 2業種のゲノムが揃った状態でローンチ
```

### なぜやるか

1. **ドッグフーディング**: 自社で使い倒すことでプロダクトの品質を担保
2. **最強のデモ**: 「自社で全部回してます」が営業トークになる
3. **限界突破**: 社長・戦略・営業・マーケ・採用・経理を1人で回すのは不可能。AIで回す
4. **MVP開発中も価値が出る**: Phase A は CLI で即日稼働。MVP完成を待たない
5. **ゲノムが2つ揃った状態で独立**: コンサル・人材紹介業 + SaaS業

### なぜ Phase A（CLI）が必要か

**Phase B（シャチョツーMVP上での稼働）にはプロダクトが必要。プロダクトはまだない。**

| | Phase A なし | Phase A あり |
|---|---|---|
| MVP開発中 | エージェント支援ゼロ。全部手動 | CLIエージェントが即日稼働 |
| MVP設計品質 | 机上の空論リスク | 実運用で検証済みの知見が設計に反映 |
| 現職での実績 | なし | 「AIエージェントで業務効率化」の実績 |

### 各エージェント仕様

```
入力: リード情報、商談メモ、業界データ、顧客の課題ヒアリングシート
```

### スクリプト仕様

Phase Aでは `scripts/` 配下に11本のシェルスクリプトを作る。

#### 共通仕様

```bash
#!/bin/bash
set -euo pipefail

# 共通: コスト上限チェック
check_cost_budget() {
    TODAY_COST=$(cat .context/logs/daily_cost.json | jq '.total_usd')
    if (( $(echo "$TODAY_COST > 15" | bc -l) )); then
        echo "ERROR: Daily cost limit exceeded ($TODAY_COST). Escalating."
        slack_notify "コスト上限超過: $TODAY_COST USD"
        exit 1
    fi
}

# 共通: 監査ログ記録
log_action() {
    local agent=$1 action=$2 output_path=$3
    jq -n \
        --arg ts "$(date -u +%Y-%m-%dT%H:%M:%SZ)" \
        --arg agent "$agent" \
        --arg action "$action" \
        --arg output "$output_path" \
        '{timestamp:$ts, agent:$agent, action:$action, output_path:$output}' \
        >> .context/logs/audit.jsonl
}
```

## 5. Phase C — 完全自律運用

### 自律レベル定義

```
Level 0: 人間が指示 → AIが実行
Level 1: AIが提案 → 人間が承認 → AIが実行
Level 2: AIが実行 → 人間に事後報告（異常時のみ介入）
Level 3: AIが実行 → ログ記録のみ（人間は監査時に確認）
```

### エージェント別の目標自律レベル

| エージェント | Phase A | Phase B | Phase C目標 |
|---|---|---|---|
| Strategy Agent | Level 0 | Level 1 | Level 1（意思決定は常に人間承認） |
| Sales Agent | Level 0 | Level 1 | Level 2（提案書生成は自動、送信は承認） |
| Marketing Agent | Level 0 | Level 1 | Level 2（コンテンツ生成は自動、公開は承認） |
| Hiring Agent | Level 0 | Level 1 | Level 2（スクリーニングは自動、面談判断は承認） |
| Accounting Agent | Level 0 | Level 0 | Level 2（仕訳は自動、支払い実行は承認） |
| Analysis Agent | Level 0 | Level 1 | Level 3（分析・レポートは完全自動） |
| CS Agent | Level 0 | Level 1 | Level 2（FAQ回答は自動、エスカレーションは承認） |
| Legal Agent | Level 0 | Level 0 | Level 1（契約は常に人間承認） |
| Product Agent | Level 0 | Level 1 | Level 2（分析は自動、ロードマップ変更は承認） |
| DevOps Agent | Level 0 | Level 1 | Level 2（監視・アラートは自動、本番操作は承認） |
| PR Agent | Level 0 | Level 1 | Level 2（ドラフトは自動、公開は承認） |

**絶対にLevel 3にしない操作:**
- 資金移動・支払い実行

## 6. エージェント規約（全フェーズ共通）

> **このセクションは全AIエージェントの行動規範。**
> エージェント起動時に必ず参照すること。

### 6.1 MCP禁止

- **MCPは一切使用しない**
- 全操作はCLI経由で行う（Claude CLI, gh, slack-cli, freee-cli, gcloud 等）
- 理由: MCPは抽象度が高すぎて、操作の透明性・再現性・デバッグ性が落ちる

### 6.2 Memory管理

- **セッション間のMemoryは使用しない**（Worktreeベースの開発のため）
- 永続化が必要な情報は**全てgit管理**する
- 設計判断 → `shachotwo/adr/`
- エージェント出力 → `.context/{agent}/`
- 運用ログ → `.context/logs/`

### 6.3 一時ファイル

- **一時ファイルは `.context/tmp/` に作成する**
- `/tmp/` や OS のテンポラリディレクトリは**使用禁止**
- 理由: Subagentや別Agentが一時ファイルを読む必要がある

### 6.4 ADR（Architecture Decision Records）

- **大きめの設計変更・技術選定は必ずADRを作成**
- 保存先: `shachotwo/adr/`

### 6.5 Plan Review

- **Planを人間に提示する前に、AIセルフレビューを実施する**
- レビュー観点: 技術的実現可能性、既存設計との整合性、スコープの妥当性

### 6.6 Fail-safe原則

- **判断に迷ったら実行しない。人間にエスカレーションする**
- エスカレーション: Slack通知 → `.context/escalation/` に記録 → タスク一時停止

### 6.7 変更の原子性

- **1タスク = 1コミット**
- コミットメッセージは `feat:`, `fix:`, `chore:`, `docs:` プレフィックス + 日本語OK

### 6.8 出力の検証義務

- **LLM出力をそのまま外部に送信しない**
- 事実確認、トーン確認、PII漏洩チェック、社内情報の意図しない露出チェック

### 6.9 コスト上限

- 通常タスク: $1.00 / タスク
- 大規模分析: $5.00 / タスク（事前承認必要）
- 月間上限: $500
- 無限ループ防止: 同一プロンプトの再試行は最大3回まで

### 6.10 秘密情報の扱い

- **ローカル環境**: 環境変数（`.env`）で管理。`.env` はgitignoreに入れる
- **本番環境**: GCP Secret Manager のみ
- APIキー・トークンをソースコードに書くこと・シークレットをログに出力すること・`.env` をコミットすることは禁止

### 6.11 並列実行の排他制御

- `.context/locks/{resource_name}.lock` で排他制御
- ロック取得できない場合は待機（最大5分）→ タイムアウトでエスカレーション

### 6.12 ログの構造化

- 全エージェントの実行ログは**JSON Lines (`.jsonl`)**で `.context/logs/` に記録
- 「なぜこの判断をしたか」を後から追跡可能にする

### 6.13 ゲノム・ナレッジのフィードバック義務

- Phase Aで得た知見は、シャチョツーのゲノムテンプレートにフィードバックする
- フィードバック先: `.context/product/feedback/` に記録 → Product Agentが集約

## 7. 運用ルール

### Human-in-the-Loop 基準

**必ず人間承認が必要な操作（例外なし）:**

| カテゴリ | 具体例 |
|---|---|
| 金銭 | 支払い実行、振込、請求書送付、契約締結 |
| 対外コミュニケーション | 顧客へのメール送信、SNS投稿の公開 |
| 採用 | 面談の合否判断、オファー送付 |
| 経営判断 | 事業方針変更、予算配分 |
| 法的行為 | 契約書の署名、規約変更 |
| インフラ | 本番環境へのデプロイ、DBスキーマ変更 |

**AIが自律実行してよい操作:**

| カテゴリ | 具体例 | 条件 |
|---|---|---|
| 情報収集 | 市場調査、競合モニタリング | 外部APIの利用範囲内 |
| ドラフト生成 | 提案書、メール、記事、JD | 送信・公開はしない |
| 分析 | KPI集計、トレンド分析 | 社内データのみ |
| 整理 | ファイル整理、タグ付け | 削除はしない |
| 通知 | Slackへの社内通知 | 社内チャネルのみ |

### 承認フロー

```
AIエージェント
  ├─ 自律実行可能 → 実行 → ログ記録

                  → 社長が確認
                      ├─ 承認 → 実行 → ログ記録
                      ├─ 修正指示 → AI修正 → 再承認フロー
                      └─ 却下 → ログ記録（却下理由）
```

## 9. 懸念事項と対策

### IP（知的財産）の帰属

現職で作ったコード・ゲノムのスピンアウト時の扱い:
- **コード**: 業務時間外に個人リポジトリで開発 or 現職と事前合意
- **ゲノム（パターン・テンプレート）**: 匿名化したパターンは持ち出し可能にする合意を取る
- **現職のデータ**: 顧客情報・候補者情報は持ち出し不可。ゲノム（匿名化済み）のみ

### 現職のデータ保護

- 顧客情報・候補者の個人情報は `.context/` に残さない
- ゲノムテンプレートには一般化・匿名化したパターンのみ格納
- `.env` に格納した認証情報は絶対にコミットしない

### 現職の理解

- 社内にAIエージェントを入れることへの事前合意が必要
- 「業務効率化の実験」として位置づける
- データの取り扱いについて透明性を確保する

## 10. 11エージェントの役割定義

> Phase Aで社内運用するCLIエージェント群。
> 各エージェントは独立したPythonスクリプト or Makefileターゲットとして実装。
> orchestratorが全体を調整する。

### エージェント一覧

| # | エージェント名 | 担当領域 | 実行頻度 | 依存先 |
|---|---|---|---|---|
| 1 | knowledge-collector | 社長との対話からナレッジを構造化 | 随時（対話発生時） | llm/, brain/extraction/ |
| 2 | qa-engine | 社内Q&A応答 | 随時（質問発生時） | brain/knowledge/, llm/ |
| 3 | process-miner | SaaS操作ログから業務フローを発見 | 日次バッチ | workers/connector/, brain/inference/ |
| 4 | bpo-executor | 定型業務の自動実行（Shadow→承認付き） | イベント駆動 | workers/bpo/, process-miner |
| 5 | report-generator | 日次/週次レポート生成 | 日次 8:00 / 週次 月曜 9:00 | brain/knowledge/, brain/twin/ |
| 6 | risk-detector | リスク検出・アラート | 6時間ごと | brain/proactive/, brain/twin/ |
| 7 | genome-builder | 業種テンプレート構築・更新 | 週次 | brain/genome/, brain/knowledge/ |
| 8 | saas-connector | SaaS API接続管理・ヘルスチェック | 15分ごと | workers/connector/ |
| 9 | audit-logger | 全操作の監査ログ記録 | 常時（イベント駆動） | security/audit.py |
| 10 | eval-runner | LLM品質評価・テスト実行 | 日次 深夜 | llm/, tests/ |
| 11 | orchestrator | 他エージェントの調整・スケジューリング | 常時 | 全エージェント |

### 各エージェントの詳細

```
■ Agent 1: knowledge-collector
  入力: 音声ファイル(.wav/.m4a) / テキスト / ドキュメント(.pdf/.xlsx)
  処理: Whisper文字起こし → LLM構造化 → ナレッジ項目生成
  出力: knowledge_items レコード（+ embedding）
  依存: llm/client.py, brain/extraction/
  CLIコマンド: shachotwo knowledge add

■ Agent 2: qa-engine
  入力: 自然言語の質問
  処理: クエリembedding → ベクトル検索 → LLMによる回答生成
  出力: 回答テキスト + 参照ナレッジ + confidence score
  依存: brain/knowledge/, llm/client.py, pgvector
  CLIコマンド: shachotwo qa ask

■ Agent 3: process-miner
  入力: SaaS操作ログ（freee, kintone, Slack等）
  処理: 操作パターン抽出 → 業務フロー推定 → 繰り返しパターン検出
  出力: department_flows レコード + ボトルネック候補
  依存: workers/connector/(ログ取得), brain/inference/
  CLIコマンド: shachotwo bpo discover

■ Agent 4: bpo-executor
  入力: 承認済みワークフロー定義 + トリガーイベント
  処理: Safety Check → (承認待ち) → SaaS API実行 → 結果検証
  出力: BPO実行ログ + 成功/失敗レポート
  依存: workers/bpo/, workers/connector/, process-miner(フロー定義)
  CLIコマンド: shachotwo bpo execute

■ Agent 5: report-generator
  入力: 会社状態スナップショット + 直近のナレッジ変更 + BPO実行結果
  処理: データ集約 → LLMによるサマリー生成 → フォーマット整形
  出力: 日次レポート（LINE通知） / 週次レポート（PDF + LINE）
  依存: brain/knowledge/, brain/twin/
  CLIコマンド: shachotwo report generate

■ Agent 6: risk-detector
  入力: 会社状態 + ナレッジ + 外部情報（法改正等）
  処理: リスクスコアリング → 閾値超過検出 → 提案生成
  出力: proactive_proposals レコード（risk_alert タイプ）
  依存: brain/proactive/, brain/twin/
  CLIコマンド: shachotwo risk scan

■ Agent 7: genome-builder
  入力: 蓄積ナレッジ + 業種テンプレートのベースライン
  処理: ナレッジ → ゲノム次元マッピング → 充足度計算 → テンプレート更新
  出力: genome_templates レコード更新 + 充足度スコア
  依存: brain/genome/, brain/knowledge/
  CLIコマンド: shachotwo genome build

■ Agent 8: saas-connector
  入力: tool_connections 設定
  処理: OAuth トークンリフレッシュ → ヘルスチェック → ログ取得
  出力: 接続ステータス + 取得ログ
  依存: workers/connector/
  CLIコマンド: shachotwo connector status

■ Agent 9: audit-logger
  入力: 全エージェントのイベントストリーム
  処理: イベント正規化 → 構造化ログ生成 → 保存
  出力: audit_logs レコード
  依存: security/audit.py
  CLIコマンド: shachotwo audit log

■ Agent 10: eval-runner
  入力: ゴールデンデータセット（Q&A: 50件, extraction: 30件）
  処理: テストスイート実行 → 精度計測 → 回帰検出
  出力: 評価レポート（precision/recall/F1 + 前回比較）
  依存: llm/client.py, tests/
  CLIコマンド: shachotwo eval run

■ Agent 11: orchestrator
  入力: スケジュール定義 + イベントトリガー
  処理: エージェント起動 → 依存関係解決 → エラーハンドリング → リトライ
  出力: 実行ログ + ステータスダッシュボード
  依存: 全エージェント
  CLIコマンド: shachotwo orchestrate run-all
```

### エージェント依存関係図

```
orchestrator (Agent 11)
  ├── knowledge-collector (1) ─→ qa-engine (2)
  ├── saas-connector (8) ─→ process-miner (3) ─→ bpo-executor (4)
  ├── genome-builder (7) ←── knowledge-collector (1)
  ├── risk-detector (6) ←── knowledge-collector (1) + process-miner (3)
  ├── report-generator (5) ←── all data sources
  ├── eval-runner (10) ── 独立（日次バッチ）
  └── audit-logger (9) ←── 全エージェント（イベント駆動）
```

## 11. CLIコマンドカタログ

> Phase Aでは全操作をCLIで実行する（ADR-005準拠）。
> GUIは Phase B 以降。CLIログがそのまま監査証跡になる。

### ナレッジ操作

```bash
# 音声ファイルからナレッジを追加（Whisper → LLM構造化）
shachotwo knowledge add --source voice --file recording.wav

# テキストから直接追加
shachotwo knowledge add --source text --content "見積もり100万以上は社長承認"

# ドキュメントから追加（PDF/Excel/Word）
shachotwo knowledge add --source document --file 業務マニュアル.pdf

# ナレッジ検索
shachotwo knowledge search "見積もりの承認フロー"

# カテゴリ別一覧
shachotwo knowledge list --category decision_rules
shachotwo knowledge list --category business_flows --department 営業部

# ナレッジの信頼度確認（Half-life Model）
shachotwo knowledge health --stale-threshold 0.5

# ナレッジ更新（リフレッシュ）
shachotwo knowledge refresh --id <knowledge_id> --content "更新内容"
```

### Q&A

```bash
# 質問
shachotwo qa ask "今月の受注判断基準は？"

# 部門スコープ指定
shachotwo qa ask "残業申請のフローは？" --department 総務部

# バッチ質問（ファイルから）
shachotwo qa batch --file questions.txt --output answers.json
```

### BPO（業務自動化）

```bash
# 業務フロー自動発見（SaaS操作ログ分析）
shachotwo bpo discover --saas freee --days 30

# Shadow Mode起動（バックグラウンドで並行実行）
shachotwo bpo shadow --workflow invoice_check --duration 7d

# Shadow Mode結果確認
shachotwo bpo shadow-report --workflow invoice_check

# ワークフロー実行（ドライラン）
shachotwo bpo execute --workflow invoice_check --dry-run

# ワークフロー実行（本番）
shachotwo bpo execute --workflow invoice_check

# ワークフロー一覧
shachotwo bpo list --status active
```

### ゲノム（業種テンプレート）

```bash
# ゲノム初期化（業種テンプレートから）
shachotwo genome init --industry construction --company-size 30

# ゲノム充足度確認
shachotwo genome status

# ゲノム手動更新
shachotwo genome update --dimension strategy --field vision --value "地域No.1の建設会社"

# ゲノムエクスポート
shachotwo genome export --format json --output my_genome.json

# 他社ゲノムとの比較（匿名化ベンチマーク）
shachotwo genome benchmark --industry construction
```

### レポート・提案

```bash
# 日次レポート生成
shachotwo report generate --type daily

# 週次レポート生成（PDF出力）
shachotwo report generate --type weekly --output weekly_report.pdf
```
