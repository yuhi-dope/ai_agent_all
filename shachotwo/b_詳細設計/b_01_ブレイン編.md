# 001: ブレイン編 — コンサルを潰す

> **このフェーズの本質**: 社長の頭の中を完全にデジタル化し、
> 「月100万のコンサルが3ヶ月かけてやること」を月3万で即日提供する。
> ナレッジが溜まるほど賢くなり、コンサルを雇う理由がなくなる世界を作る。

## このドキュメントの使い方（SMB / Enterprise 別読み方ガイド）

| | SMB（10〜200名） | Enterprise（200名以上・上場等） |
|---|---|---|
| **目的** | 社長の頭の中を整理・Q&A化 | 組織知識の体系化・部門横断ナレッジ管理 |
| **優先フェーズ** | Phase 1〜2 | Phase 1〜3全て |
| **重点機能** | Issue Tree・音声入力・Q&A | ゲノムテンプレート・ネットワーク知性・可視化 |
| **スキップ可** | §4.4（ゲノムガバナンス詳細）・§4.5（ネットワーク知性） | なし |

### SMB向け — 最初に実装する機能（Phase 1）

```
① Issue Tree-Driven知識収集（§4.1）
   → 社長との対話形式でナレッジを構造化
   → まず「会社の課題仮説ツリー」から始める

② Decision Point Capture（§4.6 Engine 1）
   → 既存SaaS操作のたびにLINEで暗黙知を自動取得
   → SaaS連携と同時に設定する

③ ナレッジ半減期モデル（§4.6 Engine 2）
   → 人名依存・価格などの情報を自動更新リマインド
   → 蓄積量が増えてから有効化

④ Q&Aエンジン（§4の knowledge/ 実装）
   → 社内検索として即日使えるようにする
```

### Enterprise向け — SMB機能に加えて実装

```
⑤ 分散ナレッジ収集（§4.6 Engine 3）
   → 従業員からのナレッジ投稿フロー（CEO承認制）
   → 100名超の組織に効果大

⑥ ゲノムテンプレート × 遺伝子ガバナンス（§4.4）
   → 業種テンプレートとの差分管理
   → 複数拠点・事業部がある場合に必須

⑦ ネットワーク知性（§4.5）
   → 業界ベンチマーク参照・因果パターン活用
   → 競合優位性の客観評価に使用
   → ⚠️ 独禁法チェック要（a_02 §4.1参照）

⑧ 可視化エンジン（§4 visualization/）
   → フロー図・意思決定ツリー・充足度マップ
   → 経営報告・取締役会向け資料自動生成
```

## 1. 潰す対象の構造理解

```
■ コンサルが売っているもの（= シャチョツーが代替するもの）

  ① 現状把握        「御社の業務フローを整理しました」
  ② 課題発見        「ここがボトルネックです」
  ③ 改善提案        「こうすれば年間○○万円削減できます」
  ④ 業界知見        「同業他社ではこうしています」
  ⑤ 意思決定支援    「この投資のROIはこうなります」

■ コンサルの弱点（= シャチョツーの勝ち筋）

  ・高い（月100-300万 × 3-12ヶ月）
  ・遅い（現状把握だけで1-2ヶ月）
  ・属人的（担当コンサルが変わると品質激変）
  ・一過性（プロジェクト終了 = 知見が消える）
  ・表面的（週1訪問では暗黙知まで届かない）
  ・業界横断知見が個人の経験に依存

■ シャチョツーの圧倒的優位

  ・安い（月3万〜）→ 部長決裁で導入可能
  ・速い（テンプレートで初日から80%完成）
  ・蓄積型（使うほど賢くなる。辞めない。忘れない）
  ・深い（日常の対話・ファイルから暗黙知も取り込む）
  ・ネットワーク知性（匿名化された全顧客データから業界知見を生成）
```

## 2. ブレイン編のゴール

```
入口: 社長が業種を選ぶ（or LINEで話しかける）
 ↓
自動: テンプレートで80%の会社モデルが即座に生成される
 ↓
蓄積: 対話・ファイル・音声から残り20%を埋めていく
 ↓
出口: 社長が何を聞いても、データに基づいた回答と提案が返ってくる
      → 「もうコンサルいらないじゃん」
```

| 指標 | 目標 | コンサル比較 |
|------|------|------------|
| 現状把握の完了速度 | 初日（テンプレ）→ 2週間（80%充足） | コンサル: 1-2ヶ月 |
| Q&A応答精度 | 85%以上（Phase 2以降） | コンサル: 担当者依存 |
| 能動提案の採用率 | 30%以上 | コンサル: 提案書の実行率10-20% |
| 月額コスト | ¥30,000〜 | コンサル: ¥1,000,000〜 |
| ナレッジ蓄積の永続性 | 永続（解約後もエクスポート可能） | コンサル: プロジェクト終了で消滅 |

### 3.1 対話型引き出し — 状態遷移マシン設計

> **対話型ナレッジ引き出し（brain/ingestion/interactive.py）の動作仕様。**
> LLMが「次に何を聞くか」をどう判断し、いつ終了するかを定義する。

#### 状態遷移図

```
[START]
  ↓
[GREETING] ← セッション開始時
  ↓ テンプレート充足度チェック
[TOPIC_SELECT] ← 未充足の部署×カテゴリから最も重要なトピックを選択
  ↓ LLMが質問を生成
[ASKING] ← ユーザーに質問を提示
  ↓ ユーザー回答 or タイムアウト
  ├── 回答あり → [EXTRACTING]
  ├── 「わからない」→ [SKIP] → [TOPIC_SELECT]（次のトピックへ）
  ├── 「終了」→ [WRAP_UP]
  └── タイムアウト(5分) → [PAUSE] → 保存して中断（後日再開可）
[EXTRACTING] ← LLMで回答を構造化
  ↓ 構造化成功
[VALIDATING] ← 矛盾チェック（既存ナレッジと照合）
  ├── 矛盾なし → [CONFIRMING]
  ├── 矛盾あり → [CONFLICT_RESOLUTION]
  └── 構造化失敗 → [CLARIFYING]（追加質問で補完）
[CONFIRMING] ← 「○○という理解で正しいですか？」
  ├── 承認 → [SAVED] → [TOPIC_SELECT]（次のトピックへ）
  └── 修正 → [ASKING]（同じトピックで再質問）
[CONFLICT_RESOLUTION] ← 「既存のルールXと矛盾しています。どちらが正しいですか？」
  ├── 既存を更新 → knowledge_items.superseded_by を設定 → [SAVED]
  ├── 新規を破棄 → [TOPIC_SELECT]
  └── 両方保持（条件付き）→ conditions を追加 → [SAVED]
[WRAP_UP] ← 終了処理
  ↓ 今回のセッションサマリーを表示
  ↓ 次回推奨トピックを提示
[END]
```

#### トピック選択アルゴリズム

```
def select_next_topic(company_id: str) -> Topic:
    """未充足の部署×カテゴリから最も価値の高いトピックを選択"""

    # 1. 充足度マップを取得（部署×カテゴリの行列）
    coverage = get_knowledge_coverage(company_id)

    # 2. 未充足セルを重要度でソート
    gaps = [
        cell for cell in coverage
        if cell.coverage_rate < 0.6  # 60%未満
    ]
    gaps.sort(key=lambda g: (
        g.business_impact,     # ビジネスインパクト（コスト/リスク系が高い）
        -g.coverage_rate,      # 充足率が低いものを優先
        g.template_priority,   # テンプレートでの優先度
    ), reverse=True)

    # 3. 直近セッションで聞いたトピックは除外（連続質問を避ける）
    recent_topics = get_recent_session_topics(company_id, hours=24)
    gaps = [g for g in gaps if g.topic not in recent_topics]

    return gaps[0] if gaps else None  # None → 全トピック充足（WRAP_UP）

```

#### 質問生成プロンプト戦略

```
■ 質問の種類（LLMが動的に選択）

  1. オープン質問: 「○○部門の業務フローを教えてください」
     → 新規トピック開始時。自由記述で幅広く収集

  2. クローズド質問: 「承認フローは3段階ですか？2段階ですか？」
     → テンプレートに選択肢がある場合。回答を構造化しやすい

  3. 深掘り質問: 「先ほどの○○ルールの例外はありますか？」
     → confidence < 0.7 の場合。条件・例外を補完

  4. 矛盾確認: 「AとBが矛盾していますが、どちらが正しいですか？」
     → CONFLICT_RESOLUTION状態

■ 終了条件（OR条件で判定）

  a. 全部署×カテゴリの充足率が80%以上
  b. セッション時間が30分経過
  c. ユーザーが「終了」と発言
  d. 3回連続で「わからない」回答
  e. 20問到達（1セッションの上限）

■ セッション構造（BPOテーマ別 + 自動compacting）

  【設計変更 2026-04】1本の長い会話ではなく、ゲノムのdepartment単位でセッションを分割する。
  LLMに渡す量を最小化し、トークン爆発を防ぐ。

  knowledge_sessions テーブル（主要カラム）:
    - bpo_theme: ゲノムのdepartment名（例: "製造", "品質管理"）
    - question_count: セッション内の質問数（10問でauto_compact発動）
    - session_status: 'active' | 'compacted' | 'closed'
    - compressed_context: auto_compact後の要約（LLMに渡す用）
    - raw_context_archive: 圧縮前の生会話履歴（アーカイブ用）

  自動compacting（COMPACT_THRESHOLD = 10問）:
    question_count >= 10 になったら:
    1. 生会話履歴をLLMで要約 → compressed_context に保存
    2. 生データはraw_context_archiveにJSON退避
    3. 以降はcompressed_context + 直近3ターンのみLLMに渡す

  テーマ進捗: GET /api/v1/knowledge/theme-progress?industry_code=xxx
    → 業種の全テーマ × coverage_rate を返す（UIの進捗バー表示用）

  セッション再開: 中断分があれば「前回の続きから始めますか？」を表示
```

#### エラーハンドリング

```
■ LLM応答エラー
  → リトライ(3回) → 「申し訳ありません、一時的にエラーが発生しました。しばらくお待ちください」
  → 3回失敗 → セッション中断（PAUSE）→ 状態保存

■ ユーザー回答が構造化不可能
  → CLARIFYING状態に遷移 → 「もう少し具体的に教えてください。例えば…」
  → 2回CLARIFYINGしても不十分 → confidence=0.4で仮保存 + 後日確認キューに追加

■ 矛盾検出の偽陽性
  → ユーザーが「両方正しい」を選択可 → conditions追加で共存
  → 矛盾検出の閾値: cosine similarity > 0.85 AND 意味的反対
```

## 3.5 データマイグレーション — 既存ナレッジの一括取り込みパイプライン

> **このセクションの本質**: オンボーディングの最大ボトルネックは「既存の業務知識をシステムに入れる作業」。
> 社長が口頭で語るだけでは不十分。Excel・Word・社内Wiki・Google Drive・Dropbox・紙資料…
> 散在する既存ナレッジを一括で取り込み、構造化する仕組みがないと「Day 1 Value」は実現しない。
>
> **統一ファイル処理エンジン（c_02 §6.5.2）はファイル単体の処理。
> この設計は「大量ファイルを一括で取り込み、相互関係まで構造化する」パイプライン。**

### 3.5.1 取り込み対象マトリクス

```
■ 中小企業に実在するナレッジの所在地（ヒアリング調査ベース）

  ┌──────────────────────┬──────────────┬──────────┬──────────┐
  │ 所在地                │ 形式          │ 推定量    │ 優先度    │
  ├──────────────────────┼──────────────┼──────────┼──────────┤
  │ ファイルサーバー/NAS   │ Excel/Word/PDF│ 100-1000件│ ★★★    │
  │ Google Drive          │ Sheets/Docs  │ 50-500件 │ ★★★    │
  │ Dropbox/Box           │ 混在          │ 50-300件 │ ★★☆    │
  │ 社内Wiki (Notion等)   │ Markdown/HTML│ 30-200件 │ ★★★    │
  │ メール（Gmail/Outlook）│ テキスト+添付 │ 大量     │ ★☆☆    │
  │ 紙資料（棚・引出し）  │ スキャン画像  │ 10-100件 │ ★★☆    │
  │ 個人のPC/デスクトップ  │ 混在          │ 不明     │ ★☆☆    │
  │ SaaS内のデータ        │ API経由       │ 多量     │ ★★★    │
  │ 社長の頭の中          │ 口頭/対話     │ ∞       │ ★★★    │
  └──────────────────────┴──────────────┴──────────┴──────────┘

  MVP: ファイルサーバー + Google Drive + SaaS内データ の3つに集中
  Phase 2: Wiki + Dropbox + メール
  Phase 3: 紙資料（OCR大量処理）+ 個人PC
```

### 3.5.2 一括取り込みパイプライン

```
■ 全体フロー

  Step 1: ソース接続（5分）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    ・Google Drive → OAuth接続（1タップ）
    ・ファイルサーバー → ZIP アップロード or フォルダ指定
    ・Notion → API連携
    ・kintone/freee → 既存コネクタ流用（マスタデータ取得）
    ↓

  Step 2: スキャン＆インベントリ（自動・数分〜数十分）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    全ファイルをスキャンし、インベントリ（台帳）を生成:
    {
      "total_files": 347,
      "by_type": {"excel": 120, "word": 85, "pdf": 90, "image": 52},
      "by_folder": {"経理": 95, "営業": 80, "総務": 60, "現場": 112},
      "estimated_processing_time": "45分",
      "estimated_cost": "¥2,400（LLM + OCR）",
      "duplicates_detected": 23,
      "too_old_files": 41  // 3年以上更新なし
    }
    → 社長に確認: 「347件のファイルが見つかりました。取り込みますか？」
    → 除外フォルダ/古いファイルの選択UI
    ↓

  Step 3: 一括構造化（バッチ処理・バックグラウンド）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    各ファイルを統一ファイル処理エンジン（c_02 §6.5.2）で処理:
    ・ファイル種別判定 → 前処理 → LLM構造化 → Embedding生成

    並列処理:
    ・10ファイル同時処理（Gemini Flash のレート制限内）
    ・進捗バー: 「処理中: 120/347件（35%）残り推定25分」
    ・エラー発生時: スキップして続行。エラーファイルリストを最後に表示

    構造化の出力:
    ・各ファイルから抽出された knowledge_items（1ファイル→1-10件）
    ・confidence スコア付き（LLMの確信度）
    ・source_file 参照（元ファイルへの紐付け）
    ↓

  Step 4: 重複検出＆統合（自動）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    ・Embedding距離 < 0.15 のペアを重複候補として検出
    ・LLMで重複判定: 「完全重複」「部分重複（片方が詳しい）」「関連するが別物」
    ・完全重複 → 新しい方を採用、古い方を自動マージ
    ・部分重複 → 社長に確認: 「この2つは同じルールですか？統合しますか？」
    ↓

  Step 5: 関係グラフ自動構築（自動）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    ・取り込んだ全ナレッジ間で knowledge_relations を自動推定
    ・LLMで関係タイプ判定: depends_on / contradicts / refines / part_of / triggers / overrides / prerequisite_for / example_of（8種。定義は c_02 §5 参照）
    ・矛盾検出: contradicts が見つかった場合、社長に確認
      「経理マニュアルでは"月末締め"ですが、営業資料では"20日締め"です。どちらが正しいですか？」
    ↓

  Step 6: ギャップ分析＆確認セッション
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    ・Genome テンプレートと比較して、不足している領域を特定
    ・「値決めのルールが見つかりませんでした。口頭で教えていただけますか？」
    ・不足ナレッジのリストを「対話型引き出しセッション」のアジェンダに自動変換
    ・優先度付き: 「BPOに必要なルールが3件不足しています（最優先）」
```

## 4. 詰めるべき設計ポイントと提案

### 4.1 イシューツリー駆動ナレッジ収集 — コンサルの思考法をAIに実装

**課題**: 現状の「対話型引き出し」は「追加質問を生成する」だけ。コンサルは「イシューツリー（論点構造）」を持って仮説検証型でヒアリングする。この思考フレームワークがAIにない。

**なぜ重要か**: ランダムに質問するのと、イシューツリーに沿って仮説検証するのでは、同じ30分でも得られるナレッジの密度が5-10倍違う。McKinseyのコンサルが初回訪問で「80%把握」できるのは、ツリーがあるから。

```
■ 提案: 仮説駆動ヒアリングエンジン

  [STEP 1] イシューツリー自動生成（テンプレート × 個社情報）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    業種テンプレートから、調査すべき論点のツリーを自動生成する。
    例: 建設業（30名規模）の場合:

    経営全体
    ├── 受注プロセス [充足率: 0%]
    │   ├── 案件獲得方法（入札 or 紹介 or 直接）
    │   ├── 見積もり作成フロー（誰が・どのツールで・何日で）
    │   ├── 値決めルール（利益率基準・例外判断）★高インパクト
    │   └── 受注判断基準（GO/NO-GOの閾値）★暗黙知ゾーン
    ├── 施工管理 [充足率: 0%]
    │   ├── 工程管理方法
    │   ├── 原価管理方法 ★高インパクト
    │   ├── 安全管理
    │   └── 品質管理
    ├── 経理・財務 [充足率: 0%]
    │   ├── 請求フロー
    │   ├── 支払フロー
    │   └── 資金繰り管理
    └── 組織・人事 [充足率: 0%]
        ├── 組織構造
        ├── 評価制度
        └── 採用方法

    ※ ★マーク = ネットワーク知性のデータから
      「この論点を押さえた企業はBPO自動化率が2倍高い」と判明した高インパクトノード

  [STEP 2] 仮説生成（ネットワーク知性 × テンプレート）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    各ノードに対して「おそらくこうだろう」という仮説を生成する。
    仮説のソース:
      ・同業種の中央値パターン（ネットワーク知性）
      ・テンプレートのデフォルト値
      ・初回ヒアリングで得た断片情報

    例:
      ノード: 「値決めルール」
      仮説: 「建設業30名規模で民間工事中心の場合、
             材料費＋労務費＋経費に諸経費率25-30%を上乗せする
             原価積み上げ方式が85%（同業42社の中央値）」

  [STEP 3] 仮説検証型質問（確認 > 自由回答）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    「御社の値決めは、材料費＋労務費＋経費に諸経費率を乗せる
     原価積み上げ方式ですか？」

    → Yes: 「諸経費率は何%ですか？案件規模で変えていますか？」
            confidence: 0.9 で保存
    → No:  「どのような方式ですか？」→ 自由回答 → 再構造化
            confidence: 0.7 で保存

    ポイント:
      ・確認質問は回答が5秒。自由質問は回答が2分。
      ・社長の負担を1/10にしながら、情報密度は維持。
      ・「当たり」の仮説は即保存。「外れ」は深掘り対象。

  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    + 0.20 × ease_of_collection -- 社長が答えやすい度合い（1回の質問で埋まるか）

      「値決めルール」: 0.35×3 + 0.25×5 + 0.20×2 + 0.20×4 = 3.50 ← 最優先

  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    以下のシグナルを検出したら、自動的に深掘りモードに入る:

    シグナル1: 「大体こんな感じ」「ケースバイケース」等の曖昧表現
      → 「具体的にはどの数字を見て判断されていますか？
         例えば直近で迷ったケースを教えてください」

    シグナル2: 同じ業務に対して異なる説明が2回以上ある（矛盾）
      → 「以前は○○とおっしゃいましたが、今回は△△ですね。
         条件によって使い分けているルールがありますか？」

    シグナル3: ネットワーク知性で「この業種の80%が持っているルール」が未登録
      → 「同業他社の多くは○○というルールを持っていますが、
         御社にも似たルールはありますか？」

    シグナル4: 社長の回答が「社員はこうやっている」と第三者視点
      → 「社長自身は、もし自分でやるならどう判断しますか？
         社員のやり方と違う部分はありますか？」
      （→ 社長と現場のギャップ = 最も価値のある暗黙知）

■ LangGraph 状態遷移
```

### 4.2 ナレッジグラフ × 矛盾解消 × イベント駆動鮮度管理

**課題**: ナレッジは独立した「カード」ではない。相互に依存・矛盾・補完し合う。個別の鮮度管理では不十分で、「Aが変わったらBも古くなる」という連鎖を追跡する必要がある。

**なぜ重要か**: コンサルは「この情報が変わると、あの判断も変わりますよ」と連鎖的に指摘できる。シャチョツーがこれをできなければ、古いナレッジに基づく提案が出て信頼を失う。

```
■ 提案: ナレッジグラフ + イベント駆動リフレッシュ

  [1] ナレッジグラフの自動構築
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    ナレッジ保存時に、LLMが既存ナレッジとの関係を自動検出する。
    knowledge_relations テーブル（既存設計）を活用:

    関係タイプ:
      depends_on:   「請求フロー」は「受注フロー」に依存する
      contradicts:  「原価率20%」と「原価率25%」が矛盾する
      refines:      「値決めは原価積上げ」→「ただし1000万以上は社長判断」
      triggers:     「受注確定」→「発注処理が開始される」
      overrides:      「担当者Aの業務手順」→「担当者Aが退職すると後任の手順で上書き」

    自動検出の実装:
      新規ナレッジ保存時:
        1. Embedding で類似ナレッジ上位10件を取得
        2. LLM に「この2つのナレッジの関係は？」を判定させる
        3. 関係を knowledge_relations に保存
        4. contradicts が検出された場合 → 即座に社長に確認

  [2] イベント駆動リフレッシュ（時間ベースではなく、イベントベース）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    時間で区切る（30日で期限切れ）のは粗い。
    ビジネスイベントに連動してリフレッシュすべき。

    イベントと影響するナレッジの対応表:

    ┌─────────────────────┬──────────────────────────────────────┐
    │ ビジネスイベント       │ リフレッシュ対象ナレッジ               │
    ├─────────────────────┼──────────────────────────────────────┤
    │ 四半期末               │ 売上目標・予算・KPI関連全て            │
    │ 人事異動（入退社）      │ 担当者が紐づく業務フロー全て           │
    │ 取引先変更             │ その取引先に関連するルール全て         │
    │ 法改正                 │ 該当法令に depends_on するナレッジ全て  │
    │ SaaS契約変更           │ そのSaaSに関連するBPOルール全て        │
    │ 新規事業・撤退         │ 対象事業の業務フロー全て               │
    │ 能動提案の採用          │ 提案内容に contradicts するナレッジ     │
    └─────────────────────┴──────────────────────────────────────┘

    検出方法:
      ・LINEでの日常会話から自動検出
        「今月から田中さんが経理に異動した」
        → イベント: 人事異動
        → 影響: 経理フロー担当者ナレッジをリフレッシュ対象に
      ・SaaSコネクタのデータ変化から検出
        freeeの従業員データに変更 → 人事イベントを検出
      ・カレンダーイベントとの連動
        「決算」「四半期レビュー」→ 関連ナレッジをリフレッシュ

  [3] 矛盾解消エンジン
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    矛盾を検出した場合の解消フロー:

    1. 矛盾ペアを検出（LLM + Embedding類似度）
    2. 各ナレッジの「鮮度」「ソース信頼度」「登録者ロール」を比較
    3. 自動解消できる場合:
       ・新しい方が正しい（明らかに更新情報の場合）→ 古い方にdeprecatedフラグ
    4. 自動解消できない場合:
       ・社長に「2つの矛盾する情報があります。どちらが正しいですか？」
       ・選択結果を学習 → 次回の自動解消精度向上

    矛盾の重大度判定:
      critical: BPOルールに影響する矛盾（実行結果が変わる）→ BPO一時停止 + 即確認
      high:     能動提案に影響する矛盾 → 提案を保留 + 24h以内に確認
      medium:   Q&A応答に影響する矛盾 → 両方の情報を併記 + 確認依頼
      low:      直接的な業務影響なし → 次回対話時にさりげなく確認
```

### 4.3 コンサルティングAIフレームワーク — 分析→診断→処方を自動化

**課題**: Q&Aと能動提案は「質問に答える」「リスクを検出する」だけ。コンサルの真の価値は**「構造的に分析し、根本原因を特定し、優先順位付きの処方箋を出す」**こと。この分析フレームワークがない。

**なぜ重要か**: 「人員不足リスクがあります」だけでは社長は動かない。「人員不足の根本原因は受注の季節変動で、対策は①繁忙期前の先行採用②協力会社プールの拡大③工程の平準化の3つ。ROI順に①が年400万、③が年200万、②が年100万」まで出して初めてコンサルの代替になる。

### 4.4 遺伝子ガバナンス — バージョニング × 品質スコア × 自動進化

#### C. テンプレート適用戦略

```
■ 自動適用階層

  Layer 1: 共通ベース（全企業に自動適用）
    → サインアップ時に自動ロード
    → オプトアウト可（項目単位で「使わない」を選択可能）

  Layer 2: 業種別スターター（業種選択時に自動ロード）
    → 業種を選択すると共通ベースに上乗せ
    → 複数業種の選択可能（建設+不動産など）

  Layer 3: 企業固有カスタマイズ（共創・利用中に蓄積）
    → テンプレートの上書き・追加・削除
    → 社長の暗黙知・例外ルール

■ カスタマイズ率の追跡

  customization_rate = (変更項目数 + 追加項目数) / テンプレート総項目数
  → 業界平均と比較して「御社の固有性」を可視化
  → 高カスタマイズ率の項目 → テンプレート改善のヒント

■ テンプレート改善フィードバックループ

  [企業がテンプレートをカスタマイズ]
    ↓
  [カスタマイズ内容を匿名集約]
    ↓
  [パターン分析: 同業3社中2社が同じ修正 → テンプレート候補]
    ↓
  [人間レビュー → テンプレート更新（バージョニング）]
    ↓
  [既存企業に差分通知: 「テンプレートが更新されました。適用しますか？」]

■ Day 1体験設計

  1. サインアップ（1分）
     → 会社名、業種、従業員数、年商レンジを入力

  2. テンプレート適用（自動、10秒）
     → 共通ベース + 業種別スターターを自動ロード
     → 「御社には○○件のルール・フローがすでに入っています」

  3. 最初のQ&A体験（30秒）
     → テンプレートだけで答えられるサンプル質問を提示
     → 「経費精算のフローは？」→ テンプレートが回答
     → 「もっと御社に合った回答にするには、カスタマイズしましょう」

  4. カスタマイズ誘導
     → 最もインパクトが大きい項目トップ5を提示
     → 「この5つを御社に合わせるだけで、精度が大幅に上がります」

■ テンプレート品質担保基準

  リリース前:
    ・業界経験者2名以上のレビュー
    ・Q&Aテスト: テンプレートだけで基本質問20問中15問以上正答
    ・法規制チェック: 業界固有の法令・規制との整合性確認

  リリース後:
    ・月次でカスタマイズ率を集計
    ・カスタマイズ率50%超の項目 → テンプレートの見直し対象
    ・ユーザーFBで「的外れ」と評価された項目 → 即時修正
```

### 4.4.2 事業間遺伝子合成（グループ版） — 実名ナレッジ転用 × M&Aゲノム統合

> §4.4の遺伝子合成は「異なる企業間の匿名集約」。
> グループ版は「同一グループ内の事業間で実名ナレッジを転用」する仕組み。
> 1事業=1シャチョツーの原則を保ちつつ、グループ経営の価値を最大化する。

#### 通常の遺伝子合成との違い

```
■ 通常の遺伝子合成（§4.4）
  対象: 異なる企業のゲノム
  匿名性: 必須（個社が特定できない形で集約）
  目的: 業種テンプレートの改善
  例: 建設業A社の安全ルール + B社 + C社 → 匿名パターン → テンプレート更新

■ グループ版遺伝子合成（本セクション）
  対象: 同一グループ内の事業（シャチョツー複数台）
  匿名性: 不要（同一グループなので実名で共有可能）
  目的: グループ内のナレッジ横展開・経営品質の均質化
  例: 建設子会社の安全管理ルール → 製造子会社に「提案」として配信
```

#### ユースケース

```
■ Case 1: ベストプラクティスの横展開
  A事業（建設）の安全管理チェックリスト → B事業（製造）の安全管理に転用
  → 構造が共通（チェックリスト形式）、内容は業種で異なる
  → 共通部分を自動マッピング、業種固有部分はカスタマイズ促進

■ Case 2: グループ共通ルールの配布
  ホールディングスが「グループ共通の経費精算ルール」を策定
  → 全子会社のシャチョツーに強制適用（group_policy: enforced）
  → 各子会社は上書き不可（コンプライアンス目的）

■ Case 3: 成功パターンの提案
  A事業で営業効率化ルールが成功（Q&A精度90%+、BPO成功率95%+）
  → B事業に「A事業でこのルールが効果的でした。適用しますか？」と提案
  → B事業の責任者が受入/拒否を判断（強制ではない）
```

#### 技術設計

```sql
-- グループ管理テーブル
CREATE TABLE company_groups (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,                    -- 例: "○○ホールディングス"

-- グループメンバーシップ
CREATE TABLE group_memberships (
  group_id UUID REFERENCES company_groups(id),
  company_id UUID REFERENCES companies(id),
  role TEXT NOT NULL CHECK (role IN ('parent', 'subsidiary', 'affiliate')),
  joined_at TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (group_id, company_id)
);

-- グループ内遺伝子転送ログ
CREATE TABLE group_gene_transfers (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  group_id UUID REFERENCES company_groups(id),
  source_company_id UUID REFERENCES companies(id),
  target_company_id UUID REFERENCES companies(id),
  transfer_type TEXT NOT NULL CHECK (transfer_type IN ('proposal', 'enforced')),
  knowledge_item_ids UUID[] NOT NULL,     -- 転送対象のナレッジ
  status TEXT NOT NULL CHECK (status IN ('pending', 'accepted', 'rejected', 'applied')),
  reviewed_by UUID REFERENCES users(id),
  reviewed_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- グループ共通ポリシー
CREATE TABLE group_policies (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  group_id UUID REFERENCES company_groups(id),
  policy_type TEXT NOT NULL CHECK (policy_type IN ('enforced', 'recommended')),
  knowledge_item_ids UUID[] NOT NULL,     -- ポリシーとして配布するナレッジ
  target_companies UUID[],                -- NULL = 全メンバー
  created_by UUID REFERENCES users(id),
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

```
■ 転送フロー

  提案型（proposal）:
    1. A事業のナレッジがグループ内共有対象としてマークされる
    2. B事業のシャチョツーに「提案」として表示
    3. B事業の責任者が内容を確認
    4. 受入 → B事業のゲノムに追加（source: "group_transfer" として記録）
    5. 拒否 → 理由を記録（今後の提案精度改善に活用）

  強制型（enforced）:
    1. ホールディングスがgroup_policyを作成
    2. 対象子会社のシャチョツーに自動適用
    3. 子会社側では上書き不可（UI上でロック表示）
    4. ポリシー変更はホールディングスのみ可能

■ 権限モデル

  ホールディングス（parent）:
    ・グループ俯瞰ダッシュボードの閲覧
    ・group_policyの作成・変更
    ・事業間転送の承認
    ・グループ全体のレポート生成

  子会社（subsidiary）:
    ・自社ゲノムの完全な管理権
    ・提案型転送の受入/拒否
    ・強制型ポリシーの閲覧（変更不可）
    ・他事業へのナレッジ共有のオプトイン/アウト
```

#### M&A時のゲノム活用

```
■ 買収時（Acquisition）
  1. 買収先がシャチョツーを使っている場合:
     → グループに「追加」→ ゲノムを即座に可視化
     → ホールディングスから「デューデリジェンス的に」ゲノムを閲覧可能
     → 既存のグループ共通ポリシーを自動適用

  2. 買収先がシャチョツーを使っていない場合:
     → 新規シャチョツーを作成 → グループに追加
     → グループ共通ポリシー + 業種テンプレートで初期ゲノム構築
     → PMI（Post-Merger Integration）のナレッジ統合ツールとして活用

■ 事業売却時（Divestiture）
  → そのシャチョツーをグループから分離
  → ゲノムは買収先に譲渡（ポータビリティ保証）
  → グループ共通ポリシーは分離時に解除
  → 分離後のゲノムはグループからアクセス不可

■ デューデリジェンス活用
  → 買収候補がシャチョツーを使っている場合、ゲノムの充足度・品質が
     その会社の「業務の標準化度合い」の指標になる
  → ゲノム完成度 0.85+ = 業務がよく整理されている会社
  → ゲノム完成度 0.50以下 = 属人化が激しい（PMIコスト高）
```

### 4.4.3 テンプレート適用戦略

> 「ゼロから育てる」ではなく「70点で始めて共創で90点に磨く」を実現する適用設計。
> 共通ベース(A-G) + 業種別スターター = Day 1で7割のナレッジが入った状態を作る。

#### 1. 自動適用階層

```
■ レイヤー構造（上が優先）

  [Layer 5] グループ enforced policy（上書き不可）
  [Layer 4] 企業固有ルール（社長が直接入力）
  [Layer 3] グループ recommended policy（受入済み）
  [Layer 2] Marketplace購入テンプレート（手動適用）
  [Layer 1] 業種別スターターテンプレート（業種選択時に自動ロード）
  [Layer 0] 共通ベーステンプレート（全社に自動適用・オプトアウト可）

■ 適用タイミング
  → アカウント作成時: Layer 0（共通ベース）を自動適用
  → 業種選択時: Layer 1（業種別スターター）を自動ロード
  → グループ参加時: Layer 3/5（グループポリシー）を自動適用
  → 手動: Layer 2（Marketplace）、Layer 4（固有ルール）

■ オプトアウト
  → Layer 0（共通ベース）: カテゴリ単位でオプトアウト可能
    例: 「経理・財務」カテゴリは自社の既存ルールがあるので不要
  → Layer 1（業種別）: 項目単位でオプトアウト可能
  → Layer 5（enforced）: オプトアウト不可（コンプライアンス用途）
```

#### 2. カスタマイズ率の追跡

```
■ カスタマイズ率 = テンプレートから変更された項目数 / テンプレート総項目数

■ 追跡対象
  → カテゴリ別カスタマイズ率（例: 経営管理 30%, 人事 55%, 経理 10%）
  → 項目別カスタマイズ率（例: 経費精算フロー 80%, 給与テーブル 90%）
  → 業種別平均カスタマイズ率（例: 建設業は安全管理を70%カスタマイズ）

■ 閾値アクション
  → カスタマイズ率 > 50% のカテゴリ
    → テンプレート改善の候補としてフラグ
    → 「同業他社も同じカスタマイズをしていますか？」を月次レポートに含める

  → カスタマイズ率 > 80% の項目
    → 「このテンプレート項目は実態と大きく異なっています」警告
    → テンプレートチームに自動レポート

  → カスタマイズ率 < 5% のカテゴリ
    → テンプレートが的確 → そのまま維持
```

#### 3. フィードバックループ

```
■ テンプレート自動改善サイクル

  1. 検出: 3社以上が同じカスタマイズを実施
     → 類似度チェック（LLMセマンティック比較 > 0.85）
     → 「建設業の3社が経費精算の承認金額を30万→50万に変更」

  2. 提案: テンプレートチームに改善提案を自動生成
     → 変更内容、該当企業数、変更理由の推定
     → 「業界の実態として30万は低すぎる可能性」

  3. レビュー: テンプレートチーム or 業界専門家が判断
     → 承認 → テンプレートのデフォルト値を更新
     → 却下 → 理由を記録（将来の参考）
     → 保留 → N社が5社を超えたら再検討

  4. 配信: テンプレート更新を全利用企業に通知
     → 既にカスタマイズ済みの企業: 「更新あり。適用しますか？」
     → テンプレートのまま使用中の企業: 自動更新（通知のみ）

■ 月次テンプレート品質レポート
  → カテゴリ別カスタマイズ率サマリー
  → 新規改善提案の一覧
  → 前月に実施したテンプレート更新の影響分析
```

#### 4. Day 1体験設計

```
■ 新規登録〜初期テンプレート適用の流れ

  Step 1: アカウント作成
    → 社名・業種・従業員規模・地域を入力

  Step 2: 共通ベーステンプレート自動適用（即時）
    → 「経営管理・人事・経理・営業・法務・IT・総務の基本ルールを入れました」
    → カバレッジスコア: 0.40 表示

  Step 3: 業種別スターターテンプレート自動適用（即時）
    → 「建設業の工程管理・安全管理・原価管理等を追加しました」
    → カバレッジスコア: 0.60-0.70 表示

  Step 4: ギャップハイライト
    → 「御社固有の情報が必要な箇所をハイライトしました」
    → 優先度順に表示（高: 経営理念・組織図・主要取引先、中: 評価制度・給与テーブル等）
    → 「まずはハイライトの項目を埋めると、Q&Aの精度が大幅に上がります」

  Step 5: 初回Q&Aテスト（テンプレートだけで回答可能）
    → 「早速質問してみましょう: 『経費精算の承認フローは？』」
    → テンプレートのデフォルト値で回答（注記: テンプレートに基づく一般的な回答です）
    → 「御社の実際のフローと違う場合は、ここから修正できます」

■ ユーザーの感覚
  → Day 0: 「もう7割入ってる。すごい」
  → Day 1: 「うちに合わないところを修正したら、もう使える」
  → Week 1: 「Q&Aの精度が上がってきた」
  → Month 1: 「カバレッジ0.85。ほぼ完成」
```

#### 業種別テンプレートの運用方針

```
■ テンプレート選択
  → アカウント作成時に業種を選択（複数選択可: 建設+不動産 等）
  → 複数業種選択時は両方のスターターを適用（重複項目は自動マージ）

■ サブ業種の扱い
  → 初版は大分類15業種。サブ業種（例: 建設→土木/建築/設備）は
    カスタマイズで対応。N社蓄積後にサブ業種テンプレートを自動生成。

■ 業種追加の基準
  → 新業種テンプレートの追加条件:
    1. 当該業種の顧客が3社以上
    2. 共通ベースとのカスタマイズ率が40%以上（十分な独自性）
    3. 業界専門家のレビューを受けられる
  → Marketplaceから業界コンサルが先に出品する可能性もあり
    → 品質が高ければ公式テンプレートに昇格

■ 遺伝子合成との関係（§4.4参照）
  → 業種別テンプレートは「遺伝子の初期値」
  → N社の蓄積により遺伝子が進化 → テンプレートにフィードバック
  → 70-90%再利用可能という遺伝子合成の前提は、このテンプレート設計で実現
```

### 4.5 ネットワーク知性の全業界展開 — 階層ベンチマーク × 因果推論

**課題**: 新規業種は母数N社が少なくベンチマークが使えない。また、現状のベンチマーク設計は「数値比較」のみで、「なぜ差があるか」「どうすれば改善できるか」の因果関係まで踏み込めていない。

```
■ 提案: 3層ベンチマーク + 因果パターン共有

  [Layer 1] 同業種ベンチマーク（N≥10社）
    「御社と同じ金属加工業50社の中央値と比較して...」

  [Layer 2] 遺伝子クラスタベンチマーク（同業種N<10）
    遺伝子共有率でクラスタリングし、「似た会社群」で比較。
    例: 金属加工(7社)は単独では不足 → 「受注生産型製造業」(35社)でクラスタ
    クラスタ計算: 遺伝子の Jaccard 係数 ≥ 0.6 で同一クラスタ

  [Layer 3] 汎用ベンチマーク（規模×地域で比較）
    「従業員50名規模の中小企業全体(200社)では...」
    粗利率/離職率/SaaS月額/承認段数 = 業種に依存しない汎用指標

  [因果パターン共有]（★核心）
    単なる数値比較ではなく、「なぜ上位企業は数値が良いのか」を共有。

    例:
      データ: 粗利率 上位25%の建設業 vs 下位25%
      因果分析結果:
        上位群の共通パターン:
          ① 値決めルールがデータベース化されている（87%）← BPO Level 2以上
          ② 協力会社を5社以上から相見積もりを取っている（76%）
          ③ 原価の週次チェックを行っている（92%）← BPO Level 1で自動化可能
        下位群の共通パターン:
          ① 値決めが社長の勘（82%）
          ② 協力会社が固定2社（71%）
          ③ 原価チェックは月末のみ（88%）

      → 「御社は下位群の③に該当しています。
         原価の週次チェックをBPO Level 1で自動化すると、
         上位群と同じ管理水準になります。推定効果: 粗利率+3%」

    これにより:
      ・ベンチマーク = 単なる「スコア表」→ 「改善の処方箋」に進化
      ・因果パターンは全業界で蓄積 → 新規業種にも「似た因果パターン」で対応可能
      ・コンサルの「業界知見」を完全にデータ化

■ ベンチマーク値には必ず以下を表示:
  ・母数N社
  ・信頼区間（N<20の場合は広い → 「参考値」と明記）
  ・データの鮮度（直近何ヶ月のデータか）
  ・自社の位置（パーセンタイル表示: 上位30%等）
```

### 4.6 ナレッジ蓄積の完全性強化 — 4つの補完エンジン

**課題**: 現状のナレッジ収集は「社長との対話」が主軸。しかし社長は多忙で、重要な意思決定の瞬間に毎回立ち止まって入力する余裕はない。また、一度入力したナレッジが陳腐化しても検知できない。組織全体の暗黙知も社長フィルターでしか入ってこない。

**なぜ重要か**: 対話型収集だけではナレッジの「抜け・漏れ・古さ」が不可避。BPO実行品質はナレッジ品質に直結するため、収集漏れ・鮮度劣化はそのまま実行精度の低下に繋がる。4つの補完エンジンを組み合わせることで、対話に頼らない常時収集・自動更新ループを実現する。

```
■ 補完エンジン全体像

  [エンジン1] Decision Point Capture ── SaaS操作の「決断瞬間」を捕捉
  [エンジン2] Knowledge Half-life Model ── ナレッジの「賞味期限」を自動管理
  [エンジン3] 分散ナレッジ収集 ── 組織全体からボトムアップで収集
  [エンジン4] BPO実行フィードバックループ ── 使用実績で信頼度を自動更新
```

```
■ エンジン1: Decision Point Capture（意思決定瞬間キャプチャ）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ねらい:
    社長がSaaSを操作する「決断の瞬間」にリアルタイムでフックし、
    最小コスト（タップ1回）でナレッジを取得する。

  トリガーの例:
    ・kintoneで案件ステータスを「商談中」→「失注」に変更した瞬間
    ・freeeで支払いを「承認」→「保留」に変更した瞬間
    ・Chatworkで特定キーワードを含むメッセージを送信した瞬間

  捕捉フロー:
    SaaS操作イベント
      ↓ Webhook / SaaS API フック
    シャチョツー検知
      ↓
    LINE / Slack にプッシュ通知（3秒以内）
      「案件『○○工事』を失注にしましたね。理由を教えてください。」
      [価格]  [競合に負けた]  [ニーズ違い]  [タイミング]  [その他]
      ↓ タップ1回で選択（テキスト入力不要）
    decision_captures テーブルに保存
      ↓
    9次元モデルの「意思決定基準」ノードを自動更新
      ↓
    類似パターンが3件蓄積 → ルールを自動生成
      「失注理由が「価格」の場合: 見積もり上限 ×× 万円を超えると失注率67%」

  なぜタップ1回か:
    操作直後5分以内に回答率が急落する。
    テキスト入力を求めると回答率は10%以下。
    選択肢形式なら回答率は80%超（類似プロダクト実績）。

  DB:
    decision_captures(
      id               UUID PRIMARY KEY,
      company_id       UUID NOT NULL REFERENCES companies(id),
      triggered_by_saas VARCHAR(50) NOT NULL, -- 'kintone' | 'freee' | 'chatwork' etc.
      action_type      VARCHAR(100) NOT NULL,  -- 'deal_lost' | 'payment_held' etc.
      selected_reason  VARCHAR(200),           -- タップで選択した理由
      raw_context      JSONB,                  -- SaaSから取得した操作コンテキスト
      captured_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
    );

  LangGraph状態:
    class DecisionCaptureState(TypedDict):
        company_id: str
        saas_event: dict          # Webhookペイロード
        action_type: str
        reason_options: list[str] # 提示する選択肢
        selected_reason: str | None
        generated_rule: dict | None  # 3件蓄積時に自動生成
```

```
■ エンジン2: Knowledge Half-life Model（ナレッジ半減期モデル）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ねらい:
    ナレッジは時間とともに陳腐化する。タイプ別に半減期を設定し、
    期限到来前に自動で「要確認」フラグを立て、BPO実行の精度劣化を防ぐ。

  ナレッジタイプ別 半減期設定:
    ┌─────────────────────┬──────────┬───────────────────────────────┐
    │ ナレッジタイプ       │ 半減期   │ 理由                          │
    ├─────────────────────┼──────────┼───────────────────────────────┤
    │ 人名依存ルール       │  6ヶ月   │ 担当者交代・退職リスクが高い   │
    │ 価格・単価情報       │  3ヶ月   │ 原材料費・市況変動が激しい     │
    │ 法制度・規制         │ 12ヶ月   │ 年度改正が主                   │
    │ 業務手順・フロー     │ 24ヶ月   │ 組織変更のペースに合わせる     │
    │ 経営判断基準         │ 18ヶ月   │ 戦略転換のサイクルに合わせる   │
    └─────────────────────┴──────────┴───────────────────────────────┘

  動作フロー:
    half_life_days 経過の 14 日前
      ↓
    BPO実行ルーティングで参照されるナレッジ → 「要確認」フラグ + LINE通知
      「このルールの確認をお願いします: 『値決め: 原価×1.3が下限』
       今も有効ですか？  [はい、まだ有効]  [変わった、更新する]」
      ↓
    確認: expires_at を半減期分延長
    更新: 新内容でナレッジ上書き + 旧バージョンをアーカイブ
      ↓
    確認されないまま期限超過 → ステータス「仮眠(dormant)」へ
      ・BPO実行では参照しない（ノイズ削減）
      ・UIの検索結果にも表示しない

  BPO実行ログとのズレ自動検知:
    BPO実行時に参照したナレッジが「3回連続で修正承認」された場合
      → confidence を 0.15 下げ
      → 「このルールはよく修正されています。現在の正しいルールを教えてください」
         と能動的に確認を促す

  DB:
    -- knowledge_items テーブルへの追加カラム:
    half_life_days  INTEGER,        -- ナレッジタイプ別デフォルト値を初期設定
    expires_at      TIMESTAMPTZ,    -- half_life_days から算出
    dormant_at      TIMESTAMPTZ,    -- 期限超過時に記録

    knowledge_decay_log(
      id                  UUID PRIMARY KEY,
      knowledge_item_id   UUID NOT NULL REFERENCES knowledge_items(id),
      decay_triggered_by  VARCHAR(50), -- 'scheduled' | 'bpo_correction' | 'manual'
      old_confidence      FLOAT NOT NULL,
```

### 8.1 ゲノムデータ収集パイプライン

#### 匿名化アルゴリズム（k-匿名化）

新規ナレッジをゲノムへ集約する際、**k=5** を最小単位として以下の匿名化を行う。
5社未満しか持っていないパターンは集約せず、個社識別リスクを排除する。

```python
from dataclasses import dataclass
from typing import Optional
import re

@dataclass
class KnowledgeItem:
    company_id: str
    content: str
    category: str
    industry_code: str
    company_size: str  # "small"(10-50名) / "medium"(50-200名) / "large"(200名+)

@dataclass
class AnonymizedPattern:
    industry_code: str
    company_size: str
    pattern_text: str
    supporting_company_count: int  # 何社のデータが元になっているか
    confidence: float

class KAnonymizer:
    """
    k-匿名化によるゲノムデータ保護。
    k=5 未満のパターンは集約しない（個社識別リスク排除）。
    """

    def __init__(self, k: int = 5):
        self.k = k
        self._pii_patterns = self._build_pii_patterns()

    def anonymize_knowledge(
        self, item: KnowledgeItem, k: int = 5
    ) -> Optional[AnonymizedPattern]:
        """
        1. PII除去
        2. 一般化（金額・固有名詞 → レンジ・プレースホルダー）
        3. k-匿名性チェック（同業種×同規模で k社以上が持つパターンか検証）
        """
        # Step 1: PII除去
        cleaned = self._remove_pii(item.content)

        # Step 2: 一般化
        generalized = self._generalize(cleaned, item.industry_code)

        # Step 3: k-匿名性チェック（DBから同パターン件数を確認）
        supporting_count = self._count_similar_patterns(
            generalized, item.industry_code, item.company_size
        )
        if supporting_count < k:
            # k未満 → 集約不可。個社の貢献データとして内部保持のみ
            return None

        return AnonymizedPattern(
            industry_code=item.industry_code,
            company_size=item.company_size,
            pattern_text=generalized,
            supporting_company_count=supporting_count,
            confidence=min(1.0, supporting_count / 50),  # 50社で confidence=1.0
        )

    def _remove_pii(self, text: str) -> str:
        """PII（個人情報・特定可能情報）を除去する"""
        result = text
        for pattern, replacement in self._pii_patterns:
            result = re.sub(pattern, replacement, result)
        return result

    def _generalize(self, text: str, industry_code: str) -> str:
        """
        一般化ルール:
          金額: ¥1,234,567 → [¥100万〜200万]
          社名: 株式会社〇〇 → [同業者]
          個人名: 田中部長 → [担当者]
          住所: 東京都渋谷区〇〇 → [都市部]
        """
        result = text
        # 金額レンジ化
        result = self._generalize_amounts(result)
        # 社名 → [同業者] or [取引先]
        result = re.sub(r'株式会社\S+|有限会社\S+|\S+株式会社', '[取引先]', result)
        # 個人名（〇〇さん / 〇〇部長 / 〇〇氏）
        result = re.sub(r'[\u4e00-\u9fff]{1,4}(さん|部長|課長|社長|氏|くん)', '[担当者]', result)
        # 住所の丁目以降
        result = re.sub(r'(東京都|大阪府|愛知県|[^\s]{2,4}[都道府県])[^\s]*', r'\1[市区町村]', result)
        return result

    def _generalize_amounts(self, text: str) -> str:
        """金額を100万単位のレンジに一般化する"""
        def to_range(m: re.Match) -> str:
            raw = m.group(0).replace('¥', '').replace(',', '').replace('円', '')
            try:
                val = int(raw)
            except ValueError:
                return m.group(0)
            unit = 1_000_000
            lo = (val // unit) * unit
            hi = lo + unit
            return f'[¥{lo // unit}M〜{hi // unit}M]'

        return re.sub(r'[¥￥]?[\d,]{4,}円?', to_range, text)

    def _count_similar_patterns(
        self, pattern: str, industry_code: str, company_size: str
    ) -> int:
        """
        Supabase pgvector で類似パターンの件数をカウント。
        実際の実装では embedding 類似度 >= 0.85 のパターン数を返す。
        （この関数はDBアクセス層に委譲）
        """
        raise NotImplementedError("DBレイヤーに委譲")

    def _build_pii_patterns(self) -> list[tuple[str, str]]:
        return [
            (r'\d{3}-\d{4}-\d{4}', '[電話番号]'),            # 電話番号
            (r'\d{3}-\d{4}', '[郵便番号]'),                   # 郵便番号
            (r'[a-zA-Z0-9._%+-]+@[a-zA-Z0-9.-]+\.[a-zA-Z]+', '[メールアドレス]'),
            (r'\d{4}年\d{1,2}月\d{1,2}日', '[日付]'),         # 特定日付
            (r'[A-Z]{2}\d{6}', '[証明書番号]'),               # 資格証番号等
        ]
```

#### PII除去チェックリスト

| カテゴリ | 除去対象 | 置換形式 |
|---|---|---|
| 人名 | 個人名（漢字・カタカナ） | `[担当者]` |
| 法人名 | 株式会社〜、有限会社〜 | `[取引先]` または `[同業者]` |
| 金額（個別） | 具体的な金額 ¥1,234,567 | `[¥1M〜2M]` などのレンジ |
| 住所 | 市区町村以降の番地 | `[都市部]` / `[地方]` |
| 連絡先 | 電話・メール | `[電話番号]` / `[メールアドレス]` |
| 日付（特定） | 特定の契約日・事故日等 | `[日付]` |
| 口座情報 | 銀行名・口座番号 | 除去のみ |

#### ゲノム集約スケジュール（週次バッチ）

```python
from datetime import datetime, timedelta
from dataclasses import dataclass, field
from typing import Literal

IndustryCategoryCode = Literal[
    "construction", "manufacturing", "dental",
    "retail", "logistics", "food_service", "other"
]
CompanySize = Literal["small", "medium", "large"]

@dataclass
class GenomeAggregationJob:
    """
    週次バッチ: 新規ナレッジをゲノムテンプレートへ集約する。
    - 業種 × 規模 × 地域（都市部/地方）でセグメント分類
    - k=5 の k-匿名化フィルターを通過したパターンのみ集約
    - 既存テンプレートとのマージ or 新パターン追加を自動判定
    """

    k_threshold: int = 5          # k-匿名化最小社数
    run_at: datetime = field(default_factory=lambda: datetime.now())

    async def run(self) -> "AggregationReport":
        # Step 1: 前回バッチ以降の新規ナレッジを取得
        new_items = await self._fetch_new_knowledge_items(
            since=self.run_at - timedelta(days=7)
        )

        # Step 2: オプトイン企業のナレッジのみに絞り込む
        opted_in_items = [i for i in new_items if await self._is_opted_in(i.company_id)]

        # Step 3: 業種 × 規模 × 地域でセグメント分類
        segments = self._classify_by_segment(opted_in_items)

        results = []
        for segment_key, items in segments.items():
            anonymizer = KAnonymizer(k=self.k_threshold)

            # Step 4: 各アイテムを匿名化（k未満は除外）
            patterns = [
                p for item in items
                if (p := anonymizer.anonymize_knowledge(item)) is not None
            ]

            if not patterns:
                continue

            # Step 5: 既存テンプレートとマージ or 新パターン追加
            existing_template = await self._fetch_template(segment_key)
            if existing_template:
                updated = await self._merge_into_template(existing_template, patterns)
                results.append(("updated", segment_key, updated))
            else:
                if len(patterns) >= self.k_threshold:
                    new_template = await self._create_template(segment_key, patterns)
                    results.append(("created", segment_key, new_template))

        return AggregationReport(
            run_at=self.run_at,
            processed_items=len(opted_in_items),
            results=results,
        )

    def _classify_by_segment(self, items: list) -> dict:
        """
        セグメントキー = industry_code + "_" + company_size + "_" + region_type
        例: "construction_small_urban", "manufacturing_medium_rural"
        """
        segments: dict = {}
        for item in items:
            region = "urban" if item.prefecture in URBAN_PREFECTURES else "rural"
            key = f"{item.industry_code}_{item.company_size}_{region}"
            segments.setdefault(key, []).append(item)
        return segments

    async def _fetch_new_knowledge_items(self, since: datetime) -> list:
        raise NotImplementedError("DBレイヤーに委譲")

    async def _is_opted_in(self, company_id: str) -> bool:
        raise NotImplementedError("DBレイヤーに委譲")

    async def _fetch_template(self, segment_key: str):
        raise NotImplementedError("DBレイヤーに委譲")

    async def _merge_into_template(self, template, patterns: list):
        raise NotImplementedError("DBレイヤーに委譲")

    async def _create_template(self, segment_key: str, patterns: list):
        raise NotImplementedError("DBレイヤーに委譲")


URBAN_PREFECTURES = {"東京都", "神奈川県", "大阪府", "愛知県", "福岡県", "北海道", "宮城県"}

@dataclass
class AggregationReport:
    run_at: datetime
    processed_items: int
    results: list
```

#### 品質評価基準と自動リリース判定

```python
from dataclasses import dataclass

@dataclass
class TemplateQualityEvaluator:
    """
    ゲノムテンプレートの品質を自動評価し、
    現行版を上回った場合のみ新バージョンをリリースする。
    """

    def compute_quality_score(
        self,
        contributing_company_count: int,
        avg_coverage_score: float,     # このテンプレートを使った企業の平均充足率
        avg_qa_satisfaction: float,    # Q&A応答の平均評価（0〜1）
        avg_customization_rate: float, # カスタマイズ率（低いほど汎用的）
        bpo_success_rate: float,       # BPO実行の成功率
    ) -> float:
        # 企業数ボーナス: 50社以上で安定、200社で最大
        volume_bonus = min(1.0, contributing_company_count / 200)

        score = (
            0.25 * avg_coverage_score
            + 0.20 * avg_qa_satisfaction
            + 0.20 * (1.0 - avg_customization_rate)  # カスタマイズ不要 = 汎用性高
            + 0.20 * bpo_success_rate
            + 0.15 * volume_bonus
        )
        return round(min(1.0, score), 4)

    def should_release(
        self,
        candidate_score: float,
        current_score: float,
        min_improvement: float = 0.02,  # 2%以上改善しなければリリースしない
    ) -> bool:
        return candidate_score >= current_score + min_improvement
```

### 8.5 ネットワーク効果の成長モデル

#### Metcalfe則の適用

ゲノムの価値はネットワーク参加企業数 n に対して **n²** に比例する。
ただし、ゲノムの実用的な精度はS字曲線に従い、**業種あたり50社** で「プロ並み」に達するという試算を持つ。

```
ゲノム価値のS字曲線（建設業の例）

  精度
  100%│                                    ╭─────── Gold (200社)
      │                              ╭────╯
   80%│                        ╭────╯ ← Silver (50社) = プロ並み
      │                  ╭────╯
   60%│            ╭────╯ ← Bronze (10社)
      │      ╭────╯
   40%│  ╭──╯ ← Seed
   20%│ ╯
    0%└──────────────────────────────────────────────
       0   10  20  30  50      100        200    参加社数

  ポイント:
    ・10社前後 → Bronze到達: 基本パターンをカバー
    ・50社到達 → Silver: 例外ケースまでカバー。コンサル品質に並ぶ
    ・200社到達 → Gold: 統計的に安定。因果推論が信頼できるレベル
    ・それ以降: 精度の改善は緩やか（既に十分高い）
```

| 業種 | 目標社数 | 到達目標月 | 達成時のゲノム品質 | 戦略 |

| 製造業 | 30社 | Month 18 | Bronze〜Silver | 業種分散が広いためセグメント絞り込みが必要 |

```sql
CREATE VIEW genome_network_health AS

    gtv.industry_code,
    gtv.quality_tier,
    gtv.contributing_company_count,

            WHEN 'dental'        THEN 50

            WHEN 'retail'        THEN 100
            ELSE 50

LEFT JOIN genome_consents gc
    ON gc.opted_in_at IS NOT NULL

GROUP BY gtv.id, gtv.industry_code, gtv.quality_tier,
         gtv.contributing_company_count, gtv.quality_score, gtv.released_at
```

### 8.1 Half-life Model 数式定義

> ナレッジには「鮮度」がある。人名依存の情報は数日で古くなり、
> 企業文化に関する情報は年単位で有効。この減衰を定量化する。

#### 基本減衰関数

```python
import math
from datetime import datetime, timedelta

def knowledge_confidence(
    confidence_0: float,   # 初期信頼度（0.0〜1.0）
    lambda_decay: float,   # 減衰定数（大きいほど速く減衰）
    days_elapsed: float,   # 最終更新からの経過日数
) -> float:
    """
    指数減衰モデルによるナレッジ信頼度計算。
    半減期 = ln(2) / λ ≈ 0.693 / λ
    """
    return confidence_0 * math.exp(-lambda_decay * days_elapsed)
```

#### カテゴリ別の減衰定数（λ）

| カテゴリ | λ（/日） | 半減期 | 例 |
|---|---|---|---|
| **人名依存** | 0.100 | ≈ 7日 | 「担当は田中さん」→ 人事異動で即無効 |
| **価格・相場** | 0.020 | ≈ 35日 | 「鉄骨単価は○円/kg」→ 月次で変動 |
| **業務プロセス** | 0.005 | ≈ 139日 | 「請求フローは5ステップ」→ 半年〜1年で変化 |
| **法規制** | 0.003 | ≈ 231日 | 「建設業法の基準は…」→ 法改正サイクル |
| **企業文化・価値観** | 0.001 | ≈ 693日 | 「うちは品質第一」→ 経営者が変わるまで安定 |
| **業界構造** | 0.0005 | ≈ 1,386日 | 「建設業の重層下請け構造」→ 産業構造は数年単位 |

#### リフレッシュ（再確認）時の信頼度回復

```python
def refresh_confidence(
    current_confidence: float,
    refresh_type: str,  # "confirmed" | "updated" | "contradicted"
) -> float:
    """
    ナレッジが再確認・更新された時の信頼度回復ロジック。
    """
    if refresh_type == "confirmed":
        # 既存内容が正しいと確認 → 信頼度を初期値近くに回復
        return min(1.0, current_confidence + (1.0 - current_confidence) * 0.8)
    elif refresh_type == "updated":
        # 内容が更新された → 新しい内容の信頼度は0.9から開始
        return 0.9
    elif refresh_type == "contradicted":
        # 矛盾する情報が入った → 信頼度を大幅引き下げ
        return max(0.1, current_confidence * 0.3)
```

#### 複合減衰（複数カテゴリにまたがる情報）

```python
def composite_decay(categories: list[dict]) -> float:
    """
    複数カテゴリに属するナレッジの減衰計算。
    最も速い減衰率が支配的になる（最弱リンクモデル）。

    例: 「田中部長が承認する購買フロー」
      → 人名依存(λ=0.1) × 業務プロセス(λ=0.005)
      → 支配的な減衰 = max(λ) = 0.1（人名が変わると無効化）
    """
    max_lambda = max(cat["lambda"] for cat in categories)
    weighted_confidence = min(
        knowledge_confidence(cat["confidence_0"], cat["lambda"], cat["days"])
        for cat in categories
    )
    return weighted_confidence
```

#### 充足度スコア計算

```python
def calculate_coverage(genome: CompanyGenome) -> float:
    """
    9次元の充足度を重み付き平均で算出。
    必須次元（戦略・プロセス・組織）の重みが大きい。
    """
    weights = {
        "dim1_strategy": 0.15,      # 経営戦略
        "dim2_processes": 0.20,     # 業務プロセス（最重要）
        "dim3_organization": 0.15,  # 組織構造
        "dim4_finance": 0.10,       # 財務
        "dim5_customers": 0.10,     # 顧客
        "dim6_technology": 0.05,    # 技術基盤
        "dim7_culture": 0.05,       # 文化
        "dim8_risk": 0.10,          # リスク
        "dim9_industry": 0.10,      # 業界特性
    }
    total = sum(
        getattr(genome, dim).score * weight
        for dim, weight in weights.items()
    )
    return round(total, 2)
```

### 8.3 セマンティックキャッシュ実装方針

> Q&Aの70%は類似質問の繰り返し。LLM呼び出しをキャッシュで削減する。

#### アーキテクチャ

```
[ユーザー質問]
  ↓
[クエリembedding生成] (Voyage AI)
  ↓
[キャッシュ検索] pgvector cosine similarity
  ↓
similarity > 0.95 → キャッシュヒット → 即座に回答返却
similarity ≤ 0.95 → キャッシュミス → LLM呼び出し → 回答生成 → キャッシュ保存
```

#### DBスキーマ

```sql
CREATE TABLE semantic_cache (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  company_id      UUID NOT NULL REFERENCES companies(id),
  query_text      TEXT NOT NULL,              -- 元の質問テキスト
  query_embedding VECTOR(1024) NOT NULL,      -- Voyage AI embedding
  response_text   TEXT NOT NULL,              -- キャッシュされた回答
  response_metadata JSONB,                    -- confidence, sources等
  hit_count       INTEGER DEFAULT 0,          -- ヒット回数
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  last_hit_at     TIMESTAMPTZ,
  invalidated_at  TIMESTAMPTZ,               -- 無効化日時（NULLなら有効）

  CONSTRAINT fk_company FOREIGN KEY (company_id) REFERENCES companies(id)
);

-- HNSW インデックス（高速近傍検索）
CREATE INDEX idx_semantic_cache_embedding
  ON semantic_cache USING hnsw (query_embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 64);

-- RLS
ALTER TABLE semantic_cache ENABLE ROW LEVEL SECURITY;
CREATE POLICY company_isolation ON semantic_cache
  USING (company_id = current_setting('app.company_id')::UUID);
```

#### 無効化（Invalidation）戦略

```
■ イベント駆動invalidation
  ・ナレッジアイテムの追加/更新/削除時:
    → そのナレッジに関連するキャッシュエントリを無効化
    → 関連判定: ナレッジembeddingとキャッシュquery embeddingの類似度 > 0.7

■ TTL（Time-To-Live）
  ・最大TTL: 7日（ナレッジ更新がなくても定期的にリフレッシュ）
  ・ヒットが多いエントリ: TTLを14日に延長

■ 目標メトリクス
  ・キャッシュヒット率: 70%以上（LLMコスト30%削減に相当）
  ・平均レイテンシ: ヒット時 < 100ms、ミス時 < 3秒
```

### 8.4 Knowledge Graph エッジスキーマ

> ナレッジアイテム間の関係（依存・矛盾・精緻化等）をグラフで管理する。

#### DBスキーマ

```sql
CREATE TABLE knowledge_relations (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  company_id    UUID NOT NULL REFERENCES companies(id),
  source_id     UUID NOT NULL REFERENCES knowledge_items(id) ON DELETE CASCADE,
  target_id     UUID NOT NULL REFERENCES knowledge_items(id) ON DELETE CASCADE,
  relation_type VARCHAR(20) NOT NULL CHECK (relation_type IN (
    'depends_on',        -- source は target に依存（target変更時にsourceも要確認）
    'contradicts',       -- source と target は矛盾（どちらかが古い可能性）
    'refines',           -- source は target をより詳細に定義
    'part_of',           -- source は target の一部
    'triggers',          -- source の条件成立時に target が実行される
    'overrides',         -- source は target を上書き（新ルールが旧ルールを置換）
    'prerequisite_for',  -- source は target の前提条件
    'example_of'         -- source は target の具体例
  )),
  -- ※ 8種の定義は c_02_プロダクト設計.md §5 が Source of Truth
  confidence    FLOAT DEFAULT 1.0 CHECK (confidence BETWEEN 0.0 AND 1.0),
  detected_by   VARCHAR(20) DEFAULT 'llm' CHECK (detected_by IN ('llm', 'user', 'system')),
  metadata      JSONB DEFAULT '{}',           -- 関係の補足情報
  created_at    TIMESTAMPTZ DEFAULT NOW(),

  CONSTRAINT no_self_relation CHECK (source_id != target_id),
  CONSTRAINT unique_relation UNIQUE (company_id, source_id, target_id, relation_type)
);

-- RLS
ALTER TABLE knowledge_relations ENABLE ROW LEVEL SECURITY;
CREATE POLICY company_isolation ON knowledge_relations
  USING (company_id = current_setting('app.company_id')::UUID);

-- インデックス
CREATE INDEX idx_kr_source ON knowledge_relations(source_id);
CREATE INDEX idx_kr_target ON knowledge_relations(target_id);
CREATE INDEX idx_kr_type ON knowledge_relations(relation_type);
```

#### グラフ走査クエリ例

```sql
-- あるナレッジに依存する全ナレッジを再帰的に取得（影響範囲分析）
WITH RECURSIVE affected AS (
  SELECT target_id AS knowledge_id, 1 AS depth
  FROM knowledge_relations
  WHERE source_id = :changed_knowledge_id
    AND relation_type = 'depends_on'
    AND company_id = :company_id

  UNION ALL

  SELECT kr.target_id, a.depth + 1
  FROM knowledge_relations kr
  JOIN affected a ON kr.source_id = a.knowledge_id
  WHERE kr.relation_type = 'depends_on'
    AND kr.company_id = :company_id
    AND a.depth < 5  -- 最大深度制限
)
SELECT DISTINCT knowledge_id, depth
FROM affected
ORDER BY depth;

-- 矛盾検出: contradicts関係にあるナレッジペアを表示
SELECT
  ki1.content AS knowledge_a,
  ki2.content AS knowledge_b,
  kr.confidence AS contradiction_confidence,
  kr.created_at
FROM knowledge_relations kr
JOIN knowledge_items ki1 ON kr.source_id = ki1.id
JOIN knowledge_items ki2 ON kr.target_id = ki2.id
WHERE kr.relation_type = 'contradicts'
  AND kr.company_id = :company_id
  AND kr.confidence > 0.7
ORDER BY kr.confidence DESC;
```

#### 自動関係検出パイプライン

```
[新規ナレッジ追加]
  ↓
[embedding生成] → pgvector で類似ナレッジ Top-10 取得
  ↓
[LLM関係推定] → 各ペアについて関係タイプを判定
  プロンプト: 「以下の2つの情報の関係を判定してください:
    A: {source_content}
    B: {target_content}
    関係: depends_on / contradicts / refines / part_of / triggers / none」
  ↓
[confidence ≥ 0.7 のみ保存] → knowledge_relations に INSERT
  ↓
[contradicts検出時] → proactive_proposals に「矛盾アラート」を生成
```
