# 003: 自社システム編 — SaaSを潰す

> **このフェーズの本質**: BPO編で「SaaSを繋いで自動化」していた世界から、
> 「そもそもSaaS要らなくない？」という世界に進化させる。
> 月5万のkintone + 月3万のfreee + 月2万のジョブカン = 月10万を、
> シャチョツーの中に全部統合してゼロにする。
> SIerに月50万払う世界も終わらせる。

## このドキュメントの使い方（SMB / Enterprise 別読み方ガイド）

| | SMB（10〜200名） | Enterprise（200名以上・上場等） |
|---|---|---|
| **主目標** | 月額SaaS費用を削減する | 基幹システム・ERPをAIで内製化 |
| **移行アプローチ** | ブラウザ拡張で透明移行（§4.5） | レガシーシステム統合 + CAB承認（§6.5） |
| **優先フェーズ** | Phase 6（ROI高い業務から） | Phase 6〜7全て |
| **スキップ可** | §4.1 Monte Carlo ROI（概算でも可）・§6.5 Enterprise部分 | なし |

### SMB向け — SaaS解約の最短ルート

```
Step 1: SaaS解約ロードマップ自動生成（§4.7）
  → 現在の月額SaaS費用を入力
  → 依存関係グラフから「解約できる順番」を自動算出
  → 例: Slack→LINE WORKS移行（月3万円削減）から開始

Step 2: ブラウザ拡張で既存SaaSをUI複製（§4.5）
  → Phase A: 既存SaaSと並行して動作
  → Phase B: 拡張機能で業務UXを再構築（フロー統合・自動化・最適化）
  → Phase C: 完全カットオーバー（SaaS解約）

Step 3: 自動保守パイプライン（§4.6）
  → ナレッジ変更時にシステムを自動パッチ
  → 「社長が方針変更 → 翌日にシステム反映」

SMBで最初に解約すべきSaaS候補:
  1. 汎用タスク管理（Asana/Notion → 自社ダッシュボードへ）
  2. 勤怠管理（汎用SaaS → 自社特化システムへ）
  3. 社内申請フォーム（Googleフォーム等 → 承認フロー内製化）
```

### Enterprise向け — SMB機能に加えて実装（§6.5参照）

```
◎ レガシーシステム統合（§6.5 パターン1〜4）
  → API連携 / DBダイレクト / ファイル連携 / iPaaS
  → SAP・Salesforce等の基幹システムとの接続

◎ CAB承認フロー（Change Advisory Board）
  → 生成コードのセキュリティレビュー → リリース承認
  → IT部門・情報セキュリティ部門を承認プロセスに組み込み

◎ 展開オプション A〜D（§6.5）
  → Option A: 共有クラウド（SMBデフォルト）
  → Option B: 専用インスタンス（中規模Enterprise）
  → Option C: 顧客GCPプロジェクト内展開
  → Option D: オンプレミス（金融・医療）

◎ Composable System Architecture（§4.2）
  → 原子コンポーネント（データ/フロー/計算/AI型）の組み合わせ
  → 業務変化に対してコンポーネント単位で差し替え可能
  → 大規模システムの保守コスト大幅削減

SLA:
  SMB: Standard 99.9%（月8.7時間停止許容）
  Enterprise: Premium 99.95%（月4.4時間停止許容）
```

## 1. 潰す対象の構造理解

```
■ SaaSが売っているもの

  ① 業務機能         kintone（DB/ワークフロー）, freee（会計）, ジョブカン（勤怠）
  ② データの保管場所  「御社のデータはクラウドに安全に保管」
  ③ 業務プロセスの型  「ベストプラクティスに沿ったワークフロー」
  ④ SaaS間連携       「freeeとSlackを連携できます」

■ SIerが売っているもの

  ① カスタム開発      「御社専用のシステムを作ります」（500万-3,000万）
  ② 保守運用          「月50万でサーバー管理します」
  ③ ヘルプデスク      「困ったら電話ください」
  ④ ベンダーロックイン 「このシステムは弊社しか触れません」

■ これらの弱点（= シャチョツーの勝ち筋）

  SaaS:
    ・高い積み上げ（kintone + freee + Slack + ... = 月10-30万）
    ・データが分散（各SaaSに情報がバラバラ）
    ・連携が面倒（iPaaS設定は素人には無理）
    ・「うちのやり方」に合わない（SaaSの型に業務を合わせている）
    ・SaaS疲れ（増えるほど管理コスト増大）

  SIer:
    ・高い（初期500万〜 + 保守月50万〜）
    ・遅い（要件定義3ヶ月→開発6ヶ月→テスト2ヶ月 = 11ヶ月）
    ・変更できない（仕様変更 = 追加見積もり）
    ・属人的（担当SE退職 = ブラックボックス化）
    ・ベンダーロックイン（他社に乗り換えられない）

■ シャチョツーの圧倒的優位

  ・SaaS統合（バラバラのSaaSを1つのBrainに統合済み）
  ・データ一元化（BPO編で全データがBrainに流れ込んでいる）
  ・「うちのやり方」が標準（ナレッジ = 業務ロジックそのもの）
  ・変更が即時（ナレッジを変更 → システムが自動適応）
  ・要件定義が不要（BPO編の実行データ = 要件定義の素材）
  ・開発速度が圧倒的（Engineer Worker + develop_agent）
```

```
入口: BPO編の実行データ（どの業務がどう自動化されたか）
 ↓
```

## 4. 詰めるべき設計ポイントと提案

### 4.1 BPO実行ログ → システム化候補自動検出 — 結晶化ポイント分析

**課題**: Engineer Workerは「社長が相談してきた時」に動く受動的な設計。BPOデータを分析して「ここ、システム作った方がいいですよ」と能動的に提案する仕組みがない。単純なパターンマッチング（高頻度タスク検出等）では「点」でしか捉えられず、「複数のBPOタスクが常にセットで実行され、データを共有し、1つのシステムに結晶化すべきクラスタ」を見逃す。

**なぜ重要か**: BPOは「既存SaaSを使って自動化する」世界。しかし本当のコスト削減は「SaaS自体を不要にする自社システム」から生まれる。BPO実行ログには「どのタスク群が常にセットで動くか」「どのSaaS間でデータが行き来するか」のトポロジーが隠れている。このトポロジーを分析すれば、**「この5つのBPOタスクは1つのシステムに統合すべき」**という結晶化ポイントが自動的に浮かび上がる。

```
■ 提案: BPO実行トポロジー分析 + モンテカルロROI試算 + 成熟度モデル

  [STEP 1] BPO実行トポロジーの構築（タスク共起グラフ）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    BPO実行ログから「タスク共起グラフ」を自動構築する。
    ノード = BPOタスク、エッジ = 共起関係（同一ワークフロー内 or 24h以内に連続実行）

    エッジの重み計算:
      edge_weight(task_A, task_B) =
        0.40 × co_execution_freq      -- 同時実行頻度（月間）
      + 0.30 × data_sharing_score     -- 共有データフィールド数 / 全フィールド数
      + 0.20 × temporal_proximity     -- 平均実行間隔の近さ（1h以内=1.0, 24h=0.3）
      + 0.10 × error_correlation      -- 片方のエラーが他方に影響する頻度

    例: ある建設会社の実行ログ（直近3ヶ月）
      ┌──────────────────────────────────────────────────────────┐
      │  タスク共起グラフ（エッジの太さ = 重み）                     │
      │                                                            │
      │  [kintone受注取得] ━━━━ 0.92 ━━━━ [freee請求書作成]        │
      │        │                                │                   │
      │       0.85                             0.78                  │
      │        │                                │                   │
      │  [原価データ集計] ━━━━ 0.88 ━━━━ [利益率計算]              │
      │        │                                │                   │
      │       0.45                             0.72                  │
      │        │                                │                   │
      │  [日報集計]                        [請求書PDF生成]           │
      │                                        │                   │
      │                                       0.91                  │
      │                                        │                   │
      │                                  [メール送付]               │
      └──────────────────────────────────────────────────────────┘

  [STEP 2] 密部分グラフ（結晶化ポイント）の検出
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    グラフから「密なクラスタ」を検出する。各クラスタ = システム化候補。

    アルゴリズム:
      1. エッジ重み ≥ 0.6 のサブグラフを抽出
      2. Louvain法でコミュニティ検出（modularity最大化）
      3. 各コミュニティに対して密度スコアを計算:
         cluster_density = Σ(edge_weights) / (n × (n-1) / 2)
         ※ n = クラスタ内ノード数
      4. cluster_density ≥ 0.7 のクラスタ = 「結晶化ポイント」

    上記グラフの場合:
      結晶化ポイント1: {kintone受注取得, freee請求書作成, 原価データ集計,
                       利益率計算, 請求書PDF生成, メール送付}
        → density: 0.81 → 結晶化候補
        → 提案: 「受注→原価→請求→送付」を一体管理するシステム

      結晶化ポイント2: {日報集計} (孤立ノード)
        → density: N/A → 単独タスクとして残す（BPOで十分）

  [STEP 3] 関連SaaS・データフローの特定
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    結晶化ポイント内のタスクが使っているSaaSとデータを分析:

    結晶化ポイント1の分析結果:
      使用SaaS: kintone（受注管理app_id:15）, freee（請求書）
      データフロー:
        kintone.受注レコード → [会社名, 金額, 工事名, 担当者]
          → freee.請求書下書き → [取引先, 金額, 品目, 税率]
            → PDF生成 → メール送付
      共有データフィールド: 会社名↔取引先, 金額, 工事名↔品目（12フィールド中8共有）
      現在のSaaS月額: kintone ¥5,000 + freee ¥3,000 = ¥8,000
      月間実行回数: 平均15回
      平均処理時間: 42分/回（BPO経由）
      エラー率: 12%（主に金額不一致）

  [STEP 4] モンテカルロROIシミュレーション
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    単純な「月額削減額」ではなく、複数シナリオの確率分布でROIを試算する。

    入力パラメータ（各々に分布を設定）:
      ・月間実行回数: Normal(μ=15, σ=3)        -- 季節変動考慮
      ・1回あたり削減時間: Uniform(20min, 40min) -- 最小〜最大
      ・時間単価: ¥3,000（固定）
      ・SaaS月額削減: ¥8,000（固定）
      ・エラー削減率: Beta(α=8, β=2)           -- 高い確率で70-90%削減
      ・エラー1件あたりの修正コスト: Lognormal(μ=¥5,000, σ=¥2,000)
      ・開発コスト: Triangular(¥30,000, ¥50,000, ¥80,000)
      ・保守コスト: ¥5,000/月（固定）

    シミュレーション（10,000回試行）:
      monthly_benefit = (exec_count × time_saved × hourly_rate)
                      + saas_saving
                      + (exec_count × error_rate_reduction × error_fix_cost)
                      - maintenance_cost

      payback_months = dev_cost / monthly_benefit

    出力:

      │  ROIシミュレーション結果（10,000回試行）               │

      │  月間効果:                                           │

      │                                                      │
      │  投資回収期間:                                        │
      │    P10（悲観）: 4.4ヶ月                               │
      │    P50（中央）: 1.8ヶ月                               │
      │    P90（楽観）: 0.8ヶ月                               │
      │                                                      │
      │  3ヶ月以内にROI達成する確率: 87%                      │

      │                                                      │
      │  ※ 類似企業12社の実績データで校正済み                  │

  [STEP 5] システム提案の成熟度モデル（4段階ゲート）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    提案は段階的に成熟させる。各ゲートに具体的な通過条件を設定:

    Stage 1: proposal（自動検出）
      通過条件:
        ・結晶化ポイントの密度 ≥ 0.7
        ・月間タスク実行回数 ≥ 10

      通過条件:
```

### 4.2 Composable System Architecture — アトミックコンポーネントによるシステム生成

**課題**: 「5つの開発パターン（A-E）」では粒度が粗すぎる。実際の中小企業システムは「データ入力+承認フロー+帳票出力+通知」のように複数パターンを組み合わせる。パターンの組み合わせ爆発に対応できない。さらに、個社の業務ルール（ナレッジ）をシステムにどう注入するかの設計がない。

**なぜ重要か**: SIerの弱点は「毎回ゼロからカスタム開発する」こと。シャチョツーの優位は「小さな部品の組み合わせで高速にシステムを構築する」こと。レゴブロックのように標準化されたアトミックコンポーネントを持ち、要件定義結果に応じてLLMがコンポーネントを組み合わせてシステムを生成する。個社固有のロジックは「ビジネスロジックプラグイン」として汎用コンポーネントに注入する。これにより、SIerの1/10の期間・1/10のコストでカスタムシステムが量産できる。

### 4.3 SaaS Migration Safety Engine — ゼロダウンタイム移行

**課題**: いきなりSaaSを止めると業務が止まる。「3段階移行」だけでは具体性が不足。データの移行方法、並行稼働時の整合性保証、自動ロールバック条件、APIのないSaaSからのデータ抽出方法が未設計。

**なぜ重要か**: SaaS移行の失敗は即座に業務停止を意味する。請求書が発行できない、勤怠が記録できない、受注データにアクセスできない — これらは中小企業にとって致命的。シャチョツーが信頼を失う最大のリスクは「移行で業務が止まった」という事態。航空機の飛行中エンジン交換のように、業務を1秒も止めずに移行を完了する安全エンジンが必須。

```
■ 提案: 5層移行パイプライン + シャドウモード + カナリアデプロイ + 自動ロールバック

  [Layer 1] データマイグレーションパイプライン
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    SaaSのデータをシャチョツーに移行するパイプライン。

    Step 1: スキーママッピング（SaaSフィールド → シャチョツーフィールド）

      自動マッピング:
        ・SaaS APIのスキーマ定義を取得
        ・シャチョツーのターゲットスキーマとフィールド名をEmbedding比較
        ・類似度 ≥ 0.85 → 自動マッピング
        ・類似度 0.5-0.85 → 候補表示 → 社長に確認
        ・類似度 < 0.5 → 新規フィールド作成を提案

      例: kintone → シャチョツー
        kintone「会社名」 → target「client_name」     (自動: 0.92)
        kintone「金額」   → target「amount」           (自動: 0.95)
        kintone「備考」   → target「notes」            (自動: 0.88)
        kintone「status」 → target「order_status」     (候補: 0.72 → 確認)
        kintone「f_001」  → ???                        (不明: 0.31 → 新規提案)

    Step 2: 変換ルール定義

      フィールドの型変換・値変換ルール:
        ・日付形式: "2024/01/15" → "2024-01-15T00:00:00+09:00"
        ・金額: 文字列"1,500,000" → integer 1500000
        ・ステータス: "受注" → "confirmed", "見込" → "prospect"
        ・計算フィールド: kintoneの自動計算 → シャチョツーの計算式に変換
        ・関連レコード: kintoneのルックアップ → シャチョツーのリレーション

    Step 3: バリデーション + インクリメンタル同期

      初回: 全件同期（フルダンプ → 変換 → バリデーション → インポート）
      以降: 差分同期（更新タイムスタンプベース → 差分のみ同期）

      バリデーションルール:
        ・件数一致チェック: SaaS件数 = シャチョツー件数
        ・合計値チェック: SaaS金額合計 = シャチョツー金額合計
        ・サンプル照合: ランダム10%のレコードを完全照合
        ・参照整合性: 外部キーの参照先が全て存在するか

  [Layer 2] シャドウモード（並行稼働 + 出力比較）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    旧SaaSとシャチョツーの新システムを並行稼働させ、出力を自動比較する。

    動作イメージ:
      1. BPO Workerが通常通り旧SaaSに操作を実行（本番）
      2. 同時に、シャチョツーの新システムにも同じ操作を実行（シャドウ）
      3. 両方の出力を比較 → 乖離を記録

    乖離率の計算:
      divergence_rate = 不一致件数 / 全比較件数

      比較対象:
        ・データ値の一致（金額・日付・テキスト）
        ・計算結果の一致（集計値・帳票の数値）
        ・ステータスの一致（承認状態・処理状態）
        ・タイミングの一致（処理完了時刻の差 ≤ 5分）

    シャドウモードの段階:
      Week 1-2: 読み取り系のみシャドウ（GET操作 → 結果比較）
        → 目標乖離率: ≤ 5%
      Week 3-4: 書き込み系もシャドウ（POST操作 → 下書き状態で比較）
        → 目標乖離率: ≤ 2%
      Week 5+:  全操作シャドウ
        → 目標乖離率: ≤ 1% を2週間維持でカナリアに進む

    乖離が検出された場合:
      ・乖離パターンを分類（型変換ミス/計算ロジック差/タイミング差/データ欠損）
      ・自動修正可能 → 変換ルールを更新 → 次回から修正
      ・自動修正不可 → 社長に報告 + 開発チームに修正チケット

  [Layer 3] カナリアデプロイ（段階的トラフィック移行）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    シャドウモード通過後、実際のトラフィックを段階的に新システムに移行。

    カナリア段階:
      Stage 1: 10% — 特定の低リスク操作のみ新システムで実行
        ・読み取り系（データ表示・レポート生成）を新システムで
        ・書き込みは引き続き旧SaaS
        ・期間: 1週間 / 昇格条件: エラー率 ≤ 0.5%

      Stage 2: 30% — 下書き・計算系も新システムで実行
        ・下書き作成・集計計算を新システムで
        ・確定操作は引き続き旧SaaS
        ・期間: 1週間 / 昇格条件: エラー率 ≤ 0.3%

      Stage 3: 70% — 確定操作の大部分を新システムで実行
        ・旧SaaSは高リスク操作（外部送信・法令関連）のみ
        ・期間: 1週間 / 昇格条件: エラー率 ≤ 0.1%

      Stage 4: 100% — 完全移行
        ・旧SaaSは読み取り専用（30日間保持後解約）
        ・移行完了レポート自動生成

  [Layer 4] 自動ロールバックエンジン
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    「元に戻すボタン」ではなく、異常を自動検出して自動ロールバック。

    自動ロールバックトリガー:
      ① エラー率急上昇:
         直近1時間のエラー率 > 5% (基準値の10倍)
         → 即座に旧SaaSに全トラフィックを戻す

      ② レスポンス劣化:
         直近30分の平均レスポンス > 3秒 (基準値の3倍)
         → 新システムのヘルスチェック → 改善なければロールバック

      ③ データ整合性異常:
         定期整合性チェック（5分間隔）で不一致検出
         → 不一致件数 > 3 → 即ロールバック + 不一致データの詳細レポート

      ④ ユーザー報告:
         社長/ユーザーから「おかしい」報告
         → 1クリックで即ロールバック（管理画面に緊急ボタン）

    ロールバック実行:
      1. 新システムへのトラフィックを停止（0.5秒以内）
      2. 旧SaaSへのトラフィックを再開
      3. 新システムで作成されたデータを旧SaaSに逆同期
      4. ロールバック原因を自動分析 → レポート生成
      5. 修正後、前のカナリアステージから再開

  [Layer 5] SaaS APIリバースエンジニアリング（エクスポートAPI非対応SaaS向け）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    エクスポートAPIがないSaaSからのデータ抽出方法。

    手法1: UI操作 + スクレイピング（Computer Use活用）
      ・Computer Use で管理画面にログイン
      ・一覧画面をスクロール → スクリーンショット → OCR → データ抽出
      ・CSV出力機能がある場合はCSVダウンロード操作を自動化

    手法2: API観察（ブラウザのNetwork通信をキャプチャ）
      ・Chrome拡張でSaaS操作時のAPIリクエストをキャプチャ
      ・リクエスト/レスポンスのパターンから非公開APIのスキーマを推定
      ・推定APIを使って直接データ取得（非公式のため、変更リスク高）
      ・⚠️ 利用規約チェック必須 → 違反の可能性がある場合は手法1に切替

    手法3: 手動エクスポート支援
      ・SaaSの画面上で「エクスポート」ボタンの位置を検出
      ・ユーザーに「この画面でCSV出力してアップロードしてください」とガイド
      ・アップロードされたCSVを自動パース → 移行パイプラインに投入
```

### 4.4 System Evolution Engine — 自律的システム改善

**課題**: システムは「作って終わり」ではない。SIerの世界では仕様変更のたびに追加見積もりが発生し、システムは陳腐化していく。シャチョツーが作るシステムも、ナレッジの変化・使用パターンの変化・外部環境の変化に自動で適応しなければ、結局SIerと同じになる。

**なぜ重要か**: シャチョツーの最大の差別化は「変更が即時」。ナレッジが変われば → ビジネスロジックプラグインが自動更新 → システムが自動適応。これがSIerの「仕様変更=追加見積もり2週間」を完全に過去のものにする。さらに、使用データを分析して「この機能は使われていない」「この処理が遅い」を自動検出し、改善提案まで行う自律進化エンジンが、SIer保守契約（月50万）を不要にする。

```
■ 提案: 4つの自律改善メカニズム

  [1] Usage Analytics → 自動改善提案
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    全操作をトラッキングし、改善ポイントを自動検出する。

    トラッキング対象:
      ・ページビュー: どの画面がどの頻度で使われているか
      ・操作パス: ユーザーが画面をどう遷移しているか
      ・滞在時間: 各画面での滞在時間（長い = 分かりにくい？）
      ・入力パターン: どのフィールドがいつも同じ値で入力されるか
      ・エラー発生: どの操作でバリデーションエラーが起きるか
      ・未使用機能: 作ったが1ヶ月以上使われていない機能

    自動検出パターンと改善提案:

      Pattern A: 同じ値の繰り返し入力
        検出: field_X の値が直近50回のうち80%以上同じ
        提案: 「この項目のデフォルト値を "○○" に設定しますか？」
        効果: 入力時間 −3秒/回 × 50回/月 = 月2.5分削減

      Pattern B: 未使用機能の検出
        検出: 特定のコンポーネントが30日間使用回数ゼロ
        提案: 「"○○レポート" 機能が1ヶ月使われていません。

  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    ナレッジが変更された時、新旧ロジックをA/Bテストで比較する。

    トリガー: ナレッジの更新イベント → ビジネスロジックプラグインの変更検出

    実行フロー:
      1. 旧プラグイン（Variant A）と新プラグイン（Variant B）を並行稼働
      2. トラフィックを50:50で分割（同一ユーザーは同じVariantに固定）
      3. 以下のメトリクスを比較:
         ・処理時間
         ・エラー率
         ・承認時の修正率（少ない方が正確）
         ・ユーザー満足度（👍/👎）
      4. 2週間後 or サンプル数50以上で自動判定:
         ・Variant Bが統計的に有意に優れている → 全トラフィックをBに
         ・有意差なし or Aが優れている → Aを維持 + ナレッジ変更を再検証

    例:
      ナレッジ変更: 「諸経費率を25%→22%に変更」
      A: 旧ルール（25%）で見積もり作成
      B: 新ルール（22%）で見積もり作成
      結果: Bの方が見積もり承認率が15%高い → Bを採用

    安全弁:
      ・金額影響が100万円超の場合は自動A/B禁止 → 社長承認が必要
      ・法令関連のロジック変更は自動A/B禁止

  [3] Self-Healing（自動修復）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    よくある障害を自動検出・自動修復する。SIerの保守契約を不要にする。

    自動修復パターン:

      ① スキーマドリフト修復
        検出: 外部SaaSのAPIレスポンススキーマが変更された
        修復: 新旧スキーマをLLMで比較 → フィールドマッピングを自動更新
        → 修復できた場合: ログ記録 + 社長に「更新しました」通知
        → 修復できない場合: データ同期を一時停止 + 開発チームに通知

      ② APIバージョン変更対応
        検出: APIリクエストが4xx/5xxを返し始めた
        修復:
          Step 1: APIドキュメントをWebFetchで取得
          Step 2: 変更点をLLMが分析
          Step 3: リクエスト形式を自動修正
          Step 4: テスト実行 → 成功なら適用
        → 月間で平均2-3回発生。SIerなら対応に1-3日。シャチョツーは5分。

      ③ パフォーマンス劣化の自動修復
        検出: 特定のクエリのレスポンス時間が前週比200%以上
        修復:
          Step 1: Supabaseのクエリプランを分析
          Step 2: インデックス不足を検出 → 自動CREATE INDEX
          Step 3: N+1クエリを検出 → JOINに書き換え
        → レスポンス改善後、変更内容をログ記録

      ④ ディスク/行数アラート
        検出: テーブルの行数が閾値超過 or Supabaseのストレージ使用率80%超
        修復: 古いログデータのアーカイブ提案 + 自動アーカイブ実行

  [4] ナレッジ連動バージョニング
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    システムバージョンをナレッジバージョンに紐づけて管理する。

    仕組み:
      ・各ビジネスロジックプラグインに「source_knowledge_version」を持つ
      ・ナレッジが更新されると:
        1. 影響を受けるプラグインを特定
        2. プラグインを自動更新（ナレッジ → LLM構造化 → 新プラグイン）
        3. システムバージョンを自動インクリメント（patch level）
        4. 変更影響レポートを生成
          「受注入力システム v1.2.3 → v1.2.4
           変更点: 諸経費率計算ルール更新（25%→22%）
           影響範囲: 見積もり画面の金額計算
           テスト結果: 全10ケース合格」

    ロールバック:
      ・任意の過去バージョンに戻せる
      ・バージョン間の差分を可視化（どのナレッジ変更がどのシステム変更を引き起こしたか）
```

### 4.5 System Pattern Network Intelligence — 全社横断パターン学習

**課題**: 業種Aで作ったカスタムシステムが業種Bでも使えるケースが多い。単純なテンプレートの横展開では「業種ごとの微妙な違い」に対応できない。さらに、「日報+在庫管理=現場管理統合」のようなパターンの組み合わせ、類似企業への先回り推薦ができていない。

**なぜ重要か**: 100社に展開した時点で、各社が個別にシステムを作っていたら100通りの保守が必要。しかし実際は80%が共通パターンの組み合わせ。パターンを抽象化・組み合わせ・業界適応することで、新規企業には「あなたの会社に必要なシステムはおそらくこの3つです。類似企業50社で実証済み」と即提案できる。これが「SIerに要件定義を依頼する」必要性を完全に消す。

```
■ 提案: パターン抽象化 + コンポジション代数 + 業界適応 + 予測推薦

  [1] Pattern Abstraction Engine（成功システムからのパターン抽出）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    稼働中のシステムから、個社固有の情報を除去してパターンを抽出する。

    抽出プロセス:
      1. 稼働中システムのコンポジショングラフを取得
      2. ビジネスロジックプラグインを「プレースホルダー」に置換
         例: {formula: "cost * 0.25"} → {formula: "cost * {overhead_rate}"}
      3. 個社データ（会社名・具体的金額・担当者名）を除去
      4. コンポーネント構成 + データフロー + プレースホルダーロジック = パターン
      5. パターンのハッシュを計算 → 既存パターンとの類似度チェック
         類似度 ≥ 0.9 → 既存パターンのバリエーションとして登録
         類似度 < 0.9 → 新規パターンとして登録

    例: 建設会社A → 「受注→請求一体化パターン」を抽出
      コンポーネント: data_input → calculation_engine → approval_flow
                     → pdf_builder → notification
      プレースホルダー:
        ・{overhead_rate}: 諸経費率（企業ごとに15-30%で異なる）
        ・{approval_stages}: 承認段数（1-3段で異なる）
        ・{invoice_template}: 請求書テンプレート（企業ロゴ等）
        ・{notification_channel}: 通知先（LINE/Slack/メール）

  [2] Pattern Composition Algebra（パターンの組み合わせ）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    パターン同士を「合成」して新しい統合パターンを作る。

    合成の種類:
      ① 直列合成（Sequential）: A → B
         例: 「受注管理」→「請求管理」= 「受注→請求一体化」
         条件: Aのoutput_schemaとBのinput_schemaに共通フィールドが存在

      ② 並列合成（Parallel）: A || B
         例: 「日報管理」||「在庫管理」= 「現場管理統合」
         条件: AとBが同じデータソースを共有（data_input コンポーネント共有）

      ③ 拡張合成（Extension）: A + Δ
         例: 「日報管理」+ Δ(写真+天候) = 「建設日報管理」
         条件: Δのフィールドを既存コンポーネントに追加可能

    互換性スコア:
      compatibility(pattern_A, pattern_B) =
        0.40 × schema_overlap        -- データスキーマの重複度
      + 0.30 × component_reuse       -- 共有可能なコンポーネント比率
      + 0.20 × workflow_alignment    -- ワークフローの整合性
      + 0.10 × user_overlap          -- 同じユーザーが使う確率

      compatibility ≥ 0.7 → 合成推薦対象

    合成の具体例:

      ├──────────────────────────────────────────────────────┤
      │                                                      │
      │  「日報管理」(利用: 45社) + 「在庫管理」(利用: 38社)  │
      │   → 「現場管理統合」(互換性: 0.82)                    │
      │                                                      │
      │  合成メリット:                                        │
      │    ・日報入力時に在庫使用量も同時記録（二重入力廃止）  │
      │    ・在庫アラートが日報の作業予定と連動               │
      │    ・コンポーネント共有: data_input(1個削減)           │
      │    ・推定効果: 入力時間 −40% / データ整合性向上       │

  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    パターンの品質を継続的に測定し、低品質パターンを改善する。

    + 0.20 × (1 - maintenance_cost) -- 月間メンテナンス工数の逆数

    + 0.15 × stability_score        -- エラー率の低さ × 稼働月数

      beta     (0.3-0.6): 3-10社で検証中。「ベータ版」と明記して推薦
      draft    (0.0-0.3): 1-2社のみ。推薦対象外。改善パイプラインへ

    改善パイプライン:
      ・beta以下のパターンに対して:
        1. カスタマイズ率30%以上のフィールドを特定
        2. 「このフィールドをパターンのデフォルトオプションに追加すべきか」をLLMが判定

  [5] Predictive Pattern Recommendation（予測推薦エンジン）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 4.5 ブラウザ拡張による透明移行エンジン — 社員が気づかない移行

**課題**: 現設計はシステムを作って「移行してください」で終わり。実際のボトルネックは社員の反発と再教育コスト。「新しいシステムです、使い方覚えてください」は必ず失敗する。

**なぜ重要か**: 移行コストがゼロなら、SaaS置換の意思決定が劇的に楽になる。社員が気づかないうちに移行が完了し、ある日「kintone解約しました」と社長が言えばいい。

#### ■ 3フェーズ移行設計

```
Phase A: 並行稼働（2〜4週間）
  - 自社システムとkintoneに同時書き込み
  - データ同期差分率を毎時計測（目標: diff_rate < 0.1%）
  - 社員は引き続きkintoneを使う → 一切の変化なし

Phase B: ブラウザ拡張によるUX再構築（2〜6週間）
  - Chrome拡張をサイレントデプロイ（管理者がGoogle管理コンソールから配布）
  - kintone.com へのアクセスをインターセプト → 自社UXを注入 → 自社APIに書き込み
  - 単なるUI差し替えではなく、業務フロー自体を再設計:
    ・複数画面遷移の1画面統合（例: 3画面で行っていた日報入力を1画面に）
    ・承認フローの自動化（社長の判断基準ベースで自動承認/エスカレーション）
    ・入力順序の最適化（業務フローに沿った自然な順序に並び替え）
    ・不要フィールドの非表示 + 必須フィールドの追加
    ・コンテキスト連動（前工程のデータを自動プリフィル）
    ・リアルタイムバリデーション（社長ルールベースの即時チェック）
  - 社員の体験: 「画面が同じ」ではなく「仕事が楽になった」
  - staff_error_rate + task_completion_time を連続モニタリング
    （error_rate 0.5%以上増加 or completion_time 20%以上悪化で自動ロールバック）

Phase C: 完全移行・解約
  - diff_rate < 0.1% かつ staff_error_rate に有意な変化なし → cutover_ready = true
  - kintoneへの書き込みを停止 → 読み取りのみに切り替え → 30日後に解約手続き
```

### 4.6 ナレッジ変更 → システム自動保守パイプライン — 作って終わりにしない

**課題**: 現設計はシステムを「作って終わり」。業務ルールが変わったとき誰が直すか設計されていない。中小企業でシステムが陳腐化する最大原因は保守コストと人材不在。

**なぜ重要か**: ブレイン編でナレッジが更新されるたびに自動でシステムに反映されれば、保守コストがほぼゼロになる。「生きているシステム」として常に最新の業務ルールを体現し続ける。

#### ■ 変更規模分類と自動対応フロー

```
ナレッジ変更トリガー
    ↓
影響システム自動検出（knowledge_item_id → system_ids マッピング）
    ↓
変更規模分類
    ├─ 軽微（表示名・閾値・メッセージ変更）
    │     → 自動パッチ適用 → テスト実行 → 通過で自動デプロイ → LINE通知
    │
    ├─ 中程度（フロー変更・条件分岐追加・API変更）

```

### 4.7 SaaS解約ロードマップ自動生成 — 「いつ何を解約するか」を設計する

**課題**: 現設計はシステムを「作る」まで。「いつ何を解約して、いくら削減できるか」のロードマップが自動生成されない。社長は「なんとなくSaaS代が減った気がする」程度の認識になりがち。

**なぜ重要か**: 解約ロードマップが可視化されると、社長の投資判断が劇的に変わる。「6ヶ月後に月¥95k削減、年間¥1.14M削減」という具体的な数字があれば、開発投資の意思決定が「コスト」から「投資」に変わる。

#### ■ SaaS依存グラフとトポロジカルソート

```python
# SaaS依存グラフ: ノード=SaaS、エッジ=データフロー方向
# 例: kintone → freee（kintoneの売上データがfreeeに連携されている）
#
# トポロジカルソートで置換順序を決定
# → データの「送り元」から先に置換する（依存元から先に置換）
#
# 具体例:
#   kintone ──→ freee ──→ マネーフォワードクラウド
#   この場合: kintone → freee → マネーフォワードの順に置換
#   理由: kintoneを先に置換しないとfreeeへのデータ連携が途絶える

def calculate_replacement_order(saas_dependency_graph: Dict) -> List[str]:
    """
    依存グラフをトポロジカルソートして置換順序を返す。
    依存元（データの送り元）が先に来る。
    """
    return topological_sort(saas_dependency_graph)
```

```
    API品質       × 0.3   # 公式APIあり=0.1、非公式=0.5、APIなし=0.9

  + 法的制約      × 0.2   # 電子帳簿保存法対応必須=0.8、なし=0.1
```

## 5. 「SaaSを潰す」キラーシナリオ

```
■ Before（SaaS + SIerの世界）

  社長: 「うちのITコスト、高くない？」
  → kintone: ¥50,000/月（30ユーザー）
  → freee:   ¥30,000/月
  → ジョブカン: ¥20,000/月
  → Slack:   ¥15,000/月（30名）
  → ANDPAD:  ¥50,000/月
  → SIer保守: ¥500,000/月
  → 合計: ¥665,000/月 = ¥7,980,000/年

  しかも:
  → 各SaaS間の連携はSIerに別途依頼（初期200万 + 月20万保守）
  → データがバラバラでレポート作成に毎月3日かかる
  → 仕様変更のたびにSIerへ追加見積もり

■ After（シャチョツーの世界 — 3年後の姿）

  社長: 「シャチョツーだけでいいんじゃない？」
  → シャチョツー: ¥80,000/月（基本料+従量課金）
    ・ナレッジ管理 + Q&A（旧kintoneの一部を代替）
    ・請求処理 Level 4 自動化（freeeの主要機能を代替）
    ・日報システム内蔵（kintoneの日報機能を代替）
    ・コミュニケーション → LINE/Slack連携（Slackは残す判断も可）
    ・勤怠データ → 専用画面（ジョブカンの基本機能を代替）
    ・建設現場管理 → 写真+日報+天候の統合管理
    ・SIerの仕事 → AIが代替（ヘルプデスク+連携+セキュリティ）

  → 月額: ¥665,000 → ¥80,000 = 年間700万削減
  → しかもデータが一元化されているのでレポートは即生成
  → 仕様変更はナレッジを変えるだけで自動反映

■ 注意: 全SaaSを潰すわけではない

  残すもの:
  ・法令で必須のもの（e-Tax連携のため最低限のfreee等）
  ・取引先指定のもの（元請指定のANDPAD等）
  ・深い専門機能（会計仕訳AIなど、シャチョツーより優れている部分）

  潰すもの:
  ・「データの入れ物」としてしか使っていないSaaS
  ・機能の20%しか使っていないSaaS
  ・連携のためだけに存在するiPaaS
  ・SIerの定型保守（ヘルプデスク+バックアップ+監視）
```

## 6.5 SMB / エンプラ 自社システム展開の設計差異

### SMBとエンプラで何が変わるか

```
■ SMB（〜300名）向け自社システム
  目的: SaaS月額費用の削減・業務最適化
  展開方式: シャチョツー共有インフラ上にデプロイ
  セキュリティ: 00_00_セキュリティ設計.md §1-§17
  開発期間: 1-4週間（アトミックコンポーネント組み合わせ）
  コスト: 初期¥0（月額プランに含む）+ インフラ従量

■ エンプラ（300名〜）向け自社システム
  目的: 上記 + 既存基幹システムとの統合・社内セキュリティポリシー準拠
  展開方式: 専有インスタンス or 顧客GCPプロジェクトへのデプロイ
  セキュリティ: 00_00_セキュリティ設計.md §1-§26（全て）
  開発期間: 2-8週間（審査・テスト工程が増える）
  コスト: 専有インスタンス費用 + 開発工数（別途見積）
```

### エンプラ向け自社システム固有の追加設計

**① 生成コードのセキュリティ審査フロー**

```
SMB:
  LLMコード生成 → Semgrep自動スキャン → デプロイ
  （自動スキャンでCritical/High未検出なら即デプロイ）

エンプラ:
  LLMコード生成
    ↓
  Semgrep自動スキャン（§16）
    ↓
  顧客の情報システム部門によるコードレビュー（5営業日）
    ↓
  ステージング環境での受け入れテスト（顧客実施）
    ↓
  顧客の変更管理委員会（CAB）承認
    ↓
  本番デプロイ（メンテナンスウィンドウ内）

  ※ CAB: Change Advisory Board（変更管理委員会）
     大企業では全システム変更にCAB承認が必要

■ 生成コードの提出物（エンプラ標準）:
  ・ソースコード（GitHubリポジトリへのアクセス付与）
  ・セキュリティスキャン結果レポート（Semgrep / Trivy）
  ・アーキテクチャ設計書（データフロー図付き）
  ・テスト仕様書 + テスト結果
  ・運用手順書（障害時・定期メンテナンス）
  ・データ項目一覧（個人情報の取り扱いを含む）
```

**② 既存基幹システムとの統合設計**

```
エンプラが持つ既存システム（中小企業にはないもの）:
  ・ERP（SAP / Oracle / 勘定奉行クラウド）
  ・グループウェア（Microsoft 365 / Notes）
  ・ID管理（Active Directory / LDAP）
  ・ファイルサーバー（オンプレミス）

統合パターン:

  [パターン1] API統合（ERP等がAPI公開している場合）
    自社システム ←→ ERP API
    データ同期: リアルタイム or 夜間バッチ

  [パターン2] DB直結（許可された場合）
    自社システム → 読み取り専用接続 → ERP DB
    ※ ERPベンダーの承認が必要

  [パターン3] ファイル連携（最も多いケース）
    ERP → CSVエクスポート → GCS → 自社システムが取り込み
    スケジュール: 毎朝6時バッチ処理

  [パターン4] iPaaS経由（Make / Zapier）
    既存システム → Make → 自社システム
    ※ BPO編 4.4のコネクタ戦略と同一設計

■ 統合時の責任分界点
  シャチョツーの責任範囲: 自社システム側の設計・開発・保守
  顧客の責任範囲: 既存システム側のAPI提供・認証情報管理・ファイル出力
  共同責任: データ形式の仕様定義・テスト実施
```

**③ エンプラ向けデプロイオプション**

```
[Option A] シャチョツー共有インフラ（デフォルト）
  ・Cloud Run（東京リージョン）
  ・Supabase（マルチテナント）
  ・月額: 通常プランに含む
  ・対象: 〜300名、機密性要件が標準レベル

[Option B] 専有インスタンス（エンプラ標準）
  ・専用Cloud Run + 専用Supabase
  ・他社データと物理分離
  ・月額: +¥150,000
  ・対象: 300名〜、「他社と同じDBは困る」要件がある場合

[Option C] 顧客GCPプロジェクトデプロイ（最高セキュリティ）
  ・顧客が所有するGCPプロジェクトにデプロイ
  ・データが顧客インフラから外に出ない
  ・初期費用: ¥500,000 + 月額保守¥100,000
  ・対象: 金融・医療・防衛・機密性最高レベル

[Option D] オンプレミスデプロイ（将来対応）
  ・Dockerコンテナで顧客データセンターにデプロイ
  ・インターネット接続なしで動作
  ・LLMはオンプレミスモデル（Llama等）に切替
  ・対象: 完全クローズド環境が必要な顧客（国防・原子力等）
```

**④ エンプラ向け保守・SLAオプション**

```
■ SMBとエンプラの保守の違い

  SMB:

    本番デプロイは定期メンテナンスウィンドウ内（毎週水曜22:00-24:00等）

■ エンプラ向けSLAオプション

  Standard（エンプラ基本）:
    稼働保証: 99.9%/月
    サポート: 平日9-18時（日本語）

```

```
    ・社長の判断基準（decision_rules）をリアルタイム適用
    ・Confidence × Impact マトリクスで自動承認/要確認/要エスカレーションを判定
    ・例: 5万円以下の経費精算 → 社長ルールに合致すれば自動承認

  3. 入力順序最適化
    ・業務フローの自然な順序に並び替え（SaaSのDB都合の順序ではなく）
    ・頻度の高いフィールドを上部に配置
    ・条件分岐: 前の入力値に応じて次のフィールドを動的表示

  5. コンテキスト連動（Auto-Prefill）
    ・前工程のデータを自動プリフィル
    ・例: 見積書作成時に顧客情報・過去単価・直近の商談メモを自動挿入
    ・関連ナレッジの自動サジェスト（入力中にリアルタイム）

  6. リアルタイムバリデーション
    ・社長ルール（decision_rules）ベースの即時チェック
    ・例: 値引率15%超 → 警告「社長承認が必要です（ルールR-042）」
```

```python
class DataFlowEdge(BaseModel):
    from_saas: str                         # データ送信元
    to_saas: str                           # データ受信先

    frequency: str                         # realtime/daily/weekly/monthly
```

## 7. 3編の全体フロー（まとめ）

```
001_ブレイン編（コンサルを潰す）
  ↓ ナレッジが溜まる
  ↓ 業務フロー・判断基準・暗黙知が構造化される
  ↓
002_BPO編（業務代行・人材紹介を潰す）
  ↓ ナレッジが「実行」に変換される
  ↓ 実行データ（成功/失敗/ボトルネック）が溜まる
  ↓
003_自社システム編（SaaSを潰す）
  ↓ 実行データが「システム要件」に変換される
  ↓ 個社専用システムが自動構築される
  ↓
  ↓ ─── そしてデータネットワーク効果 ───
  ↓
  ↓ 全社のナレッジ × 実行データ × システムパターン
  ↓ → 匿名化 → 業界標準化 → 新規企業に還元
  ↓ → 使えば使うほど、全員が賢くなる
  ↓ → これが競合が追いつけない最大の堀（Moat）

最終形: シャチョツー = 中小企業の「IT部門 + コンサル + BPO」のオールインワン
        月3-8万で、年間1,000万以上のコスト削減を実現
```
