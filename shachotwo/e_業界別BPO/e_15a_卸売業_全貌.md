## 1. 卸売業の現実（業界人なら「分かる」話）

### 1.1 中小卸の現場

従業員10-300名の中小卸売業、約25万社（経産省 商業統計調査）。
食品、建材、機械、繊維、医薬品、化学品。取扱商品は違っても共通する現実がある。

**社長の1日（Before）**

```
AM 7:00  出社。FAX機から昨夜と今朝の注文書がドサッと出ている。30枚
         → パートさんが1枚ずつ確認して販売管理システムに手入力
         → 1枚10-15分。30枚で5-7時間。今日も入力で1日終わる
AM 7:30  得意先の居酒屋から電話「いつもの日本酒3ケース、明日朝までに頼む」
         → 「いつもの」が何を指すか、ベテラン営業しか知らない
         → 在庫あるのか、倉庫に電話して確認。「えーと、2ケースしかない」
         → 仕入先に緊急発注。間に合うか不明
AM 8:00  営業A君がメールで見積依頼を受けた。掛率表をExcelで探す
         → 得意先ごとに掛率が違う。A社は定価の70%、B社は65%、C社は72%
         → 特売品は別掛率。数量割引もある。間違えたら赤字
AM 9:00  パートさんから「この注文、商品名が略称で何かわかりません」
         → FAXの手書き文字。「サバ缶」→サバの水煮缶か味噌煮缶か。サイズは？
         → 得意先に電話確認。出ない。折返し待ち
AM 10:00 経理から「先月の請求書と仕入先の請求書が合わない。3件差異あり」
         → 納品書と突き合わせ。数量違い1件、単価違い1件、未納品1件
         → 仕入先に連絡。「あ、すみません、単価改定のお知らせ出してました」
PM 1:00  得意先D社の入金が遅れている。3回目。与信限度額を超えそう
         → 営業部長「D社、最近ちょっと危ないかも」→ 感覚的な判断
         → 売掛金3,000万円。もし焦げたら…
PM 2:00  仕入先の展示会。新商品の情報を仕入れる
         → 帰社後、商品マスタに登録…するのは来週。目の前の受注処理が優先
PM 3:00  在庫の棚卸。実在庫とシステムの数字が合わない。毎月このズレ
         → 入出庫のたびに手入力。忙しいと「後で入れよう」→ 忘れる
PM 4:00  来月のメーカー発注。需要予測は営業の「肌感覚」
         → 「そろそろ焼肉のタレが動く季節だから多めに…」
         → 過剰在庫でキャッシュフローが苦しくなった前科あり
PM 5:00  今日入力した注文の確認。2件入力ミス発覚
         → 商品コード違い→誤出荷の手前で気づいた。ヒヤリハット
PM 7:00  退社。今日も受注処理と在庫確認とトラブル対応で終わった。
```

### 1.3 業界構造変化（存在意義が問われている）

**メーカー直販化の加速**

```
従来: メーカー → 卸（1次卸→2次卸→3次卸） → 小売 → 消費者
現在: メーカー → 直販EC → 消費者（卸を飛ばす「中抜き」）

事例:
  ・食品メーカーのD2Cブランド増加（ベースフード、完全メシ等）
  ・建材メーカーの直販サイト開設
  ・P&Gの「日用品サブスク」

中小卸の存在意義:
  ① 多品種少量の品揃え（メーカー1社では揃わない）
  ② 小口配送（メーカーはケース単位。卸はバラ出荷対応）
  ③ 与信・代金回収（零細小売の信用を卸が担保）
  ④ 情報提供（新商品情報、市場動向、棚割提案）

→ ①②③は物流+金融機能。④がAIで代替されたら？
→ 「配送だけなら運送会社でいい」となるリスク
→ 卸の価値 = 顧客への提案力 + 在庫リスクの引受け + 小口対応の便利さ
```

**EC・BtoB ECの台頭**

```
BtoB ECの市場規模: 約420兆円（経産省 2023年）。前年比+10.7%
モノタロウ、Amazon Business、アスクル → 中小卸の競合になりつつある

ただし:
  ・食品卸の温度帯管理はECで代替困難
  ・建材卸の現場直送・カット加工はECで代替困難
  ・医薬品卸の法規制対応はECで代替困難
  → 「物理的なオペレーション」に卸の価値がある業界は生き残る
  → デジタルで済む商品を扱う卸は淘汰される
```

**インボイス制度・電子帳簿保存法のインパクト**

```
インボイス制度（2023年10月〜）:
  ・適格請求書の発行・保存義務
  ・免税事業者からの仕入税額控除が段階的に縮小
  → 中小卸の仕入先（零細農家、個人事業主）が対応できない問題
  → 卸がインボイス発行を代行するケースも

電子帳簿保存法（2024年1月〜義務化）:
  ・電子取引のデータ保存義務
  ・FAXで受けた注文書も電子保存が必要（ただし紙出力保存の猶予あり）
  → 「紙のFAXを紙のまま保管」がNGになる方向
  → いずれFAX→電子化の流れは避けられない
```

**結論:** 卸売業は「存在意義」を問われている。単なる中間流通から「データを持つ、提案ができる、品揃えが最適化されている」卸に変わらなければ淘汰される。**受発注のデジタル化は生存条件**。

### 2.2 食品卸の特殊性（最優先ターゲットの深掘り）

```
食品卸が最もBPO効果が高い理由:

1. FAX依存率80%超
   飲食店・小売店からの発注がFAX。特に個人店は「手書きFAX」
   → OCR + LLM の最大活用ポイント

2. 賞味期限管理（日付逆転防止）
   先入先出（FIFO）の徹底。期限の1/3ルール（製造日から賞味期限の1/3で納品）
   → 在庫管理AIの必須機能

3. 温度帯管理（3温度帯）
   常温（ドライ）、冷蔵（チルド 0-10℃）、冷凍（フローズン -18℃以下）
   → 同じ得意先への納品でも3温度帯で別配送になることがある

4. 「いつもの」注文
   得意先の80%はリピート注文。「いつもの」= 前回と同じ内容
   → 注文パターン学習AIの最大活用ポイント

5. 欠品=即取引打ち切りリスク
   飲食店は「今日使う食材」を注文。欠品=メニューが出せない=大問題
   → 需要予測+安全在庫の精度が生命線

6. 荷姿の多様性
   ケース/バラ/kg/リットル/尾/本/枚…。単位変換が複雑
   → 「サバ缶24缶入り1ケース」と「サバ缶24個」は同じ。AIが理解する必要がある
```

## 3. 受発注の全貌（これが分かってないと話にならない）

### 3.1 受発注の基本フロー

```
Step 1: 得意先から注文を受ける（FAX/電話/メール/LINE/EDI）
  │      ← FAXが60-80%。手書きも多い。同じ得意先でもフォーマットがバラバラ
  │
Step 2: 注文内容の読取・確認
  │      ← 商品名の略称・型番揺れの解読（「サバ水煮」「鯖水」「マルハサバ」→同じ商品）
  │      ← 数量の単位確認（ケース？バラ？kg？）
  │      ← 納品日・配送先の確認
  │
Step 3: 在庫確認
  │      ← 在庫あり → 引当て
  │      ← 在庫なし → 仕入先に緊急発注 or 得意先に代替品提案 or 欠品連絡
  │
Step 4: 販売管理システムに入力
  │      ← 商品コード・数量・単価・納品日・配送先を手入力
  │      ← ★ここが最大のボトルネック。月500-2,000件×10-15分
  │
Step 5: ピッキング指示→出荷→納品→納品書発行
  │
Step 6: 月次で請求書発行→入金確認→消込
```

### 3.2 掛率（かけりつ）の仕組み

得意先ごとに商品の販売価格が異なる。これが卸売業の利益構造の根幹。

```
掛率 = 販売価格 / 定価（メーカー希望小売価格）

例:
  メーカー定価 ¥1,000 の商品
  A社への販売: 70掛（¥700）  ← 大口得意先。薄利だが量で稼ぐ
  B社への販売: 75掛（¥750）  ← 中口得意先
  C社への販売: 80掛（¥800）  ← 小口得意先
  D社への販売: 85掛（¥850）  ← 新規得意先。まだ信用が浅い

仕入原価: 55掛（¥550）の場合
  A社の粗利: ¥150（粗利率21.4%）
  D社の粗利: ¥300（粗利率35.3%）

→ 大口ほど掛率が低い（薄利）。小口ほど掛率が高い（厚利）
→ 大口得意先を失うと売上激減。小口を増やすと手間が増える
→ この「掛率の最適化」が卸の経営そのもの
```

**掛率の複雑さ:**

```
基本掛率: 得意先ランクで決まる（S/A/B/C）
  + 商品カテゴリ別掛率: カテゴリごとに異なる場合がある
  + 数量割引: 10ケース以上→-2%、100ケース以上→-5%
  + 特売掛率: メーカー販促期間中の特別価格
  + 期間限定掛率: 新商品導入時の初回割引
  + 年間リベート: 年間購入額に応じた事後値引き

→ 1つの商品の「正しい売価」を出すのに、5-6個の条件を確認する必要がある
→ 間違えたら赤字販売 or 得意先の不信（「前と値段が違うんだけど」）
```

**会社ごとに得意先別の掛率テーブルを持っている。** このマスタデータの精度が、シャチョツーの価格表・見積作成AIの精度を決める。

### 3.3 商品マスタの地獄

```
食品卸の商品マスタ: 5,000-30,000 SKU
  ・メーカー100社以上の商品を取り扱い
  ・1メーカーで50-500アイテム
  ・同じ商品でも容量違い（100g/200g/500g/1kg）
  ・同じ商品でもケース入数違い（6本入り/12本入り/24本入り）
  ・JANコード+自社コード+得意先コードの3重管理
  ・新商品: 月10-30アイテム追加。廃番: 月5-20アイテム

建材卸の商品マスタ: 10,000-50,000 SKU
  ・品番体系がメーカーごとに全く異なる
  ・TOTOの品番とLIXILの品番の規則が違う
  ・色違い・サイズ違いで品番が別
  ・旧品番→新品番の切替が頻繁

共通の問題:
  ・得意先が使う商品名 ≠ 正式品名 ≠ メーカー品名
    例: 「ハイター」→ 花王キッチンハイター？ 花王ハイター？ カネヨブリーチ？
  ・略称、通称、俗称が乱立
  ・FAXの手書きで「判読不能」な商品名が日常的
```

### 3.4 受発注の「例外」（ここがAI化の難所）

| パターン | 内容 | AI対応 |
|---|---|---|
| **略称・揺れ** | 同じ商品を10通りの名前で呼ぶ | 商品マスタのファジーマッチ+得意先別辞書 |
| **「いつもの」** | 前回と同じ。リピート注文の80% | 得意先別の注文パターン学習→予測入力 |
| **単位変換** | ケース→バラ、kg→個 | 商品マスタの入数/重量情報で自動変換 |
| **欠品対応** | 在庫なし→代替品提案 | 類似商品DB+得意先の過去購入履歴 |
| **返品** | 賞味期限切れ、破損、誤出荷 | 返品理由の記録+仕入先への請求自動連携 |
| **得意先別帳票** | 納品書のフォーマットが得意先指定 | テンプレートDB+自動生成 |
| **複数配送先** | 1注文で3店舗に分納 | 配送先別のピッキング指示分割 |
| **価格交渉** | 「今回だけ安くして」 | 臨時掛率の承認WF+実績記録 |
| **セット販売** | 5種×各2ケースのアソートセット | セット構成マスタの自動展開 |
| **先日付注文** | 来月の特売に合わせて事前発注 | 納品日指定の在庫引当+発注連携 |

## 4. シャチョツーが解決する6つの課題

### 4.1 モジュール一覧

| # | モジュール | 課題 | AI化の内容 | BPO価値/月 |
|---|---|---|---|---|
| ① | 受発注AI | FAX手入力で月100-500時間 | FAX/メールOCR→商品マスタ照合→販売管理自動転記 | ¥10-20万 |
| ② | 請求書照合AI | 仕入先との照合で月40-80時間 | 請求書OCR→発注・納品データ突合→差異アラート | ¥5-10万 |
| ③ | 在庫問い合わせ対応AI | 「在庫ある？」に月100-300回対応 | チャット/電話→商品特定→在庫DB照会→即回答 | ¥3-5万 |
| ④ | 価格表・見積作成AI | 掛率計算ミスで赤字販売 | 得意先別掛率×数量割引→自動見積書生成 | ¥3-5万 |
| ⑤ | 与信管理AI | 貸倒れで年500-3,000万円の損失 | 支払遅延パターン検知→与信限度アラート→取引条件提案 | ¥2-3万 |
| ⑥ | 発注最適化AI | 肌感覚の発注で過剰在庫 or 欠品 | 売上データ→需要予測→安全在庫→発注点アラート | ¥3-5万 |

料金: BPOコア ¥250,000/月（①-④）+ 追加モジュール ¥100,000/個/月（⑤-⑥）

### 4.2 受発注AI — キラーフィーチャー

```
Layer 0: LLMデフォルト
  FAX画像をOCRしてテキスト化。LLMが商品名・数量・得意先を推定。
  精度: 商品特定率 60-70%（略称対応が弱い）
  Confidence: 0.3-0.5
  作業: ゼロ

Layer 1: ゲノム設定（JSON）
  商品マスタ（品名/JANコード/入数/単位）、得意先マスタ、
  商品略称辞書（「サバ水」→マルハニチロ サバ水煮 190g）を定義。
  精度: 商品特定率 85-90%
  Confidence: 0.6-0.8
  作業: マスタ連携（1-2日）

Layer 2: 実績学習
  得意先別の注文パターンを学習。「A店は毎週火曜にサバ缶24缶を注文」
  → 火曜のFAXで「サバ」とだけ書いてあっても正確に特定可能。
  精度: 商品特定率 95-98%
  Confidence: 0.85-0.95
  作業: 3ヶ月の実績蓄積

+ FAXの手書き文字認識精度向上 → Confidence: 0.95+
  （確信度85%未満の項目はHITL=人間確認）
```

**解決順序（各データ項目ごと）:**

```

```

## 5. 競合比較

| | シャチョツー | CO-NECT | Bカート | COREC | 弥生/PCA |
|---|---|---|---|---|---|
| **ポジション** | BPO全域（経営OS） | BtoB受発注プラットフォーム | BtoB EC構築 | 受注管理SaaS | 販売管理パッケージ |
| **受注自動化** | ◎（FAX OCR+LLM+パターン学習） | ○（Web受注。FAX非対応） | ○（EC経由。FAX非対応） | ○（Web受注+メール取込） | ×（手入力前提） |
| **FAX対応** | ◎（OCR+商品マスタ照合） | ×（Webへの移行が前提） | × | △（メール受注のみ） | × |
| **請求照合** | ○（AI照合） | △ | × | × | △（手動照合） |
| **在庫連携** | ○（在庫DB+問い合わせAI） | △ | ○ | × | ○ |
| **与信管理** | ○（AI分析+アラート） | × | × | × | △（残高管理のみ） |
| **需要予測** | ○（AI予測+発注提案） | × | × | × | × |
| **強み** | FAX対応+6モジュール統合+学習 | UI/UX。Web受発注の使いやすさ | EC機能。カタログ型 | シンプル。低価格 | 会計連携。実績 |
| **弱み** | 新参。実績少ない | FAX対応なし（移行コスト大） | 卸専用ではない | 機能が限定的 | AI機能なし |
| **月額** | ¥3-40万 | ¥3-10万 | ¥5-30万 | ¥1-5万 | ¥1-5万 |

**シャチョツーの独自ポジション:**

1. **既存受発注SaaSは「得意先にWeb発注させる」前提。FAXのまま対応できるのはシャチョツーだけ。** 中小卸の得意先（個人飲食店等）にWeb発注を強制できない現実がある。FAXのままでいい、がキラーメッセージ。
2. **受発注だけでなく、請求照合+在庫+与信+需要予測まで一気通貫。** 別々のSaaSを4つ入れる必要がない。
3. **使うほど精度が上がる。** 得意先の注文パターン・略称・発注リズムが蓄積され、3ヶ月で「ベテラン受注担当の頭の中」をAIが再現する。
4. **ゲノム駆動。** 食品卸も建材卸も機械卸も、ゲノムJSONの切替で1つのエンジンが対応。

**正直な弱み:**
- 新参。導入実績がまだ少ない。パイロット企業での実証フェーズ
- FAXの手書き文字認識精度は80-90%。崩し字のFAXは人間確認が必要
- 既存の販売管理システムとの連携が前提。API非対応の古い基幹システムだとRPA連携が必要
- EDI（流通BMS等）対応はPhase 2+

## 6. 法令・規制環境

### 6.1 下請法（下請代金支払遅延等防止法）

```
対象: 資本金3億円超の親事業者 → 3億円以下の下請事業者
  （卸売業では、大手小売→中小卸の関係で適用されるケースあり）

禁止行為:
  ・受領拒否
  ・下請代金の支払遅延（60日以内）
  ・下請代金の減額
  ・返品
  ・買いたたき

シャチョツーの対応:
  ・支払期日の自動管理+遅延アラート
  ・取引条件の記録・エビデンス保管
```

### 6.2 独占禁止法（優越的地位の濫用）

```
対象: 取引上優越的な地位にある事業者（大手メーカー、大手小売）
  → 卸は「板挟み」。メーカーからの押し込み販売、小売からの不当返品

シャチョツーの対応:
  ・取引条件の可視化（不当な返品・値引き要求の記録）
  ・公正取引委員会への相談のためのエビデンス蓄積
```

### 6.3 食品衛生法（HACCP義務化）

```
2021年6月〜: 全食品事業者にHACCP対応義務化
  卸売業も「食品の保管・配送」工程でHACCP対応が必要

対応項目:
  ・温度管理の記録（倉庫・配送車両）
  ・衛生管理計画の策定
  ・記録の保管

シャチョツーの対応:
  ・温度記録の自動保存（IoT温度計連携）
  ・衛生管理記録のデジタル化
  ・HACCP文書の自動生成
```

### 6.4 酒税法（酒類卸の場合）

```
酒類販売業免許が必要。
  ・卸売業免許（全酒類/ビール/洋酒/輸入酒 etc.）
  ・仕入先・販売先の記帳義務
  ・酒類の流通経路管理

シャチョツーの対応:
  ・酒類取引の自動記帳
  ・免許更新の期限管理
```

### 6.5 インボイス制度・電子帳簿保存法

```
インボイス制度（2023年10月〜）:
  ・適格請求書の発行義務
  ・登録番号の記載
  ・税率区分ごとの消費税額の記載
  → 請求書AIで自動対応

電子帳簿保存法（2024年1月〜）:
  ・電子取引データの保存義務
  ・タイムスタンプ or 訂正削除履歴
  → 受発注AIで電子保存+検索機能を自動対応
```

## 8. 技術アーキテクチャ（概要）

### 8.1 ゲノム駆動型アーキテクチャ

```
卸売業ゲノム（JSON）
  ├─ 業務フロー定義（受注→在庫確認→出荷→請求）
  ├─ 業界用語辞書（掛率、建値、リベート、先日付、口座…）
  ├─ 商品マスタ構造（JANコード/品名/入数/単位/温度帯）
  ├─ 帳票フォーマット（注文書/納品書/請求書のパターン）
  ├─ 掛率計算式（基本掛率×数量割引×特売×リベート）
  └─ 業種別カスタマイズ（食品/建材/機械/繊維…）

共通エンジン: FAX OCR → 商品特定 → 在庫確認 → 受注確定 → ピッキング指示

新しい業種の追加:
  Layer 0: 即日（OCR+LLMで基本的な注文内容を構造化）
  Layer 1: 1-2日（ゲノムJSON+商品マスタ連携）
  Layer 2: 3ヶ月（得意先別注文パターンの学習）
```

### 8.2 既存パイプラインとの共通性

```
卸売業モジュール → 再利用元
  受発注AI      → construction/estimation（OCR→抽出→構造化）: 80%共通
  請求書照合AI  → construction/billing（帳票照合）: 90%共通
  在庫問い合わせ → brain/knowledge/qa（RAG Q&A）: 70%共通
  価格表・見積  → manufacturing/quoting（見積計算）: 60%共通
  与信管理      → 新規（支払パターン分析）
  発注最適化    → 新規（需要予測+在庫シミュレーション）

→ 6モジュール中4つは既存エンジンで70-90%カバー可能
→ 卸売業固有の開発は与信管理と発注最適化の2つ
```

### 8.3 学習ループ

```
1. FAX注文をAIが読取・構造化（Confidence 0.5-0.7）
2. 受注担当が確認・修正（「この略称は○○のこと」「数量はバラではなくケース」）
3. 修正データをDBに保存
4. 次回同得意先×同略称 → 過去の正解データを自動適用
5. 得意先50件以上の実績蓄積 → Confidence 0.9+
6. 「いつもの」パターン学習 → リピート注文は確認不要レベルに
```

使えば使うほど、その会社専用の受発注AIに育つ。

## 9. 導入フロー

### Phase 1（Month 1）: データ吸収

```
Week 1: アカウント開設・初期設定
  ・ブレイン基本情報入力（会社情報・倉庫・配送エリア）
  ・商品マスタ連携（既存販売管理システムからCSVエクスポート）
  ・得意先マスタ連携（得意先ランク・掛率テーブル）
  ・仕入先マスタ登録

Week 2-3: 過去データの取り込み
  ・過去3ヶ月のFAX注文書50-100枚をアップロード
  ・AIが略称辞書を自動構築（商品名の揺れパターンを学習）
  ・得意先別の注文パターンを分析

Week 4: シャドーモード開始
  ・新規FAX注文に対してAIが受注ドラフトを生成
  ・受注担当が従来通り手入力
  ・両者を比較して精度を検証
```

### Phase 2（Month 2-3）: 並走運用

```
・AIが受注ドラフト → 受注担当が確認・修正 → 販売管理に転記
・修正内容がフィードバックされ、略称辞書・パターン学習が向上
・目標: リピート注文（80%）はAI完結。新規注文は確認+修正
・月次精度レポートで改善状況を可視化
・請求書照合AIの稼働開始
```

### Phase 3（Month 4+）: 自走

```
・リピート注文はAI完結（受注確定→ピッキング指示まで自動）
・新規注文・イレギュラーのみ人間が確認
・在庫問い合わせ対応AIの稼働（チャット+電話）
・与信管理+発注最適化の段階的追加
・月次: 得意先別粗利率+商品別回転率+与信スコア分析
・四半期: 掛率見直し提案+仕入先評価+在庫最適化レポート
```
