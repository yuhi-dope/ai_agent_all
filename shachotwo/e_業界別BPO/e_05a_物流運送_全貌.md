## 1. 物流・運送業の現実（業界人なら「分かる」話）

### 1.1 中小運送会社の現場

車両10-200台、ドライバー15-250名の中小一般貨物運送事業者、約6.3万社（国交省 自動車運送事業経営実態調査）。
長距離幹線輸送、地場配送、冷凍冷蔵、危険物。業態は違っても共通する現実がある。

**社長の1日（Before）**

```
AM 5:30  出社。配車マンと今日の配送予定を確認（ホワイトボード+手書き配車表）
AM 6:00  荷主A社から電話「今日14時に追加で4パレット、急ぎで千葉まで頼めない？」
         → 配車マン「うーん、2号車の帰り荷がなければ…確認します」
         → この判断が配車マンの頭の中にしかない
AM 6:30  出庫前点呼。アルコール検知器でチェック。記録は紙の台帳
         → 30人分の点呼で1時間かかる。しかも法定保存義務がある
AM 7:00  ドライバーが出発。運行指示書を1枚ずつ手書きで渡す
         → 書き間違えても出てから気づく
AM 8:00  荷主B社のドライバーが指定時間に遅刻。電話でクレーム
         → GPSなし。ドライバーに電話しても運転中で出ない
AM 9:00  配車マンから「鈴木さん、今月の残業があと20時間しかない」
         → 手書きの出勤簿を見て計算。合っているか不安
         → 2024年4月から年960時間超えたら事業停止命令
AM 10:00 燃料高騰。今月の軽油代が先月より50万円増えている
         → 荷主にサーチャージを請求したいが、交渉の根拠データがない
PM 1:00  車検が来週の3号車の代車手配を忘れていた
         → 点検・車検・保険の期限が一覧になっていない
PM 2:00  荷主C社への月次請求書を作成。運賃体系が荷主ごとに全部違う
         → 距離制、個建て、パレット建て、時間制…。1社の請求書に3時間
PM 3:00  事故報告。12号車が接触事故。警察対応、保険会社連絡、荷主への報告
         → 安全教育の記録が揃っていない。Gマーク更新に影響する
PM 5:00  明日の配車を組む。配車マンが2時間かけてパズルを解く
         → 「7号車の佐藤さん、明日は12時間拘束だから翌日は短めに…」
         → 改善基準告示の制約を頭の中で全部計算している
PM 7:00  退社。今日も電話対応と書類作成で終わった。経営は何もできなかった。
```

### 1.3 2024年問題（業界最大のインパクト）

**改善基準告示の改正（2024年4月施行）**

```
                改正前          改正後
拘束時間（1日）  最大16時間      原則13時間（例外15時間、週2回まで）
拘束時間（月）   最大322時間     原則284時間（例外310時間）
休息期間         継続8時間以上   継続11時間以上（最低9時間）
連続運転         4時間→30分休   4時間→概ね30分の合計休憩
時間外上限       上限なし        年960時間
```

**影響の試算（国交省「物流の2024年問題」）:**
- 何も対策しなければ、2024年に14%、2030年に34%の輸送力が不足
- ドライバー1人あたりの走行可能距離が縮小
- 「今まで1人で回していた配送を2人に分けなければならない」
- 長距離運行（片道450km超）は中継輸送が必須に

**違反した場合の罰則:**
- 時間外上限超過: 6ヶ月以下の懲役 or 30万円以下の罰金
- 改善基準告示違反: 事業停止命令（=トラックが走れない=売上ゼロ）
- 国交省の行政処分（車両停止日車数の加算）

**結論:** 残業時間・拘束時間の管理が「できたらいいね」から「やらなければ事業が止まる」に変わった。手書きの出勤簿で管理している会社は、法令違反に気づかないまま処分を受けるリスクがある。

### 1.4 業界構造トレンド

**標準的な運賃告示制度**

2020年に国交省が告示。荷主に対して「適正運賃の交渉」が法的に後押しされた。
しかし現実は、告示運賃と実勢運賃の乖離が大きく（告示の70-80%水準で取引されているケースが多い）、交渉のためのデータ整備（原価計算）ができていない中小が多い。

**M&A・事業統合の加速**

ドライバー不足 × 後継者不在 × 低利益率 → 中小運送のM&Aが急増。
2025年以降、生き残る会社は「データで運営できる会社」。買い手が見るのは車両台数ではなく運営の可視化レベル。

**荷主の物流子会社のアウトソーシング**

大手メーカーの物流子会社が次々に3PL化。中小運送にとっては新規荷主獲得のチャンスだが、品質要求（GPS追跡、温度管理、定時率データ等）が高い。データ提供能力のない運送会社は取引先として選ばれない。

### 2.4 傭車（ようしゃ）の実態

```
傭車 = 自社で対応できない荷量を、他の運送会社に委託すること

中小運送の傭車比率:
  小規模（10-30台）: 10-20%
  中規模（30-100台）: 20-40%
  繁忙期: 40-60%に跳ね上がることも

課題:
  ・傭車先の品質が読めない（遅刻・破損リスク）
  ・傭車運賃が不透明（相見積もりしない。なじみの会社に電話1本）
  ・傭車と自社便の損益が混在して原価が見えない
  ・帰り荷のマッチングが属人的（配車マンの人脈依存）

シャチョツーの価値:
  ・傭車先の実績DB（品質スコア・単価・対応エリア）
  ・自社便 vs 傭車の損益シミュレーション
  ・繁忙期の傭車手配の自動化
```

## 3. 配車計画の全貌（これが分かってないと話にならない）

### 3.1 配車計画の基本フロー

```
Step 1: 荷主から配送依頼を受ける（電話/FAX/メール/EDI）
  │
Step 2: 配送依頼の一覧化（荷主×届先×荷量×車種×時間指定×特記事項）
  │      ← 現状: 紙のメモ or Excelに手入力。ここで既に1時間
  │
Step 3: ドライバーの稼働状況を確認
  │      ← 現状: 配車マンの記憶。出勤簿と照らし合わせる
  │      ← ポイント: 残業時間の残り、前日の拘束時間、休息期間の確保
  │
Step 4: 車両の割当
  │      ← 荷量→積載量、荷物の種類→車種、車検・整備予定の確認
  │
Step 5: ルートの組み立て
  │      ← 配送順序の最適化。時間指定の考慮。高速道路の使用判断
  │      ← 帰り荷がある場合はルートに組み込む
  │
Step 6: 改善基準告示の制約チェック
  │      ← 拘束時間13時間以内？連続運転4時間以内？休息11時間？
  │      ← 月間拘束時間の累計は？年間残業時間の累計は？
  │      ← ★ここを間違えると事業停止命令
  │
Step 7: 運行指示書の作成・交付
  │      ← 法定記載事項: 運行の開始/終了地点・日時、経由地、休憩地
  │
Step 8: ドライバーへの指示・出庫前点呼
```

### 3.2 運賃体系の仕組み

荷主ごとに運賃計算方法がバラバラ。これが請求業務を地獄にしている。

| 運賃体系 | 計算方法 | 使われるケース |
|---|---|---|
| **距離制** | 基本運賃 + km単価 × 距離 | 長距離幹線輸送 |
| **重量制** | 重量ランク × 距離ランク | 特積み（路線便） |
| **個建て** | 1個あたり単価 × 個数 | 食品・飲料の配送 |
| **パレット建て** | 1パレット単価 × パレット数 | 倉庫間輸送 |
| **車建て（チャーター）** | 1台あたりの定額 | 建材・引越し |
| **時間制** | 基本料金 + 時間超過料金 | 市内配送、雑貨 |

```
追加料金の例:
  時間指定料:      +5-15%
  休日・祝日割増:  +20-30%
  深夜・早朝割増:  +30-50%
  冷凍冷蔵割増:    +10-30%
  危険物割増:      +20-50%
  待機料:          ¥1,500-3,000/30分（実際には請求できないケースが多い）
  燃料サーチャージ: 基準軽油価格との差額を転嫁（計算式が荷主ごとに異なる）
```

**会社ごとに荷主別の運賃テーブルを持っている。** 同じ荷主でも年次契約更新で条件が変わる。
この運賃マスタの精度が、シャチョツーの運賃計算・請求AIの精度を決める。

### 3.3 改善基準告示の制約条件（配車AIのコア）

```
■ 1日の拘束時間
  原則: 13時間以内
  例外: 15時間まで延長可。ただし14時間超は週2回まで
  計算: 出庫〜帰庫（含む点呼・荷役作業・待機時間）

■ 1日の休息期間
  原則: 継続11時間以上
  例外: 9時間以上（ただし11時間未満は週3回まで）
  計算: 拘束終了〜次の拘束開始

■ 1ヶ月の拘束時間
  原則: 284時間以内
  例外: 労使協定で310時間まで（年6ヶ月まで）
  年間上限: 3,300時間

■ 連続運転時間
  上限: 4時間以内
  休憩: 概ね連続30分以上（分割可：1回10分以上で合計30分以上）

■ 時間外労働
  上限: 年960時間
  計算: 1日8時間 or 1週40時間を超える部分の累計

■ 長距離運行の特例
  片道450km超: 中継輸送を推奨。1運行16時間以内に努める
```

**配車AIは、この制約条件を全ドライバーについてリアルタイムで計算し、違反の可能性がある配車を自動でブロックする。** これが手計算で不可能な理由は、「今日の配車が明日以降の制約に影響する」からで、ドライバー30人分を5日先まで見通せる人間はいない。

### 3.4 配車の「例外」（ここがAI化の難所）

| パターン | 内容 | AI対応 |
|---|---|---|
| **当日の追加依頼** | 荷主から当日追加。配車の組み直しが必要 | リアルタイム再最適化 |
| **ドライバーの急な欠勤** | 体調不良、家族の事情。代わりのドライバー手配 | 代替候補を制約条件で自動フィルタ |
| **車両故障** | 出庫後に故障。別車両への積替え | 近隣の空き車両自動検索+顧客通知 |
| **帰り荷の突発確保** | 行きの配送後に帰り荷案件が入る | 位置×時間×車種でマッチング |
| **荷主の荷量変動** | 「10パレットって言ったけど15だった」 | 積載超過チェック+分割配送提案 |
| **渋滞・天候** | 高速通行止め、台風 | リアルタイムルート変更+到着時間再計算 |
| **待機時間の発生** | 荷受け先で1-2時間待ち（荷主都合） | 待機時間の記録→荷主への請求根拠データ |
| **季節変動** | 年末年始、お盆、引越しシーズン | 過去実績から荷量予測→傭車の事前手配 |
| **特殊車両の指定** | ユニック、ウイング、パワーゲート | 車種制約を加味した配車 |
| **同乗研修** | 新人ドライバーの横乗り期間 | 研修スケジュールと配車の統合管理 |

### 4.2 配車計画AI — キラーフィーチャー

```
Layer 0: LLMデフォルト
  改善基準告示の基本制約 + 一般的な配車ロジック。
  精度: 制約条件のチェックのみ。最適化は限定的。
  Confidence: 0.3-0.5
  作業: ゼロ

Layer 1: ゲノム設定（JSON）
  車両マスタ（車種/積載量）、ドライバーマスタ（資格/運行可能エリア）、
  荷主別制約（時間指定/車種指定/温度帯）を定義。
  精度: 制約遵守+基本的なルート最適化
  Confidence: 0.6-0.8
  作業: 初期設定（1-2日）

Layer 2: 実績学習
  過去の配車実績→所要時間の実績値（荷積み/荷卸し時間含む）、
  荷主別の荷量パターン、曜日/季節の変動パターンを学習。
  精度: 実態に即した最適配車
  Confidence: 0.8-0.95
  作業: 3ヶ月の実績蓄積

+ 荷主データ連携（EDI/API）→ Confidence: 0.95+
```

**解決順序（各制約ごと）:**

```
ドライバー稼働: 勤怠DB → ゲノム設定 → LLM推定
所要時間:       過去実績 → Google Maps API → LLM推定
荷量予測:       荷主EDI → 過去パターン → LLM推定
コスト最適化:   実績データ → 運賃マスタ → LLM推定
```

**シャチョツーの独自ポジション:**

1. **既存AI配車（LYNA等）は大手向け。中小10-100台に最適化されたAI配車はまだない。** シャチョツーはこの空白を取る。
2. **配車だけでなく、2024年問題（労務管理）+ 運行管理書類 + 請求まで一気通貫。** 別々のSaaSを3つ入れる必要がない。
3. **使うほど精度が上がる。** 配車実績・所要時間・荷主の荷量パターンが蓄積され、3ヶ月で「配車マンの頭の中」をAIが再現する。
4. **ゲノム駆動。** 地場配送も長距離も冷凍も、ゲノムJSONの切替で1つのエンジンが対応。

**正直な弱み:**
- 新参。導入実績がまだ少ない。パイロット企業での実証フェーズ
- リアルタイム位置追跡はGPS連携（Cariot等）に依存。自社でIoTは持たない
- 数理最適化の精度はLYNA等の専門ツールに及ばない（LLM+ヒューリスティクスで実用十分レベルを目指す）

## 7. 技術アーキテクチャ（概要）

### 7.1 ゲノム駆動型アーキテクチャ

```
物流・運送業ゲノム（JSON）
  ├─ 業務フロー定義（配車→運行→帰庫→請求）
  ├─ 業界用語辞書（傭車、積合せ、パレット建て、帰り荷…）
  ├─ 制約条件（改善基準告示の全項目）
  ├─ 車両マスタ（車種/積載量/特殊装備）
  ├─ 運賃計算式（距離制/個建て/車建て…）
  └─ 業態別カスタマイズ（冷凍/危険物/長距離…）

共通エンジン: 配送依頼入力 → 制約チェック → 配車最適化 → 運行指示書出力

新しい業態の追加:
  Layer 0: 即日（改善基準告示の制約チェックだけでも価値あり）
  Layer 1: 1-2日（ゲノムJSON設定。車両・ドライバー・荷主マスタ）
  Layer 2: 3ヶ月（実績蓄積で所要時間・荷量パターンを学習）
```

### 7.2 学習ループ

```
1. AIが配車計画を生成（Confidence 0.4-0.7）
2. 配車マンが修正（「この荷主の荷積みは1時間かかる」「この道は渋滞する」）
3. 修正データをDBに保存
4. 次回同荷主×同エリア → 過去実績の所要時間を自動適用
5. 3ヶ月分の実績蓄積 → Confidence 0.8+
6. 季節パターン学習 → 年末繁忙期の荷量予測が可能に
```

使えば使うほど、その会社専用の配車AIに育つ。

## 8. 導入フロー

### Phase 1（Month 1）: データ吸収

```
Week 1: アカウント開設・初期設定
  ・ブレイン基本情報入力（会社情報・営業所・車両一覧）
  ・車両マスタ登録（車種/積載量/車検日/特殊装備）
  ・ドライバーマスタ登録（資格/運転可能車種/住所エリア）
  ・荷主マスタ登録（運賃体系/時間指定/車種指定）

Week 2-3: 過去データの取り込み
  ・過去3ヶ月の配車実績（Excelでも可）
  ・荷主別運賃テーブル
  ・ドライバーの勤怠実績

Week 4: シャドーモード開始
  ・新規配送依頼に対してAIが配車ドラフトを生成
  ・配車マンが従来通りの配車を作成
  ・両者を比較して精度を検証（改善基準告示の遵守状況も比較）
```

### Phase 2（Month 2-3）: 並走運用

```
・AIが配車ドラフト → 配車マンが確認・修正 → 確定
・修正内容がフィードバックされ、精度が継続的に向上
・目標: 定型配送（固定ルート）はAI完結。変動配送はAIドラフト+人間修正
・月次精度レポートで改善状況を可視化
・労務管理モジュール稼働開始（改善基準告示のリアルタイム監視）
```

### Phase 3（Month 4+）: 自走

```
・定型配送はAI完結。配車マンは例外対応と品質管理に集中
・運賃計算・請求AIの稼働（月次の請求業務を自動化）
・車両管理・安全管理モジュールの段階的追加
・月次: 車両稼働率+ドライバー稼働率+荷主別収益分析
・四半期: 運賃交渉用データ+傭車比率最適化提案
```
