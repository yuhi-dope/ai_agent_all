# d_03 — 国際化設計（概要）

> **このドキュメントは方針と構造のみ。詳細設計はPhase 2以降で実施する。**
> 現時点では日本市場に集中し、国際展開時に手戻りが発生しない設計方針を定めておく。

---

## 1. 国際化の段階戦略

```
Phase 1: 日本市場制覇（現在〜Year 3）
  ├── ターゲット: 日本のSMB（10-300名）
  ├── 言語: 日本語のみ
  ├── 法域: 日本法のみ
  ├── 通貨: 円のみ
  ├── インフラ: GCP asia-northeast1（東京）
  └── ゴール: ARR ¥1B、1,000社、テンプレート15業種

Phase 2: APAC展開（Year 3〜5）
  ├── ターゲット: 韓国・台湾・シンガポール・タイのSMB
  ├── 言語: +韓国語、+繁体中国語、+英語（APAC共通）、+タイ語
  ├── 法域: +韓国PIPA、+台湾PDPA、+シンガポールPDPA、+タイPDPA
  ├── 通貨: +KRW、+TWD、+SGD、+THB
  ├── インフラ: +GCP asia-northeast3（ソウル）、+asia-southeast1（シンガポール）
  └── ゴール: APAC ARR ¥500M、3,000社

Phase 3: 英語圏展開（Year 5〜7）
  ├── ターゲット: US/UK/AU のSMB + Enterprise
  ├── 言語: +英語（US/UK/AU バリアント）
  ├── 法域: +GDPR（UK）、+CCPA/CPRA（US-CA）、+州法対応（US）、+Privacy Act（AU）
  ├── 通貨: +USD、+GBP、+AUD
  ├── インフラ: +GCP us-central1、+europe-west2（ロンドン）、+australia-southeast1
  └── ゴール: 英語圏 ARR ¥2B

Phase 4: グローバル（Year 7〜）
  ├── ターゲット: 全世界
  ├── 言語: 主要20言語
  ├── 法域: 各国データ保護法に逐次対応
  ├── インフラ: GCPマルチリージョン（10+リージョン）
  └── ゴール: グローバル ARR ¥10B+
```

### なぜAPACからか

```
1. 文化的近接性
   → 日本・韓国・台湾は「社長が全て決める」中小企業文化が共通
   → シャチョツーの「社長の頭の中をデジタル化」が刺さる土壌がある
   → 欧米は「経営チーム」文化 → ツインの対象が異なる（Phase 3で対応）

2. 法規制の類似性
   → APAC各国のデータ保護法はGDPRより軽い → 対応コストが低い
   → 業種規制も日本と類似点が多い（建設業法の構造等）

3. 市場特性
   → SMB比率が高い（韓国: 99%、タイ: 97%、台湾: 98%）
   → DX浸透率が低い → ブルーオーシャン
   → 日系企業の現地法人がアーリーアダプター候補

4. オペレーション効率
   → 時差が小さい（±2時間）→ サポート体制の共有が可能
   → ソウル・シンガポールにGCPリージョン既存
```

## 2. マルチ言語アーキテクチャ方針

### 設計原則: 「構造は1つ、表現は多言語」

```
■ Phase 1で守るべき設計制約（今すぐ）

  1. ハードコードされた日本語テキストを排除
     → UI文字列は全てi18nキー経由
     → LLMプロンプトのテンプレートは言語パラメータを受け取る構造に
     → エラーメッセージ・通知もi18nキー経由

  2. テンプレートの構造と内容の分離
     → テンプレート構造（フィールド定義・型・バリデーション）: 言語非依存
     → テンプレート内容（ラベル・説明文・デフォルト値）: 言語別ファイル
     → genome_templates テーブルに locale カラムを追加（Phase 2で有効化）

  3. 文字コード・フォーマット
     → 全てUTF-8
     → 日付: ISO 8601（表示時にロケール変換）
     → 数値: 内部は小数点、表示時にロケール変換（1,000 vs 1.000）

■ Phase 2で実装するもの

  1. LLMプロンプトの多言語化
     → system_prompt に {language} パラメータ
     → 言語別のプロンプトチューニング（日本語と英語でLLMの得意不得意が異なる）
     → Gemini: 日本語○ / 韓国語○ / タイ語△ → 言語別モデル選択ロジック

  2. Q&A応答の言語自動検出・切替
     → ユーザー入力の言語を自動検出（langdetect）
     → ナレッジは元言語で保持 + 回答時にユーザー言語に翻訳
     → 翻訳品質の担保: 業界用語辞書（言語ペア別）

  3. テンプレート翻訳パイプライン
     → 日本語テンプレート → LLM翻訳 + 現地専門家レビュー → 公開
     → 翻訳メモリ（TMX）の蓄積で品質・速度を向上
```

## 3. マルチ法域対応方針

### テンプレートの法域レイヤー

```
テンプレート構造:
  共通ベース（法域非依存）
    → 経営管理・組織・業務フローの「構造」は万国共通
    → 例: 「経費精算フロー」の構造（申請→承認→支払い）は同じ

  + 法域レイヤー（国別）
    → 承認閾値・税率・届出先・書式が異なる
    → 例: 日本の源泉徴収 vs 米国のW-2 vs 韓国の甲種勤労所得

  + 業種×法域レイヤー（国×業種）
    → 例: 建設業 × 日本 = 建設業法 / 建設業 × 韓国 = 건설산업기본법
```

### データ保護法対応マトリクス

| 法律 | 対象国 | Phase | 主な要件 | 影響範囲 |
|---|---|---|---|---|
| 個人情報保護法 | 日本 | 1 | 利用目的特定、同意取得、安全管理措置 | 対応済み（a_02） |
| PIPA | 韓国 | 2 | 同意取得、データローカライゼーション | DB韓国リージョン必須 |
| PDPA (TW) | 台湾 | 2 | 同意取得、越境移転制限 | 契約条項追加 |
| PDPA (SG) | シンガポール | 2 | DPO任命、DPIA実施 | 組織体制追加 |
| PDPA (TH) | タイ | 2 | 同意取得、データ主体の権利 | UI追加（権利行使機能） |
| GDPR | EU/UK | 3 | DPO、DPIA、72h通知、忘れられる権利 | アーキテクチャ変更大 |
| CCPA/CPRA | US-CA | 3 | オプトアウト権、データ販売禁止 | UI追加 + ポリシー変更 |

### データレジデンシー方針

```
原則: ユーザーデータは契約リージョンに物理保存

Phase 1: 東京リージョン（全データ）
Phase 2:
  → 韓国企業 → ソウルリージョン
  → ASEAN企業 → シンガポールリージョン
  → 日本企業 → 東京リージョン（変更なし）
Phase 3:
  → US企業 → US-Centralリージョン
  → UK/EU企業 → ロンドンリージョン

実装方針:
  → Supabase: リージョン別プロジェクト（Phase 2まで）
  → GCP移行後: Cloud SQL + リージョンルーティング
  → テナント作成時にリージョンを確定、変更不可（移行は特別対応）
```

## 4. マルチ通貨・会計基準

```
■ 通貨対応

  内部表現: 全て最小単位の整数（円=1、ドル=セント、ウォン=1）
  為替: 表示用のみ。テナントの基準通貨で管理（通貨変換は行わない）
  テンプレートの金額: 通貨パラメータ化
    → 「経費精算の閾値: {amount: 10000, currency: "JPY"}」
    → 国別テンプレートで適切な金額に設定

■ 会計基準

  Phase 1: 日本基準のみ
  Phase 2: 韓国K-IFRS、台湾TIFRS、シンガポールSFRS、タイTFRS
  Phase 3: US-GAAP、UK-GAAP、IFRS

  実装: 会計テンプレートを基準別に用意
    → 勘定科目体系が異なる（日本: 5分類 / IFRS: 異なる分類体系）
    → 決算スケジュールが異なる（日本: 3月決算多 / 米国: 12月決算多）
    → 税務カレンダーが完全に異なる

■ Phase 1で守るべき制約
  → 金額フィールドにcurrencyを持たせる（デフォルト"JPY"）
  → 会計関連テンプレートに accounting_standard フィールドを持たせる
  → 税務カレンダーをハードコードしない（テンプレートデータとして管理）
```

## 5. ゲノムの国際展開

### テンプレート階層の国際版

```
Layer 0: Universal（全世界共通）
  → 経営の本質: 戦略策定、組織運営、財務管理、顧客管理
  → 業務フローの「構造」: 受注→生産→納品→請求の流れは万国共通
  → これが共通ベーステンプレートの骨格

Layer 1: Region（地域共通）
  → APAC共通: 年功序列的な組織文化、オーナー経営
  → 欧米共通: ジョブ型雇用、フラット組織
  → これがテンプレートカスタマイズの方向性を決める

Layer 2: Country（国別）
  → 法規制、税制、商慣習、言語
  → 例: 日本の「判子文化」、韓国の「빨리빨리（パルリパルリ）文化」

Layer 3: Industry × Country（業種×国）
  → 業種固有の規制が国ごとに異なる
  → 例: 建設業 × 日本 = 建設業法 / 建設業 × タイ = Building Control Act

Layer 4: Company（企業固有）
  → 現在のゲノムカスタマイズと同じ
  → 国際版でも変わらない
```

### 遺伝子合成の国際版

```
国内合成（現在）:
  日本の建設業A社 × 日本の製造業B社 → 共通パターン抽出

国際合成（Phase 2以降）:
  日本の製造業 × タイの製造業 → 「製造業の普遍的ベストプラクティス」抽出
  → 日本のカイゼン文化 → タイの製造業テンプレートに反映
  → タイの多国籍労働者管理 → 日本の外国人技能実習生管理テンプレートに反映

価値: 「日本で成功したパターンをASEANに輸出する」
  → 日本の製造業は世界最高水準 → そのナレッジがテンプレートとして他国に展開される
  → これが「日本発のグローバルプラットフォーム」としてのストーリーになる
```

## 6. 技術インフラ方針

### マルチリージョン展開

```
Phase 1: 単一リージョン
  → GCP asia-northeast1（東京）
  → Supabase（東京リージョン）
  → 全テナントが同一インフラ

Phase 2: 3リージョン
  → asia-northeast1（東京）: 日本テナント
  → asia-northeast3（ソウル）: 韓国テナント
  → asia-southeast1（シンガポール）: ASEAN テナント
  → リージョン間でテンプレート・匿名データのみ同期
  → テナントデータは越境しない

Phase 3: 5リージョン
  → +us-central1（US テナント）
  → +europe-west2（UK/EU テナント）

Phase 4: 10+リージョン
  → 需要に応じて追加
```

### レイテンシ要件

```
Q&A応答: < 3秒（LLM応答含む）
  → ユーザーの最寄りリージョンで処理
  → LLM APIは各リージョンから最寄りのエンドポイントを呼ぶ

BPO実行: < 10秒（SaaS API呼び出し含む）
  → SaaS APIのリージョンに依存するため、制御限界あり

テンプレート同期: < 1時間
  → テンプレート更新は非同期でグローバル配布
```

### Phase 1で守るべきインフラ設計制約

```
1. テナントIDにリージョン情報を含めない
   → 将来のリージョン移行を可能にするため
   → リージョンはテナントメタデータとして別管理

2. 外部サービス呼び出しをリージョン非依存に
   → LLM APIのエンドポイントを設定ファイルで管理
   → ハードコードしない

3. タイムゾーンの明示管理
   → 全てUTC保存、表示時にテナントのTZで変換
   → cron系処理もUTC基準

4. ログ・監査証跡のリージョン対応
   → ログにリージョン情報を含める
   → 将来のリージョン別監査に備える
```

## 7. Phase 1での実施事項（今やること）

上記の国際化設計を踏まえ、Phase 1（日本市場集中）で**今から守るべき設計制約**：

| # | 制約 | 理由 | コスト |
|---|---|---|---|
| 1 | UI文字列のi18nキー化 | Phase 2でのUI翻訳を容易にする | 低（開発習慣の問題） |
| 2 | 金額フィールドにcurrency属性 | マルチ通貨対応の基盤 | 極低（カラム1つ追加） |
| 3 | テンプレートにlocale属性 | 多言語テンプレートの基盤 | 極低（カラム1つ追加） |
| 4 | LLMプロンプトのlanguageパラメータ化 | 多言語Q&Aの基盤 | 低（変数1つ追加） |
| 5 | 日付/時刻のUTC保存 | マルチTZ対応の基盤 | 低（設計方針の問題） |
| 6 | テナントメタデータにregion属性 | マルチリージョンの基盤 | 極低（カラム1つ追加） |

> これら6つの制約を守るだけで、Phase 2での国際化コストが1/10になる。
