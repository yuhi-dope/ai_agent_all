## 1. Level 3: Agent Economy（エージェント経済圏）

### 1.1 Agent Marketplace

エージェント同士が取引・連携する場。

```
建設エージェント: 「鉄筋100t必要」
    ↓ Agent Marketplace
卸売エージェント: 「在庫あり。見積¥9,500,000」
    ↓
建設エージェント: 「発注します」→ 自動決済
```

#### 取引フロー

```
① 需要発信: エージェントが「○○が必要」をMarketplaceに投稿
② マッチング: 対応可能なエージェントを自動検索
③ 見積: 供給側エージェントが見積を自動生成
④ 交渉: 価格・納期の自動交渉（ルール範囲内）
⑤ 合意: 双方のエージェントが合意 → 人間承認（金額閾値超の場合）
⑥ 実行: 発注→納品→検収→決済を自動実行
⑦ 評価: 取引結果をTrust Networkに記録
```

### 1.2 通信タイプ

| タイプ | 説明 | Phase |
|---|---|---|
| 同一顧客内リレー | 積算→経理→支払のタスク連鎖 | Phase 2（d_04 §5） |
| 同一業界ベンチマーク | 匿名化された業界平均共有 | Phase 3 |
| 異業界間取引 | 建設←→卸売、製造←→物流 | Phase 3 |
| 外部システム連携 | 他社AIシステムとの標準プロトコル接続 | Phase 4+ |

## 2. Genome Marketplace

業界ゲノムの売買・共有・進化の場。

### 2.1 ゲノムの流通

| 種類 | 説明 | 例 |
|---|---|---|
| 公式ゲノム | シャチョツー本体が提供 | construction.json, wholesale.json |
| パートナーゲノム | 子会社が作成・販売 | 特殊建設業（トンネル工事）向けゲノム |
| コミュニティゲノム | ユーザーが作成・共有 | 小規模工務店向けの簡易版 |

### 2.2 品質保証

- ゲノムの品質スコア（ベンチマーク精度で算出）
- レビュー・評価制度
- 公式認定制度（シャチョツー品質認定マーク）

### 2.3 収益モデル

- ゲノム販売: 作成者に売上の70%
- プラットフォーム手数料: 30%
- サブスクリプション: 継続利用で月額課金

## 3. Trust Network（信頼ネットワーク）

### 3.1 エージェント認証

```
認証情報:
├── エージェントID: {company_id}/{industry}/{agent_type}
├── 信頼スコア: 0-100（取引実績から算出）
├── 精度証明: 直近90日の精度データ
├── 稼働率: 直近90日の稼働率
└── 認定: シャチョツー認定 / パートナー認定 / 未認定
```

### 3.2 信頼スコアの算出

```
信頼スコア =
  取引成功率(40%) +
  精度(30%) +
  稼働率(15%) +
  紛争なし率(15%)
```

### 3.3 紛争解決

| 紛争レベル | 対応 |
|---|---|
| Level 1 | 自動仲裁（ルールベース） |
| Level 2 | AI仲裁（過去事例ベース） |
| Level 3 | 人間仲裁（エスカレーション） |

## 4. Level 4: Agent Governance（自律的統治）

### 4.1 独禁法準拠の監視

- 同業種エージェント間で価格情報を共有していないか
- 匿名化が本当に匿名か（k-匿名性検証）
- 特定の子会社に有利なルールになっていないか

### 4.2 システミックリスク防止

- 1社のAgent OS障害が経済圏全体に波及しないか
- エージェント間の依存度が高すぎないか（集中リスク監視）
- Circuit Breakerの経済圏レベル版

### 4.3 進化の方向制御

- Constitutional AI的な原則を経済圏全体に適用
- 「顧客利益最優先」「法令遵守」「公正取引」の3原則
- エージェントが利益最大化に最適化しすぎないよう制約

## 5. Level 5: Agent Evolution（自己進化）

### 5.1 新ゲノムの自動生成

```
トリガー: 既存ゲノムのどれにも合わない業務パターンが一定数蓄積
  ↓
分析: 「この業務群は既存の建設ゲノムのA部分と卸売ゲノムのB部分の組み合わせ」
  ↓
生成: 新ゲノムのドラフトを自動作成
  ↓
テスト: ベンチマークで精度検証
  ↓
提案: 人間に「新しい業種ゲノムを作成しました」と提案
```

### 5.2 新エンジンタイプの自動設計

- 既存5エンジンでカバーできない処理パターンを検出
- 新エンジンの仕様を自動設計
- コード生成→テスト→人間レビュー

### 5.3 アーキテクチャの自己改善

- メモリGC頻度の自動チューニング
- 品質ゲートの閾値自動調整
- コスト最適化パラメータの自動学習

### 5.4 前提条件

- 12ヶ月以上の運用データ
- 100社以上の顧客データ（パターンの多様性）
- OS Level 3の精度が95%以上で安定
- LLMがコード生成+テスト生成を高精度で実行可能

## 7. 自社Agent OS設計

シャチョツー社自身がAgent OSを使って運営する（ドッグフーディング）。

### 7.1 8エージェント体制

| エージェント | 役割 | 使うパイプライン |
|---|---|---|
| CEO Agent | KPIダッシュボード・What-if・競合監視 | brain/visualization |
| Sales Agent | リード→アウトリーチ→商談→契約 | sales/* パイプライン |
| CS Agent | ヘルススコア→介入→問い合わせ→解約防止 | cs/* パイプライン |
| Product Agent | 精度監視→改善→A/Bテスト→ゲノム進化 | evaluation engine |
| Dev Agent | バグトリアージ→修正→テスト→デプロイ | Claude Code連携 |
| Finance Agent | 請求→入金→PL/BS→税務 | common/expense |
| HR Agent | 採用→スクリーニング→日程→勤怠 | common/attendance |
| Meta Agent | 全エージェント統治・矛盾解決・自己進化 | agent_os/* |

### 7.2 同時並行ビルド戦略

| シャチョツー本体 | 自社Agent OS |
|---|---|
| Phase 1: ゲノム駆動化 | Finance Agent（請求自動化） |
| Phase 2前半: Harness | CS Agent（ヘルススコア） |
| Phase 2後半: OSコア | Sales + Dev + Product Agent |
| Phase 3: Economy | Meta + CEO + HR Agent |

共有基盤（Harness/OS/ゲノムエンジン/オブザーバビリティ）は1回作れば両方で使える。
