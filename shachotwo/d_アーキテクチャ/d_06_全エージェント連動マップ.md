## 1. 7層アーキテクチャ総覧

> パイプライン層を「全社共通基盤（Layer 3a）」と「業界特化プラグイン（Layer 3b）」に分離。
> 詳細設計: d_08 Section 0

```
Layer 1: API Router（25個）
  ユーザー / Webhook → FastAPIエンドポイント → BPOTask生成

Layer 2: BPO Manager（6個）
  TaskRouter（2層レジストリ）→ 承認エンジン → 実行制御

Layer 3a: 全社共通基盤 Base OS（36本）← 全テナント自動有効化
  営業SFA/CRM/CS 12本 + バックオフィス 24本

Layer 3b: 業界特化プラグイン Industry Plugin ← 業界選択で有効化
  建設8本 / 製造8本 / 医療 / 介護 / 不動産 / 物流 + 凍結11業種

Layer 4: Micro Agent（20個）← 業界中立の原子操作のみ
  OCR / 抽出 / 生成 / 計算 / 判定 / 通知 etc.

Layer 5: Connector（8個）← 外部SaaS必須3件 + オプション同期5件
  必須: CloudSign / Bank API / gBizINFO
  オプション: freee / SmartHR / kintone / Slack / Gmail

Layer 6: Brain（10モジュール）
  知識抽出 / 検索 / ゲノム / 推論 / ツイン / 能動提案 etc.

Layer 7: DB（Supabase + pgvector）
  既存22テーブル + スタンドアロン強化47テーブル = 69テーブル
```

## 3. Layer 2: BPO Manager（司令塔 — 6エージェント）

| エージェント | ファイル | 役割 | 連動先 |
|---|---|---|---|
| **TaskRouter** | `manager/task_router.py` | パイプライン登録・実行ディスパッチ・同時実行制御（max 3/tenant） | 全52パイプライン |
| **EventListener** | `manager/event_listener.py` | Webhook/イベント → BPOTask変換。lead_created, contract_signed 等を監視 | TaskRouter |
| **ScheduleWatcher** | `manager/schedule_watcher.py` | Cron式スケジュール → BPOTask変換 | TaskRouter |
| **ProactiveScanner** | `manager/proactive_scanner.py` | 現在状態からknowledge_itemsの条件を評価し次アクション提案。低リスク提案は自動実行 | TaskRouter + Brain |
| **ConditionEvaluator** | `manager/condition_evaluator.py` | knowledge_relationsの連鎖ルール評価 | TaskRouter |
| **ApprovalWorkflow** | `engine/approval_workflow.py` | 信頼度スコア計算（直近100件のbpo_approvals）+ 承認要否判定 | TaskRouter |

### 実行レベル制御

```
Level 0: READ_ONLY        → 常に自動実行（通知のみ）
Level 1: DATA_COLLECT      → 常に自動実行（データ集計）
Level 2: NOTIFY/DRAFT      → impact > 0.7 なら承認必須
Level 3: APPROVAL_GATED    → 常に承認必須
Level 4: AUTONOMOUS        → trust_score ≥ 0.95 なら承認不要
```

### 信頼度スコア（Trust Score）

```
入力: 直近100件の承認/却下/修正記録
計算: 承認率 × 0.4 + 連続成功 × 0.3 + (1-修正率) × 0.3
出力: weighted_score 0.0〜1.0 + level 0〜4
効果: score上昇 → 承認不要範囲が拡大 → 自律度UP
```
