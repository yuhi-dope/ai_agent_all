# ADR-007: RAGパイプライン強化（クエリ拡張 + 再ランキング）

## ステータス

**承認** (2026-03-20)

## 日付

2026-03-20

## コンテキスト

現在のhybrid_search（pgvector HNSW + ILIKE fallback）は2022年水準のRAGに留まっている。
業務ナレッジQ&Aにおける具体的な問題:

1. **埋め込みミスマッチ**: ユーザーの質問クエリと、ナレッジとして登録されたテキストの語彙・表現が異なるケースで検索精度が下がる（例: 「足場の費用は？」→ ナレッジには「仮設工事 鋼管枠組足場 ¥1,200/m²」と登録）
2. **top-1信仰**: ANN検索のtop-1が正解でないケースでも、そのまま回答生成に使われる
3. **Eval不在**: 精度が下がってもモニタリングできない。パイロット中に静かに劣化する

Anthropic/OpenAI水準の本番RAGパイプライン:
```
クエリ → クエリ拡張(HyDE) → ANN検索(top-10) → 再ランキング → top-k → LLM回答
```

## 検討した選択肢

### 選択肢A: 現状維持（hybrid_searchのまま）
- メリット: 実装コストゼロ、レイテンシ低
- デメリット: Q&A精度の上限が低い。パイロット企業が「答えが的外れ」と感じてチャーン

### 選択肢B: Cross-Encoderによる再ランキング（外部モデル）
- メリット: 業界標準の高精度再ランキング
- デメリット: 別のモデルAPIが必要（ms-marco等）。日本語対応が弱い

### 選択肢C: LLMベースのクエリ拡張 + LLM再ランキング（採用）
- メリット: 既存のLLMクライアントで完結。日本語業界用語に対応。実装がシンプル
- デメリット: 1クエリあたり追加LLMコスト（+約¥1〜2）。レイテンシ+500ms程度

### 選択肢D: ColBERTスタイルのlate interaction
- メリット: 理論上最高精度
- デメリット: Phase 1で過剰。日本語モデルが成熟していない

## 決定

**選択肢C**を採用する。

具体的な実装:
1. `expand_query()`: クエリから仮想的な回答テキストをLLM生成 → 元クエリと結合してembedding
2. `rerank_results()`: top-10の検索結果をLLMで関連度スコアリング → top-kに絞る
3. `enhanced_search()`: 上記2つを組み合わせた新しいエントリポイント
4. `answer_question(use_enhanced_search=True)`: デフォルトで拡張検索を使用
5. `eval.py`: ゴールデンデータセットによる精度測定フレームワーク

## 影響

- **Q&Aコスト増加**: 1クエリあたり+¥1〜3（クエリ拡張: FAST tier × 1回 + 再ランキング: FAST tier × 1回）。月100クエリ/テナントなら+¥300程度。許容範囲
- **レイテンシ増加**: +500ms〜1秒（LLM2回追加）。ユーザー体験上許容範囲
- **後方互換**: `hybrid_search`は変更なし。`use_enhanced_search=False`で旧動作を維持可能
- **Eval基盤**: `eval.py`によりパイロット後のQ&A精度劣化を週次で検出可能
