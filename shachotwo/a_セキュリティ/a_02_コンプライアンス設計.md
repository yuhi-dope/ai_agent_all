```

  → 不正競争防止法（営業秘密の集積）
```

## §2. BPO操作系法令

### §2.1 電子署名法・電子記録

**問題の本質:**

> BPOが「社長の代わりに」契約書送付・発注確定・採用通知を実行する場合、
> その操作の法的主体は「社長」か「シャチョツー（AI）」か？

```
法的リスク分類:
┌──────────────────┬──────────────────────────────────────────────┐
│ 操作タイプ        │ 法的リスクと対策                              │
├──────────────────┼──────────────────────────────────────────────┤
│ 情報の読み取り    │ リスク低 — 参照のみ、承認不要               │
│ (Level 0-1)      │                                              │
├──────────────────┼──────────────────────────────────────────────┤
│ 下書き作成        │ リスク低 — 最終確認は人間                    │
│ (Level 2)        │ 対策: 送信前に必ず人間が目視確認             │
├──────────────────┼──────────────────────────────────────────────┤
│ 社内承認フロー    │ リスク中 — 社内手続きの代行                  │
│ の実行 (Level 3) │ 対策: LINE承認 + 操作ログ = 社長意思の証跡   │
├──────────────────┼──────────────────────────────────────────────┤
│ 対外的な法的行為  │ リスク高 — 契約締結・手形振出等             │
│ (Level 3-4)      │ 対策: 「確認済み」の電子証跡 + 公証可能な形式 │
│                  │ → 電子署名法§3: 本人による電子署名の効力     │
│                  │ → LINE承認 = 二要素認証 = 本人確認とみなせる │
└──────────────────┴──────────────────────────────────────────────┘
```

**BPO承認の証跡設計（法的証拠能力）:**

```python
# BPO承認レコードに法的証拠能力を持たせる設計
class BPOApprovalRecord:
    """
    電子署名法 §3 への対応:

    signature_hash: str     # SHA256(company_id + approver_id + action_detail + timestamp)

    retention_until: date
```

### §2.2 電子帳簿保存法（2024年完全義務化対応）

**シャチョツーBPOが扱う対象書類:**

```
電子帳簿保存法の対象 × BPOの操作:
┌───────────────────┬────────────────┬──────────────────────────────┐
│ 書類種別           │ BPO操作        │ 保存要件                      │
├───────────────────┼────────────────┼──────────────────────────────┤
│ 請求書（受取）      │ 自動仕訳入力   │ 7年保存、改ざん防止、検索可能  │
│ 請求書（発行）      │ 自動作成・送付 │ 7年保存、タイムスタンプ        │
│ 契約書             │ 送付・署名依頼 │ 7年保存、真正性確保            │
│ 領収書             │ 経費精算入力   │ 7年保存、スキャン保存対応      │
│ 見積書             │ 自動作成       │ 5年保存                       │
│ 注文書・注文請書    │ 自動作成       │ 7年保存                       │
└───────────────────┴────────────────┴──────────────────────────────┘
```

**実装要件:**

```python
# brain/ingestion/document_archive.py

DOCUMENT_RETENTION_RULES = {
    "invoice_received":   {"years": 7, "timestamp_required": True},
    "invoice_issued":     {"years": 7, "timestamp_required": True},
    "contract":           {"years": 7, "timestamp_required": True},
    "receipt":            {"years": 7, "timestamp_required": True},
    "estimate":           {"years": 5, "timestamp_required": False},
    "purchase_order":     {"years": 7, "timestamp_required": True},
}

class ElectronicBookkeepingCompliance:
    """
    電帳法 第7条「電子取引データ保存」対応
    要件: ① 真実性確保 ② 可視性確保 ③ 検索機能
    """

    async def archive_document(
        self,
        company_id: str,
        doc_type: str,
        content: bytes,
        metadata: dict
    ) -> ArchiveResult:
        rule = DOCUMENT_RETENTION_RULES[doc_type]

        # ① 真実性確保: タイムスタンプ + ハッシュ
        doc_hash = hashlib.sha256(content).hexdigest()
        timestamp = await self.tsa_client.get_timestamp(doc_hash)  # RFC 3161準拠

        # ② 可視性確保: PDF変換
        pdf_content = await self.to_pdf(content)

        # ③ 検索機能: OCRテキスト + メタデータインデックス
        ocr_text = await self.ocr_client.extract(content)

        # Supabase Storage に保存 (7年 = retention_until)
        retention_until = date.today() + timedelta(days=365 * rule["years"])

        return await db.document_archives.insert({
            "company_id": company_id,
            "doc_type": doc_type,
            "content_hash": doc_hash,
            "timestamp_token": timestamp,
            "ocr_text": ocr_text,
            "metadata": metadata,
            "retention_until": retention_until,
        })
```

```sql
-- 電子帳簿保存法対応アーカイブテーブル
CREATE TABLE document_archives (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    company_id      UUID NOT NULL REFERENCES companies(id),
    doc_type        TEXT NOT NULL,
    storage_path    TEXT NOT NULL,    -- Supabase Storage パス
    content_hash    TEXT NOT NULL,    -- SHA256 (改ざん検知)
    timestamp_token TEXT,             -- RFC 3161 タイムスタンプトークン
    ocr_text        TEXT,             -- 全文検索用
    metadata        JSONB,            -- 取引先名・金額・日付等
    retention_until DATE NOT NULL,    -- 保存期限
    created_at      TIMESTAMPTZ DEFAULT now(),

    -- 削除不可（保存期限まで）
    CONSTRAINT no_early_delete CHECK (retention_until > CURRENT_DATE)
);
CREATE INDEX idx_doc_archives_search ON document_archives
    USING GIN(to_tsvector('japanese', ocr_text));
ALTER TABLE document_archives ENABLE ROW LEVEL SECURITY;
```

### §2.3 資金決済法・銀行法

**問題:**

> BPOが「振込指示」を自動実行する場合、これは「為替取引の媒介」にあたる可能性があり、
> 銀行業の免許または資金移動業の登録が必要になりえる。

```
法的解釈の境界線:
┌────────────────────────────────────────────────────────┐
│ 「振込指示の実行代行」≠「資金移動」                     │
│                                                        │
│ シャチョツーの設計:                                     │
│   ✅ 「社長の代わりにインターネットバンキングの           │
│       ボタンを押す」= 操作代行（代理）                   │
│   ✅ 資金の保管・移転を行わない                         │
│   ✅ 振込指示は社長が承認したもののみ実行               │
│   ❌ 振込資金の一時預かりは行わない                     │
│   ❌ 複数社の資金をプールしない                         │
│                                                        │
│ → 「電子計算機使用詐欺罪」リスク回避のため              │
│   BPO実行前に必ず承認者の意思確認を記録                 │
└────────────────────────────────────────────────────────┘
```

**安全設計（弁護士確認推奨）:**


### §2.4 下請法（下請代金支払遅延等防止法）

**シャチョツーが建設業・製造業顧客に提供するBPO操作との関係:**

```
下請法 適用チェック (BPO自動化の際に確認):
□ 発注書の交付義務 — 書面(電磁的記録)での発注書作成BPOは下請法§3準拠
□ 支払期日の確認 — 給付受領日から60日以内の支払期日設定チェック
□ 減額・返品の禁止 — BPO処理で不当な値引き操作をしないロック
□ 書類の保存義務 — 発注書・受領書を2年間保存 (document_archives テーブル)
```

```python
# workers/bpo/payment_validator.py
async def validate_subcontract_payment(payment: Payment) -> ValidationResult:
    """下請法コンプライアンスチェック"""
    issues = []

    # 支払期日チェック（受領日から60日以内）
    if payment.due_date > payment.receipt_date + timedelta(days=60):
        issues.append(ComplianceIssue(
            law="下請法 第2条の2",
            severity="high",
            message=f"支払期日が受領日から60日を超過しています: {payment.due_date}"
        ))

    # 不当減額チェック
    if payment.amount < payment.original_amount * 0.97:
        issues.append(ComplianceIssue(
            law="下請法 第4条1項3号",
            severity="high",
            message=f"3%超の減額は下請法違反の可能性があります"
        ))

    return ValidationResult(passed=len(issues) == 0, issues=issues)
```

```

    ・パートナー経由の顧客が直販に切り替え → パートナーへの30日前通知
```

## §11. インボイス制度（適格請求書等保存方式）

### 11.1 制度概要とシャチョツーへの影響

> 2023年10月1日より完全義務化。BPOで請求書処理を行う場合、
> 適格請求書（インボイス）の形式チェックと登録番号の確認が必須。

```
インボイス制度とBPOの関係:

① BPOで「請求書を受け取って仕訳入力する」場合
   → 適格請求書か否かの確認が必須
   → 非適格請求書は仕入税額控除が受けられない
   → 顧客の税務リスクに直結

② BPOで「請求書を発行・送付する」場合
   → 顧客が適格請求書発行事業者か確認
   → 適格請求書の必須記載事項を自動チェック

③ 免税事業者の顧客（年商1,000万円以下）
   → インボイス登録していない可能性
   → シャチョツーからのガイダンスが必要
```

### 11.2 適格請求書の必須チェック項目

```python
# workers/bpo/invoice_validator.py

INVOICE_REQUIRED_FIELDS = {
    "qualified_invoice": {  # 適格請求書（インボイス）
        "required": [
            "issuer_name",              # 発行者の氏名・名称
            "registration_number",       # 適格請求書発行事業者登録番号 (T+13桁)
            "transaction_date",          # 取引年月日
            "transaction_description",   # 取引内容
            "amount_by_tax_rate",        # 税率別合計額
            "tax_amount_by_rate",        # 適用税率・消費税額
            "recipient_name",            # 宛名（書類の交付を受ける事業者）
        ],
    },
    "simplified_invoice": {  # 簡易インボイス（小売等）
        "required": [
            "issuer_name",
            "registration_number",
            "transaction_date",
            "transaction_description",
            "amount_by_tax_rate",
            "tax_amount_by_rate",
            # 宛名不要（簡易の場合）
        ],
    },
}

REGISTRATION_NUMBER_PATTERN = r"^T\d{13}$"  # T + 13桁数字

async def validate_invoice_compliance(
    invoice: Invoice,
    company_id: str,
) -> InvoiceValidationResult:
    """
    BPOで請求書を受け取った際の適格性チェック
    """
    issues = []

    # 登録番号形式チェック
    reg_num = invoice.registration_number
    if not re.match(REGISTRATION_NUMBER_PATTERN, reg_num or ""):
        issues.append(InvoiceIssue(
            field="registration_number",
            severity="high",
            message="適格請求書発行事業者登録番号の形式が不正です",
            impact="仕入税額控除が受けられない可能性があります",
        ))
    else:
        # 国税庁適格請求書発行事業者公表サイトでの確認
        is_valid = await check_nta_registration(reg_num)
        if not is_valid:
            issues.append(InvoiceIssue(
                field="registration_number",
                severity="high",
                message=f"登録番号 {reg_num} は国税庁公表サイトに登録されていません",
                impact="仕入税額控除の否認リスクあり",
            ))

    # 税率別金額チェック
    if not invoice.amount_by_tax_rate:
        issues.append(InvoiceIssue(
            field="amount_by_tax_rate",
            severity="medium",
            message="軽減税率（8%）と標準税率（10%）の区分記載がありません",
        ))

    # 消費税額チェック（計算誤差 ±1円以内）
    calculated_tax = sum([
        amount * rate
        for amount, rate in invoice.tax_breakdown.items()
    ])
    if abs(calculated_tax - invoice.stated_tax) > 1:
        issues.append(InvoiceIssue(
            field="tax_amount",
            severity="medium",
            message=f"消費税額が計算値と一致しません: 記載{invoice.stated_tax}円 vs 計算{calculated_tax:.0f}円",
        ))

    return InvoiceValidationResult(
        is_qualified=len([i for i in issues if i.severity == "high"]) == 0,
        issues=issues,
        recommended_action="適格請求書として処理可能" if not issues else "発行者に再発行を依頼してください",
    )
```

### 11.3 国税庁API連携

```python
# 国税庁 適格請求書発行事業者公表サイト Web-API
NTA_API_URL = "https://web-api.invoice-kohyo.nta.go.jp/pub/v1/api/invoice/T{registration_number}"

async def check_nta_registration(registration_number: str) -> bool:
    """
    国税庁公表APIで登録番号の有効性を確認
    キャッシュTTL: 24時間（日次更新）
    """
    cache_key = f"nta_invoice:{registration_number}"
    cached = await redis.get(cache_key)
    if cached:
        return cached == "valid"

    async with httpx.AsyncClient() as client:
        response = await client.get(
            NTA_API_URL.format(registration_number=registration_number[1:])  # T除く
        )
        is_valid = response.status_code == 200 and response.json().get("count", 0) > 0

    await redis.setex(cache_key, 86400, "valid" if is_valid else "invalid")
    return is_valid
```

## §13. 景品表示法（AI能力の誇大広告）

### 13.1 規制概要

> 景品表示法 §4 「不当表示の禁止」
> 優良誤認表示（品質・性能を実際より著しく優良と示す表示）は禁止。
> AI製品の「完全自動化」「100%精度」等の表現が特に問題になりやすい。

### 13.2 禁止・要注意表現リスト

| 表現 | リスク | 代替表現 |
|---|---|---|
| ❌「完全自動化」 | 実際はLevel 0-4段階があり完全ではない | ✅「段階的な自動化を実現」 |
| ❌「AIが全業務を代行」 | 人間の承認なしには動かない | ✅「人間の承認のもとAIが業務を支援」 |
| ❌「99%の精度」 | 根拠のない数値 | ✅「一般的なBPOタスクで高精度を実現（弊社検証）」 |
| ❌「コンサルタントが不要に」 | 複雑な案件は専門家が必要 | ✅「日常的な判断・作業をAIが支援」 |
| ❌「瞬時に社長の代わりに判断」 | 実際は知識蓄積が必要 | ✅「蓄積した経営知識をもとにAIが提案」 |
| ❌「導入したその日から全自動」 | 初期学習・設定が必要 | ✅「段階的に自動化範囲を拡大」 |
| ❌「業界No.1の自動化率」 | 比較根拠が必要 | ✅「業界最高水準を目指した設計」(根拠なければ使用不可) |

### 13.3 マーケティング資料の事前チェックフロー

```python
# 広告・マーケティング資料のコンプライアンスチェック
AD_COMPLIANCE_RULES = {
    "forbidden_without_evidence": [
        r"No\.?1",           # No.1表示（調査根拠必要）
        r"\d+%の精度",       # 数値精度（検証根拠必要）
        r"完全自動",         # 完全表現
        r"全て.*自動",
        r"100%",
        r"完璧",
    ],
    "requires_disclaimer": [
        r"自動化",           # 「承認が必要な場合があります」注記
        r"AI.*判断",         # 「最終判断は人間が行います」注記
        r"代行",             # 「ユーザーの承認のもと」注記
    ],
    "required_disclaimers": {
        "bpo_features": "※BPO機能は社長・担当者の承認のもと動作します。",
        "ai_accuracy": "※精度は業務内容・データ量により異なります。",
        "automation_scope": "※完全自動化には一定の学習期間と設定が必要です。",
    },
}

async def review_marketing_content(content: str) -> MarketingReviewResult:
    issues = []
    required_disclaimers = set()

    for pattern in AD_COMPLIANCE_RULES["forbidden_without_evidence"]:
        if re.search(pattern, content):
            issues.append(f"根拠なし表現の可能性: '{pattern}'")

    for pattern in AD_COMPLIANCE_RULES["requires_disclaimer"]:
        if re.search(pattern, content):
            required_disclaimers.add(
                AD_COMPLIANCE_RULES["required_disclaimers"].get(pattern, "注記が必要")
            )

    return MarketingReviewResult(
        approved=len(issues) == 0,
        issues=issues,
        required_disclaimers=list(required_disclaimers),
    )
```

## §16. サイバー保険戦略

### 16.1 保険の必要性とリスク評価

```
シャチョツーの主要リスクと保険の対応:

   推定損害額: 売上損失 + 復旧費用
   → 事業中断保険でカバー

   → 賠償責任保険でカバー
```

### 19.1 GDPR Article 22 — 自動化意思決定への対応

BPOワークフローにおける自動実行はGDPR Article 22（プロファイリングを含む自動化処理のみに基づく決定）の対象となる可能性がある。「法的効果または重大な影響を与える」自動化決定には、データ主体が人間の判断を介在させる権利（Right to Human Review）を保証しなければならない。

#### 19.1.1 適用可能性の判定フロー

```
自動実行ワークフロー
       ↓
① 自然人（個人）に影響するか？
  NO → Article 22 対象外（法人間取引のみの場合）
  YES ↓
② 「法的効果」または「重大な影響」を及ぼすか？
  NO → Article 22 対象外（ただし透明性義務は残る）
  YES ↓
③ 人間の判断が実質的に介在するか？
  YES → Article 22(2)(a): 人間が最終決定者 → 許容
  NO  ↓
④ 明示的同意を取得しているか？
  YES → Article 22(2)(c): 同意ベース → 許容（異議申立権の付与必須）
  NO  ↓
  → Article 22(1) 違反リスク → 実行ブロック + 人間レビューキューに送付
```

#### 19.1.2 高影響ワークフロー判定実装

```python
from dataclasses import dataclass, field
from enum import Enum
from typing import Optional
from datetime import datetime
import uuid


class Article22Risk(str, Enum):
    NONE = "none"                  # 法人間のみ・個人への影響なし
    LOW = "low"                    # 個人に影響するが「重大」ではない
    HIGH = "high"                  # 法的効果・重大影響あり → 人間レビュー必須


@dataclass
class Article22Check:
    risk_level: Article22Risk
    requires_human_review: bool
    must_explain_logic: bool
    right_to_contest: bool
    gdpr_article: str
    blocking_reason: Optional[str] = None


class GDPRArticle22Compliance:
    """
    GDPR Article 22: 人に対して法的効果や重大な影響を与える
    自動化された決定には、人間の判断介在の権利を保証する。

    BPOワーカーが実行する前に必ずこのチェックを通過させること。
    """

    # BPO実行が「法的効果・重大影響」に該当するワークフロー種別
    # GDPR Art.22(1) および Recital 71 に基づく分類
    HIGH_IMPACT_WORKFLOW_TYPES: list[str] = [
        "employment_decision",      # 雇用・解雇・降格・評価関連
        "credit_decision",          # 融資・信用供与・与信評価
        "contract_termination",     # 契約解除・取引停止
        "payment_withhold",         # 支払い停止・債権回収
        "salary_change",            # 給与・報酬変更
        "insurance_assessment",     # 保険引受・保険金支払い判定
        "legal_notice_send",        # 法的通知・督促状の送付
    ]

    # 個人への影響は軽微だがリスク管理として記録が必要なワークフロー
    MEDIUM_IMPACT_WORKFLOW_TYPES: list[str] = [
        "appointment_cancel",       # 予約・アポイントキャンセル（医療等）
        "access_revoke",            # システムアクセス権限剥奪
        "notification_send",        # 個人への通知・連絡
        "crm_status_update",        # 顧客ステータス変更
    ]

    async def check_article22_applicability(
        self,
        workflow_type: str,
        affects_natural_person: bool,
        has_explicit_consent: bool = False,
        has_human_in_loop: bool = False,
    ) -> Article22Check:
        """
        ワークフロー実行前の Article 22 適用可能性チェック。

        Args:
            workflow_type: 実行しようとするワークフロー種別
            affects_natural_person: 自然人（個人）への影響有無
            has_explicit_consent: GDPR Art.22(2)(c) 明示的同意の有無
            has_human_in_loop: 人間が最終承認者として介在するか

        Returns:
            Article22Check: リスク評価と必要な対応アクション
        """
        if not affects_natural_person:
            return Article22Check(
                risk_level=Article22Risk.NONE,
                requires_human_review=False,
                must_explain_logic=False,
                right_to_contest=False,
                gdpr_article="N/A（自然人への影響なし）",
            )

        if workflow_type in self.HIGH_IMPACT_WORKFLOW_TYPES:
            if has_human_in_loop:
                # Art.22(2)(a): 人間の決定者が実質的に介在 → 許容
                return Article22Check(
                    risk_level=Article22Risk.HIGH,
                    requires_human_review=True,   # 記録として残す
                    must_explain_logic=True,
                    right_to_contest=True,
                    gdpr_article="Article 22(2)(a)（人間介在）",
                )
            elif has_explicit_consent:
                # Art.22(2)(c): 明示的同意ベース → 許容（ただし異議申立権付与必須）
                return Article22Check(
                    risk_level=Article22Risk.HIGH,
                    requires_human_review=False,
                    must_explain_logic=True,       # Recital 71 要求
                    right_to_contest=True,         # 必須
                    gdpr_article="Article 22(2)(c)（明示的同意）",
                )
            else:
                # Art.22(1) 違反リスク → 実行ブロック
                return Article22Check(
                    risk_level=Article22Risk.HIGH,
                    requires_human_review=True,
                    must_explain_logic=True,
                    right_to_contest=True,
                    gdpr_article="Article 22(1)（違反リスク）",
                    blocking_reason=(
                        f"ワークフロー '{workflow_type}' は自然人に重大な影響を与える自動化決定です。"
                        "人間レビュー（Art.22(2)(a)）または明示的同意（Art.22(2)(c)）なしに実行できません。"
                    ),
                )

        if workflow_type in self.MEDIUM_IMPACT_WORKFLOW_TYPES:
            return Article22Check(
                risk_level=Article22Risk.LOW,
                requires_human_review=False,
                must_explain_logic=True,   # 透明性原則（Art.5(1)(a)）
                right_to_contest=False,
                gdpr_article="Article 5(1)(a)（透明性原則）",
            )

        return Article22Check(
            risk_level=Article22Risk.NONE,
            requires_human_review=False,
            must_explain_logic=False,
            right_to_contest=False,
            gdpr_article="N/A",
        )
```

#### 19.3.2 転送影響評価（TIA: Transfer Impact Assessment）

GDPR Art.46 の SCC に基づく移転は、TIA（移転影響評価）の実施が実務上の必須要件とされている（EDPB Recommendations 01/2020 に準拠）。

**TIA実施項目（3ステップ）:**

1. **移転先国の法令評価** — 当該国の政府機関によるデータアクセス権限（米国: CLOUD Act, FISA Section 702 等）の有無・範囲を評価する
2. **実際のリスク評価** — 移転するデータの性質（機微性）・量・処理目的・暗号化措置を踏まえた実際のリスク水準を評価する
3. **補完的措置の検討** — 技術的措置（転送前暗号化・仮名化）、契約的措置（追加契約条項）、組織的措置（アクセス制御）を決定する

### 19.4 保持・削除ポリシー（法令準拠）

#### 19.4.1 データ種別ごとの保持期間マトリクス

| データ種別 | 保持期間 | 根拠法令 | 削除方式 | 優先順位 |
|---|---|---|---|---|
| 監査ログ（audit_logs） | 7年 | 国税通則法第70条（税務調査時効） | 法令期間終了後に物理削除 | 保持優先 |
| BPO実行ログ（bpo_executions） | 5年 | 商法第33条（商業帳簿） | 期間終了後に物理削除 | 保持優先 |
| BPO自動化決定説明記録 | 5年 | 商法第33条 / GDPR Art.22(3) | 期間終了後に物理削除 | 保持優先 |
| 電子帳簿（請求書・契約書） | 7〜10年 | 電子帳簿保存法・法人税法第74条 | 法令期間終了後に物理削除 | 保持優先 |
| 従業員の個人情報 | 契約期間中 + 60日 | GDPR Art.17 / 個情法第35条 | 論理削除→物理削除 | 削除優先 |
| 同意記録（consent_records） | 同意期間 + 3年 | 証明責任対応 | 期間終了後に物理削除 | 保持優先 |
| DPA締結記録 | 契約期間 + 10年 | 民法第166条（契約関連10年） | 期間終了後に物理削除 | 保持優先 |
| ナレッジアイテム（knowledge_items） | 契約期間中 + 60日 | GDPR Art.17 / 個情法第35条 | 論理削除→物理削除 | 削除優先 |
| 音声データ（Whisper処理前） | 処理完了後即時削除 | データ最小化原則（GDPR Art.5(1)(c)） | 即時物理削除 | 削除最優先 |
| TIA記録 | 有効期間 + 5年 | GDPR Art.46 / 説明責任原則 | 期間終了後に物理削除 | 保持優先 |

> **法令競合の解決原則**: 削除優先データであっても、特定の法令が保持を義務付けている場合（監査ログ等）はその部分のみ保持し、個人識別性のある部分を仮名化・削除する「部分削除」アプローチを採る。

#### 19.4.2 解約時完全削除プロセス

```python
from enum import Enum
from datetime import datetime, timedelta


class DeletionPhase(str, Enum):
    LOGICAL = "logical"       # 論理削除（アクセス不可・データ残存）
    PHYSICAL = "physical"     # 物理削除（データ完全消去）
    PARTIAL = "partial"       # 部分削除（個人識別部分のみ削除・集計データ保持）
    RETAINED = "retained"     # 法令保持（保持義務あり・最終削除スケジュール設定済み）


@dataclass
class DeletionReport:
    company_id: str
    deletion_initiated_at: datetime
    logical_deletion_at: Optional[datetime]
    physical_deletion_at: Optional[datetime]
    retained_records: list[dict]    # 法令上保持が必要なレコードの種別・件数・保持期限
    deletion_certificate_url: str   # 削除完了証明書のStorage URL
    notified_at: Optional[datetime] # 顧客への通知日時


class DataDeletionManager:
    """
    解約時の段階的データ削除プロセス。
    GDPR Art.17「忘れられる権利」/ 個人情報保護法第35条（利用停止・消去）対応。

    削除プロセスは2段階:
      Phase 1 (解約後30日以内): 個人データの論理削除（アクセス不可化）
      Phase 2 (解約後60日以内): 個人データの物理削除（完全消去）
      ※ 法令保持義務のあるデータは上記に関わらず保持し、保持期間終了後に物理削除する。
    """

    # 解約後の削除タイムライン
    LOGICAL_DELETION_DAYS: int = 30   # GDPR Art.17 / 個情法第35条
    PHYSICAL_DELETION_DAYS: int = 60

    # 法令上の保持義務データ（削除対象外 → 保持期間後に自動削除）
    LEGALLY_RETAINED_TABLES: dict[str, timedelta] = {
        "audit_logs": timedelta(days=365 * 7),         # 国税通則法: 7年
        "bpo_executions": timedelta(days=365 * 5),     # 商法: 5年
        "bpo_decision_explanations": timedelta(days=365 * 5),
        "customer_dpas": timedelta(days=365 * 10),     # 民法: 10年
        "consent_records": timedelta(days=365 * 3),    # 証明責任: 3年（同意終了後）
        "transfer_impact_assessments": timedelta(days=365 * 5),
    }

    async def process_account_deletion(
        self, company_id: str, cancelled_at: datetime
    ) -> DeletionReport:
        """
        解約時の完全削除プロセスを開始し、スケジュールをDBに登録する。

        処理フロー:
        1. 解約受付と同時に削除スケジュールをDBに登録
        2. 30日後: Supabase Edge Function が論理削除を実行（アクセスブロック）
        3. 60日後: 物理削除ジョブが実行（pgvectorのembeddingベクトルを含む全データ）
        4. 法令保持テーブルは別途保持スケジュールで管理
        5. 全削除完了後、削除完了証明書をPDFで生成・顧客にメール送信
        """
        logical_at = cancelled_at + timedelta(days=self.LOGICAL_DELETION_DAYS)
        physical_at = cancelled_at + timedelta(days=self.PHYSICAL_DELETION_DAYS)

        retained = []
        for table, retention_delta in self.LEGALLY_RETAINED_TABLES.items():
            retained.append({
                "table": table,
                "retention_until": (cancelled_at + retention_delta).isoformat(),
                "legal_basis": self._get_retention_basis(table),
            })

        # deletion_schedules テーブルに登録（非同期ジョブが実行）
        # ... INSERT INTO deletion_schedules ...

        # 削除完了証明書を生成（物理削除完了後）
        certificate_url = await self._generate_deletion_certificate(
            company_id, logical_at, physical_at, retained
        )

        # 顧客担当者にメール通知（削除スケジュール + 法令保持の説明）
        await self._notify_customer(company_id, logical_at, physical_at, retained)

        return DeletionReport(
            company_id=company_id,
            deletion_initiated_at=datetime.utcnow(),
            logical_deletion_at=logical_at,
            physical_deletion_at=physical_at,
            retained_records=retained,
            deletion_certificate_url=certificate_url,
            notified_at=datetime.utcnow(),
        )

    def _get_retention_basis(self, table: str) -> str:
        bases = {
            "audit_logs": "国税通則法第70条（7年）",
            "bpo_executions": "商法第33条（5年）",
            "bpo_decision_explanations": "商法第33条 / GDPR Art.22(3)（5年）",
            "customer_dpas": "民法第166条（10年）",
            "consent_records": "証明責任対応（同意終了後3年）",
            "transfer_impact_assessments": "GDPR Art.46 / 説明責任原則（5年）",
        }
        return bases.get(table, "不明（要法務確認）")

    async def _generate_deletion_certificate(
        self,
        company_id: str,
        logical_at: datetime,
        physical_at: datetime,
        retained: list[dict],
    ) -> str:
        """
        削除完了証明書（PDF）を生成してSupabase Storageに保存する。
        証明書には削除日時・対象テーブル・法令保持情報を含める。
        """
        # Supabase Storage: private bucket "deletion-certificates/{company_id}/"
        # 証明書自体はAES-256-GCMで暗号化して保存
        return f"https://storage.supabase.co/deletion-certificates/{company_id}/certificate.pdf"
```

### 19.5 データ侵害通知プロセス（GDPR Art.33/34 / 個人情報保護法第26条）

#### 19.5.1 通知義務の発生要件と期限

| 通知先 | 発生要件 | 通知期限 | 根拠条文 |
|---|---|---|---|
| 所管監督機関（EU DPA） | 個人データ侵害が発生した場合（権利・自由へのリスクが低い場合を除く） | 認知から72時間以内 | GDPR Art.33(1) |
| 影響を受けた個人（データ主体） | 「高いリスク」が個人の権利・自由に生じる可能性がある場合 | 不当な遅延なく（as soon as possible） | GDPR Art.34(1) |
| 個人情報保護委員会（日本PPC） | 要配慮個人情報・財産的被害・不正目的・1,000件以上の漏洩等 | 速報: 3〜5日以内 / 確報: 30日以内 | 個情法第26条 |
| 顧客企業（管理者） | 処理者としての義務（Art.28） | 認知後、遅延なく（実務上24時間以内） | GDPR Art.33(2) |

> **72時間の起算点**: シャチョツー社員（またはシステム）が侵害を「認知した」時点から起算する。調査完了を待つ必要はなく、利用可能な情報で速報し、情報が揃い次第追報する（Art.33(4)の段階的通知）。

#### 19.5.2 データ侵害対応実装

```python
from dataclasses import dataclass, field
from datetime import datetime, timedelta
from typing import Optional
from enum import Enum


class BreachSeverity(str, Enum):
    LOW = "low"         # 個人への具体的リスクなし（例: 暗号化データの紛失）
    MEDIUM = "medium"   # 限定的リスク（例: 社内向けメタデータの誤表示）
    HIGH = "high"       # 高いリスク（例: 個人情報・業務データの外部流出）
    CRITICAL = "critical"  # 重大リスク（例: 要配慮情報・金融情報の大規模漏洩）


class BreachType(str, Enum):
    CONFIDENTIALITY = "confidentiality"  # 機密性侵害（不正アクセス・漏洩）
    INTEGRITY = "integrity"              # 完全性侵害（改ざん）
    AVAILABILITY = "availability"        # 可用性侵害（消去・アクセス不能）


@dataclass
class DataBreach:
    breach_id: str
    company_id: str                  # 影響を受けた顧客テナント（複数可）
    breach_type: BreachType
    severity: BreachSeverity
    discovered_at: datetime
    description: str
    affected_data_categories: list[str]  # 漏洩した個人データの種別
    estimated_affected_count: int        # 影響を受けた個人の推定人数
    affects_eu_residents: bool
    affects_japan_residents: bool
    is_high_risk_to_individuals: bool    # GDPR Art.34 の個人通知が必要か
    containment_measures: list[str]      # 封じ込め措置の内容
    root_cause: Optional[str] = None     # 根本原因（調査後に記入）
    dpa_notified_at: Optional[datetime] = None      # EU DPA通知日時
    ppc_notified_at: Optional[datetime] = None      # 日本PPC通知日時
    individuals_notified_at: Optional[datetime] = None


class DataBreachNotificationProcess:
    """
    データ侵害通知プロセス。
    GDPR Art.33/34 および個人情報保護法第26条に完全準拠。

    通知期限の自動計算と通知内容のテンプレート生成を担う。
    実際の送信はメール（監督機関・個人）とポータル通知（顧客企業）で行う。
    """

    NOTIFICATION_DEADLINES = {
        "eu_supervisory_authority": timedelta(hours=72),  # GDPR Art.33(1)
        "affected_individuals": None,    # 「不当な遅延なく」— 原則24時間以内を社内目標に
        "japan_ppc_speed_report": timedelta(days=5),      # 個情法第26条 速報
        "japan_ppc_full_report": timedelta(days=30),      # 個情法第26条 確報
        "customer_controller": timedelta(hours=24),       # GDPR Art.33(2) 実務目標
    }

    async def initiate_breach_notification(
        self, breach: DataBreach
    ) -> "BreachNotificationPlan":
        """
        侵害を認知した時点で即座に呼び出す。
        通知計画（期限・送信先・内容テンプレート）を返す。
        """
        plan = BreachNotificationPlan(
            breach_id=breach.breach_id,
            discovered_at=breach.discovered_at,
        )

        # EU監督機関（所管DPA）への通知
        if breach.affects_eu_residents and breach.severity != BreachSeverity.LOW:
            plan.eu_dpa_deadline = (
                breach.discovered_at + self.NOTIFICATION_DEADLINES["eu_supervisory_authority"]
            )
            plan.eu_dpa_notification = await self._prepare_eu_dpa_notification(breach)

        # 日本PPC への通知
        if breach.affects_japan_residents and self._requires_japan_ppc_notification(breach):
            plan.japan_ppc_speed_deadline = (
                breach.discovered_at + self.NOTIFICATION_DEADLINES["japan_ppc_speed_report"]
            )
            plan.japan_ppc_full_deadline = (
                breach.discovered_at + self.NOTIFICATION_DEADLINES["japan_ppc_full_report"]
            )

        # 影響を受けた個人への通知（GDPR Art.34: 高リスクの場合）
        if breach.affects_eu_residents and breach.is_high_risk_to_individuals:
            plan.individual_notification_required = True
            plan.individual_notification = await self._prepare_individual_notification(breach)

        # 顧客企業（データ管理者）への通知
        plan.customer_deadline = (
            breach.discovered_at + self.NOTIFICATION_DEADLINES["customer_controller"]
        )

        # 通知計画をDBに保存（証拠として）
        await self._store_breach_record(breach, plan)

        return plan

    def _requires_japan_ppc_notification(self, breach: DataBreach) -> bool:
        """
        個情法第26条の報告義務発生要件の確認:
        (1) 要配慮個人情報を含む
        (2) 財産的被害のおそれ
        (3) 不正目的による侵害
        (4) 1,000件以上の個人情報
        """
        return (
            "要配慮個人情報" in breach.affected_data_categories
            or "医療情報" in breach.affected_data_categories
            or "歯科情報" in breach.affected_data_categories
            or breach.estimated_affected_count >= 1000
            or breach.breach_type == BreachType.CONFIDENTIALITY
        )

    async def _prepare_eu_dpa_notification(self, breach: DataBreach) -> dict:
        """
        GDPR Art.33(3) が要求する監督機関通知の記載事項:
        (a) 侵害の性質・影響を受けた個人のカテゴリーと概数
        (b) DPO（またはその他連絡先）の氏名・連絡先
        (c) 侵害の可能性のある結果
        (d) 管理者が講じた・講じる予定の措置
        """
        return {
            "breach_nature": breach.description,
            "data_categories_affected": breach.affected_data_categories,
            "estimated_individuals_affected": breach.estimated_affected_count,
            "dpo_contact": "privacy@shachotwo.com",
            "likely_consequences": self._assess_consequences(breach),
            "measures_taken": breach.containment_measures,
            "measures_planned": [],  # 調査後に追記
            "notification_type": "initial",  # initial / follow_up / final
        }
```

## §20 データライフサイクル・保持削除ポリシーの完全設計

> **背景**: シャチョツーは電子帳簿保存法（7年保存義務）とGDPR/個人情報保護法（削除権）が直接衝突する。

#### フリーユーザー離脱時のデータ削除タイムライン

```

Day 14      : トライアル期間終了
            → 本登録を促すメール通知（Day 12・14 の2回送信）
```

### 20.3 電子帳簿保存法 vs GDPR削除権 衝突解決設計

```python

from typing import Optional


class DeletionAction(str, Enum):
    DELETED = "deleted"
    ANONYMIZED = "anonymized"
    REJECTED = "rejected"
    DEFERRED = "deferred"

    action: DeletionAction
    legal_basis: Optional[str] = None
    gdpr_article: Optional[str] = None
    message_to_user: Optional[str] = None
    scheduled_deletion_date: Optional[str] = None

    電子帳簿保存法の7年保存義務 vs GDPR Art.17 削除権の衝突を解決する。

    解決原則:

         「保存義務の維持」と「削除権の充足」を両立させる。
      2. 匿名化が技術的に不可能な場合 → 削除権の例外（Art.17(3)(b)）を根拠に
         保持期間終了まで削除を延期し、ユーザーに通知する。
      3. 法令保持義務がないデータ → 即時物理削除で削除権に完全対応する。
    """

        self,
        company_id: str,
        data_type: str,
        requested_by: str,
    ) -> DeletionDecision:

            if can_anonymize:
                await self.anonymize_personal_fields(company_id, data_type)
                return DeletionDecision(
                    action=DeletionAction.ANONYMIZED,
                    legal_basis="電子帳簿保存法第10条 / 商法第33条",
                    gdpr_article="Art.17(3)(b)",
                    message_to_user=(
                        "対象データは法令上の保存義務があるため物理削除できません。"
                        "個人を特定できる情報（氏名・住所等）を匿名化処理しました。"
                        "保存義務期間終了後に完全削除します。"

                )

                    action=DeletionAction.DEFERRED,
                    legal_basis="電子帳簿保存法第10条",
```

#### 利用規約に明記すべき禁止事項

以下の禁止事項を利用規約 §X.X「禁止行為」に明文化する：

1. **競合製品開発目的での利用禁止**
   シャチョツーのゲノムテンプレート・業務フローテンプレート・AIプロンプト設計を競合製品の開発・改善目的で利用・収集することを禁ずる。

2. **業種・企業規模の虚偽申告禁止**
   登録時の業種・従業員規模・事業内容を虚偽申告することを禁ずる。虚偽が判明した場合は即時アカウント停止し、適用プランの差額を請求できる。

3. **自動化ツールによるゲノムテンプレート収集禁止**
   スクレイピングツール・botその他の自動化ツールを用いたテンプレート・知識ベースの大量取得を禁ずる。

4. **1企業が複数アカウント保有禁止（Enterprise Multi-tenancy除く）**
   同一企業・同一事業主体が複数のフリートライアルアカウントを開設することを禁ずる。EnterpriseプランのMulti-tenancy機能を通じた複数組織管理は除く。

5. **ナレッジベースへの第三者個人情報の無断登録禁止**
   取引先・顧客・従業員等の個人情報を、当該個人の同意なくナレッジベースに登録することを禁ずる。

### 22.3 AIバイアス監査プロセス

EU AI Act Article 9（リスクマネジメントシステム）および
Article 10（データガバナンス）への対応として、四半期ごとのバイアス監査を実施する。

```python
from dataclasses import dataclass
from datetime import datetime
from typing import Optional
import uuid


@dataclass
class BiasAuditReport:
    audit_quarter: str          # '2026-Q1'
    dimension: str
    bias_detected: bool
    p_value: float
    sample_size: int
    mitigation_action: Optional[str]
    auditor: str
    completed_at: datetime


class AIBiasAuditProcess:
    """
    四半期ごとのバイアス監査。
    特にゲノムデータに業種・規模・地域バイアスが含まれていないかを確認する。
    高リスク機能（BPO支払・HR補助）は年次で第三者レビューを実施。
    """

    AUDIT_DIMENSIONS = [
        "industry_bias",       # 建設業偏重になっていないか
        "company_size_bias",   # 大企業のパターンが中小を圧倒していないか
        "regional_bias",       # 首都圏偏重になっていないか
        "gender_pay_gap",      # 給与・評価関連のルールに性差別的パターンがないか
    ]

    # カイ二乗検定でp < 0.05 を偏り検出の閾値とする
    BIAS_DETECTION_THRESHOLD = 0.05

    async def run_quarterly_audit(self) -> list[BiasAuditReport]:
        """
        監査フロー:
        1. ゲノムテンプレートの上位100パターンをサンプリング
        2. 各次元でカイ二乗検定（p < 0.05 で偏り検出）
        3. 偏りが検出された場合: 対象パターンを除外または重み調整

        for dimension in self.AUDIT_DIMENSIONS:
            report = await self._audit_dimension(dimension)
            reports.append(report)
            await self._save_audit_result(report)
        return reports

        raise NotImplementedError

    async def _save_audit_result(self, report: BiasAuditReport) -> None:
```

### 22.4 説明可能性（Explainability）設計

GDPR Article 22（自動意思決定に関する権利）および
EU AI Act Article 13（透明性・情報提供義務）に対応し、
すべてのAI判断・提案に「なぜそう判断したか」を付与する。

```python
from dataclasses import dataclass, field
from typing import Optional


@dataclass
class AIDecision:
    decision_id: str
    feature: str
    outcome: str
    confidence: float
    knowledge_sources: list[str]
    top_reasons: list[str]
    alternative_considered: Optional[str] = None


class ExplainableAIResponse:
    """
    すべてのAI判断・提案に「なぜそう判断したか」を付与する。
    GDPR Article 22 + EU AI Act Article 13（transparency）対応。

    ユーザーが「異議申し立て」を選択した場合:
    - 修正内容を correction_requests テーブルに記録
    - Correction-Driven Learning（§CDL）で次回以降の精度向上に活用
    """

    def format_explanation(self, decision: AIDecision) -> str:
        sources_summary = ", ".join(decision.knowledge_sources[:3])
        reasons = "\n".join(
            f"        {i + 1}. {reason}"
            for i, reason in enumerate(decision.top_reasons[:3])
        )
        return f"""
        【判断の根拠】
        参照したルール: {sources_summary}
        信頼度: {decision.confidence:.0%}

        【判断理由（上位3つ）】
{reasons}

        【この判断に異議を申し立てる場合】
        → [修正を提案する] ボタンで修正内容を入力してください
           修正内容はAIの学習に使用され、次回以降の精度向上に活用されます
        """

    def format_bpo_explanation(self, decision: AIDecision) -> dict:
        """BPO操作に特化した説明（法的証跡用）"""
        return {
            "decision_id": decision.decision_id,
            "outcome": decision.outcome,
            "confidence": decision.confidence,
            "knowledge_sources": decision.knowledge_sources,
            "top_reasons": decision.top_reasons[:3],
            "human_approval_required": decision.confidence < 0.95,
            "explanation_text": self.format_explanation(decision),
            "generated_at": "ISO8601タイムスタンプ",
        }
```

### 22.5 AIリスク分類と承認マトリクス

AIが実行するアクションを5段階に分類し、リスクに応じた承認要件と監査ログ保持期間を定める。

| AIアクション | リスクレベル | 承認要件 | 説明義務 | 監査ログ保持期間 |
|---|---|---|---|---|
| Q&A参照回答 | 最小 | 不要 | AI利用の開示 | 90日 |
| 能動提案（情報提供） | 限定 | 不要（開示のみ） | AI生成明示 | 1年 |
| BPO Shadow Mode（提案のみ） | 中 | 不要 | 根拠3件表示 | 3年 |
| BPO Approval Mode（人間承認後実行） | 高 | マネージャー以上 | 全根拠・操作詳細 | 7年 |
| BPO Auto Mode（自動実行） | 重大 | CEO事前承認 + 暗号署名 | 全根拠・操作詳細 | 7年 + 暗号証明 |
| 採用判断補助 | 高 | HRレビュー必須 + バイアス監査 | 評価軸全件・根拠 | 10年 |
| 給与・評価補助 | 高 | CFOレビュー | 評価軸全件・根拠 | 10年 |

```python
from enum import Enum


class AIActionRiskLevel(str, Enum):
    MINIMAL  = "minimal"    # Q&A参照
    LIMITED  = "limited"    # 能動提案・Shadow Mode
    MEDIUM   = "medium"     # BPO Shadow Mode
    HIGH     = "high"       # BPO Approval Mode・HR補助・給与補助
    CRITICAL = "critical"   # BPO Auto Mode


# 監査ログ保持期間（日数）
AUDIT_LOG_RETENTION_DAYS: dict[AIActionRiskLevel, int] = {
    AIActionRiskLevel.MINIMAL:  90,
    AIActionRiskLevel.LIMITED:  365,
    AIActionRiskLevel.MEDIUM:   365 * 3,
    AIActionRiskLevel.HIGH:     365 * 7,
    AIActionRiskLevel.CRITICAL: 365 * 7,  # 暗号証明を別途保存
}
```

### 22.7 Human-in-the-Loop（HitL）設計原則

EU AI Act Article 14（人間による監視）への対応。
高リスクAIシステムは人間が介入・修正・停止できる設計を義務付ける。

```python
from dataclasses import dataclass
from typing import Callable, Awaitable


@dataclass
class HitLConfig:
    """
    各AIアクションに対するHitL設定。
    confidence_threshold: この値を下回る場合、自動的にApprovalモードに降格
    auto_mode_requires_ceo: TrueのときCEO事前承認が必要
    override_window_seconds: 承認後にユーザーがキャンセルできる猶予時間
    """
    confidence_threshold: float
    auto_mode_requires_ceo: bool
    override_window_seconds: int
    escalation_role: str           # 承認が必要な最低ロール


HITL_CONFIGS: dict[str, HitLConfig] = {
    "bpo_shadow":    HitLConfig(0.0,  False, 0,    "employee"),   # 提案のみ。実行なし
    "bpo_approval":  HitLConfig(0.90, False, 30,   "manager"),    # 承認後30秒キャンセル可
    "bpo_auto":      HitLConfig(0.99, True,  0,    "ceo"),        # CEO事前承認必須
    "hr_screening":  HitLConfig(0.0,  False, 0,    "hr"),         # 常に人間レビュー必須
    "salary_review": HitLConfig(0.0,  False, 0,    "executive"),  # 常にCFO/Executiveレビュー
}
```

#### HitL運用ルール

1. **停止権限**: CEO・Executiveはすべての自動AIアクションを即時停止できる（緊急停止ボタン）
2. **降格権限**: 任意のAIアクションを上位承認モードに手動で引き上げられる
3. **監査**: HitL介入・停止のすべての操作は `audit_logs` に記録し、7年保持
4. **定期レビュー**: Auto Modeの動作ルールは四半期ごとにCEOが承認更新

## 23. DPA（Data Processing Agreement）必須条項

> LLMベンダー（Google, Anthropic, OpenAI）との間で締結すべきDPAの必須条項。
> 顧客データをLLM APIに送信する以上、データ処理委託の法的根拠が必要。

### 23.1 LLMベンダー向け DPA 必須条項一覧

```
■ 条項1: データ処理の範囲と目的
  ・処理対象: テキスト（ナレッジアイテム、Q&A入出力、構造化結果）
  ・処理目的: テキスト生成、構造化、要約、Q&A回答のみ
  ・処理場所: 明示的に指定（日本リージョン優先、不可の場合は米国）
  ・禁止事項: 当社データをモデル学習に使用しないことの保証

■ 条項2: モデル学習不使用保証
  ・API入出力データを、モデルの訓練・改善・評価に使用しない
  ・サブプロセッサーにも同等の制約を課す
  ・違反時の損害賠償条項
  ・確認方法: ベンダーのDPA/利用規約上のOpt-out設定

  現状のベンダー対応:
    Google (Gemini API): デフォルトでモデル学習不使用（有料API）
    Anthropic (Claude API): デフォルトでモデル学習不使用
    OpenAI (API): デフォルトでOpt-out済み（API利用時）

■ 条項3: サブプロセッサーの開示
  ・LLMベンダーが利用するサブプロセッサー（クラウドインフラ等）の一覧開示
  ・サブプロセッサー変更時の事前通知義務（30日前）
  ・当社が異議を申し立てる権利

■ 条項4: データ削除・返却
  ・API呼び出し後、ベンダー側での入出力データの保持期間明示
  ・当社の要求により30日以内に完全削除
  ・削除完了の書面確認

■ 条項5: セキュリティインシデント通知
  ・ベンダー側でデータ漏洩が発生した場合、72時間以内に当社に通知
  ・通知内容: 影響範囲、漏洩データの種類、対応状況
  ・当社からの監査要求への対応義務（年1回）

■ 条項6: 越境データ移転
  ・日本→米国（EU経由なし）: APPI（個人情報保護法）の越境移転規定に準拠
  ・SCC（Standard Contractual Clauses）の適用可否
  ・GDPR対象顧客がいる場合: EU SCCs 2021を付属
```

## 25. データ保持・削除ポリシー

| データ種別 | 保持期間 | 根拠 | 削除方法 | 削除確認 |
|---|---|---|---|---|
| **監査ログ** | **7年** | 電子帳簿保存法・税法 | 自動アーカイブ(3年後) → 物理削除(7年後) | 年次棚卸し |
| **ナレッジデータ** | **契約期間+90日** | 顧客契約 | 論理削除(即時) → 物理削除(90日後) | 削除証明書発行 |
| **LLM入出力ログ** | **30日** | 品質改善・デバッグ | 自動削除(cron) | 月次確認 |
| **ユーザーセッション** | **24時間** | セキュリティ | 自動失効(Redis TTL) | — |
| **BPO実行ログ** | **1年** | 業務監査 | 自動アーカイブ(6ヶ月後) → 物理削除(1年後) | 四半期確認 |
| **エクスポートファイル** | **24時間** | 一時データ | 自動削除 | — |
| **バックアップ** | **90日** | 災害復旧 | 自動ローテーション | 月次確認 |
| **請求データ** | **7年** | 税法 | アーカイブ → 物理削除 | 年次棚卸し |

### 削除実行の手順

```
■ 自動削除（cron job）
  ・毎日深夜 3:00 JST に削除ジョブ実行
  ・対象: TTL超過のセッション、30日超のLLMログ、24h超のエクスポートファイル
  ・実行ログを audit_logs に記録

■ 手動削除（顧客要求時）
  Step 1: 削除リクエスト受領 → audit_logs に記録
  Step 2: 対象データの特定・一覧化
  Step 3: 論理削除（deleted_at にタイムスタンプ設定）
  Step 4: 90日後に物理削除（DELETE + VACUUM）
  Step 5: バックアップからも削除
  Step 6: 削除完了証明書発行（PDF）
```

| 監査ログ | 全操作記録（§24準拠） | **SOC2 Type II対応**レベルの詳細ログ |
| ログ保持 | 7年（§25準拠） | **カスタム保持期間** + 外部SIEM連携 |
| ログエクスポート | JSON（§24.1形式） | **リアルタイムストリーム**（Splunk/Datadog/Sentinel） |

### 26.2 SIEM連携仕様

```
■ 対応SIEM

  Splunk:
    ・接続方式: HEC（HTTP Event Collector）
    ・フォーマット: JSON（§24.1の監査ログ形式準拠）
    ・認証: HEC Token（顧客が発行→シャチョツーに設定）

    ・接続方式: OCSF（Open Cybersecurity Schema Framework）形式
    ・認証: IAM Role（クロスアカウントアクセス）

■ ストリーム対象イベント

  認証イベント:
    ・ログイン/ログアウト/MFA検証/SSO認証
    ・ログイン失敗（ブルートフォース検知用）

  データアクセスイベント:
    ・knowledge_items の CRUD
    ・genome_templates の閲覧・編集
    ・機密データへのアクセス（PII含むレコード）

  管理操作イベント:
    ・ユーザー追加/権限変更/設定変更
    ・SSO/SCIM設定の変更
    ・API Key の作成/無効化

  BPO実行イベント:
    ・操作内容/承認/結果/ロールバック
    ・外部SaaS API呼び出しログ

  異常検知イベント:
    ・通常と異なるアクセスパターン（時間帯/IP/操作頻度）
    ・大量データエクスポート検知
    ・権限昇格の試行

■ 配信設定

  遅延: リアルタイム（イベント発生から30秒以内に配信）
  再送: 配信失敗時は最大3回リトライ（指数バックオフ）
  バッファ: SIEM側が一時不達の場合、72時間分をバッファリング
  フィルタ: 顧客がイベント種別ごとに配信ON/OFFを設定可能
```

## §27 グループ間データ共有のコンプライアンス対応

> グループ管理機能（c_02 §12）で企業間ナレッジ共有を行う際の法的整理・同意取得・監査証跡。
> 日本法を主軸に、将来の国際展開（GDPR/CCPA等）も見据えた設計。

### 27.1 グループ内データ共有の法的整理

#### 日本法（個人情報保護法）

グループ企業間であっても、個人データの提供は原則「第三者提供」に該当する。
ただし**「共同利用」（法27条5項3号）** の要件を満たせば第三者提供に非該当。

```
■ 共同利用の5要件（全て充足が必要）
  1. 共同利用する旨の通知・公表
     → シャチョツー: グループ作成時に「共同利用規約」への同意を必須化
     → 公表方法: 各社のプライバシーポリシーに記載 + シャチョツー上で表示

  2. 共同利用するデータ項目の特定
     → シャチョツー: 共有対象ナレッジのカテゴリを事前に選択
     → 例: 「業務フロー」「判断基準」は共有OK、「従業員個人情報」は共有NG等

  3. 共同利用する者の範囲の特定
     → シャチョツー: group_membershipsテーブルで明示的に管理
     → グループに所属する企業のみ（範囲は画面上で確認可能）

  4. 利用目的の特定
     → シャチョツー: 「グループ経営の効率化・ナレッジ共有」を標準目的として定義
     → カスタム目的の追加も可能（group_policiesのmetadataに記録）

  5. 管理責任者の特定
     → シャチョツー: グループの親会社（parent_company_id）が管理責任者
     → 親会社の管理者ユーザーが法的責任者として登録
```

#### GDPR（EU一般データ保護規則）— 将来の国際展開時

```
■ グループ企業間データ移転の法的根拠
  → 「正当な利益」（Art.6(1)(f)）: グループ内管理目的は正当な利益として認められうる
  → ただし「データ主体の利益・権利・自由」とのバランステストが必要

■ 必要な対応
  1. グループ内データ処理契約（Intra-Group Data Processing Agreement）
     → テンプレートを事前準備（d_03 国際化設計と連携）
  2. DPIA（データ保護影響評価）の実施
     → 大規模なプロファイリング・自動化処理に該当する可能性
  3. データ主体への通知（プライバシーノーティス更新）
  4. EU域外移転の場合はSCC（標準契約条項）が必要
     → 日本はGDPR十分性認定済みだが、今後の状況変化に注意

■ シャチョツーでの実装方針（国際展開Phase 2以降）
  → company.regionフィールドでGDPR対象企業を自動判定
  → GDPR対象企業を含むグループでは追加同意フローを挿入
```

#### APPI（改正個人情報保護法 2022年施行分）

```
■ 個人関連情報の第三者提供規制
  → Cookie・閲覧履歴等の「個人関連情報」がナレッジに含まれる場合
  → 提供先で個人データとなることが想定される場合、本人同意が必要
  → シャチョツー: ナレッジのPIIスキャン結果に「個人関連情報」フラグを追加

■ 外国にある第三者への提供
  → グループ内に外国法人がある場合（将来の国際展開時）
  → 「外国にある第三者への提供」として追加の情報提供義務
  → シャチョツー: company.countryフィールドで自動判定
```

### 27.2 同意取得フロー

```
■ グループ作成時（1回のみ）
  ┌──────────────────────────────────────┐
  │ 1. 親会社管理者が「グループ共同利用規約」に同意  │
  │    → 規約内容: 共同利用の5要件を網羅           │
  │    → 同意記録: consent_recordsテーブルに保存     │
  │                                              │
  │ 2. 子会社追加時に、子会社管理者も同規約に同意     │
  │    → 同意なしではグループに参加不可              │
  │    → 同意撤回 = グループからの離脱              │
  │                                              │
  │ 3. 共同利用の対象データ項目・目的・管理責任者を    │
  │    設定画面で確認・保存                         │
  └──────────────────────────────────────┘

■ ナレッジ共有時（共有ごと）
  ┌──────────────────────────────────────┐
  │ 1. 共有対象ナレッジのPIIスキャン（自動）         │
  │                                              │
  │ 2. PIIなし → そのまま共有（追加同意不要）        │
  │                                              │
  │ 3. PIIあり → 以下の選択肢を提示:               │
  │    a. PIIをマスクして共有                      │
  │       → PII部分を[MASKED]に置換               │
  │       → マスク前後の対応表は共有元のみ保持       │
  │    b. 共同利用同意を確認して共有（PIIそのまま）   │
  │       → 共同利用規約の範囲内か自動チェック       │
  │       → 範囲外なら追加の個別同意を要求          │
  │    c. 共有をキャンセル                         │
  │                                              │
  │ 4. 選択結果をaudit_logsに記録                  │
  │    → event_type: 'group_share_consent'         │
  │    → metadata: {pii_detected, action, basis}    │
  └──────────────────────────────────────┘

■ 従業員データを含むナレッジの特別処理
  → 人事評価・給与・勤怠等は「要配慮個人情報」に該当しうる
  → 要配慮個人情報の共同利用には「本人同意」が原則必要（共同利用の例外が使えない）
  → シャチョツーの実装:
    1. ナレッジに「要配慮」タグを自動/手動で付与
    2. 要配慮タグ付きナレッジの共有時に警告表示
    3. 共有する場合は「従業員への通知済み」チェックボックスを必須化
    4. 通知方法の記録（メール/社内掲示/個別説明等）
```
