### 1.1 エージェント分離の理由

| | 企業リサーチエージェント | アポ取りエージェント |
|---|---|---|
| **責務** | 企業を見つけて調べてリストに追加 | リストの企業にアプローチしてアポを確定 |
| **実行頻度** | 毎日（新規企業を継続追加） | 毎日（400件/日送信） |
| **入力** | 業種×地域の検索条件 | スプレッドシートの未送信行 |
| **出力** | スプレッドシートに企業+LP URL追加 | 送信結果 + 日程確定 + Gmail通知 |
| **使うAPI** | 求人サイト / gBizINFO / LLM | フォーム送信 / Gmail / Calendar / LLM |
| **接続点** | **スプレッドシート（書き込み側）** | **スプレッドシート（読み取り側）** |

## 2. AI社員の職務経歴書 — カスタマイズ設計

### 2.1 カスタマイズの判断軸

企業ごとにAI社員の「できること」を決めるための3つの軸：

```
軸①: 業種（何の業務ができるか）
  └→ e_00_業界別BPO業務フロー総覧.md の32業務から選択

軸②: 企業の痛み（何を一番解決したいか）
  └→ Webリサーチ（求人情報・ニュース・IR）から推定

軸③: 企業規模（どのレベルの提案が刺さるか）
  └→ 従業員数・売上規模で提案トーンを変える
```

#### 軸①: 業種 → BPO業務マッピング

| 業種 | AI社員の「得意業務」トップ3 | 訴求ポイント |
|------|--------------------------|-------------|
| **建設業** | 積算AI / 安全書類自動作成 / 出来高請求 | 「積算に3日→半日。現場監督の事務2h→15分」 |
| **製造業** | 見積AI / 生産計画 / 在庫最適化 | 「FAX受発注の手入力ゼロ。在庫回転率20%改善」 |
| **歯科** | レセプトAI / 予約最適化 / リコール管理 | 「レセプト返戻率50%減。予約キャンセル率30%減」 |
| **介護** | ケアプラン書類 / 介護報酬請求 / シフト管理 | 「記録業務2h→30分。離職原因No.1を解消」 |
| **士業** | 顧問先Q&A / 月次巡回準備 / 届出期限管理 | 「同じ質問に何度も答える問題を解消」 |
| **不動産** | 物件査定AI / 契約書自動作成 / 賃料収支管理 | 「管理物件200→500でも人員増なし」 |

#### 軸②: 企業の痛み → リサーチで推定する項目

| リサーチ元 | 取得できる情報 | 痛みの推定 |
|-----------|---------------|-----------|
| **求人サイト** | 募集職種・人数・急募フラグ | 人手不足の部署・業務を特定 |
| **企業HP** | 事業内容・拠点数・従業員数 | 規模感・主要事業の把握 |
| **ニュース/プレスリリース** | 新規事業・M&A・受賞 | 成長フェーズ・注力分野 |
| **Googleマップ口コミ** | 顧客満足度・クレーム傾向 | 現場オペレーションの課題 |
| **SNS（X/Facebook）** | 社長の発信内容・関心事 | 社長個人の課題意識 |
| **業界ニュース** | 法改正・市場動向 | 業界共通の課題（2024年問題等） |

**推定ロジック例（建設業）:**
```
IF 求人で「施工管理技士」を急募
  → 痛み = 現場監督の負荷過大
  → 訴求 = 「安全書類・日報をAIが自動作成。現場監督の事務作業を月40時間削減」

IF 従業員50名以上 AND 公共工事受注あり
  → 痛み = 積算の属人化・入札対応の負荷
  → 訴求 = 「積算AIで拾い出し70%自動化。入札準備を3日→半日に」

IF 求人で「経理」「事務」を募集
  → 痛み = バックオフィス人手不足
  → 訴求 = 「出来高請求・原価管理をAIが自動集計。経理の残業ゼロへ」
```

#### 軸③: 企業規模 → 提案トーン

| 規模 | 従業員数 | 提案トーン |
|------|---------|-----------|
| **零細** | 1〜10名 | 「社長が全部やってる」→「社長の右腕」訴求 |
| **小規模** | 11〜50名 | 「人を雇うより安い」→ コスト比較訴求 |
| **中規模** | 51〜300名 | 「属人化リスク」→ ナレッジ標準化訴求 |

### 3.3 日程選択画面の生成ロジック

```
① Google Calendar APIで自分の空き枠を取得
   - 範囲: 翌営業日〜5営業日後
   - 時間帯: 9:00〜18:00（土日祝除外）
   - 既存予定と重複しない30分枠を抽出
        ↓
② 候補を最大6枠に絞る
   - 午前2枠 + 午後2枠 × 2〜3日分
   - 偏りがないように分散
        ↓
③ LLMで丁寧な文面に整形
   - 相手の企業名・担当者名を入れる
   - ビジネスメール風のトーン
   - 各枠をクリック可能なボタンとして表示
```

**ポイント: Calendlyのようなカレンダー UI ではなく、人間の営業が書いたメール風の文面の中にボタンが埋まっている。** ツール感がなく、丁寧な印象になる。

## 4. 顧客フォーム設計

### 4.1 「話を聞きたい」フォーム — 入力項目

| 項目 | 必須 | 理由 |
|------|------|------|
| **企業名** | ✅ | こちらから架電する際に必要 |
| **担当者名** | ✅ | 電話で「○○様お願いします」に必要 |
| **電話番号** | ✅ | 架電先。これがないとアポ取れない |
| **メールアドレス** | ❌ 不要 | LP URLに企業IDが紐づいているので追跡可能。入力項目増でCV率低下リスク > メリット |

**3項目のみ。** メールアドレスは以下の理由で不要：
- LP URL自体に企業IDが埋まっているので、どの企業かは既に分かっている
- フォローメールを送りたい場合は、フォーム営業時に使った企業の公開メアドに送ればいい
- **入力項目が1つ増えるとCVRが約10%下がる**（一般的なフォーム最適化の知見）

### 4.2 資料DLフォーム — 入力項目

| 項目 | 必須 | 理由 |
|------|------|------|
| **メールアドレス** | ✅ | PDF送付先。これだけ |

**1項目のみ。** 資料DLはハードル最低にする。企業情報はLP URLのIDから取得済み。

## 8. フォーム営業 — スプレッドシート起点の適応型送信設計

### 8.1 全体像

**スプレッドシートが営業リストの一元管理ツール。** エージェントはシートを読んで、フォーム送信して、結果をシートに書き戻す。

```
Google Spreadsheet（営業リスト）
    ↓ エージェントが読み取り
企業HPからフォーム検出 → 解析 → 入力 → 送信
    ↓ 結果をシートに書き戻し
✅ 送信済みチェック or ⚠️ 備考に不足項目を記載
```

### 8.2 スプレッドシートのカラム設計

```
| A: 企業名 | B: 業種 | C: HP URL | D: 送信済 | E: 送信日時 | F: 送信方法 | G: 備考 | H: LP URL | I: LP閲覧 | J: 面談希望 | K: 担当者名 | L: 電話番号 |
|-----------|---------|-----------|-----------|------------|------------|---------|-----------|-----------|------------|------------|------------|
| ○○建設    | 建設業  | https://… | ✅        | 3/15 10:23 | フォーム    |         | https://… |           |            |            |            |
| △△製作所  | 製造業  | https://… |           |            |            | ⚠️ 必須項目不足:「部署」「導入時期」 |  |  |  |  |  |
| □□歯科    | 歯科   | https://… | ✅        | 3/15 10:45 | メール      | フォームにCAPTCHAあり→メール送信 | https://… |  |  |  |  |
```

#### カラム詳細

| 列 | カラム名 | 入力者 | 説明 |
|----|---------|--------|------|
| A | **企業名** | 人間 | 営業対象の企業名 |
| B | **業種** | 人間 or AI | 業種。AIがHPから自動判定も可能 |
| C | **HP URL** | 人間 | 企業のホームページURL |
| D | **送信済** | AI | ✅チェックボックス。送信成功時にAIがチェック |
| E | **送信日時** | AI | フォーム or メール送信した日時 |
| F | **送信方法** | AI | 「フォーム」or「メール」 |
| G | **備考** | AI / 人間 | 不足項目・エラー・メモ |
| H | **LP URL** | AI | 生成した職務経歴書LPのURL |
| I | **LP閲覧** | AI | LP閲覧があったら日時を記入 |
| J | **面談希望** | AI | 「話を聞きたい」ボタンが押されたら日時を記入 |
| K | **担当者名** | AI | 面談希望フォームから取得 |
| L | **電話番号** | AI | 面談希望フォームから取得 |

### 8.3 エージェントの処理フロー

```
① スプレッドシートを読み込み
   → D列（送信済）が空の行を上から処理
        ↓
② C列（HP URL）にアクセス
   → 問い合わせフォームを自動検出
        ↓
③ フォーム解析
   HTMLパースして全フィールドを抽出:
   - label / name属性 / placeholder / required
   - input type（text / email / tel / select / textarea）
   - selectの選択肢
        ↓
④ フィールドマッピング（LLM）
   フォーム項目 → 用意している値にマッチング
        ↓
⑤ 判定分岐
   ├→ 全必須項目OK → フォーム送信
   │   → D列 ✅ / E列に日時 / F列「フォーム」
   │
   ├→ 必須項目不足 → 送信しない
   │   → D列 空のまま / G列「⚠️ 必須項目不足:「部署」「導入時期」(選択肢: すぐに/3ヶ月以内/半年以内/情報収集中)」
   │
   ├→ CAPTCHAあり → メール送信にフォールバック
   │   → D列 ✅ / F列「メール」/ G列「CAPTCHAあり→メール送信」
   │
   └→ フォームなし → メール送信にフォールバック
       → D列 ✅ / F列「メール」/ G列「フォームなし→メール送信」
```

### 8.5 不足項目の備考記載フォーマット

送信できなかった場合、G列（備考）に以下の形式で書き込む:

```
⚠️ 必須項目不足:
- 「部署名」(テキスト入力)
- 「導入検討時期」(選択肢: すぐに / 3ヶ月以内 / 半年以内 / 情報収集中)
- 「従業員数」(選択肢: 1-10名 / 11-50名 / 51-100名 / 101名以上)
```

**ポイント:**
- 項目名だけでなく**入力タイプと選択肢も記載**する → 人間が何を入れればいいかすぐ分かる
- select/radioの場合は選択肢を全部列挙する

### 8.6 不足項目の追加方法 — 3つのやり方

#### 方法①: スプレッドシートの「デフォルト値」シートに追加（推奨）

同じスプレッドシートに **「デフォルト値」シート** を用意する。

```
シート名: デフォルト値
| A: 項目パターン | B: 入力値 | C: 備考 |
|----------------|-----------|---------|
| 部署           | 営業企画部 | 「部署」「ご所属」「所属部署」等にマッチ |
| 導入時期        | 情報収集中 | selectの場合、最も近い選択肢を自動選択 |
| 役職           | 代表       |         |
| 従業員数        | 11-50名   | selectの場合、最も近い選択肢を自動選択 |
| 予算           | 未定       |         |
```

**使い方:**
1. 備考に「⚠️ 必須項目不足:「部署名」」と出る
2. 「デフォルト値」シートに `部署 | 営業企画部` を1行追加
3. 次回実行時、「部署」「ご所属」「所属部署」等を全て「営業企画部」で自動入力
4. **過去に送信失敗した企業も自動で再送信**（D列が空 & G列に⚠️がある行）

#### 方法②: 備考欄に直接書く（その企業だけ）

```
G列の備考:
⚠️ 必須項目不足:「部署名」「導入時期」
↓ 人間が書き換え
→ 部署名=営業企画部、導入時期=情報収集中
```

エージェントは次回実行時に:
1. G列に `→` で始まる行を検出
2. その値でフォーム再送信
3. 成功したらD列 ✅、G列を「再送信成功（部署名・導入時期を手動指定）」に更新

**この方法は「この企業だけ特別な値を入れたい」場合に使う。**

#### 方法③: 企業リストの行に追加カラムで個別指定

特定の企業だけ違う値を入れたい場合、スプレッドシートにカラムを足す。

```
| M: 部署名（個別） | N: 導入時期（個別） |
|-------------------|-------------------|
|                   |                    |  ← 空ならデフォルト値シートの値を使う
| 代表取締役室       | すぐに             |  ← この企業だけ個別値
```

### 8.7 フィールドマッチングのロジック

フォーム側の項目名は表記揺れが激しいので、LLMでセマンティックマッチングする。

```
パターン例:
  「御社名」「貴社名」「会社名」「企業名」「法人名」 → company_name
  「お名前」「氏名」「ご担当者名」「Name」         → sender_name
  「部署」「ご所属」「所属部署」「部門」            → department（デフォルト値シート参照）
  「ご用件」「お問い合わせ種別」「カテゴリ」        → inquiry_type
```

**select/radio の選択肢マッピング:**
```
デフォルト値シート: 導入時期 = 情報収集中

フォームの選択肢: [すぐに導入したい / 3ヶ月以内 / 半年以内 / 情報収集段階]
    ↓ LLMが「情報収集中」に最も近い選択肢を判定
→ 「情報収集段階」を選択
```

### 8.8 送信できないケース

| ケース | スプレッドシートへの記載 | 自動対応 |
|--------|----------------------|---------|
| **必須項目不足** | D列: 空 / G列: ⚠️ 不足項目詳細 | デフォルト値追加後に自動再送信 |
| **CAPTCHA** | D列: ✅ / F列: メール / G列: CAPTCHAあり→メール送信 | メールにフォールバック |
| **フォームなし** | D列: ✅ / F列: メール / G列: フォームなし→メール送信 | メールにフォールバック |
| **送信エラー** | D列: 空 / G列: ❌ 送信エラー（3回リトライ失敗） | 次回実行時に再試行 |
| **HPアクセス不可** | D列: 空 / G列: ❌ HPにアクセスできません | — |

### 8.9 エージェントの成長サイクル

```
Week 1:  200社送信 → 120社成功 / 60社項目不足 / 20社CAPTCHA
```

## 10. 1日400件送信 & AIマーケター（自動PDCA）

### 10.1 400件/日を実現するためのインフラ

**Gmail単体では無理。** Gmail API の上限は2,000通/日だが、フォーム営業は別の課題がある。

```
400件/日の内訳:
├── フォーム営業: 200件/日（メインチャネル）
├── メール送信:   200件/日（フォールバック + フォロー）
└── 合計:         400件/日
```

**フォーム200件/日を実現する仕組み:**

| 対策 | 内容 |
|------|------|
| **IP分散** | プロキシローテーション（住宅用プロキシ推奨）。1 IP あたり30件/日に制限 |
| **送信間隔** | 30秒〜2分のランダム間隔（人間らしい挙動） |
| **並列処理** | Playwright 5並列で同時処理（1件あたり平均1分 → 5並列で200件=40分） |
| **User-Agent** | ブラウザフィンガープリントをランダム化 |
| **時間帯分散** | 9:00〜12:00 に分散送信（一気に送らない） |

**メール200件/日を実現する仕組み:**

| 対策 | 内容 |
|------|------|
| **複数ドメイン** | 3〜5ドメインを用意（例: shachotwo.com / shachotwo.jp / shachotwo-ai.com） |
| **各ドメイン50件/日** | 4ドメイン × 50件 = 200件/日 |
| **ウォームアップ** | 新ドメインは Week 1: 10件/日 → Week 2: 30件/日 → Week 3: 50件/日 |
| **SPF/DKIM/DMARC** | 全ドメインに設定必須 |
| **送信間隔** | 1〜3分のランダム間隔 |
