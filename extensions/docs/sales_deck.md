## [SLIDE 8] 御社がコントロールできる仕組み

### 確認モード（推奨）

```
Phase 1: 指示 → ジャンル判定 → 要件定義書の作成 → 【ここで停止】
                                                    ↓
                                         ダッシュボードで確認
                                           「これでOK？」
                                                    ↓ 承認ボタン
Phase 2: 実装 → テスト → レビュー → 自動デプロイ → 完了通知
```

- 要件定義書は**日本語のMarkdown**で表示。技術知識不要で読める
- 問題があればコメントして修正指示を出せる
- **「担当者が承認して進めた」**という記録が残る

### 全自動モード（慣れてきたら）

- ワンクリックで切り替え可能
- 指示を出したら完成まで全自動
- 信頼関係が構築されてからの利用を推奨

> **スピーカーノート:**
> 「最初は確認モードで始めてください。AIの仕事ぶりに信頼が持てたら、全自動モードに切り替えるだけです。トグル1つで切り替わります。」

## [SLIDE 9] セキュリティ — 「事故ゼロ」を技術で担保

### 5重のセキュリティ防壁

```
 第1壁              第2壁              第3壁
┌─────────┐    ┌─────────────┐    ┌─────────────┐
│ 秘密鍵    │    │ Docker隔離   │    │ コマンド     │
│ 自動検知  │    │ 実行環境     │    │ ホワイトリスト │
│ (Push前に │    │ (メモリ制限・ │    │ (危険操作を   │
│  即ブロック)│    │  ネットワーク  │    │  物理的にブロック│
│           │    │  完全遮断)    │    │  rm/chmod等)  │
└─────────┘    └─────────────┘    └─────────────┘

 第4壁                          第5壁
┌─────────────┐            ┌─────────────┐
│ 段階テスト    │            │ 全操作       │
│ (Lint→単体    │            │ 監査ログ     │
│  →E2E の順で  │            │ (誰が・いつ・  │
│  品質チェック)  │            │  何をしたか   │
│              │            │  全記録)      │
└─────────────┘            └─────────────┘
```

### 監査ログで「何をやったか」が全て追跡可能

| 記録項目 | 内容 |
|---------|------|
| 操作者 | AI社員（自動記録） |
| 日時 | タイムスタンプ付き |
| 操作内容 | ファイル作成・コマンド実行・テスト結果の全て |
| 承認者 | ダッシュボードで承認した担当者 |

**→ 監査法人への説明にそのまま使えます。**

> **スピーカーノート:**

## [SLIDE 10] 使うほど、御社専用のAIに進化します

### Self-Improving（自己改善）の仕組み

```
  1回目の依頼                     10回目の依頼
┌────────────────┐          ┌────────────────────┐
│ 汎用的な知識で    │          │ 御社の業務を熟知した  │
│ 設計・実装       │    →     │ 専門AIとして          │
│                 │          │ 設計・実装            │
│ 修正: 数回必要   │          │ 修正: ほぼ不要        │
└────────────────┘          └────────────────────┘

     ↑                              ↑
  一般的なAIツール              AI社員（本サービス）
  は、ここから進化しない          毎回の実行結果から学習し
```

- 同じ業種・業務の案件は**回を追うごとに精度が向上**

## [SLIDE 18] 技術基盤（情シス・技術責任者向け）

### アーキテクチャ概要

| レイヤー | 技術 | 説明 |
|---------|------|------|
| AI基盤 | Gemini Pro / Flash | 設計はPro（高精度）、実装はFlash（高速・低コスト） |
| オーケストレーション | LangGraph | 状態管理付きマルチエージェント制御 |
| 隔離実行 | Docker Sandbox + MCP | 生成コードをコンテナ内で安全に実行・テスト |
| データベース | Supabase (PostgreSQL) | RLS対応。将来のマルチテナント拡張可能 |
| インフラ | Google Cloud (Cloud Run) | サーバーレス。オートスケール |
| CI/CD | GitHub Actions | 自動テスト・自動デプロイ |
| 監査 | 構造化JSON監査ログ | 全操作を記録しSupabaseに永続化 |

### セキュリティ仕様

| 対策 | 実装 |
|------|------|
| コード実行隔離 | Docker: read-only FS, no-network, メモリ512MB, PID 256 |
| 秘密鍵漏洩防止 | 正規表現スキャン（Push前に検知・拒否） |
| 危険操作防止 | コマンドホワイトリスト（rm, chmod等をブロック） |
| 品質担保 | Lint → Unit → E2E の段階的テスト |
| 監査証跡 | 全操作のJSON構造化ログ + 承認者記録 |
| タスク上限 | 1タスクあたり $0.50 以下、変更量 200行以下 |

> **スピーカーノート:**
