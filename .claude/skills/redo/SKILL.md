# Redo — 原因究明→改善策→実装フロー

対象: $ARGUMENTS

---

## Step 1: Diagnose（原因特定）

以下を順番に確認し、**何が問題だったかを特定**する。

1. 直前のエラーログ・スタックトレースがあれば読む
2. 対象ファイルを Read して現状のコードを把握する
3. 設計書（`shachotwo/` 配下）と照合し、意図とのズレを確認する
4. テストが存在する場合は実行して失敗内容を確認する

```bash
# テストがある場合
cd shachotwo-app && python -m pytest tests/ -x -q --tb=short 2>&1 | tail -30
```

---

## Step 2: Root Cause（根本原因の言語化）

Diagnose の結果を踏まえ、**根本原因を1〜2行で明確に言語化**する。

フォーマット:
```
【根本原因】〇〇が△△だったため、□□が起きた。
```

表面的な症状（エラーメッセージ等）ではなく、**なぜそうなったか**を掘り下げること。

---

## Step 3: Options（改善策の提示）

根本原因に対する改善策を **2〜3案** 提示する。

各案には以下を明記:
- **方針**: 何をどう変えるか（1行）
- **メリット**: 解決できること
- **デメリット/リスク**: 副作用・追加コスト
- **推奨度**: ◎ / ○ / △

推奨案を明示した上で、**そのまま実装に進む**（ユーザーから明示的な指示がない限り最推奨案を採用）。

---

## Step 4: Implement（実装）

推奨案で実装する。

- 修正範囲は根本原因に限定する。便乗リファクタはしない
- 型ヒント・RLS・async/await 等のコーディング規約を守る
- フロントエンド変更の場合は `/ui-review` を後続で実行

---

## Step 5: Verify（検証）

実装後、以下を実行して問題が解消されたことを確認する。

```bash
# Python の場合
cd shachotwo-app && python -m pytest tests/ -x -q --tb=short 2>&1 | tail -20
```

```bash
# 差分確認
git diff --stat && git diff
```

- テストが存在しない場合は、手動での動作確認手順を明示する
- 根本原因が解消されたことを Root Cause と照合して明記する

---

## 完了レポート

```
【根本原因】（Step 2 の再掲）
【採用した改善策】（Step 3 の推奨案）
【変更ファイル】（git diff --stat の結果）
【確認結果】テスト pass / 差分が意図通り
```
