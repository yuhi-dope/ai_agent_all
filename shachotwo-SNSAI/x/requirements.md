# X (旧Twitter) アカウント運用 半自動化システム 要件定義書

> **最終更新**: 2026-03-15
> **アカウント**: @shachotwo
> **目的**: 経営者層からのDMリード獲得（月5件以上を初期目標）

---

## 1. プロジェクト概要

「AIエージェントによる業務改善」をテーマにしたXアカウントの運用システム。
高額なX APIは使用せず、**Apify（スクレイピング） + Google スプレッドシート（DB） + Google Apps Script (GAS) + Gemini API（生成）** を組み合わせた高コスパアーキテクチャを構築する。

**このシステム自体が「FDEによる業務自動化」の生きたポートフォリオ**であり、発信内容と発信方法の両方がマーケティングとして機能する。

## 2. ターゲット定義

### 2.1 ターゲットペルソナ

| 属性 | 詳細 |
|---|---|
| **役職** | 社長・代表取締役・事業責任者・役員（最終意思決定者） |
| **企業規模** | 従業員10〜300名の中小企業 |
| **業種（優先）** | 建設業・製造業（食品加工含む）・歯科クリニック・介護福祉・士業事務所・不動産管理 |
| **課題感** | 属人化、ナレッジ流出、DXの進め方がわからない、AI活用に興味はあるが具体策がない |
| **行動特性** | X上で情報収集している、経営者コミュニティに参加、日経やNewsPicksを読む |
| **心理状態** | 「AI使わなきゃまずい」と焦っている or「何から始めればいいかわからない」 |

### 2.2 ターゲット外（明示的に除外）

- エンジニア向けの技術論（彼らはDMしてこない）
- 大企業のIT部門（意思決定が遅くリード化しにくい）
- AI初心者向けの入門コンテンツ（ペルソナと合わない）

### 2.3 DM到達までの心理ステップ

```
認知（タイムラインで発見）
  ↓ 「この人、自分の業界のこと分かってるな」
興味（プロフ訪問・固定ポスト確認）
  ↓ 「GASとかAIで業務自動化してる人か」
信頼（複数ポストを読む・フォロー）
  ↓ 「具体的で再現性がある。実際にやってる人だ」
欲求（自社課題との接続）
  ↓ 「うちの〇〇も自動化できないかな…」
行動（DM送信）
  ↓ 「ちょっと相談したいんですが…」
```

**各ステップに対応するコンテンツを意図的に配置する**（Section 3参照）。

### 3.1 アカウントのペルソナ

```

肩書:     FDE（Forward Deployed Engineer）× 業務改善コンサルタント

          ・政治・宗教・センシティブな話題
```

### 3.3 コンテンツ4本柱（Content Pillars）

全ポストは以下4カテゴリに分類。**比率を守ることでフォロワーの期待値を安定させる**。

| # | 柱 | 比率 | 目的（心理ステップ） | 例 |
|---|---|---|---|---|
| P1 | **実績公開（Build in Public）** | 35% | 信頼→欲求 | 「今日作った自動化：請求書OCR→freee自動入力。月40時間→2時間に」 |
| P2 | **業種別あるある課題 × 解決策** | 30% | 認知→興味 | 「建設業の社長が毎日やってる"積算確認"、AIに任せたら精度99%だった話」 |
| P3 | **図解・フレームワーク** | 20% | 認知→信頼 | Mermaid図解で「中小企業AI導入の5ステップ」「属人化を解消する構造」 |
| P4 | **引用RT・時事コメント** | 10% | 認知（リーチ拡大） | バズポスト/日経記事に対する専門家コメント |
| P5 | **記事ダイジェスト（note誘導）** | 10% | note流入→アフィリ | note記事の前半50%抜粋 +「全文はnoteで」 |

> **詳細テンプレート**: `content_pillars.md` 参照

### 3.4 X記事（ダイジェスト版）— note連携

X記事機能を使い、note記事の**ダイジェスト版**を投稿する。

**ルール**:
- note記事の**前半50%**（2,000〜2,500字）を抜粋
- 末尾に「全文はnoteで → {URL}」を必ず付ける
- note記事と**完全同一にしない**（重複コンテンツ回避）
- noteにしかない要素: 後半の詳細手順・アフィリエイトリンク・FAQ

**テンプレート（記事ダイジェスト版の末尾）**:
```
---

ここまでが概要です。

▼ 具体的な設定手順・使ったツールの比較・テンプレートはnoteにまとめました:
{note記事URL}

#AI業務改善 #中小企業DX
```

## 5. 機能要件（詳細）

### 5.1 長文ポスト自動下書き生成

**目的**: 毎朝、確認するだけで投稿できる状態の下書きを用意する。

| 項目 | 仕様 |
|---|---|
| **トリガー** | GAS時間トリガー 毎日 6:00 |
| **入力** | `業務改善ネタ` シートから `ステータス=未処理` の最古1件 |
| **処理** | ネタ本文 + `頭のDB` から関連タグのエントリ5件 → Gemini API |
| **出力** | ① 長文ポスト本文（280〜1000文字） ② 図解Mermaidコード ③ ハッシュタグ候補3つ |
| **プロンプト** | `prompts/long_post.md` 参照 |
| **後処理** | `投稿管理` シートに書き込み + ダッシュボードに未確認表示 |

**長文ポストの構造テンプレート**:

```
【フック（1行目）】← ここで止まるかスクロールするかが決まる
具体的な数字 or 逆説 or 問いかけ

【課題提示（2-3行）】
ターゲットの「あるある」を描写

【解決策（3-5行）】
具体的なツール名・手順・数字を含む

【結果（1-2行）】
Before → After を数字で

【CTA（最終行）】
「DMで相談受付中」or「続きはリプで」or「保存して後で見返して」
```

### 5.2 引用リポスト自動キュレーション & 下書き生成

**目的**: バズっているポストやニュースに、専門家として価値あるコメントを素早く出す。

| 項目 | 仕様 |
|---|---|
| **トリガー** | Apify Webhook → GAS `doPost()` |
| **入力** | Apifyが検出したバズポスト or ニュース記事 |
| **フィルタ** | 以下を満たすもののみ処理: ①いいね50以上 ②日本語 ③24時間以内 ④AI/DX/業務改善関連 |
| **処理** | 元ポスト/記事 + `頭のDB` 全文 → Gemini API |
| **出力** | 引用RTテキスト（100〜280文字） |
| **プロンプト** | `prompts/quote_rt.md` 参照 |
| **後処理** | `投稿管理` シートに書き込み + ダッシュボードに未確認表示（Urgency: 高/中/低を付与） |

**引用RTの品質基準**:

- ✅ 元ポストに「新しい視点」を追加している（単なる同意ではない）
- ✅ 具体的な経験・数字・ツール名が含まれている
- ✅ ターゲット（経営者）にとって有益
- ❌ 「その通りですね」だけの浅いコメント
- ❌ 元ポストの否定・批判（炎上リスク）
- ❌ 宣伝色が強すぎる

### 5.3 図解（Mermaid）生成

| 項目 | 仕様 |
|---|---|
| **トリガー** | 長文ポスト生成時に同時生成 or 手動リクエスト |
| **入力** | ポスト本文 or テーマ |
| **出力** | Mermaidコード（graph TD / flowchart / sequenceDiagram） |
| **画像化** | Mermaid Live Editor (https://mermaid.live) で手動変換 → PNG保存 → X投稿時に添付 |
| **スタイル** | 白背景・日本語ラベル・3〜7ノード（複雑すぎない） |

### 5.4 パフォーマンス記録 & 週次レポート

| 項目 | 仕様 |
|---|---|
| **日次** | 投稿後にXアナリティクスを確認 → `パフォーマンス` シートに手動記録 |
| **週次** | GASが月曜9:00にダッシュボードの週間KPIを自動集計 |
| **レポート内容** | 総imp / 総エンゲージメント / フォロワー増減 / DM数 / 最もパフォーマンスが良かったポストとその特徴 |
| **改善サイクル** | 月次でコンテンツ比率・投稿時間・クエリを調整 |

### 5.5 リード管理

| 項目 | 仕様 |
|---|---|
| **トリガー** | DMが来たら手動で `リード管理` シートに記録 |
| **管理項目** | 相手情報 / 企業名 / 業種 / 課題 / ステータス / 次アクション / 期限 |
| **ステータス遷移** | `初回DM` → `ヒアリング済` → `提案中` → `成約` or `失注` |
| **対応SLA** | DM受信後 **24時間以内** に返信（初回） |

### 6.2 フェーズ別KPI

## 8. リスク管理

| リスク | 影響度 | 発生確率 | 対策 |
|---|---|---|---|
| **シャドウバン** | 高 | 中 | 手動投稿を徹底。短時間での大量投稿を避ける。1日の投稿上限を5件に制限 |
| **Apifyのスクレイピング失敗** | 中 | 中 | GASで受信確認 → 6時間データなしならダッシュボードにアラート表示。手動でのネタ収集もバックアップとして維持 |
| **Geminiの品質低下** | 中 | 低 | 全下書きは人間が確認。品質基準を満たさない場合は却下 → 手動作成 |
| **X ToS変更** | 高 | 低 | 自動投稿はしていないので直接のBAN risk は低い。Apifyのスクレイピングが禁止された場合は手動キュレーションにフォールバック |
| **炎上** | 高 | 低 | コンテンツポリシー（3.1の禁止事項）を厳守。政治・宗教・批判系は一切触れない |
| **ネタ枯渇** | 中 | 中 | `業務改善ネタ` シートに常時20件以上ストック。実業務から得たネタを週次で追加 |
| **DM対応の負荷増大** | 低 | 低 | 嬉しい悲鳴。テンプレ返信を準備し、初回返信の工数を削減 |
