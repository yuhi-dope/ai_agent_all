## 1. エグゼクティブサマリー

本計画は、従来の「システム開発」や「SaaS導入」の概念を捨て、**「優秀なバックオフィス・経営企画人材（AI）を派遣する」**というアプローチで、中堅・中小企業（SMB）の成長エンジンそのものを提供するものである。

顧客企業の現場（アナログ業務）に入り込み、業務をAIに学習・代行させることで、単なる効率化を超え、「松下幸之助やスティーブ・ジョブズのような経営判断ができるNo.2（参謀）」を各社に配備する。
これにより、労働集約型のBPOや人材紹介業界のシェアを奪取し、売上150億円・営業利益60億円（利益率40%）を達成、時価総額1,000億円企業の実現を目指す。

## 2. 戦略的フレームワーク：AI Staffing

### 2.1 コア・コンセプト：SaaSでもSIerでもない「第3の選択肢」

- **Against SaaS**: SaaSは「ベンダーが決めた仕様」に人が合わせるものだが、本サービスは「企業の独自文化・業務」をAIが学習し、企業ごとの最適解（専用の脳みそ）を構築する。
- **Against 人材紹介**: 人間は退職リスクがあり、教育コストがかかる。本サービスのAI人材は、24時間365日稼働し、情報を蓄積すればするほど賢くなり、絶対に辞めない。

### 2.2 3段階の進化機能（The Trinity Agents）

顧客には「システム」ではなく、以下の役割を持つ「AI社員」として紹介・導入する。

- **【手足】事務エージェント（The Executor）**
    - ミーティング内容や業務指示を投げるだけで、業務フローを整理し、必要なSFA/CRM/会計処理を実行するWEBシステムを自律的に構築・運用する。
- **【参謀】No.2 エージェント（The Brain）**
    - 偉大な経営者（松下幸之助、孫正義、ジョブズ等）の思考回路と自社データを統合。「社長、ジョブズなら製品ラインをこう絞り込みますが、どうしますか？」といった経営提言を行う。
- **【戦略】M&A エージェント（The Strategist）**
    - Web情報をクローリングし、競合比較やM&A候補企業の選定、デューデリジェンスの初期判断を行い、自社の成長に直結する買収・バリューアップ案を提案する。

### 2.4 BPOエージェントの専門化（Specialist Branching）

事務エージェントの中核である bpo_agent は、BPO実務を通じて各職種の業務ルールを自動蓄積し、**汎用BPOから職種別専門エージェントへ自律的に分岐**する設計とする。

```
初期状態:  汎用 bpo_agent（全ジャンル対応・汎用ルール）
   │
   │  BPO実務でルール蓄積 → learned/ にルールが溜まる
   │
   ▼  ジャンル別の learned ルールが閾値（成熟度スコア）を超えた時点で分岐
分岐後:
   ├─ 会計専門 bpo_agent  （freee/MFの深い業務知識・仕訳パターン・月次締めフロー）
   ├─ 営業専門 bpo_agent  （Salesforce/kintoneの商談管理・パイプライン最適化）
   ├─ 人事専門 bpo_agent  （SmartHR/KOTの入退社・勤怠・年末調整フロー）
   └─ ...（実績が溜まったジャンルから順次分岐）
```

**分岐の判定基準（成熟度スコア）:**

| 指標 | 閾値 | 根拠 |
| --- | --- | --- |
| ジャンル別 learned ルール数 | ≥ 10 件 | 十分なパターン蓄積 |
| 同ジャンルのタスク成功率 | ≥ 85% | ルールの品質保証 |
| 同ジャンルの累計タスク数 | ≥ 50 件 | 統計的有意性 |
| plan_confidence の平均値 | ≥ 0.8 | 計画の安定性 |

**専門化による効果:**
- **プロンプト精度向上**: 汎用ルールではなくジャンル特化ルールが優先され、LLMの計画精度が向上
- **営業訴求**: 「御社には会計専門のAI社員を配属します。freeeの仕訳パターンを200社分学習済みです」
- **単価根拠**: 汎用BPO（月額15万）→ 専門BPO（月額30-50万）への自然なアップセル

### 2.5 SaaS構造ナレッジとテンプレート出力

BPO業務中にAI社員が操作するSaaSの構造（フィールド定義・レイアウト・ビュー・ワークフロー）を自動的にナレッジとして蓄積する。このナレッジは、顧客が「自社システムを作りたい」となった際の**テンプレート**として即座に活用できる。

```
BPO実務中:
  kintoneアプリ操作 → フィールド定義・レイアウト・ビューを自動スナップショット
  Salesforce操作   → オブジェクト構造・項目定義・プロセスビルダーを記録
  freee操作        → 勘定科目体系・仕訳パターン・承認フローを記録
       │
       ▼ saas_structure_knowledge テーブルに蓄積
       │
  顧客が「自社システムを作りたい」と決断
       │
       ▼ ダッシュボードで「どのSaaSの構造をベースにしますか？」と選択
       │
       ├─「kintoneの顧客管理アプリっぽく」→ kintoneのフィールド構造 + レイアウトをテンプレート化
       ├─「Salesforceの商談管理っぽく」    → SFの商談オブジェクト構造をテンプレート化
       └─「freeeの経費精算っぽく」         → freeeの経費UIフローをテンプレート化
       │
       ▼ bpo_agent の code_tools にテンプレートを入力として渡す
       │
  自社システム生成（Phase 3: 独立）※ 同じ bpo_agent が一貫して実行
```

**事業的意義:**
- **Phase 1→3 の移行がスムーズ**: 「今使っているSaaSと同じ見た目・操作感のシステムが手に入る」ためユーザーの学習コストゼロ
- **SaaS解約の意思決定が容易**: 「既存SaaSと同等の機能が自社システムで動く」ことをテンプレートで事前に可視化できる
- **横展開**: 同業種N社のSaaS構造を匿名化して統合することで「製造業の標準kintone構成」のような業種テンプレートが自動生成される

### 2.3 10ジャンル体制と統合ダッシュボード

上記 Trinity Agents を実務レベルで具現化するため、顧客業務を以下の **10ジャンル** に分類し、それぞれに専門AIエージェントを配備する。顧客には **1つの統合ダッシュボード（Next.js + Supabase）** を提供し、全ジャンルのエージェントアウトプットを一元管理する。

| ジャンル | 表示名 | 主な対象業務 |
| --- | --- | --- |
| sfa | SFA/営業エージェント | 商談管理・パイプライン・見積書・受発注 |
| crm | CRMエージェント | 顧客情報管理・問い合わせ・顧客分析 |
| accounting | 会計エージェント | 請求書・仕訳・経費精算・財務分析 |
| legal | 法務エージェント | 契約書・稟議・コンプライアンス |
| admin | 事務エージェント | 日報・勤怠・スケジュール・備品管理 |
| it | 情シスエージェント | インフラ・ヘルプデスク・IT資産管理 |
| marketing | マーケティングエージェント | 集客・広告・施策効果測定 |
| design | デザインエージェント | UI/UX・ブランディング・制作物管理 |
| ma | M&Aエージェント | 買収候補・DD・企業価値分析 |
| no2 | No.2/経営エージェント | KPI・経営分析・偉人ペルソナ・戦略提言 |

要件テキストからジャンルを **AI が自動分類**（Genre Classifier）し、適切な専門ルール・テンプレートを適用してシステムを生成する。これにより、PMは業務要件をNotionに書くだけで、ジャンル判定から実装までが自律化される。

## 3. デリバリーモデル：BPO entry to AI Automation

「システム提供」ではなく、**「業務を巻き取る（BPO）」**として入り込むのが最大の差別化要因。
全フェーズを**単一の bpo_agent が一貫して遂行**する。SaaS操作もコード生成もAI社員にとっては「業務ツール」の一つであり、別システムへのバトンタッチは発生しない。

1. **BPOフェーズ（アナログハック）**:
    - 人間のオペレーター（AI Handler）が現場に入り込み、アナログな情報の流れを特定・掌握する。
2. **AI社員の配備（bpo_agent 統一アーキテクチャ）**:
    - AI社員（bpo_agent）が顧客企業に配属され、SaaS操作・コード生成・業務分析を**1つのエージェント**として遂行する。顧客からは「優秀なスタッフが爆速で業務を片付けている」ように見える。
    - bpo_agent は3種類のケイパビリティ（ツール群）を持つ:
        - **saas_tools**: kintone, Salesforce, freee 等のSaaS操作
        - **code_tools**: コード生成・Docker Sandbox内テスト・GitHub push（旧 develop_agent の機能）
        - **analysis_tools**: パターン検出・SaaS構造スナップショット・ナレッジ蓄積
    - **承認チェックポイント**: 全てのタスクは計画段階で一旦停止し、担当者がダッシュボードで確認・承認してから実行に進む。
    - **Docker Sandbox 隔離**: コード生成・実行は Docker コンテナに完全隔離。セキュリティ事故ゼロを技術的に担保。
3. **進化フェーズ（寄生→理解→独立）**:
    - **Phase 1（寄生）**: 顧客の既存 SaaS に接続し、AI社員が saas_tools で BPO を代行。ジャンル別ルールを自動蓄積し、汎用→専門エージェントへ分岐。SaaS構造をナレッジとして記録。
    - **Phase 2（理解）**: analysis_tools で業務パターンを自動検出。企業固有の業務モデルを構築。SaaS構造ナレッジからテンプレートを自動生成。
    - **Phase 3（独立）**: 同じ bpo_agent が code_tools を使い、蓄積したSaaS構造テンプレートから個社最適の自社システムを生成。データ移行まで bpo_agent が一貫して実行。SaaS 依存ゼロへ。
4. **自律化・高度化フェーズ**:
    - 業務データが溜まるにつれ、AIエージェントの精度が向上。最終的に人間は抜け、AIのみが顧客企業に残る（純粋な月額課金モデルへ移行）。

```
bpo_agent（唯一のエージェント = AI社員）
  │
  ├─ capabilities
  │   ├─ saas_tools:     kintone, salesforce, freee, slack, smarthr, google
  │   ├─ code_tools:     spec生成, コード生成, lint/test, git push (Docker Sandbox)
  │   └─ analysis_tools: pattern_detect, schema_snapshot, knowledge_capture
  │
  ├─ pipeline（全タスク共通）
  │   plan → approve → execute(tools) → verify → report → learn
  │
  └─ rules（4層合成 + 専門化）
      general → platform → genre → learned
```

## 7. 実行アクションプラン（直近6ヶ月）

### 7.1 プロダクト開発

- **[Month 1-2]**: bpo_agent 統一アーキテクチャへのリファクタリング。SaaSアダプタの Typed Function Tools 化。CLIデバッグツール自動生成。（技術詳細は [develop_agent.md](develop_agent.md) を参照）
- **[Month 3-4]**: bpo_agent に code_tools（旧 develop_agent）を capability として統合。SaaS構造ナレッジ自動蓄積機能。専門化成熟度スコア基盤。
- **[Month 5-6]**: 偉人経営者の思考モデル（RAG）のプロトタイプ作成。SaaSテンプレート出力機能。汎用→専門エージェント分岐の実運用開始。

#### 完了済みインフラ整備

- **10ジャンル自動分類**: Genre Classifier ノードの実装。要件テキストからAIがジャンルを自動判定し、専門ルールを適用する仕組みを構築。
- **統合ダッシュボード標準構造**: クライアント向け Next.js アプリの標準構造（10ジャンルカード・左サイドバー・ジャンル別ページ）を定義。
- **コード品質ルール整備**: Coder / Spec / Review / Fix 各エージェントのルールを大幅強化（型安全・エラーハンドリング・RLS・AIセマンティックレビュー等）。
- **本番デプロイ基盤**: Dockerfile（Cloud Run対応）、GitHub Actions CI/CD（ruff + pytest + Cloud Run自動デプロイ）を構築。
- **DBマイグレーション**: Supabaseスキーマの冪等マイグレーションランナー（`server/migrate.py`）を構築。
- **Spec Review Checkpoint（要件確認チェックポイント）**: パイプラインを要件定義フェーズと実装フェーズに分割。ダッシュボードの auto_execute トグルで切替可能。OFF 時は要件定義完了後に停止し、IT担当者がダッシュボード上で要件書を確認・承認してから実装を開始できる。これにより「AIが勝手に作った」から「担当者が承認して進めた」に変わり、中小企業のIT担当者の信頼と安心感を確保。
- **Docker Sandbox + MCP（コード実行隔離）**: 生成コードの実行を Docker コンテナに完全隔離。非root・ネットワーク無効・メモリ/CPU/PID 制限・危険コマンドブロック・パス横断攻撃防止。MCP 経由の構造化 JSON インターフェースで全操作を監査ログに記録し Supabase に永続化。「AIが生成したコードが暴走してもホスト環境に影響ゼロ」を技術的に担保し、セキュリティ事故 0 件の KPI を裏付ける。
- **テストスイート拡充**: Sandbox MCP サーバー・クライアント・Guardrails・Spec Checkpoint の 4 カテゴリ、合計 38 テストパス。品質保証の自動化基盤を確立。
- **LangGraph グラフ分離**: Phase 1（要件定義）と Phase 2（実装）のグラフを独立ビルドし、`invoke_spec()` / `invoke_impl()` で個別起動可能に。各フェーズに独立タイムアウト設定。
- **10ジャンル専門ルール・テンプレート完備**: 全10ジャンル（SFA/CRM/会計/法務/事務/情シス/マーケティング/デザイン/M&A/No.2経営）に専門ルールファイルとDBスキーマテンプレートを整備。ジャンル分類後に Spec Agent / Coder Agent が自動ロードし、ドメイン固有のビジネスルール（商談パイプライン遷移、複式簿記の貸借一致、契約承認フロー等）・エンティティ定義・受入条件テンプレート・実装指針を注入。「ジャンル名を知っているだけの汎用エージェント」から「各業務ドメインの専門知識を持つ専門エージェント」へ進化。
- **SaaS MCP アダプタ基盤（v4.0）**: 6 SaaS アダプタ（Salesforce/freee/Slack/Google Workspace/kintone/SmartHR）を `server/saas_mcp/` に実装。`@register_adapter` パターンで新 SaaS は 1 ファイル追加のみ。OAuth トークン管理・監査ログ・テナント分離を標準装備。
- **Pattern Detector（v4.0）**: `server/pattern_detector.py` で操作パターンの自動検出を実装。4 種類の分析（操作頻度・シーケンス検出・フィールドマッピング推定・スキーマ差分検出）を実行し、`operation_patterns` テーブルに蓄積。Phase 3 の自社システム自動生成のデータ基盤。
- **マルチテナント RLS 自動適用（v4.0）**: `develop_agent/utils/rls_validator.py` で生成コードの SQL に対して RLS ポリシーの存在を自動検証。company_id カラムを持つ全テーブルに SELECT/INSERT/UPDATE ポリシーを必須化。不足時は自動注入機能あり。Review Guardrails に統合済み。
- **Streamable HTTP MCP（v4.0）**: STDIO に加えて Streamable HTTP トランスポートをサポート。Sandbox MCP サーバー（`sandbox/mcp_server.py`）と HTTP クライアント（`develop_agent/sandbox/http_client.py`）を実装。SaaS MCP アダプタの 24h 常時接続に対応。
- **Trivy コンテナスキャン（v4.0）**: CI パイプラインに Trivy 脆弱性スキャンを追加。CRITICAL 脆弱性でビルドを停止し、スキャン結果を GitHub Security タブにアップロード。デプロイワークフローでも Push 前にスキャンを実行。
- **マルチチャネル対応**: Slack・Chatwork・Google Drive からの要件受付に対応。Channel Adapter パターンで統一的なメッセージ処理。
- **OAuth フレームワーク**: Notion / Slack / Google / GitHub の OAuth 2.0 フローを統合。トークンの暗号化保存（`server/crypto.py`）と自動リフレッシュ（`server/token_refresh.py`）。
- **テナント管理**: 企業 CRUD・メンバー招待・ロール管理（owner/admin/member）・法人番号検索・オンボーディング自動プロビジョニング。
- **i18n 対応**: `server/i18n.py` で日本語/英語の多言語対応基盤。

### 7.2 営業・組織

- **[Month 1]**: 「AI社員の職務経歴書」を模した営業資料の作成。
- **[Month 2-3]**: モデル企業へのBPO形式での先行導入（実データの収集）。
- **[Month 1-3]**: AI Handler（AIを使いこなす現場人材）の採用と研修。
